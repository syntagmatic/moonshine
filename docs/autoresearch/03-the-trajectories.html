The Trajectories

How different D3 blocks respond to iterative improvement

The proposer-evaluator loop generates candidate revisions, scores them, and keeps only the ones that are shorter and at least as good. Each "experiment" is one propose-evaluate cycle. Over dozens of experiments, patterns emerge.

Not all code compresses equally. Verbose, Sonnet-generated blocks with non-idiomatic D3 can shrink 25% while improving quality. Already-compact blocks hit diminishing returns within a few experiments. The trajectory shape tells you whether to keep running or move on.

What the shapes tell us

1. The ideal candidate: 18-collapsible-tree-search

This block dropped from 694 to 522 lines, a 25% reduction with a 77% keep rate. The curve is steep early and flattens gradually. Composite score actually rises from 7.2 to 7.7. This is what a verbose, Sonnet-generated block looks like when the loop finds real waste to cut.

2. Diminishing returns: 04-bee-swarm-census

Starting at 265 lines, this block was already reasonably compact. It reached 221 (17% reduction) but the deltas tell the story: -9, -2, -2, -5, -5, -3, -3, -6, -3, -7, -3, -4. Small bites. Many discards. The loop is working hard for modest gains.

3. Volatile quality: hierarchy-bundles

The longest run at 20 experiments. It achieved a 26% reduction (1012 to 748) but the composite score bounces between 5.2 and 7.0. This block started at low quality and the loop struggled to improve both size and quality simultaneously. Notice experiment 17: a sudden -46 line jump after a plateau of discards. The proposer occasionally finds a structural refactor that unlocks a new compression regime.

4. The resistant block: 03-violin-plot-orchestra

Barely moves. 350 to 338, mostly discards. Six of nine experiments were thrown away. This block was already well-written. Its composite started at 7.5 and the loop could not reliably improve it. The trajectory is nearly flat.

The best ROI comes from blocks that start verbose and low-quality. If a block's trajectory is flat after five experiments, it is probably near its optimum. Move on.

Convergence and stopping

The stopping rule is simple: three consecutive discards ends the run. Look at the violin plot trajectory. After the single keep at experiment 5, it hits four straight discards (experiments 6, 7, 8, 9). The loop correctly recognized there was nowhere left to go.

Contrast this with hierarchy-bundles: experiments 14, 15, and 16 were all discards, but the composite hovered around 5.3 to 6.3. The block had room to improve in quality but the proposer kept generating revisions that traded quality for size. Experiment 17 finally broke through with a -46 line drop at composite 6.5. Patience sometimes pays off, but the stopping rule would have ended the run after experiment 16.

There is a tension here. Strict stopping saves compute but risks missing late breakthroughs. The three-discard rule is a reasonable default. For blocks with low composite scores, a longer leash might be warranted.

Three trajectory archetypes

After running enough blocks through the loop, you start seeing three shapes repeat. They are distinct enough to name, and the name tells you what to expect before the run finishes.

Steep decline

The trajectory drops fast in the first few experiments, then gradually flattens. This is the shape you see on verbose, LLM-generated code with redundant variables, overly explicit style setups, and inline constants that belong in a scale. The loop finds genuine waste and removes it. Keep rates are high (70% or above) because almost every proposal finds something real to cut. The collapsible-tree block is a textbook example: 694 to 522 lines, most of the compression happening in the first eight experiments. These blocks are ideal candidates. They reward iteration cheaply.

Gradual plateau

The trajectory drifts downward with small, irregular steps. Each kept experiment shaves two to six lines. The keep rate hovers around 50%: half the proposals are rejected because the remaining code is tightly enough written that most changes break something or fail to improve the score. The bee-swarm block shows this pattern. It started reasonably compact and the loop scraped out 17% through persistent small bites. Whether those last few experiments are worth running depends on your budget. The per-experiment cost is the same regardless of how many lines it saves.

Flat refusal

Nearly flat. The line barely moves. Most experiments are discards. The block was already near its optimum, or its code is too tightly coupled for the proposer to find safe, isolated changes. The violin-plot block is the clearest case: 350 to 338 lines, six of nine experiments thrown away. The composite score actually dropped on most proposals, meaning the code was written well enough that shortening it made it worse. A flat trajectory is not failure. It is a signal that the code does not need this kind of help.

What the shape tells you about code quality

The trajectory is a diagnostic. Steep decline means the original code had structural bloat: redundant declarations, verbose patterns that D3 idioms handle more concisely, dead branches. Gradual plateau means the code was competent but not tight. Flat refusal means it was already well-written, or at least well-coupled enough that every part depends on every other part.

There is a subtlety here. "Well-coupled" is not the same as "well-written." A block can be tightly coupled because the author mixed concerns, not because the design demanded it. In that case the trajectory is flat not because the code is good but because the proposer cannot safely touch any piece without breaking another. The composite score helps distinguish: if the flat trajectory has a high composite (above 7.5), the code is genuinely good. If the composite is low (below 6), the code is probably a tangled mess that the loop cannot untangle.

Compression ratio as a quality signal

Think of it this way: if a block compresses 25%, it had 25% waste. That waste might be verbose variable declarations, duplicated style assignments, non-idiomatic patterns, or dead code. The compression ratio is a rough but honest measure of how much unnecessary material the original author included.

This reframes what the loop is doing. It is not just "making code shorter." It is measuring the gap between what the code does and how concisely it could do it, then closing that gap experiment by experiment. A block that compresses 5% was already close to its minimum viable size. A block that compresses 30% was carrying significant overhead. Neither number is inherently better. But if you are writing D3 blocks from scratch and your code consistently compresses 25% or more, that is a signal to study what the loop removes and internalize those patterns.

Cost efficiency

Each experiment in the loop costs roughly $0.05 to $0.10 in API calls, depending on block size and whether the proposer generates a long chain-of-thought. A 13-experiment run on the collapsible-tree costs about $0.65 to $1.30. A 20-experiment run on hierarchy-bundles costs $1.00 to $2.00. These are small numbers individually, but they accumulate across a library of 100+ blocks.

The question is not "how much did it cost" but "what did you get for the money." The collapsible-tree spent ~$1 and removed 172 lines while improving its composite from 7.2 to 7.7. That is roughly $0.006 per line removed, and the quality went up. The violin-plot spent ~$0.60 and removed 12 lines, most of which came from a single lucky experiment. That is $0.05 per line removed, nearly 10x worse. Hierarchy-bundles is somewhere in the middle: high absolute savings (264 lines) but the 20-experiment run and volatile quality make the per-line cost moderate.

The useful metric is cost per line saved, but only when quality holds or improves. Removing 50 lines while dropping the composite from 7.5 to 6.0 is not savings. It is damage.

Amber: the diminishing-returns trap

The last five lines are the most expensive. Once a block reaches its gradual plateau, each additional experiment is unlikely to find meaningful compression. But the loop does not know that. It keeps proposing, the evaluator keeps scoring, and the API meter keeps running. On the bee-swarm block, the final four experiments (10 through 13) saved a total of 11 lines across two keeps and two discards. That is roughly $0.30 for 11 lines, compared to $0.15 for the first 9 lines removed in experiment 2 alone.

The "last 5 lines" trap is a pattern that shows up across many blocks. The proposer, having exhausted the obvious waste, starts attempting changes that are either too small to matter or too risky to keep. You see it in the data: a sequence of -2, -3, discard, -2, discard. Each experiment costs the same as the first one did, when it was removing 30 lines at a time. The marginal cost per line saved climbs steeply at the tail.

This is a budget leak. If you run 100 blocks and each one wastes three to five experiments at the tail, that is 300 to 500 extra API calls with minimal return. At $0.075 each, that is $22 to $37 spent on the code equivalent of wringing out an already-dry towel.

Trajectory shape does not predict when to stop early enough. The three-discard stopping rule catches the flat-refusal blocks reasonably well, but it is poor at detecting the transition from steep decline into gradual plateau. By the time you see three discards in a row on a plateau block, you have already spent five to eight experiments in the low-return zone. A smarter stopping rule would track the rolling average of lines saved per kept experiment and halt when it drops below a threshold. That rule does not exist yet in the current implementation.

There is also a quality trap. Some blocks oscillate in composite score at the tail: 7.7, 7.3, 7.7, 7.3. The loop keeps going because it occasionally finds a keep, but the quality is not converging. It is bouncing. The block has reached a local optimum where different trade-offs produce similar overall scores. Continuing to run experiments at this point is spending money on lateral moves, not improvements.
