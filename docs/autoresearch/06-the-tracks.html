The Tracks

Same loop, different decision functions, different behaviors

Any artifact that can be automatically modified and automatically evaluated can be iterated. That is the core idea behind the autoresearch loop. Read the artifact. Propose a change. Render the result. Audit the output. Decide whether to keep or discard. Repeat.

The loop structure never changes. What changes is the decision function: what you optimize, what you constrain, and how you determine keep vs. discard. Swap out the decision function and the same loop produces very different behavior.

The decision function is the soul of each track. It encodes what you value. Everything else in the loop is shared infrastructure.

Three Tracks

The autoresearch system runs three distinct tracks. Each targets a different optimization objective. The figure below lays them side by side. Click a track to see its decision logic and a sample experiment.

The Decision Function

Each track's character comes from what it asks at the Decide stage. The block track asks: did the code get shorter without hurting quality? The prompt track asks: did the generation get faster without losing features? The redesign track asks: did quality improve without bloating the code?

These are simple predicates. There is no weighting, no multi-objective Pareto frontier, no learned reward model. Just a boolean: keep or discard. The simplicity is the point. A clear decision rule makes the loop predictable and easy to debug.

Block track: conservative compression

The block track targets a single HTML file and tries to make it shorter. It keeps a revision only when the line count drops and the composite quality score does not fall by more than 0.3. This is a conservative rule. It would rather discard a good compression than risk degrading the visualization.

Prompt track: aggressive speed

The prompt track targets the prompt text that generates a block. It rewrites the prompt to produce faster generation while preserving required D3 features. The bar is high: the rewritten prompt must generate at least 15% faster. Features are checked by grep, so they are either present or not. No partial credit.

Redesign track: creative improvement

The redesign track targets the same HTML file as the block track, but with the opposite priority. It optimizes for quality and tolerates modest code growth. Up to 20 additional lines are acceptable if they buy at least 0.1 composite score improvement. This makes it the most adventurous track. The proposer can add features, refine layouts, and improve interactions.

The Deferred Track

There is a fourth track not yet built: the skill track. It would iterate on SKILL.md files to improve the quality of blocks generated using those skills. This is the hardest variant because each experiment requires generating multiple blocks (four to six) to get a stable quality signal. A single block is too noisy. The cycle time balloons to 15 or 20 minutes per experiment.

The decision function would look something like: keep if the median composite across generated blocks improves by at least 0.2 and no individual block drops below 6.0. But that is speculative. The signal-to-noise ratio at that scale is still an open question.

Beyond D3

Nothing about this pattern is specific to D3 visualizations. The same loop could iterate on CSS stylesheets (optimize for file size, constrain visual regression tests). It could iterate on SQL queries (optimize for execution time, constrain result correctness). It could iterate on prose (optimize for readability score, constrain factual accuracy).

The requirements are modest. You need an artifact that a machine can modify. You need a quality signal that a machine can compute. And you need a decision function that encodes your values. The loop handles the rest.

The Redesign Constraint in Practice

The redesign track's +20 line budget is not arbitrary. It comes from watching what happens without it. Unconstrained quality optimization balloons code relentlessly. The proposer adds tooltips, legends, responsive breakpoints, accessibility labels. Each addition is defensible on its own. In aggregate, a 200-line block becomes a 400-line block, and the added complexity makes further iteration harder because the proposer has more code to reason about and more ways to break things.

Twenty lines is tight enough to force tradeoffs. If the proposer wants to add a tooltip (typically 15 to 25 lines), it has to remove something else or write the tooltip more concisely than it normally would. This produces a useful pressure: quality improvements that survive the constraint tend to be structural rather than additive. A better color scale, a cleaner axis layout, a smarter data join. These changes improve the composite score without growing the file because they replace weaker code rather than piling on top of it.

The 0.1 composite floor matters too. Without it, the proposer could make trivial changes that technically grow the code by zero lines and pass by default. The floor forces the proposer to attempt real improvements. Combined with the line cap, the decision function says: get meaningfully better, but do not get bigger. That is a harder optimization problem than either constraint alone, which explains the 38% keep rate. Most proposals either improve quality but exceed the line budget, or stay within budget but fail to clear the 0.1 bar.

Why the Prompt Track Is Hardest to Evaluate

The block and redesign tracks work with deterministic artifacts. You read a file, modify it, render it, and measure. The HTML is the same every time you render it. The prompt track lacks this stability because its artifact is a prompt, and the output of a prompt is stochastic. Generate from the same prompt twice and you get different code, different render times, different feature sets. The measurement is noisy by nature.

Generation time is the primary signal, and it varies widely between runs. A prompt that averages 98 seconds might produce a 75-second run and a 130-second run back to back. The 15% speedup threshold was chosen because it is large enough to stand above this noise most of the time, but there is no way to make a single-sample measurement reliable. Running multiple generations per experiment would reduce variance but would multiply cycle time from 3 minutes to 10 or 15, approaching the cost of the skill track.

Feature checking is the secondary signal, and it is brittle in a different way. The system uses grep to confirm that required D3 methods appear in the generated code. This catches gross failures (the prompt forgot to mention treemap) but misses subtle ones. A prompt might produce code that calls d3.treemap() but configures it incorrectly. The feature check passes. The visualization is wrong. No current guard catches this gap.

These two problems compound. Noisy timing means some good prompts get discarded because they happened to generate slowly on the test run. Brittle feature checks mean some bad prompts get kept because they happened to include the right method names despite producing broken layouts. The prompt track works, but it works with lower confidence per experiment than the other tracks. You need more iterations to trust the trajectory.

The Skill Track: Highest Leverage, Longest Cycle

The skill track is deferred because it is expensive, not because it is unimportant. If anything, it is the most important track. A SKILL.md file influences every block generated using that skill. Improve the skill file by even a small amount and the improvement propagates across dozens of future blocks. That is leverage the other tracks cannot match: block and redesign iterate on one file at a time, prompt iteration affects one prompt at a time, but skill iteration affects an entire class of outputs.

The cost comes from the evaluation side. To know whether a skill file improved, you have to generate multiple blocks with it and measure the distribution. A single block is too noisy. The proposer might have gotten lucky. Or the block might happen to be one the skill already handles well, masking a regression on other block types. Generating four to six blocks per experiment and computing the median composite score is the minimum viable signal. At roughly three minutes per block generation and audit, that is 12 to 18 minutes per experiment. A five-experiment run takes over an hour and costs several dollars in API calls.

The decision function would need to be conservative to justify that cost. A threshold of +0.2 median composite improvement with no individual block falling below 6.0 is a starting point, but even that raises questions. Which blocks do you generate? A fixed test set risks overfitting. A random sample risks missing regressions on specific block types. The sampling strategy is as important as the decision function, and no clear answer exists yet. This is why the track remains deferred: the design space is large, the experiments are slow, and a bad design wastes real money.

Composing Tracks

The tracks described above run independently. Each takes a block and iterates on it under one optimization objective. But there is no reason they must stay independent. Running tracks sequentially on the same block could combine their strengths: first compress, then improve quality on the compressed version. Or first improve quality, then compress the improved version.

The most natural composition is block track first, redesign track second. The block track strips out dead weight: redundant variables, verbose selections, unnecessary wrapper functions. This produces a leaner starting point for the redesign track. The redesign track then has a tighter code budget to work within (the +20 line cap applies to the already-compressed file), which forces it toward structural improvements rather than additive ones. The combined result should be both shorter and higher quality than either track alone would produce.

The reverse order also works but produces different results. Running redesign first means the quality improvements happen on a verbose codebase. The proposer has more room to maneuver, more lines to swap. Then the block track compresses the improved version, but it has to be careful not to undo quality gains. A line that looks redundant to the block track might be load-bearing for a tooltip the redesign track added. The block track's composite floor (-0.3) should protect against this, but the margin is tighter.

Composition order matters. Compress-then-redesign and redesign-then-compress produce different final artifacts. The order encodes a priority: which optimization gets to act on the cleanest starting state.

Amber: Risks in the Track Design

Composition order is not commutative. Compact-then-redesign produces a different artifact than redesign-then-compact. In the first ordering, the redesign track works with a tight file and makes surgical improvements. In the second, the block track compresses a file that already contains quality additions and may strip out improvements it does not recognize as valuable. The composite floor catches gross regressions, but subtle quality losses (a carefully chosen color scale replaced by a default, a readable variable name shortened to a single letter) can slip under the -0.3 threshold. There is no established best ordering, and the right choice likely depends on the block.

The prompt track optimizes for speed, not readability. A prompt rewritten for fast generation tends to be terser, more directive, and less explanatory. The generated code may be correct and fast to produce but harder for a human to understand. If the purpose of the block is to teach (and for many moonshine blocks it is), then optimizing generation speed may work against the block's reason for existing. The prompt track has no mechanism to detect this tradeoff. It checks features and timing, never clarity.

The skill track is expensive by design. Generating four to six blocks per experiment means four to six full generation and audit cycles. At roughly $0.075 per cycle, each skill experiment costs $0.30 to $0.45 before you know whether the change was an improvement. A 10-experiment run on one skill file costs $3 to $4.50. Across a library of 15 to 20 skills, a full skill optimization pass could cost $45 to $90 with no guarantee of meaningful improvement. The leverage is real but so is the bill, and the signal-to-noise ratio at this scale remains unproven.

No track optimizes for the reader. The block track optimizes for conciseness. The prompt track optimizes for generation speed. The redesign track optimizes for composite score. The composite score approximates visual quality, but visual quality is not the same as comprehension. A block can score 8.5 and still confuse the reader if the code is dense, the variable names are cryptic, or the visualization buries its main insight under decorative elements. The ultimate goal of a moonshine block is to help someone understand something. None of the tracks measure that directly. They measure proxies, and the gap between the proxies and the goal is the deepest risk in the whole system.
