The Threshold

How autoresearch decides what to keep

The decision function has two gates. Both must pass for an experiment to be kept. 

keep if: lines_after < lines_before
     AND composite_after - composite_before >= -0.3
discard otherwise

The first gate asks: did the code get shorter? The second asks: did quality hold up? That is the entire policy. Two simple checks. Everything else follows from the threshold value.

The quality gate

Why not require composite_delta >= 0? Because audits are noisy. Run the same block through the evaluator twice and you might see a 0.2-point swing. A tolerance of zero would reject experiments that actually preserved quality, just because the audit landed on the low side.

The value -0.3 was chosen empirically. It absorbs audit jitter while still catching real regressions. Below that line, something genuinely got worse.

The LOC gate

The code must actually shrink. Even by one line. This is a hard constraint, not a tunable parameter. Autoresearch exists to compress blocks. An experiment that does not reduce line count has failed at its primary objective, regardless of quality impact.

The tradeoff

The threshold is a dial between aggression and safety. Slide it toward zero and autoresearch becomes cautious: it only keeps experiments where quality held or improved. Slide it toward -1.0 and it becomes greedy: it accepts bigger quality drops in exchange for shorter code.

At -0.3, the system keeps about two-thirds of its experiments. That is enough to make steady progress on compression without letting quality erode over many rounds.

You just turned the dial yourself. The chart above recalculated every decision in real time. That is the same calculation the system runs after every experiment. One comparison. One threshold. Keep or discard.

An empirical constant

The value -0.3 is not derived from first principles. There is no theorem that says -0.3 is the right tolerance for a four-dimension audit composite scored on a 1-to-10 scale. It was chosen by running the system at several thresholds and observing which one produced the best outcomes over 20 iterations: blocks that were both shorter and not visibly degraded.

The honest account is that -0.3 was the threshold where the keep rate stayed high enough to make progress (roughly 65% of experiments kept) while catching the changes that actually broke something. Stricter values like -0.1 rejected too many good experiments. Looser values like -0.5 let regressions accumulate. The number stuck because it worked, not because it was proven optimal.

Audit noise

The evaluator is not a deterministic instrument. Run the same block through the same audit prompt twice and the composite score can shift by 0.2 to 0.5 points. Sometimes more. The LLM evaluator has its own variance: it notices different things on different runs, weights subjective criteria differently, and occasionally hallucinates problems that do not exist.

This noise means that the measured delta between "before" and "after" is itself noisy. An experiment with a true quality delta of zero might measure at +0.3 on one run and -0.2 on the next. A threshold of 0.0 would keep the experiment half the time and reject it half the time, depending on which evaluation the system happened to see. That is not a decision function. That is a coin flip.

Setting the threshold at -0.3 gives the system a noise margin. It says: I will accept that measured quality dropped by up to 0.3 points, because that drop might be measurement error rather than a real regression. Only when the composite falls further than that do I trust the signal over the noise.

This is the same logic behind statistical significance thresholds in experimental science. You pick a boundary that accounts for the known variability of your measurement instrument. The evaluator's variability is roughly plus or minus 0.25, so -0.3 sits just outside that noise band.

The conjunction gate

The two gates form a conjunction: the code must get shorter and quality must hold. Not one or the other. Both. This matters more than it seems.

A disjunction (shorter or quality holds) would be useless. Every experiment that does not shrink the code but happens to preserve quality would pass. The system would keep no-ops. A disjunction in the other direction (quality improves or code shrinks) is nearly as bad: it lets the system accept massive quality drops as long as the code got one line shorter.

The conjunction is strict. It means there are experiments that improve quality by a significant margin but get discarded because they added two lines of code. That feels wasteful. A block that jumps from 6.5 to 8.0 on the composite but grows by three lines will be thrown away. The system does not care why the code grew. It does not distinguish between "added a useful comment" and "introduced redundant logic." Shorter is shorter. Longer is discard.

This rigidity is a feature. Autoresearch exists to compress blocks. A system that sometimes accepts longer code would need a second mechanism to re-compress it later, which doubles the complexity of the control loop. The hard LOC gate keeps the objective function monotonic: every kept experiment moves the line count in one direction. The quality gate ensures that direction does not come at the cost of a broken visualization.

What happens at other thresholds

The experiment data in the scatter plot above comes from real runs. We can replay that history under different threshold values and see what would have happened. The chart below does exactly that: it simulates the full experiment sequence under five thresholds and tracks three outcomes.

Three metrics, five thresholds. The keep rate tells you how aggressively the system filters. The cumulative LOC reduction tells you how much compression survived all gates. The final composite tells you whether quality held. A good threshold maximizes LOC reduction without letting the composite drift too far below zero.

At 0.0, the system is maximally conservative. Only experiments where the composite did not drop at all get through. The keep rate hovers around 45%, and cumulative LOC savings are modest because half the compression opportunities are thrown away. Quality holds steady, but the system is leaving compression on the table.

At -0.1, the keep rate climbs to about 55%. A few more experiments survive, and cumulative LOC savings improve. Quality still holds well. This is a reasonable setting if you trust your evaluator and want to be cautious.

At -0.3, the production threshold, the keep rate reaches roughly 65%. This is where the system makes the most progress per iteration. LOC savings are substantially higher than at -0.1, and the cumulative composite delta stays within a tolerable range. Most of the kept experiments at this level are genuine improvements or neutral changes measured through noise.

At -0.5, the system starts accepting experiments with real quality drops. The keep rate reaches 75%, and LOC savings jump, but the cumulative composite tilts negative. Over 20 iterations, the block's quality has measurably degraded. The extra compression is not free.

At -1.0, nearly everything passes. The keep rate exceeds 90%. LOC savings are maximal because almost no compression is rejected. But the cumulative composite delta drops sharply. The block is shorter, yes, but it has also lost significant quality. Run this for 20 rounds and you get a block that is half its original size and noticeably worse.

Per-dimension thresholds

The production system applies one threshold to the composite score. But the composite is a weighted average of four dimensions: visual_critic (0.30), encoding_integrity (0.25), cognitive_load (0.25), and stress_test (0.20). A single threshold treats all dimensions as equally important to protect. They are not.

Visual quality is the dimension users notice first. A block that looks wrong fails regardless of how clean its encoding is. Stress test failures, on the other hand, only manifest under edge cases that most viewers never encounter. This suggests that the threshold for visual_critic should be tighter (closer to zero) than the threshold for stress_test (which can afford to be looser).

The panel below lets you set independent thresholds for each dimension. Instead of checking composite_delta >= -0.3, the system checks each dimension against its own gate. An experiment is kept only if every dimension passes its own threshold (and the code got shorter). Try tightening visual_critic to -0.1 while loosening stress_test to -0.8. Some configurations produce higher keep rates with the same or better quality outcomes.

The interesting finding is that a configuration with a tight visual_critic threshold (-0.1) and a loose stress_test threshold (-0.8) produces nearly the same keep rate as the uniform -0.3 threshold, but with better visual quality outcomes. The system discards the experiments that damage appearance while accepting the ones that only affect edge-case robustness. Whether that tradeoff is acceptable depends on the block and its audience.

Per-dimension thresholds add complexity. Instead of one number to tune, there are four. The interaction effects are not obvious: loosening cognitive_load while tightening encoding_integrity can change which experiments survive in non-intuitive ways. The production system uses a single composite threshold because it is easier to reason about, easier to debug, and good enough for most blocks. Per-dimension thresholds are worth exploring if you have blocks where one dimension consistently degrades while others improve.

Risks

The threshold is a global constant. Every block, regardless of its structure, complexity, or starting quality, faces the same -0.3 gate. A block that starts at a composite of 9.2 has very little room to drop before the regression becomes visible. A block that starts at 5.0 can absorb a 0.3 drop without anyone noticing. The same threshold means different things for different blocks.

Audit noise makes the boundary fuzzy. Experiments with true deltas between -0.1 and -0.5 sit in a zone where the keep/discard decision is effectively random. The evaluator might score them above -0.3 on one run and below it on the next. Roughly 15% of experiments land in this ambiguous zone. For those, the threshold is not a principled gate. It is a lottery.

The conjunction gate blocks quality-improving growth. An experiment that improves the composite by +1.5 but adds four lines of code will be discarded. The system cannot distinguish between "the code grew because the change was sloppy" and "the code grew because the change added a necessary feature that was missing." Both are rejected equally. Over many iterations, this creates a bias toward compression at the expense of completeness.

The composite hides dimension-level damage. A change that raises visual_critic by +1.0 while dropping stress_test by -1.0 can produce a composite delta near zero and pass the threshold. The composite says "quality held." The block's stress resilience disagrees. Per-dimension thresholds address this, but the production system does not use them.
