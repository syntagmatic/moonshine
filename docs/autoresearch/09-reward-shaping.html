Reward Shaping

Designing objective functions for the autoresearch iterate loop

The composite score is a lie you tell yourself to make a scalar out of a vector.

Four dimensions go in: visual_critic at 0.30, encoding_integrity at 0.25, cognitive_load at 0.25, stress_test at 0.20. One number comes out. The system compares this number to the previous number, checks whether the delta exceeds -0.3, and decides: keep or discard. Simple. Legible. And fundamentally lossy.

Consider what the composite hides. A change that raises visual_critic from 6 to 8 while dropping stress_test from 8 to 5 will produce a composite delta near zero. The composite says "nothing changed." The reality is that something changed quite a lot, in two opposite directions, and the weighted sum happened to cancel them out. The system kept the experiment. A human reviewer would have questions.

The problem with a single number

Fixed weights encode a permanent opinion about which dimensions matter most. But "which dimension matters most" depends entirely on where the block currently stands. A block scoring 4 on visual_critic and 9 on stress_test has a clear weakness. Improving visual quality by 2 points is far more valuable than improving stress_test by 2 points. The fixed-weight composite treats them as nearly equal (0.30 vs 0.20 weight).

Worse, the composite invites dimension collapse: the optimizer learns to pump one high-weighted dimension at the expense of others. If visual_critic carries 30% of the score, an experiment that gains 1 point on visual quality but loses 0.5 on encoding integrity will show a positive composite delta. Over many rounds, the block drifts toward a local optimum that looks good on one axis and progressively worse on the rest. 

current weights:
  0.30 x visual_critic       # highest weight, most pumpable
  0.25 x encoding_integrity
  0.25 x cognitive_load
  0.20 x stress_test         # lowest weight, most neglectable

Three alternatives

There are at least three ways to build a better objective function. Each solves a different failure mode, and each introduces its own costs.

1. Pareto dominance

Instead of reducing the four dimensions to a scalar, treat the score as a vector. An experiment is Pareto-dominant if it improves at least one dimension without worsening any. If it improves visual quality but degrades stress test robustness, it is not strictly better. It sits on a different point of the Pareto frontier: the set of all configurations where you cannot improve one dimension without hurting another.

Under Pareto selection, the system would keep an experiment only if it dominates the current state: every dimension at least as good, at least one strictly better. This prevents silent tradeoffs. The human sees the frontier and decides which tradeoff to accept.

2. Adaptive weighting

Keep the composite, but let the weights shift. The intuition: weight each dimension inversely to its current score. A dimension scoring 4/10 gets more weight than one scoring 9/10. The system automatically focuses on the weakest link. 

adaptive_weight(dim) = base_weight(dim) * (10 - score(dim)) / normalizer

When visual_critic is at 4 and stress_test is at 9, the adaptive weights might be 0.42 for visual and 0.08 for stress. The optimization pressure concentrates where the block needs it most. As visual quality climbs, its weight drops, and the next-weakest dimension takes priority. This is a soft version of curriculum learning.

3. Threshold constraints

Set hard floors. Stress_test must stay at 6 or above. Encoding_integrity must stay at 5 or above. Any experiment that drops a dimension below its floor is discarded, regardless of gains elsewhere. Above the floors, optimize freely using whatever objective you like.

This separates "acceptable" from "optimal." The floors prevent catastrophic regression in any single dimension. The optimization above the floors can be aggressive because the worst outcomes are gated out.

Curriculum: weakest-first scheduling

Curriculum learning takes adaptive weighting to its logical conclusion. Instead of blending all dimensions with shifting weights, focus on one dimension at a time. Identify the weakest dimension. Optimize for it specifically. Once it crosses a threshold, move to the next weakest.

This avoids the blurriness of a composite entirely. Each iteration has a single, clear objective. The proposer knows exactly what it is trying to improve. The audit knows exactly what to scrutinize. The decision function is simple: did the target dimension improve without the others collapsing?

The downside is scheduling. How do you pick the thresholds for advancing? What if improving dimension A necessarily hurts dimension B, so the curriculum oscillates between them? These are hyperparameters of the curriculum, and they themselves need tuning.

LOC as a proxy metric

Lines of code is the compaction track's primary gate. Code must get shorter. But shorter code and better code overlap without coinciding. A 200-line visualization can be worse than a 300-line one if the extra hundred lines are meaningful interaction code, accessible labels, or responsive layout logic. The LOC gate cannot tell the difference between removing dead code and removing necessary complexity.

The same proxy problem applies to every metric in the composite. A stress_test score of 8 means the evaluator found no rendering failures under the test conditions it checked. It does not mean the block is robust. The score is a proxy for robustness, not robustness itself. The composite is a proxy of proxies.

Asymmetric regression penalties

The current system applies a uniform -0.3 tolerance. But regressions are not equal. A 1-point drop in visual_critic is immediately visible to any viewer. A 1-point drop in stress_test may only manifest under edge-case data. A 1-point drop in encoding_integrity might mean a missing axis label that 80% of viewers will never notice.

An asymmetric penalty schedule would weight regressions differently: 

regression_penalty:
  visual_critic:       -0.15  # tight: regression is obvious
  encoding_integrity:  -0.25  # moderate
  cognitive_load:      -0.25  # moderate
  stress_test:         -0.40  # loose: less visible, noisier signal

This acknowledges that not all regressions are created equal. The system tolerates more noise in dimensions where the signal itself is noisy, while guarding dimensions where even small drops are perceptible.

Tradeoffs in every direction. Pareto optimization is slower to converge. It keeps more experiments because partial improvements are not rejected, but it also means less aggressive pruning. Adaptive weights can oscillate: improving the weakest dimension shifts the weights, making a different dimension weakest, pulling the optimizer back. Curriculum scheduling adds hyperparameters (thresholds, ordering rules) that themselves need tuning. There is no free lunch here. Each strategy trades one failure mode for another.

What changes in practice

The composite works. It has shipped 107 blocks through compaction. But it works the way a hammer works: reliably, bluntly, without knowing what it is hitting. The approaches described here are scalpels. They cut more precisely, but they require more setup and more judgment about where to cut.

If the autoresearch loop were running thousands of iterations per block, the composite would be fine. Noise averages out at scale. But the loop typically runs 5 to 15 iterations per block. At that scale, every decision matters. A single bad keep (dimension collapse) or bad discard (rejecting a Pareto improvement because the composite dipped) can shape the entire trajectory.

The practical path forward is probably layered: threshold constraints as hard gates, adaptive weighting as the optimization signal, Pareto tracking as a diagnostic. Not one strategy replacing the composite, but three strategies working together to compensate for its blind spots.
