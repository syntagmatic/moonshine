The Loop

How d3-power-tools uses autoresearch to iteratively improve visualizations.

Andrej Karpathy described a pattern he called autoresearch: let an AI agent modify code, run an experiment, check a metric, keep or discard, and loop. In his framing the experiment is usually a training run and the metric is a loss number.

The idea is simple. Make a change. Measure whether it helped. If yes, keep it. If not, revert. Repeat.

What makes the loop work is not cleverness in any one step. It is the strict separation between the part that proposes changes and the part that judges them. They run in separate processes with separate contexts, so the judge cannot be swayed by the proposer's intent.

Adapting for Visualization

Training loss is a single number. Visualization quality is not. A chart can be correct but ugly, or beautiful but misleading. So the "experiment" in d3-power-tools is not a loss curve. It is a render step followed by a multi-dimensional audit.

Four inspection skills run in parallel on every rendered screenshot. Each scores a different aspect of the block. Their scores combine into a single composite number that the loop can compare against the previous iteration.

The proposer's job is to make the block shorter and cleaner. The auditor's job is to make sure quality did not drop. Together they try to compress a block down to its essential form.

Why Separation Matters

The proposer and auditor run in different claude -p subprocesses. The auditor never sees the proposer's explanation of what it changed. It only sees the rendered result: a screenshot.

This prevents a subtle failure mode. When a judge reads a description like "consolidated color scale definitions for clarity," it is tempted to accept the change because the intent sounds reasonable. That is motivated reasoning. The description biases the evaluation.

By keeping the auditor blind to intent, d3-power-tools tries to make the evaluation honest. The auditor scores what it sees, not what the proposer meant to do. If a "cleanup" change accidentally breaks a tooltip, the auditor catches it because it inspects the actual render, not the diff.

This is the same principle behind double-blind peer review. The judge evaluates the work, not the author's cover letter.

In practice this means the loop is slow. Each iteration takes 30 to 90 seconds because it has to render, screenshot, and score from scratch. But slow and honest beats fast and credulous.

The Origin Story

In early 2025 Andrej Karpathy posted a short thread describing what he called "autoresearch." He had been running experiments on small language models, and noticed that the hardest part was not coming up with ideas for what to try next. It was remembering to actually measure each change in isolation, record the result, and decide cleanly whether to keep it. The cognitive overhead of bookkeeping swamped the creative part.

His solution was to hand the bookkeeping to a script. The script would read the current code, make one change, run an evaluation, compare the metric to the previous baseline, and commit or revert. The human set the direction. The machine handled the discipline. The key realization was that the agent did not need to be brilliant at any one step. It just had to be consistent and tireless. A mediocre change that gets tested and measured is more useful than a brilliant change that never gets evaluated because the researcher got distracted halfway through.

Karpathy's original framing was about training loss on neural networks. The metric was a single number. The change was a hyperparameter tweak or a small architectural modification. But the pattern generalizes to any domain where you can render an artifact and score it. d3-power-tools adapted the same loop for visualization: the artifact is a rendered SVG screenshot, the score is a multi-dimensional audit, and the change is a code edit.

Why One Change at a Time

The single-change constraint is not a limitation. It is the mechanism that makes the loop trustworthy.

When you change two things at once and the score goes up, you do not know which change helped. Maybe one improved things and the other was neutral. Maybe both individually hurt but together they cancel out. You have learned nothing reusable. The next iteration has no signal to build on.

This is the same principle behind controlled experiments in science. You vary one factor and hold everything else constant. If the outcome changes, you can attribute it to the factor you varied. If you vary three factors simultaneously, you are left with a confounded result. It might replicate. It might not. You cannot tell why.

In the autoresearch loop, the "one change" discipline means every kept iteration represents a causally clean data point. The change was X, the effect was Y, and nothing else moved. Over thirty iterations this builds a legible history. You can look back and see which kinds of changes reliably improve scores and which kinds reliably do not.

Single-change isolation trades speed for legibility. You could try five changes at once and move faster. But you would learn nothing about which changes matter, and you would have no way to debug a regression when one eventually arrives.

Separation as Blinding

Clinical drug trials use double-blinding for a reason. If the doctor knows which patient got the real drug, the doctor's expectations leak into the assessment. Subtle things shift: how long they spend on the examination, which symptoms they probe, how they interpret ambiguous results. The bias is not conscious. It is structural. The information is present, so it influences the judgment.

The proposer-auditor separation in the autoresearch loop is a form of blinding. The auditor subprocess receives a screenshot and nothing else. It does not know what the proposer changed. It does not know whether the proposer expected the change to improve things. It does not know whether the last three experiments were kept or discarded. It evaluates the image as if seeing it for the first time, because it is.

This removes a specific failure mode that occurs when the same agent both writes code and evaluates it. The agent remembers its own reasoning. It knows that it moved the legend to reduce occlusion, so when it evaluates the result, it looks at the legend placement and thinks "yes, that is better." But maybe the legend now overlaps a data label. The agent does not notice because it was not looking for that problem. It was looking for confirmation that its change worked.

Blinding the auditor forces a fresh evaluation every time. The auditor has no hypothesis to confirm. It just describes what it sees and scores it. This is not a guarantee of correctness. The auditor can still make mistakes. But at least those mistakes are uncorrelated with the proposer's intent.

The Loop as Evolution

There is a clean mapping between the autoresearch loop and evolutionary algorithms. A population of code variants corresponds to a generation. Each variant has a fitness (the composite score). In each generation, one variant mutates (the proposer makes a change). The mutant gets evaluated (the auditor scores the screenshot). If the mutant is fitter than the weakest member, it replaces it. Otherwise it is discarded. Over many generations, the population's average fitness climbs.

The d3-power-tools loop is a simplified version of this: a population of size one. There is only the current baseline. A mutation either beats it or does not. This is equivalent to a (1+1) evolutionary strategy, the simplest possible evolutionary algorithm. It has no crossover, no population diversity, no selection pressure beyond "better than the current best."

Isolation Architecture

The loop runs inside a git worktree. This is an underappreciated detail that makes the whole system safer.

A git worktree is a second checkout of the same repository, pointed at a different branch. The main branch stays clean. All experiments happen on an iterate branch in the worktree directory. If the experiments go badly, you delete the worktree and nothing on main was ever touched. If they go well, you squash-merge back to main: one clean commit summarizing all the improvements.

This solves a practical problem. During iteration, the proposer is writing and rewriting code rapidly. Intermediate states are often broken. If those writes happened on the main branch, any interruption (a crash, a timeout, the user hitting Ctrl-C) could leave main in a broken state. The worktree isolation means the worst case is that you lose the experimental branch. Main is always stable.

The flow is: branch off main into a worktree, run N iterations on the branch, squash-merge the final state back to main. The figure below animates this sequence.

The squash merge is important. It means main's history is clean. Each commit on main represents a net improvement, not the thirty intermediate steps that got there. If you need to bisect a regression later, you have a small number of meaningful commits to test, not hundreds of micro-edits.

Risks and Limitations

Amber risk: The loop has structural blind spots that no amount of iteration will fix.

  • Garbage in, garbage out. The loop assumes the auditor is reliable. If audit scores are noisy or systematically biased, the loop optimizes for the wrong thing. A flawed auditor that consistently overscores cluttered layouts will drive the loop toward cluttered layouts. The loop cannot detect this because it has no external reference point. It trusts the auditor the way a gradient descent trusts its loss function.
  • No synergy detection. Single-change isolation cannot find improvements that require two simultaneous changes. Suppose moving the legend left and widening the chart together produce a better layout, but either change alone makes things worse. The loop will reject each change individually and never discover the pair. This is the same limitation as coordinate descent in optimization: it follows one axis at a time and cannot traverse diagonal ridges.
  • Greedy search. The loop never accepts a regression. It never walks downhill to reach a higher peak on the other side of a valley. If the current block is at a local optimum, the loop will stay there forever, rejecting every change. There is no simulated annealing, no exploration budget, no mechanism for escaping local optima. The loop is purely greedy.

These limitations are real but bounded. Noisy auditors can be improved by calibration (running the auditor on known-good and known-bad examples and adjusting). Synergy blindness matters less in practice because most useful code improvements are independent. Greedy search works well when the fitness landscape is smooth, which it usually is for code cleanup tasks where you are removing redundancy rather than restructuring architecture.

The honest summary: the loop is a workhorse for incremental improvement. It is not a replacement for human judgment about whether the overall direction is right. It compresses code and maintains quality along the dimensions the auditor can measure. It cannot invent a new design, discover a missing feature, or question the auditor's criteria. Those remain human jobs.
