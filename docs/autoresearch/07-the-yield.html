The Yield

Where autoresearch works and where it doesn't

Some code wants to be shorter. Some doesn't. Autoresearch can shave 25% off a verbose, machine-generated block in a few dozen experiments. But hand it a tightly-coupled animation state machine and it will burn through API calls producing nothing usable.

The difference is not about size. It is about structure. Verbose code with redundant patterns has slack in it. The proposer can find that slack the same way a human editor would: collapsing repeated selectors, merging similar event handlers, lifting shared constants. But when every line is load-bearing, there is no slack to find.

The sweet spot

The best candidates for autoresearch share a pattern. They started as Sonnet output with structural redundancy: duplicate DOM queries, verbose event handlers, inline styles that could be constants. The proposer sees these patterns and compresses them reliably.

Collapsible Tree Search went from 694 lines to 522. Twenty of twenty-seven experiments were kept. That is a 74% keep rate with 25% compression. The code was full of repeated d3.select calls and duplicated toggle logic that collapsed cleanly.

The pattern: verbose, machine-generated code with non-idiomatic D3 compresses easily. 25% reduction is typical. Keep rates above 60% are common. These blocks are the sweet spot for autoresearch.

The dead zone

At the other end sit blocks where the proposer cannot help. Hierarchy Morphs ran four experiments and kept zero. Every attempted simplification broke a transition. Shape morphing animation is tightly coupled: change one interpolation and three others fall out of sync.

Blockbuilder Explorer is a different flavor of resistance. At 1,120 lines with crossfilter over 34K rows, the screenshot-based audit cannot verify whether interactions still work. The stress test consistently fails. The code did not shrink at all.

Canvas rendering pipelines, crossfilter indexes, KDE calculations. These are dense, load-bearing structures. The proposer can suggest changes, but the evaluator catches the breakage. Keeps go to zero.

The leaderboard

The diagnostic

If keep rate drops below 30% in the first six experiments, stop. You are spending API budget fighting an optimum. The code is already near its floor, or its structure resists compression. Either way, more experiments will not help.

Violin Plot Orchestra is the clearest example. Nine experiments, one keep, seven discards. The code is already near-optimal. Changes tend to break KDE calculation or violin rendering. The evaluator correctly rejects them. But each rejection costs the same as a success.

The spectrum

The scatter plot reveals the relationship. Large files tend to land lower on the Y axis (more compression). But size alone does not determine yield. Blockbuilder Explorer is the largest file and compressed zero percent. Hierarchy Morphs is mid-sized and also stuck at zero.

What separates them from Hierarchy Bundles (1,012 to 748 lines, a 26% reduction) is structure. Bundles had clear layering. Animation code sat on top of layout code, and the proposer could compress each layer independently. Morphs had one inseparable layer.

The signal

Knowing where your code sits on this spectrum is itself a useful signal about code quality. A high keep rate means the code had redundancy in it. A zero keep rate means every line was doing real work.

Neither is inherently good or bad. But if you feed autoresearch a block and it keeps nothing, that tells you something. The code is tight. Maybe it is well-written. Maybe it is just fragile. The yield tells you which, if you pay attention to the "why."

The 30-in-6 rule

Six experiments is enough to read a block's character. If keep rate sits below 30% after those first six, stop. You are fighting a local optimum. The code is either already near its floor or its structure resists the kinds of transformations the proposer can generate. Continuing past that point burns API budget on experiments that will be discarded.

Violin Plot Orchestra is the textbook case. After six experiments, one was kept and five were discarded. Running three more experiments produced zero additional keeps. The first six experiments carried the entire signal. The last three produced only cost.

Contrast that with Collapsible Tree Search. After six experiments, five were kept. The signal was unmistakable: this code has slack. Running twenty-one more experiments was justified because each one had a reasonable chance of landing.

The heuristic: keep rate below 30% after 6 experiments means further spending has negative expected value. Cut the block loose and move budget to a better candidate.

Budget allocation

Not all blocks deserve equal investment. A block with 80% keep rate after its first few experiments will convert almost every dollar spent into LOC saved. A block with 10% keep rate will waste nine out of ten dollars. Rational allocation follows from this asymmetry: spend more on high-yield blocks, spend less on resistant ones, and skip dead zones entirely.

The tricky part is that keep rate is not fixed. It can shift as the code tightens. Collapsible Tree Search started at 80% keep rate but will eventually hit a floor where further compression breaks things. Budget allocation is not a one-time decision. It is a rolling estimate, updated after each batch of experiments.

In practice, the allocation looks like a power law. The top three blocks by keep rate absorb most of the budget. The bottom three get a token allocation or nothing. The middle four get moderate investment proportional to their estimated yield. This is not a formal optimization. It is a triage heuristic.

The portfolio view

Zoom out from individual blocks and look at the whole collection. Across 100+ blocks, autoresearch has a total cost and a total yield. The relevant metric is not compression on any single block. It is aggregate ROI: how many lines of code were saved per dollar spent.

A rough accounting: each experiment costs roughly $0.15 in API calls (proposer + evaluator). Across the blocks shown here, the total experiment count is around 260. That is roughly $39. Total lines saved across all blocks is around 440. That puts the cost at about $0.09 per line removed. Whether that is a good deal depends on what those lines were doing. Removing 170 lines of redundant DOM queries from a block you will maintain for years is a better investment than removing 12 lines from a block you will never touch again.

The portfolio also reveals concentration risk. A handful of high-yield blocks account for most of the total savings. Hierarchy Bundles alone saved 264 lines. If that single block had been resistant instead, the portfolio-level ROI would drop significantly. Diversifying across many blocks hedges against this, but only if you are willing to accept that most blocks will yield modest returns.

Amber risk: Three assumptions behind this yield analysis are shakier than they look.

  • Yield prediction is unreliable. A block that looks resistant in the first three experiments may break through on experiment four. The 30-in-6 rule is a heuristic, not a guarantee. Cutting a block early saves budget but occasionally abandons a block that was one experiment away from unlocking. The false-negative rate is unknown because we do not run the counterfactual.
  • The portfolio view assumes independence. In reality, blocks share skills and patterns. An insight discovered while iterating on Collapsible Tree Search (collapsing redundant selectors) transfers to other tree-layout blocks at zero marginal cost. The portfolio model treats each block as an isolated investment, but the real returns are correlated. A breakthrough on one block can shift the yield curve for a dozen others.
  • Optimizing for yield produces mediocrity. If you allocate budget strictly by projected ROI, you spend heavily on blocks where cheap compression is easy. These tend to be verbose, structurally redundant files where the proposer shaves off boilerplate. The result: lots of cheaply compacted, middling-quality blocks. The blocks that actually need investment (complex animations, dense algorithms, tightly-coupled state machines) get starved because their yield profile looks bad. Yield optimization and quality optimization pull in different directions.
