Transfer Learning Across Blocks

How insights from iterating one visualization can accelerate iteration on similar ones

Every block starts from scratch. The 50th block the system iterates learns the same lessons the 5th one did. "Consolidating color scales breaks the density encoding." "Removing the charge force collapses the layout." "Merging tick handlers kills the animation." Each of these lessons was paid for in API budget and experiment time. And then forgotten.

The experiment JSONs from 100+ iterated blocks contain thousands of proposer explanations and audit notes. This is a corpus of what works and what fails for D3 visualization improvement. But today, none of that accumulated knowledge flows from one block to the next. Every new iteration run starts cold.

Transfer learning is the idea that it does not have to work this way. A lesson learned on one bee swarm visualization should carry over to the next bee swarm. A color scale fix discovered on a parallel coordinates block should apply to any block with a similar color setup. The question is how.

Four strategies

Transfer can happen at different levels of abstraction. The strategies below go from narrow (one skill type) to broad (any block pair that shares a code pattern).

  1. Skill-level transfer. All blocks generated by the same skill share common structure. Parallel coordinates blocks share axis drawing code, brushing logic, path generators. A lesson from iterating one parallel-coords block ("do not remove the brush extent clamp; it prevents selection from escaping the axis") applies to every other parallel-coords block. This is the narrowest form of transfer: same skill, same patterns, same pitfalls.
  2. Pattern-level transfer. Across skills, blocks share code patterns even when the visualizations look nothing alike. Tooltip placement code appears in bee swarms, network graphs, and treemaps. Color scale setup appears everywhere. Axis formatting is universal. When the proposer learns that a particular tooltip refactoring breaks hover interaction, that lesson applies to any block that uses the same tooltip pattern, regardless of skill type.
  3. Failure library. A structured database of "this change failed because..." indexed by pattern type. Before proposing a change, the proposer checks: has a similar change on a similar block failed before? If the library says "removing redundant d3.select calls from axis setup breaks update transitions" and the current block has the same axis pattern, skip that proposal. The library turns individual failures into collective memory.
  4. Meta-prompts. After iterating 10 blocks of a given type, distill the accumulated lessons into a prompt fragment that guides future proposers. "When iterating force-directed graphs, avoid: removing the charge force, consolidating tick handlers, inlining the drag behavior." This is the most compressed form of transfer: hundreds of experiments condensed into a few sentences.

Skill-level transfer in practice

Consider the three parallel coordinates blocks in the figure. All three share a common structure: parallel axes, polyline paths, optional brushing. When the system iterated the first one, it discovered that consolidating the axis scale setup into a shared function reduced lines but broke per-axis domain clamping. That experiment was discarded.

The second parallel-coords block hit the same wall. And the third. Three blocks, three wasted experiments, the same lesson learned three times. With skill-level transfer, the second and third blocks would have started with the knowledge: "do not consolidate per-axis scale setup."

The unit of skill-level transfer is the iteration record. Each experiment in the JSON log contains the proposer's intended change, the audit result, and the keep/discard outcome. Filter these records by skill tag, group by failure mode, and you have a per-skill transfer corpus.

Pattern-level transfer

Skill-level transfer is clean but narrow. Most blocks share patterns that cross skill boundaries. Tooltip placement logic shows up in 70% of blocks. Color scale setup is nearly universal. Responsive resize handlers recur across every skill type.

Pattern-level transfer requires a different index. Instead of grouping by skill tag, group by code pattern. A lightweight approach: extract function signatures and D3 method chains from each block, then compute pairwise similarity. When two blocks share a tooltip pattern, lessons about that pattern transfer, even if one is a bee swarm and the other is a force-directed graph.

A heavier approach: embed each block's code as a vector (using the code itself, the audit profile, or both) and find nearest neighbors in embedding space. The nearest iterated block becomes the source of transfer. Its successful changes become candidate proposals. Its failed changes become warnings.

The failure library

The failure library inverts the usual framing. Instead of asking "what worked?" it asks "what broke?" Every discarded experiment is a data point: a change that seemed reasonable, applied to a specific code pattern, that the audit caught as a regression.

Structure it as a lookup table. Key: the type of change (e.g., "consolidate color scales," "inline drag handler," "remove redundant selections"). Value: the blocks where this change was tried, the pattern context, and the audit dimensions that regressed. Before proposing, the proposer queries the library. If the proposed change matches a known failure pattern on a structurally similar block, the proposer skips it or adjusts.

The failure library is cheap to build. The experiment JSONs already contain the raw material. It just needs indexing. The cost is maintaining it as the codebase evolves and making sure the index stays current.

Meta-prompts

After enough blocks of a given type have been iterated, the accumulated lessons can be distilled into a prompt fragment. This is the most compressed form of transfer. A meta-prompt for force-directed graph iteration might read:

Example meta-prompt. "When iterating force-directed graphs: (1) never remove d3.forceCollide or d3.forceManyBody without verifying layout stability; (2) tick handler consolidation breaks when nodes have heterogeneous update logic; (3) extracting link styling into CSS saves lines but breaks dynamic coloring; (4) drag behavior is load-bearing and should not be inlined."

Meta-prompts are human-readable, auditable, and easy to inject into the proposer's context window. They represent the final stage of transfer: raw experiment data, compressed through the failure library, distilled into natural language guidance.

The skill track

The autoresearch system already has the concept of tracks: the block track compresses individual files, the prompt track improves the iteration prompts. There is a deferred track that has not been built yet: the skill track.

The skill track closes the loop. It measures quality across all blocks generated with a given skill, identifies the skill's systemic weaknesses, and updates the skill text. Transfer learning is the mechanism: aggregate experiment outcomes by skill, find recurring failure patterns, and rewrite the skill to avoid them.

This makes the skill a living document. It starts as a static set of instructions. After the first ten blocks are iterated, the skill absorbs what was learned. After fifty blocks, it encodes a deep map of what works and what does not for that visualization type. The skill becomes the canonical location for transferred knowledge.

Embedding similarity

The strategies above rely on explicit structure: skill tags, pattern signatures, failure categories. Embedding similarity offers a complementary approach that works without any manual indexing.

Represent each block as a vector, derived from its source code, its audit profile (the four dimension scores), or a combination. When a new block enters iteration, find its nearest neighbors among already-iterated blocks. Pull the top experiments from those neighbors. Successful experiments become candidate proposals. Failed experiments become warnings.

The advantage is generality. Embedding similarity can discover transfer relationships that no one anticipated. A treemap and a sunburst might share more code structure than two treemaps with different interaction patterns. The embeddings surface this without anyone having to define a taxonomy.

The risk is noise. Two blocks that are close in embedding space might share superficial features (both use d3.scaleOrdinal) without sharing deep structural patterns. The transferred lesson might not apply. This is the false transfer problem.

Risks

False transfer. A lesson from one block type does not always apply to another, even when they look similar. "Removing the charge force collapses the layout" is true for force-directed graphs but meaningless for radial dendrograms. If the similarity index is noisy, the proposer inherits bad advice and wastes experiments on inapplicable changes. Worse, it might apply a change that passes the audit on a superficially similar block but degrades a subtle interaction.

Cold start. Transfer learning requires a corpus. For well-represented skill types (parallel coordinates, tree layouts), there are enough iterated blocks to build a meaningful transfer set. For novel or rare block types, there is nothing to transfer from. The system falls back to starting from scratch, which is exactly where it is today. The gap between "has transfer data" and "does not" creates two tiers of iteration quality.

Maintenance cost. The failure library and meta-prompts are not free. The library needs indexing, deduplication, and periodic pruning as skills evolve and code patterns change. Meta-prompts can go stale: a lesson that was valid six months ago might not apply after a skill rewrite. If the maintenance burden exceeds the transfer benefit, the system accumulates outdated advice.

Staleness. Meta-prompts crystallize lessons at a point in time. As skills evolve, blocks change, and the proposer improves, those lessons may become obsolete or actively misleading. A meta-prompt that says "avoid inlining drag behavior" might become wrong after a refactor that makes drag handlers composable. The system needs a mechanism to retire or update stale guidance.

What this changes

Today, the iteration pipeline treats every block as independent. The total cost of iterating 100 blocks is 100 times the cost of iterating one. Transfer learning bends that curve. Each subsequent block benefits from the blocks that came before it. The 50th bee swarm iteration starts with the distilled knowledge of the first 49.

The practical path is probably incremental. Start with skill-level transfer, because it is the lowest-risk form: same skill, same patterns, high confidence that lessons apply. Build the failure library as a byproduct of normal iteration (the data already exists in the experiment JSONs). Introduce meta-prompts when a skill has accumulated enough iteration data to make distillation worthwhile. Save embedding similarity for later, when the infrastructure to compute and index vectors is justified by scale.

The skill track is the natural home for all of this. It already exists conceptually. Building it means wiring transfer learning into the place where it can do the most good: the instructions that shape every block a skill generates from the start.

The cold start problem

Transfer learning works once you have data. But the first block of a new type has no transfer data. The first parallel coordinates plot in the system has never seen another parallel coordinates plot succeed or fail. It starts from zero: no failure library, no distilled strategy, no hit-rate table. Every experiment is a fresh guess.

This is the cold start problem, and it compounds in the early rounds. Without transfer data, the proposer falls back on generic heuristics: remove dead code, clean up formatting, simplify nesting. These are safe but low-value. The first five experiments of a new block type tend to produce modest gains because the proposer has no signal about which ambitious strategies are worth trying. A proposer with transfer data from 10 prior bee swarm blocks knows that "extract the force simulation parameters into a config object" has a 75% keep rate. A cold-start proposer does not know this, so it sticks to whitespace cleanup.

The gap is measurable. Across our first 50 blocks, the first block of each skill type averaged 3.2 experiments before its first kept ambitious change (anything beyond dead code or formatting). The fifth block of the same type averaged 1.1 experiments. Transfer data cuts the exploration phase by roughly two-thirds.

Bootstrapping strategies

There are three ways to warm a cold start. The first is cross-skill priming: even if you have never iterated on a parallel coordinates block, you have iterated on other multi-axis visualizations. Lessons about axis label placement, color scale management, and responsive breakpoints transfer across skill boundaries. The transfer is weaker than within-skill, but it is better than nothing. A proposer primed with "axis label changes have a 40% discard rate across all skills, usually due to contrast or overlap" will approach its first parallel coordinates block more carefully than one with no priors at all.

The second is synthetic seeding. Before the first real iteration, run a small batch of synthetic experiments: propose 3 to 5 changes covering the major strategy categories, audit them, and use the results as a starter transfer set. This costs 5 iterations of budget, but it populates the failure library and strategy hit-rate table from day one. The proposer on experiment 6 already has 5 data points. The investment pays off if it prevents the proposer from repeating a failure that synthetic seeding would have caught.

The third is manual priors. A human who has built parallel coordinates plots before can write a short briefing: "force layout parameters are sensitive to node count; avoid merging the brush and axis drag handlers; the highlight-on-hover pattern is safe to refactor." This is cheap, takes five minutes, and gives the proposer domain knowledge that would otherwise take 10 experiments to discover. The downside is that it requires a human who both knows the visualization type and knows the iteration system well enough to phrase advice in terms the proposer can use.
