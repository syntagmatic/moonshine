Context Distillation

Part 11 of the Autoresearch series. Compressing experiment history into effective proposer context.

The proposer learns from history. But only the history that fits in its context window.

After one experiment, the proposer's context is lean: the current file, the current audit scores and notes, and a single past result. It all fits easily. After five experiments, there is more signal: five sets of scores, five diffs, five explanations of what the proposer tried and why. Still manageable. After twenty experiments, the history alone might run to 50,000 tokens. And the context window has not grown.

Something has to give. The question is what you choose to lose.

The budget

The proposer's context has four sections, each competing for space: 

System prompt       ~300 tokens   (fixed)
Current file        ~500-2000     (fixed, this is the artifact being improved)
Audit scores+notes  ~500          (fixed, current round only)
History             ~???          (variable, grows with each experiment)

The system prompt, the file, and the audit scores are non-negotiable. They define the task. History is the only section that can flex, and it is the only section that grows. With an 8K context window, the history budget is roughly 5,000 to 7,000 tokens depending on file size. A single experiment's record (diff, scores, explanation) runs about 200 to 400 tokens. So naive inclusion of all history works for maybe 15 to 20 experiments before it crowds out the file itself.

Strategy 1: Naive history

The simplest approach. Every past experiment goes into the context: its diff, its scores, whether it was kept or discarded, and the proposer's explanation. At experiment 5, this is helpful. At experiment 15, it is getting crowded. At experiment 22 or so, the history section exceeds the budget and starts pushing out the file. The proposer can no longer see the full artifact it is trying to modify.

The failure mode is predictable: as history grows, the bottom of the file gets truncated. The proposer starts making changes that conflict with code it cannot see. Quality drops. Experiments get discarded. The discards add more history, which pushes out more of the file. A feedback loop in the wrong direction.

Strategy 2: Windowed history

Keep only the last N experiments. With N=5, the history section stays at roughly 1,500 tokens. The context window never overflows. But the window slides forward, and early lessons fall off the edge.

Consider this scenario: experiment 3 tried consolidating three color scale definitions into one shared scale. The auditor caught a visual_critic regression because the distinct hues for different hierarchy levels became indistinguishable. The proposer learned this lesson and avoided color consolidation for the next several rounds. But by experiment 15, that lesson has scrolled out of the window. The proposer has no memory of the failure and tries the same consolidation again. Another discard. The loop has forgotten.

The windowed approach trades completeness for budget discipline. It guarantees the context will never overflow, but it also guarantees the proposer will eventually repeat old mistakes.

Strategy 3: Distilled history

After each experiment, a separate LLM call reads the full experiment history and produces a compressed "lessons learned" summary. This summary replaces the raw history in the proposer's context. A 20-experiment history that would normally consume 6,000+ tokens gets compressed to roughly 200 tokens of high-signal prose.

The summary might look like this: 

Lessons from 20 experiments (12 kept, 8 discarded):
- Color scale consolidation causes visual_critic regressions (tried 2x, failed 2x)
- Dead code removal is reliably safe (5/6 kept)
- Pattern consolidation is moderate risk (2/4 kept, regressions in layout logic)
- Tooltip refactoring is safe when scoped to positioning only
- Avoid merging event handlers: interaction complexity breaks stress_test
- cognitive_load gains come from reducing nesting, not from shortening names

This captures the essential patterns without the verbosity. The proposer sees decades of institutional memory in a paragraph. And the budget stays flat: whether you have 5 experiments or 50, the summary takes roughly the same space.

Strategy 4: Failure catalog

The most aggressive compression: only remember what failed and why. The current file already embodies everything that worked. The proposer does not need a record of successful experiments because their results are visible in the artifact. What it needs is a record of dead ends. 

Failed approaches:
- Color scale consolidation: visual_critic -1.2 (hierarchy levels lost distinction)
- Event handler merging: stress_test -0.8 (interaction conflicts under rapid input)
- Variable name shortening: cognitive_load -0.5 (lost semantic clarity)
- Layout grid extraction: encoding_integrity -0.9 (broke responsive breakpoints)

This is even more compact than the distilled summary, and arguably higher signal. Every line is a concrete warning. The proposer knows exactly what to avoid.

Strategy tags

All of these compression strategies benefit from tagging experiments by type. If the proposer categorizes each change (dead code elimination, pattern consolidation, design refinement, tooltip cleanup, color adjustment), the distilled summary can report hit rates per category. 

Strategy hit rates (20 experiments):
  dead_code_elimination:   5/6 kept  (83%)
  pattern_consolidation:   2/4 kept  (50%)
  design_refinement:       0/3 kept  (  0% - visual_critic regressions)
  tooltip_cleanup:         3/3 kept  (100%)
  color_adjustment:        1/2 kept  ( 50%)
  event_handler_merge:     1/2 kept  ( 50%)

This gives the proposer a menu of approaches ranked by historical success. It can lean into high-yield strategies and avoid categories with poor track records. The tags turn raw history into a decision-support tool.

The costs

Distillation is lossy. The LLM summarizing the history might drop a critical detail. If experiment 7 revealed a subtle interaction between tooltip positioning and SVG viewBox clipping, the summary might reduce that to "tooltip refactoring is risky" without capturing the specific mechanism. The proposer then avoids all tooltip work, even the safe kind.

Distillation adds latency and cost. Each experiment now requires an additional LLM call to update the summary. At 30 experiments, that is 30 extra calls. The summary step is fast (small input, small output), but it adds 5 to 10 seconds per round and a few cents per experiment. Over a long run, this adds up.

Failure catalogs bias toward conservatism. A proposer that only sees failures will develop an increasingly long list of things it must not try. After enough experiments, the negative space grows large enough that the proposer becomes timid. It sticks to the safest possible changes (removing whitespace, deleting comments) because everything else resembles some past failure. The loop still runs, but the proposals lose ambition.

There is no free compression. Each strategy trades one kind of error for another. Naive history is complete but overflows. Windowed history fits but forgets. Distilled history remembers but might remember wrong. Failure catalogs are high-signal but breed caution. The right choice depends on how many experiments you expect to run and how much you trust the distillation step.

In practice

The current autoresearch implementation uses a hybrid: windowed raw history for the last few experiments (so the proposer sees exact diffs for recent rounds) plus a distilled summary of everything older. This gives the proposer both precision for recent context and coverage for long-term patterns. The summary is updated every 5 experiments rather than every round, reducing distillation cost.

The failure catalog is appended to the summary as a separate section. This way the proposer gets both the strategic overview (what categories of change tend to work) and the tactical warnings (specific things that broke). The combined format runs about 300 to 400 tokens and stays roughly flat regardless of experiment count.

The context window is the proposer's working memory. Context distillation is the act of choosing what to remember. Get it wrong and the proposer either forgets critical lessons or drowns in irrelevant detail. Get it right and twenty experiments of trial and error compress into a paragraph that makes experiment twenty-one smarter than experiment one.
