The Vision Auditor

Auditing what the viewer actually sees, not what the code says

The current autoresearch auditor receives two inputs: the block's HTML source and a screenshot of the rendered output. In theory it judges the visualization. In practice it reads the code. The screenshot is a formality. The auditor finds duplicate d3.select calls and verbose event handlers because those live in the text it knows how to parse. It does not notice that two bars in a grouped chart are the same shade of blue, or that a label sits directly on top of a data point.

This is the blind spot. The auditor can evaluate code quality but it cannot evaluate visual quality. It scores what it reads, not what the viewer sees. And the gap between those two things is exactly where the worst bugs hide.

The gap

Consider a bar chart where the code specifies fill: "#3b82f6" for one series and fill: "#60a5fa" for another. Those hex values look different in a text file. On screen, at normal viewing distance, they are nearly indistinguishable. A text-based auditor will see two distinct color assignments and move on. A viewer will see one muddled bar chart.

The same class of problem appears in overlapping labels (the code positions them at different coordinates, but the rendered text boxes collide), truncated axes (the domain is set correctly but the SVG viewport clips the last tick), and invisible elements (a tooltip div exists in the DOM but renders behind a higher z-index panel). The code is correct. The picture is wrong. The current auditor cannot tell the difference.

The vision-first pipeline

A vision-first auditor flips the information hierarchy. Instead of reading code and glancing at a screenshot, it analyzes the rendered image without access to the source. The pipeline is straightforward:

The key constraint: the vision model receives only the screenshot and a description of what the visualization should show (the data summary, the intended chart type, the expected number of series). It does not receive the HTML. This forces it to evaluate the output purely as a viewer would.

d3-power-tools already has the rendering infrastructure for this. The test-viz.py screenshot pipeline launches a headless browser, renders each block, and captures a PNG. That same PNG, routed to a vision model instead of a text model, becomes the input for a visual audit.

The existing text-based auditor does not go away. It still catches code quality issues: duplication, dead code, non-idiomatic patterns. The vision auditor runs in parallel, scoring a different set of dimensions. Two auditors, two perspectives, one composite judgment.

Comparison auditing

Absolute quality scores are noisy. Ask a vision model "rate this chart from 1 to 10" and you will get inconsistent answers across runs. The model lacks a stable internal caliber for what a 7 looks like versus an 8.

Relative judgments are more reliable. Show the model two screenshots and ask "did the visualization improve?" This is a comparison task, not a rating task, and vision models handle it better. The model can point to specific differences: "the labels no longer overlap," "the axis now shows the full range," "the legend moved out of the data region." These are concrete observations, not abstract scores.

In an autoresearch loop, this means comparing each iteration's screenshot to the previous best. The auditor does not need to assign a number. It needs to answer a binary question with supporting evidence. Did it get better, and where?

Spatial grounding

When a text-based auditor says "the labels may be hard to read," it is guessing based on font-size values in the code. A vision auditor can be more specific: "the label 'Manufacturing' at coordinates (340, 28) overlaps the top of the tallest bar." This is spatial grounding. The feedback references a concrete region of the image rather than an abstract concern.

Spatially grounded feedback helps the proposer make targeted fixes. Instead of "improve label readability" (which could mean anything), the proposer receives "move the label at (340, 28) upward by 15 pixels to clear the bar top." The fix is surgical. The proposer does not need to guess which label or how far.

This also enables the reference image approach. At the start of an autoresearch run, capture a "golden" screenshot of the initial render. On each subsequent iteration, compare the new screenshot to the golden reference. If a previously visible element disappears, or a layout region shifts dramatically, that is a regression. The vision model can flag exactly which region changed and whether the change looks intentional or accidental.

Spatial grounding turns audit notes from vague suggestions into precise coordinates. The proposer can act on "overlap at (340, 28)" directly. It cannot act on "consider improving readability" without further interpretation.

What this changes about autoresearch

The text-based auditor catches code-level problems. The vision auditor catches rendering-level problems. Together they cover a wider surface area. A block that passes text audit with high marks but fails vision audit is one where the code is clean but the output is broken. That is exactly the category of bug that slips through the current pipeline.

In practice, the highest-value catches will be mundane. Colors that are too similar. Axis labels cut off by the viewport. Legends that occlude data. Tick marks that render at the wrong density for the axis range. These are not exotic failures. They are the same things a human reviewer would catch in two seconds of looking at the chart. The vision auditor just automates that two-second glance.

The comparison mode is where the real leverage lives. Currently, the autoresearch loop uses composite score deltas to decide keep or discard. Adding a vision comparison layer means the loop can ask: "the score went up by 0.1, but did the chart actually look better?" Sometimes a score improvement reflects code cleanup that did not change the visual output at all. Sometimes a score regression masks a visual improvement that the text dimensions missed. The vision auditor provides a second opinion that grounds the decision in what the viewer actually sees.

Amber risk: Vision models hallucinate spatial locations. A model may report an overlap at coordinates that do not correspond to any element in the image. Resolution dependence is real: bugs visible at 2x rendering may vanish at 1x, and vice versa. Each vision API call costs more than a text call, so adding a vision audit to every iteration increases the per-experiment budget. And there is an uncanny valley of almost-right visual feedback, where the model identifies a real region but mischaracterizes the problem, leading the proposer to make a "fix" that introduces a new bug.
