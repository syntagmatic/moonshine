Proposer Strategies

The proposer is not random. It is a strategist with a limited toolkit.

The autoresearch proposer receives the current HTML file, its audit scores, the auditor's notes, and a history of recent discards. Its job: make one change that improves the code. But "one change" covers an enormous range of actions. Removing a dead variable is a change. Rewriting the entire rendering pipeline is also a change. The two carry very different risks, very different rewards, and work best at very different points in the iteration lifecycle.

What follows is a taxonomy of proposer strategies. Not all of them are implemented today. Some are patterns we observe the proposer already using; others are directions it could be steered toward. The question underneath all of them is the same: given the current state of the code and the audit feedback, which type of change is most likely to be kept?

The six strategies

  • Dead code elimination
    Find and remove unused variables, unreachable branches, redundant calculations. Safe, high keep rate, shallow impact. LOC drops; scores hold steady.
  • Pattern consolidation
    Identify repeated code patterns and extract into shared functions. Moderate risk. The abstraction might not fit, or it might obscure intent if the patterns were not truly identical.
  • Algorithm improvement
    Replace an O(n²) scan with a lookup map, swap a manual sort for a smarter data structure. High impact when it applies, but it requires understanding the code's logic, not just its shape.
  • Design refinement
    Adjust colors, spacing, typography, or layout based on what the auditor flagged. Targets the visual_critic dimension directly. LOC stays stable; quality goes up.
  • Interaction hardening
    Add .interrupt() calls, debounce handlers, null guards, resize observers. Targets stress_test scores. Adds a few lines but prevents regressions under rapid input.
  • Restructure
    Reorganize the code flow: reorder functions, flatten nested callbacks, change the rendering pipeline. High risk, high reward. Can unlock large LOC drops or cause regressions across multiple dimensions.

Strategy selection and timing

Not every strategy makes sense at every stage. Early in the iteration lifecycle, there is usually low-hanging fruit: dead code, obvious redundancies, variables declared and never used. Dead code elimination is the right opener. It tightens the file, costs almost nothing, and gives the evaluator less noise to score.

In the middle rounds, the easy wins are gone. Now the proposer needs to consolidate patterns or improve algorithms. The risk goes up, but so does the potential payoff. A good consolidation can remove 20 lines in one move and make subsequent changes easier.

Late in the lifecycle, the code is already fairly tight. The remaining gains come from targeted fixes: the auditor says "label contrast is too low," so the proposer adjusts colors (design refinement). Or the auditor flags "tooltip flickers on rapid hover," so the proposer adds a debounce (interaction hardening). These are surgical changes driven by specific feedback.

Think of it as triage. Early rounds reduce the surface area. Middle rounds restructure what remains. Late rounds polish the details the evaluator still complains about.

The temperature analogy

This phasing resembles simulated annealing. In annealing, you start with high temperature (bold, exploratory moves) and cool down to low temperature (conservative, local refinements). The idea is that early exploration finds the right region of the search space, and late conservatism locks in the gains.

You could run it the opposite direction: start safe with dead code elimination, then escalate to restructuring when convergence stalls. This is closer to what happens today, since the proposer naturally tries the easiest changes first and only reaches for bigger moves when the small ones stop producing keeps.

Either way, the core principle is the same. Strategy should vary over time. A proposer that always uses the same type of change will plateau early or take unnecessary risks late.

Context-aware proposals

Strategy scheduling is one lever. The other is targeting. Today, the proposer receives the audit notes and the per-dimension scores, but it decides for itself what to change. A context-aware proposer would read the scores and target the weakest dimension. If stress_test is at 4 and visual_critic is at 7, the proposer should reach for interaction hardening, not design refinement.

This sounds obvious. In practice it means the proposer needs to do two things: diagnose (which dimension is the bottleneck?) and then select a strategy that addresses that dimension specifically. The mapping is not always one-to-one. A restructure might help cognitive_load by simplifying the code flow, or it might hurt stress_test by breaking event handler cleanup. Strategy selection with targeting is a two-dimensional problem.

The control problem. Strategy scheduling adds complexity, and the proposer may not follow the intended strategy. It is an LLM, not a deterministic program. You can instruct it to "focus on dead code elimination this round," but it might decide a restructure looks more promising and do that instead. Evaluating strategy effectiveness requires running many experiments per strategy to measure keep rates, which is expensive. The cost of systematic strategy evaluation may exceed the value of the insights it produces.

There is a deeper issue. The proposer's strategy is invisible to the evaluator. The evaluator scores the result, not the method. Two proposals that both improve the composite by 0.3 are equally good from the evaluator's perspective, whether one used dead code elimination and the other used restructuring. Strategy only matters insofar as it affects the probability of a keep. And measuring that probability requires the kind of controlled experimentation that autoresearch was designed to avoid: running the same block through the same round multiple times with different strategy instructions.

So the honest conclusion is this: strategy scheduling is a plausible optimization. It has the right shape. But proving it works, and tuning the schedule, costs more iteration budget than just letting the proposer do what it thinks is best. For now, the best evidence is anecdotal. Early rounds that remove dead code tend to be kept. Late rounds that attempt restructuring tend to be discarded. Whether formalizing that observation into a schedule produces better outcomes at scale remains an open question.

Measuring strategy effectiveness

The only honest way to evaluate a strategy is to track its keep rate across many blocks. Anecdotal evidence ("dead code removal usually works") is not evidence. It is pattern-matching on a biased sample. A strategy that was tried 3 times and kept 3 times has a 100% hit rate and a confidence interval wide enough to drive a truck through.

The measurement protocol is simple in concept: tag every proposal with its strategy type, record whether it was kept or discarded, and accumulate a hit-rate table over time. After 50 or 100 blocks, the table has enough data to say something real. Dead code elimination might show 82% across 38 attempts. Pattern consolidation might show 55% across 22 attempts. Restructuring might show 28% across 14 attempts. Now you have a basis for scheduling: lean into dead code early, use pattern consolidation when scores plateau, and reserve restructuring for cases where the other strategies have dried up.

But the keep rate is not uniform across block types. Dead code elimination might be 90% on simple bar charts and 60% on complex force-directed networks, because the network blocks have more entangled code where "dead" paths turn out to be load-bearing. The useful signal is not the global keep rate but the keep rate per strategy per block type. This is a matrix, not a vector, and it takes longer to populate. A system that has iterated on 100 blocks might have 5 to 8 data points per cell. Enough to spot trends, not enough for precision.

When strategy metrics mislead

Keep rate is a proxy for value, and like all proxies, it lies in specific ways. A strategy with a 90% keep rate that produces 0.1-point gains per keep is less valuable than a strategy with a 40% keep rate that produces 0.8-point gains per keep. The timid strategy survives more often but moves the needle less. Expected value, not keep rate, is the right metric. Expected value is (keep rate) times (average gain when kept). A strategy table that reports both columns tells a more complete story.

There is also a selection effect. Strategies that are tried more often accumulate better statistics, which makes them look more reliable, which causes them to be tried even more. Rare strategies never get enough data to prove themselves. This is the explore-exploit tradeoff from bandit theory, applied to strategy selection. A system that always picks the highest-keep-rate strategy will converge on a local optimum and never discover that restructuring, attempted at the right moment, produces the largest single-round gains in the entire catalog.
