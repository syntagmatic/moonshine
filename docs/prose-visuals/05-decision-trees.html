Decision Trees and Random Forests

How recursive splitting creates interpretable classifiers, and how ensembles overcome their weaknesses.

What question best splits these groups?

That is the entire idea behind a decision tree. You have data. The data belongs to different categories. You want to find a sequence of yes-or-no questions that separates the categories as cleanly as possible. Each question splits the data into two groups, and you keep splitting until the groups are pure, or close enough.

A doctor diagnosing a patient does this intuitively. Fever above 101? Yes. Cough present? Yes. Rash? No. Each answer narrows the possibilities. A decision tree automates this reasoning, choosing the questions that reveal the most information at each step.

Information Gain: Measuring the Quality of a Question

Not all questions are equal. A question that splits 50 blue points from 50 red points into perfectly pure groups is excellent. A question that puts 48 blue and 47 red on one side and 2 blue and 3 red on the other is nearly useless. We need a way to measure how good a split is.

The standard measure is Gini impurity. A node that contains only one class has Gini impurity of 0 (perfectly pure). A node with a 50/50 mix has the maximum impurity of 0.5. The information gain of a split is the parent's impurity minus the weighted average of the children's impurities. Higher gain means a better split.

A good split does not need to be perfect. It just needs to reduce uncertainty. Even a modest gain at each level compounds as the tree grows deeper, and each subsequent split only has to handle a smaller, more homogeneous subset.

Growing the Tree: Recursive Splitting

A single split rarely separates the classes. So the algorithm recurses: for each child region, find the best split again. And again. The result is a tree of decisions. Each internal node asks a question ("is x < 3.2?"), each branch is an answer (yes or left, no or right), and each leaf assigns a class label by majority vote of the points that landed there.

The tree and the decision boundary are two views of the same structure. A split in the tree corresponds to a line on the scatter plot. A leaf in the tree corresponds to a rectangular region. Watch both views build simultaneously.

Each split is locally optimal: the algorithm picks the single feature and threshold that maximizes information gain right now. It does not look ahead. This greedy strategy is fast, but it means the tree might miss a combination of splits that would be globally better. That is a tradeoff we accept for interpretability and speed.

When to Stop: Overfitting and Pruning

If you never stop splitting, the tree will eventually give every single training point its own region. Training accuracy hits 100%. The decision boundary becomes a jagged mess that memorizes noise rather than learning the true pattern. This is overfitting.

The telltale sign: training accuracy keeps climbing, but test accuracy peaks and then drops. The model gets better at the data it has seen and worse at everything else.

Figure 3. Drag the depth slider. At depth 1: underfitting, one coarse split. At depth 5: a reasonable boundary that captures the pattern. At depth 15: overfitting, the boundary contorts to wrap around every training point. The accuracy chart below shows training and test accuracy diverging.

In practice, you control tree depth with stopping rules (maximum depth, minimum samples per leaf) or by pruning: grow the full tree, then remove branches that do not improve performance on a held-out validation set. Both approaches trade a small loss in training accuracy for a large gain in generalization.

Overfitting is not a bug in the algorithm. It is the natural consequence of an expressive model given too much freedom. Every machine learning method faces this tension between fitting the training data and generalizing to new data. Decision trees just make it unusually visible.

The Forest: Many Weak Learners, One Strong Ensemble

A single decision tree is unstable. Change a few training points and the whole tree can rearrange itself. The splits are greedy and brittle. Random forests fix this by building many trees and letting them vote.

Each tree in the forest sees a different bootstrap sample (random subset with replacement) of the training data. At each split, it only considers a random subset of features. These two sources of randomness make the individual trees different from each other. Some will be wrong in one region, others wrong elsewhere. But when they vote together, the errors tend to cancel out.

This is the central insight of ensemble methods: combining many imperfect models can produce a result better than any individual model. The individual trees do not need to be deep or accurate. They just need to be different from each other. The randomness is not a weakness. It is the mechanism that creates diversity, and diversity is what makes the ensemble work.

Decision trees are the only machine learning model you can explain to your manager in one sentence: "It asks a series of yes-or-no questions about the data and follows the answers to a conclusion." Random forests are the sentence you add when the manager asks why you need more than one: "We ask five different people the same question and go with the majority."
