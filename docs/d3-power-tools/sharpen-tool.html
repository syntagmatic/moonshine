Sharpen Tool

Skills accumulate bloat just like code. The sharpen tool audits a SKILL.md, triages every section, and compresses what remains until only signal is left.

A new skill gets written in a burst. You add a section for every pattern you can think of, every pitfall you have hit, every variant that might matter. This is the right instinct at first. But over time the file grows past the point where it helps. Sections that sounded important when written turn out to be niche, or duplicated in another skill, or derivable from the core patterns. The LLM reading this skill at generation time has a finite context window, and every low-value token displaces a high-value one. Bloated skills do not just waste tokens. They dilute the signal that actually drives output quality.

The sharpen tool works in three phases. First, triage: classify every section as core (keep in full), compress (valuable but verbose), reference (covered better in another skill), or cut (niche, derivable, or rarely needed). Second, compress: tighten the survivors by collapsing variants, extracting shared scaffolding, and stripping boilerplate. Third, verify: check that no implementable pattern was lost and that compression did not introduce errors like undefined variables or dangling function references.

The key judgment: how often does a developer actually need this section? Weekly patterns are core. Patterns you can derive from core knowledge are cut candidates. The goal is not a shorter file. The goal is a higher ratio of useful tokens to total tokens.

The Three Phases in Detail

Triage is the hardest phase because it requires judgment about frequency, not importance. A section on IE11 fallback patterns might be technically correct and carefully written. But if the target audience stopped supporting IE11 two years ago, every token in that section is wasted context. The triage step builds a section inventory: for each ## heading, note its line count, classify it as core, compress, reference, or cut, and write a one-sentence justification. Show this table to the user and get approval before changing anything. Triage without buy-in leads to arguments later.

The classification is not always obvious. A section that covers a pattern you use once a quarter still matters if that pattern is hard to derive from first principles. Responsive resize recipes fall into this category: you do not need them often, but when you do, getting the ResizeObserver teardown wrong causes memory leaks that are painful to debug. The question is not "how often" alone but "how often, and how costly is getting it wrong without guidance."

Compress targets the sections classified as valuable but verbose. The most common compression: a skill shows five variants of a pattern (bar, grouped bar, stacked bar, diverging bar, population pyramid) as five full code blocks. After compression, you show one canonical form in full and describe each variant's delta in two or three lines of code plus one line of prose. The reader can reconstruct any variant from the base pattern plus its delta. This preserves all the information while cutting tokens by 60-70% in the best cases.

Other compression moves: extract shared DOM scaffolding that appears in every code block (SVG setup, margin convention, scale definitions) into a single "setup assumed" block at the top. Strip role and aria-label attributes from examples where accessibility is not the point of the section. Collapse prose that says "Note that you should be careful to..." into a single direct sentence. Every word that does not teach a pattern or warn about a pitfall is a candidate for removal.

Verify catches the damage that compression inevitably causes. When you delete a wrapper function to save 15 lines, every call site that referenced it becomes a dangling reference. When you inline a helper, the variable it returned may still appear three sections later. The verify phase walks every code block and checks: is every variable defined or clearly external (data, svg, width, d3.*)? Is every function called also defined? Does the prose still match what the code shows? Verify also builds a coverage diff: list every implementable pattern from the original and confirm each one is still present or explicitly cross-referenced. If a developer cannot build the pattern from the compressed version alone (or by following one pointer), something was lost.

Token Budget

Every skill loaded into a generation prompt consumes context window. The context window is finite, and skills are not the only thing competing for it. The user's prompt takes tokens. The data description or sample rows take tokens. The conversation history takes tokens. And the model needs room for its own reasoning and output. Skills sit in the middle of this budget, squeezed between the input the user controls and the output the model produces.

A skill at 2,000 tokens is a rounding error in a 200K context window. A skill at 8,000 tokens is noticeable. Load three skills at 8,000 tokens each and you have burned 24,000 tokens before the model has seen a single line of the user's request. That budget pressure is invisible: you do not get an error message when skills are too large. The model just has less room to reason, and the outputs get subtly worse. Shorter chain-of-thought. Fewer alternative approaches considered. More pattern-matching from the skill text and less genuine problem-solving.

The sharpening target is not "as small as possible." It is "small enough that loading this skill alongside two others still leaves room for the model to think." In practice, that means each skill should aim for under 4,000 tokens. Past 6,000 you should be asking whether the skill is trying to do two jobs.

Skill Length and Output Quality

Longer skills do not always produce better output. This is counterintuitive. If a skill contains more patterns, more examples, and more guidance, the model should do better with it. And for short skills, that is true: going from 500 tokens to 2,000 tokens typically improves output because you are adding genuine signal. The model has more patterns to draw from and more pitfalls to avoid.

But past a certain length, quality plateaus and then declines. The mechanism is attention dilution. When the model reads a 10,000-token skill, it cannot attend to all of it with equal weight. The sections at the beginning and end get more attention (the well-documented primacy and recency bias in long-context models). Sections in the middle get less. If your most important pattern happens to land in the middle third of a bloated skill, it may receive less attention than a marginal pattern near the top.

There is also the copy problem. Long skills with many code examples tempt the model to copy blocks wholesale instead of adapting them to the specific prompt. A compact skill that shows one canonical pattern and describes variants in prose forces the model to synthesize. That synthesis step is where the model applies the pattern to the user's actual data and requirements, which is exactly the work you want it to do.

When to Split a Skill

Compression has limits. If you have sharpened a skill down to its core patterns and it is still 7,000 tokens, the problem is probably not verbosity. The skill is covering two distinct capabilities that happen to share a name.

The hierarchy-layouts skill is a good example. It covers treemap, pack, tree, cluster, and partition, each with its own layout function, its own tiling or sizing strategy, its own labeling challenges. Compressing all five into one tight skill still produces a large file, because each layout is genuinely distinct. The alternative is to split: one skill for space-filling layouts (treemap, partition) and one for node-link layouts (tree, cluster), with pack going wherever it fits best. Each resulting skill is focused enough to sharpen below the 4,000-token target.

The test for splitting: if two sections in a skill never appear in the same generation prompt, they belong in separate skills. A user asking for a treemap will not also need the tree layout in the same chart. Keeping both in one skill means every treemap generation loads the tree layout patterns for no benefit. Splitting eliminates that waste.

The test against splitting: if sections share infrastructure (the same data join pattern, the same scale setup, the same interaction model), splitting forces you to duplicate that infrastructure in both files. Duplication across skills is worse than a slightly long single skill, because now you maintain two copies of the same pattern that can drift apart. When in doubt, compress first. Split only when compression cannot bring the file below target and the sections are genuinely independent.

Versioning and Rollback

Sharpening is destructive. You are removing content from a file that may have taken weeks to write. If the compression turns out to be too aggressive, you need a way back. The simplest approach: commit the pre-sharpen version before making any changes. Not a branch, not a stash, just a clean commit with a message like "pre-sharpen snapshot: cartography skill." Then sharpen on top. If the compressed version causes output regressions, git diff shows you exactly what was removed and you can restore specific sections.

For skills that go through multiple rounds of sharpening, a version comment at the top of the file helps: <!-- v3: compressed from 6,200 to 3,800 tokens, cut IE11/resize sections -->. This is not for the model (the model does not read HTML comments in a SKILL.md). It is for the human who opens the file six months later and wonders why a section they remember writing is gone.

The calibrate tool is the other half of versioning. After sharpening, run a calibration: generate blocks with the old skill and the new skill, evaluate both blind. If the compressed version scores the same or better on all four audit dimensions (visual critic, encoding integrity, stress test, cognitive load), the sharpening succeeded. If any dimension drops, inspect the diff to find what was lost. Sometimes a single sentence of guidance, restored from the pre-sharpen version, recovers the lost quality without undoing the compression.

When This Skill Hurts

Over-compression removes edge cases that matter. A section that fires 10% of the time might cover the exact case a user hits today. The sharpen loop optimizes for average-case quality because it triages by frequency. But the user who needs the IE11 fallback or the right-to-left text layout or the accessibility teardown pattern does not care that 90% of users never needed it. Cutting a section that covered their case means they get no guidance at all, and the model falls back to generic training data, which is worse than a verbose but correct skill section.

The sharpen loop degrades edge-case quality while improving average-case density. Each round of triage asks "how often is this needed?" and cuts the least frequent sections. Over multiple rounds, the skill converges toward the top five patterns and loses everything else. The result looks great on aggregate metrics (higher signal density, fewer tokens, faster generation) but fails silently on unusual inputs. A skill sharpened three times may handle bar charts perfectly and produce broken output for polar area charts, because the polar section was cut in round two as "niche."

Compression favors code over judgment. When a section contains both a code pattern and three paragraphs explaining when to use it, when to avoid it, and why the obvious alternative is wrong, the compress phase naturally keeps the code (concrete, dense, directly useful) and cuts the prose (wordy, abstract, seems derivable). But the prose is the "why" behind the pattern. Without it, the model applies the pattern in every situation, including the ones where the deleted prose would have said "not here." Code without judgment is a template. Judgment without code is still useful, because the model can generate code from principles. When in doubt, keep the paragraph that says "do not use this when..." and cut the third variant of the code example.

LLMs generating terse skills that fail silently. If you ask an LLM to sharpen a skill, it will produce something impressively compact. Variable names shortened, prose stripped to sentence fragments, examples reduced to the happy path. The result reads well in a review because it is concise and covers the main patterns. But edge cases vanish without trace. The LLM does not flag what it removed. It does not say "I cut the section about handling missing data in treemaps because it seemed niche." It just produces a shorter file, and you discover the gap weeks later when a generated treemap crashes on null values. Always diff the LLM's output against the original and review every deletion, not just the sections that remain.
