Shape Morphing

Proving that two views show the same data by transforming one into the other.

A bar chart and a pie chart can show the same numbers. But placed side by side, they feel like two separate graphics. Morph the bars into arcs, though, and the connection becomes visceral. You see each bar curl into its wedge. The values do not change, only the geometry. That continuity is the point: the transition proves the two representations are equivalent.

This works because morphing asserts identity. When a bar slides, stretches, and bends into an arc, your eye tracks it as one object changing shape, not two objects replacing each other. Object constancy, the principle that a visual element should persist across state changes, is what makes the morph legible instead of chaotic.

Morphing between visual states shows continuity. The same data, presented two different ways, with the transition proving they are equivalent. When the claim of sameness is false, the morph misleads.

The three demos below each use a different morphing strategy. The bar-to-pie morph interpolates arc parameters (angles and radii). The circle-to-rectangle morph tweens cornerRadius. The sort morph interpolates positions while key functions maintain object constancy. All three rely on the same idea: find numeric parameters that both the source and target share, and let D3 interpolate between them.

Figure 1. Both views encode the same eight values. In bar mode, equal-width arc segments use radius to encode value. In pie mode, a constant radius uses angle instead. The transition interpolates all four arc parameters (innerRadius, outerRadius, startAngle, endAngle) so you see each bar curl into its wedge.

Figure 2. A rounded rectangle with rx = width/2 is a circle. Tweening rx, width, and height morphs between the three states with no path interpolation needed.

Figure 3. Same bars, different order. Key functions tie each bar to its data item so D3 moves each one to its new position rather than recoloring bars in place. Watch any single color track across the reorder.

Path Interpolation via Point Resampling

The three figures above morph between shapes that share numeric parameters: arc angles, corner radii, positions. But what about shapes that have nothing in common geometrically -- a star and a circle, a country outline and a rectangle? There is no set of shared parameters to interpolate. The shapes are defined by entirely different SVG path commands.

The solution is to discard the path commands and work with raw points instead. Sample both the source and target paths at N evenly-spaced positions using getPointAtLength(), producing two arrays of [x, y] coordinates with exactly the same length. Now interpolation is trivial: each point in the source array travels in a straight line to its corresponding point in the target array. The intermediate shapes are polygons connecting those interpolated points.

The critical step is rotation alignment. Both paths are sampled starting from their first point, but the "first point" of a star and the "first point" of a circle have no natural correspondence. Without alignment, the morph produces a collapsing, spinning artifact as points on one side of the source travel to the opposite side of the target. The fix: try each possible rotation of the source array (shifting the start index) and pick the one that minimizes total squared distance to the target. This is an O(N2) operation, but it runs once before the animation starts, not per frame.

Point count matters. Below about 64 points, circles and arcs look visibly polygonal during the morph. At 128 points, most shapes appear smooth. Going higher is rarely needed and costs more in per-frame interpolation. The figure below uses 128 points and lets you see the morph between a five-pointed star and a circle.

Figure 4. Three shapes with incompatible SVG path commands, morphed via point resampling (128 points each). The source and target paths are sampled to equal-length [x, y] arrays, rotation-aligned to minimize travel distance, then interpolated per frame. The intermediate shapes are always polygons -- notice the slight flattening of the circle's curves at mid-transition.

The output of point resampling is always a polygon. A circle becomes a 128-gon. This is visually acceptable but mathematically imprecise. When both shapes can be expressed as arcs or rounded rects, parametric interpolation preserves true curves at every frame. Resampling is the fallback for when parametric is impossible, not a general replacement for it.

Projection Transitions

Map projections are functions from spherical coordinates to screen coordinates. Morphing between projections means interpolating those functions: for every point on every country outline, compute where it lands under projection A and where it lands under projection B, then lerp between the two screen positions. The geographic data does not change. The math that maps it to pixels does.

The implementation wraps the interpolation inside d3.geoProjection(), which takes a raw projection function (longitude, latitude in radians to [x, y] in pixels) and returns a proper projection object with .stream() support. At each frame, the interpolated projection recomputes every path. This is expensive -- a world map with detailed coastlines might have tens of thousands of vertices, all reprojected every 16 milliseconds. Simplify geometry before animating. TopoJSON's presimplify or mapshaper can reduce vertex counts by 90% with minimal visual loss.

Figure 5. The same landmass geometries under three projections. Switching between them interpolates the raw projection function point-by-point. The coastlines deform continuously -- Greenland shrinks as it moves from Mercator to Equal Earth, and wraps around the horizon for Orthographic. Geometry is simplified to ~800 vertices for smooth 60fps animation.

Projection transitions reveal something that side-by-side maps cannot: the distortion each projection introduces. Watching Greenland shrink as you move from Mercator to Equal Earth makes the area distortion visceral in a way that a static comparison does not. This is morphing at its best -- the transition itself communicates, not just the endpoints.

Arc Parameter Tweening in Detail

The bar-to-pie morph in Figure 1 works because both bars and pie wedges are arcs. A D3 arc is defined by four parameters: innerRadius, outerRadius, startAngle, and endAngle. A "bar" is an arc with equal angular width and value-driven radius. A "pie wedge" is an arc with constant radius and value-driven angular width. The morph interpolates all four parameters simultaneously, and d3.arc() renders a valid arc at every intermediate state.

The key implementation detail is where to stash the current arc parameters. D3's .data() method replaces each element's datum with the new bound object. If you store the current arc state on the datum, it vanishes on rebind, and d3.interpolate(undefined, nextParams) snaps instantly instead of animating. The fix: stash the parameters on the DOM element itself (this.__arcParams = params) so they survive data rebinding. The attrTween callback reads from this.__arcParams, computes the interpolator, and writes back the intermediate state on every frame.

This pattern generalizes. Any morph that uses attrTween with a custom interpolator should stash state on the element, not the datum. It applies to donut-to-sunburst transitions (interpolating inner and outer radii for nested arcs), to gauge animations (interpolating endAngle), and to any situation where the visual state and the data state need to evolve independently during the transition.

When Morphing Helps vs. When It Confuses

Morphing communicates one specific claim: these two views represent the same data. When that claim is true, the transition proves it. When it is false, the transition lies.

Morphing helps when the source and target are genuinely equivalent representations. A bar chart and a pie chart of the same values. A Mercator map and an Equal Earth map of the same countries. A scatterplot sorted by one variable, then re-sorted by another. In each case, the viewer can track individual elements across the transition and verify that nothing was added, removed, or swapped. The motion carries information.

Morphing confuses when it implies continuity between unrelated states. Morphing a revenue bar chart into a headcount bar chart suggests a continuous relationship between dollars and people. Morphing one country's outline into another's is geometric spectacle with no meaningful correspondence -- point 47 of France does not "become" point 47 of Brazil. Morphing a bar chart into a map is impressive but incoherent: the intermediate state is neither a bar chart nor a map, and the viewer cannot extract information from it.

The test is simple: can the viewer track a single element through the transition and understand what happened to it? If yes, the morph communicates. If the elements dissolve into an amorphous intermediate state before re-forming, the morph is decoration. A crossfade or hard cut is more honest because it correctly signals that the two states are unrelated.

The Uncanny Valley of Imperfect Morphs

There is a middle ground that is worse than either a clean morph or a clean cut: the almost-right morph. Bars that sort into roughly the right positions but with a few swapped. Paths that deform smoothly but with one vertex that spirals wildly. Arcs that interpolate correctly except for one that collapses to zero width at t=0.5 before expanding to its target.

These near-misses are more disorienting than no animation at all. The viewer's visual system commits to tracking the motion, then gets contradicted by the errors. A static swap would have been processed in 200 milliseconds. An imperfect morph forces the viewer to watch an 800-millisecond animation, notice something is wrong, and then re-examine the final state to figure out what actually happened. The animation consumed attention without providing clarity.

The practical consequence: if you cannot get the morph exactly right -- correct keys, correct alignment, correct interpolation for every element -- do not morph. A hard cut is always available as a fallback, and it never lies.

When This Skill Hurts

Meaningless morphs. Morphing should demonstrate that two views represent the same data. LLMs use morphing for visual spectacle between unrelated states -- morphing a bar chart into a map, or a pie chart into a network graph -- because the animation looks impressive in a demo. But the intermediate state is nonsense: it is neither one chart type nor the other, and the viewer cannot extract information from it. The morph implies a correspondence that does not exist. If the source and target do not share data identity, use a crossfade or a hard cut. The discontinuity correctly signals that the data changed.

Path interpolation garbage. d3.interpolateString on SVG path d attributes matches numbers positionally. When the source path is M 0 0 L 100 100 and the target is M 0 0 C 50 0 50 100 100 100, the command sequences differ in length and structure. The interpolator produces intermediate strings with mangled commands -- a cubic Bezier with linearly interpolated control points that were never control points in the source. The result is a path that flickers, collapses, or traces an absurd trajectory. LLMs reach for interpolateString because it exists in D3's API and "seems like it should work." The fix is always to resample both paths to equal point counts first, bypassing the path command layer entirely.

Performance on complex paths. Morphing paths with 500+ resampled points creates visible jank during the transition. Each frame must interpolate 500 coordinate pairs and rebuild the path string. On a 60fps display that is 30,000 interpolations per frame, plus the SVG path parsing overhead. For a single shape this is manageable; for a map with 50 country outlines morphing simultaneously, it is not. The skill should simplify paths before morphing -- TopoJSON's presimplify or Douglas-Peucker reduction can cut vertex counts by 80-90% with minimal visual loss. LLMs tend to morph raw, unsimplified geometry because simplification is an extra step that tutorials rarely mention.

Object constancy failure. Without proper key functions in .data(data, d => d.id), D3 matches elements to data by index. When the data is reordered, element 0 gets datum 0 regardless of identity. The bars animate smoothly to their new positions, but the wrong bar travels to each position -- the red bar that was "Tech" is now animating to where "Energy" should be. The chart looks like it is working. The data mapping is broken. LLMs omit key functions more often than they include them, because the animation still "looks right" without keys. It just tells the wrong story. Figure 3 above uses d => d.id as the key. Remove it and the sort animation still runs, but the colors no longer track their labels.
