Time Series

Time is not just another continuous axis. It has structure that a chart should reveal, not hide.

Most charting libraries treat time as a number that happens to format as a date. Plot the points, connect them with lines, done. But time has internal structure that generic line charts ignore: weekly cycles where activity dips every weekend, seasonal trends that repeat each year, gaps where nothing was recorded. A time series visualization should surface these patterns, not bury them under a single squiggly line.

The data below is synthetic: two years of daily values (~730 points) built from a weekly cycle (weekends are lower), a seasonal trend (summer peaks, winter troughs), and random noise. Drag across the overview bar to select a time window. The main chart zooms to show that window in detail, and the axis ticks adapt: months at broad zoom, individual days when you zoom in tight. This is the overview+detail pattern, built with d3.brushX and d3.scaleUtc.

Time-aware tick formatting matters. When you zoom in to a single month, you want to see day labels. When you zoom out to two years, you want months. The d3.scaleUtc axis handles this automatically, picking the right interval for the visible span. Use scaleUtc over scaleTime unless you specifically need local-timezone labels; DST creates days with 23 or 25 hours that distort durations silently.

Toggle to the horizon chart view and the same data takes a fraction of the vertical space. A horizon chart folds the area chart into layered bands: values near zero get a light fill, higher values stack darker bands on top. The result is that you can compare many series in the space where a line chart would only fit one or two. The tradeoff is legibility: horizon charts take a moment to learn, and untrained viewers may mistake band boundaries for real features in the data. Two or three bands keep things readable; beyond four, accuracy drops.

The brush on the overview bar converts pixel coordinates to dates via scaleUtc.invert(), then updates the main chart's domain. The main axis re-ticks itself because D3's time scales know about calendar structure. That calendar awareness is the whole point: scaleUtc does not space ticks at uniform millisecond intervals, it picks meaningful boundaries (midnight, month start, year start) depending on the visible span.

scaleTime vs scaleUtc

Both d3.scaleTime and d3.scaleUtc map dates to pixels, but they disagree about what "a day" means. scaleUtc lives in UTC, where every day is exactly 86,400 seconds. scaleTime uses the browser's local timezone, where daylight saving time creates days with 23 or 25 hours.

The difference is invisible most of the year. On the two DST transition days, it is not. With scaleTime, the pixel distance between 1:00 AM and 3:00 AM on a spring-forward Sunday is half the normal two-hour span, because that hour never existed locally. On a fall-back Sunday, 1:00 AM to 1:59 AM happens twice, and both occurrences map to the same pixel range. The chart lies about duration, and the lie is hard to spot because the shape still looks like a smooth line.

Use scaleUtc as the default. Switch to scaleTime only when the axis labels must read "9 AM" in the viewer's local time, such as a work-hours dashboard. Even then, compute durations in milliseconds ((end - start) / 3.6e6 for hours) rather than counting calendar hours, because millisecond arithmetic is DST-safe.

A common LLM mistake: using scaleTime because the data contains timestamps, without considering whether local-time labels matter. The chart renders, the axis looks fine, and nobody notices the data shifted by the timezone offset until someone in a different timezone gets different results from the same dataset. If your data is in UTC (most server-side data is), use scaleUtc.

Gap Handling

Real time series have holes. Sensors go offline. Markets close for weekends. Servers reboot. When a gap appears in the data, the line generator has two options: connect across it (implying continuity that does not exist) or break the line (honestly showing the absence).

D3's line and area generators support a .defined() accessor that controls this. When .defined() returns false for a data point, the generator lifts the pen. Without it, the generator draws a straight segment from the last real value to the next, which can cross minutes, hours, or days of missing data as if nothing happened. For monitoring dashboards, that false continuity can mask an outage. For financial data, it implies trading occurred over a weekend.

The first approach is more honest. The second preserves the familiar line-chart feel but requires the viewer to notice the gap.

Horizon Charts in Depth

The horizon chart is a space-compression trick. Start with a standard area chart. Divide the y-range into bands (typically 2 to 4). Cut the area chart at each band boundary, then collapse all bands into the same vertical space, using color intensity to encode which band a value falls in. The result takes one-third to one-quarter the height of a standard area chart while preserving the ability to read magnitude.

The technique comes from a CHI 2009 study by Heer, Kong, and Agrawala, which found that 2-band horizon charts match standard line chart accuracy once viewers learn to read them. At 3 bands, accuracy is still high. Beyond 4 bands, the encoding becomes ambiguous: viewers confuse band 2 of a 5-band chart with band 3, because the color steps are too subtle to distinguish reliably.

Horizon charts earn their space when you need to compare many series at once. A monitoring dashboard with 20 services stacked vertically is unreadable as 20 line charts but workable as 20 horizon charts, each 30 pixels tall. The tradeoff is onboarding cost. Viewers who have not seen the format before need a moment to learn it, and some never fully trust it. For a general audience, default to line charts and save horizon charts for expert dashboards or when vertical space is genuinely constrained.

A practical detail: horizon charts handle negative values by mirroring. Positive values fold upward in blue (or your positive hue), negative values fold upward in red (or your negative hue). This diverging color scheme lets you spot sign changes instantly, which makes horizon charts especially useful for metrics that fluctuate around a baseline, such as temperature anomalies or profit/loss.

Swimlanes and Gantt Charts

Not all time series are continuous measurements. Event data (deployments, incidents, task durations) is better shown as bars on a timeline. A swimlane chart assigns each category its own horizontal row and draws event bars from start to end. A Gantt chart is a swimlane chart where the rows are tasks and the bars show scheduled duration.

The layout challenge is overlap. When two events in the same lane overlap in time, you need to stack them into sub-rows. The greedy algorithm is straightforward: sort events by start time, maintain an array of per-sub-row end times, and assign each event to the first sub-row whose last event ends before this one starts. If no sub-row fits, create a new one. The event's sub-row index becomes its y-offset within the lane.

Swimlanes pair well with line charts. Place the line chart above showing a continuous metric (CPU usage, request latency) and the swimlane below showing categorical events (deployments, incidents). Align both to the same scaleUtc x-axis so the viewer can visually correlate "latency spiked here" with "this deployment happened at the same time." This combination answers causal questions that neither chart answers alone.

Cycle Plots for Seasonal Decomposition

A line chart of two years of data shows the weekly pattern as a rapid zigzag superimposed on a slow seasonal wave. You can see that something cycles weekly, but you cannot easily answer: "Is Sunday always the lowest day, or just usually?" A cycle plot answers this by splitting the data into one panel per cycle position (one per day of week, or one per month of year) and plotting all observations for that position in sequence.

Each panel in Figure 2 shows every occurrence of one day of the week across the full two-year span. The line traces the value over time (left to right within the panel), and the dashed red line marks the mean for that day. The pattern jumps out: Saturday and Sunday sit well below the weekday panels, and you can see the seasonal wave within each day's panel. This decomposition is impossible to read from a standard line chart.

Figure 2. Cycle plot decomposing the same synthetic data by day of week. Each panel shows all values for one day (Sunday through Saturday) across two years. The dashed red line marks the mean. The weekly pattern is explicit: weekends are consistently lower, and the seasonal trend is visible within each day.

Cycle plots assume a meaningful recurring period exists. If the mean values across cycle positions show no variance, the plot adds nothing. Test this before committing to the chart: compute d3.mean per cycle position and check that the range across positions is large relative to the within-position variance. If Tuesday's mean is 51.2 and Wednesday's is 51.4, a cycle plot will show seven nearly identical panels.

Real-Time Streaming Patterns

Streaming charts receive data continuously and must render it without growing memory or blocking the main thread. The naive approach, push every point into an array and redraw, works until it does not. At 1 Hz for an hour, you have 3,600 points. At 10 Hz for an hour, 36,000. At some point the array is large enough that redraw takes longer than the interval between points, and the chart falls behind.

The fix is a circular buffer. Allocate a fixed-size array, write new points at buffer[head % capacity], and advance the head. When reading, iterate from (head - capacity) to head, wrapping around. This gives O(1) insertion with no array copying, and the memory footprint is constant.

The second fix is requestAnimationFrame gating. When data arrives faster than the screen refreshes (typically 60 Hz), batch incoming points and flush them on the next animation frame. This prevents the chart from attempting 100 redraws per second when the screen can only show 60. Without this gate, high-frequency data sources cause jank even with a small buffer.

A third concern is cleanup. A streaming chart that outlives its DOM element (because the user navigated away but the component was not unmounted) keeps running its animation loop and processing WebSocket messages into nothing. Always pair requestAnimationFrame with cancelAnimationFrame, close WebSocket connections on teardown, and disconnect ResizeObserver instances. In framework contexts, this means a cleanup function in useEffect or onDestroy. In vanilla JS, it means a destroy() method that the caller is responsible for invoking.

If data arrives slower than 1 Hz, skip streaming architecture entirely. Just re-render on each new point. The circular buffer, rAF gating, and teardown machinery are only worth their complexity when the data rate is high enough to cause performance problems without them.

LTTB Downsampling for Large Datasets

A 1920-pixel-wide chart has at most 1920 distinct x-positions. Plotting 100,000 points on it means roughly 50 points share each pixel column. The extra points cost rendering time but add nothing visible. Downsampling reduces the point count to something proportional to the pixel width while preserving the visual shape of the line.

LTTB (Largest Triangle Three Buckets) is the standard algorithm for this. It divides the data into buckets, and for each bucket selects the point that maximizes the triangle area formed with the selected points from the neighboring buckets. The result is a subset that preserves peaks, valleys, and slopes, the features that define the visual shape. Gentle curves keep fewer points; sharp spikes keep more.

Figure 3 lets you see this in action. The slider reduces the displayed point count from the full 730 down to as few as 20. Watch how the algorithm preserves the major peaks and the weekly zigzag pattern even at aggressive downsampling levels. The shape holds remarkably well down to about 100 points. Below that, the weekly cycle disappears and only the seasonal envelope survives.

The core tradeoff: LTTB preserves what the chart looks like but not what the data says. Peak locations shift slightly. Frequency content is distorted. Exact values at specific timestamps are lost. This is fine, because LTTB is a rendering optimization, not a data transformation. The viewer sees the same shape; the underlying data remains untouched for any analytic queries.

For even larger datasets (millions of points), min-max bucketing is simpler and faster: keep the minimum and maximum value per pixel column. This guarantees that every visual peak and valley is represented. Use LTTB when the downsampled line needs to look smooth; use min-max when you just need to ensure no extrema are hidden.

Brushed Time Selection with Overview + Detail

The overview+detail pattern in Figure 1 is the canonical time-series interaction, and it is worth understanding the mechanics. The overview bar shows the full time range at low fidelity. The brush selects a window. The main chart zooms to that window at high fidelity. The axis re-ticks to match the new span.

The implementation hinges on two copies of the x-scale. xOverview maps the full date range to the overview bar's pixel width and never changes. xMain starts as a copy but its domain updates on every brush event. The brush's selection is a pair of pixel coordinates in overview space; xOverview.invert() converts them to dates, and those dates become xMain's new domain. Every element bound to xMain (the line, the area, the axis) re-renders with the new domain.

The axis re-ticking is automatic because d3.scaleUtc knows about calendar structure. At a two-year span, it picks month boundaries. At a two-month span, it picks weeks. At a two-week span, individual days. At a single day, hours. This is not a custom tick format; it is built into the scale. The only thing to be careful about is the tick count: pass a target count based on available width (Math.floor(innerW / 120) works well) so labels do not collide.

For programmatic brush control ("Last 7 days", "Last 30 days" buttons), call brushG.transition().call(brush.move, [x0, x1]) where x0 and x1 are pixel positions computed via xOverview(date). Do not also call the update function manually, or you will double-render: the brush's on('end') handler fires from the programmatic move.

When This Skill Hurts

Timezone confusion. scaleTime uses local time. scaleUtc uses UTC. LLMs mix them, and the resulting chart usually looks correct because the shape of the data does not change, only the x-position shifts by the timezone offset. The failure mode is subtle: a chart labeled "9 AM" actually shows 9 AM UTC, which is 1 AM Pacific. Nobody notices until someone in a different timezone gets different axis labels from the same code, or until a deployment timestamp lines up with the wrong event on the chart. The skill warns about this, but the warning is easy to skip when the chart renders without errors.

Gap mishandling. Missing data points should break the line, not connect across gaps. The mechanism is .defined() on the line generator, which lifts the pen when a point is absent. LLMs following the skill often omit .defined() entirely, producing a line that sails smoothly across hours or days of missing data as if nothing happened. For monitoring dashboards, this is actively dangerous: an outage appears as a gentle slope between two known values instead of a visible break. For financial data, it implies trading occurred on weekends. The fix is simple, but you have to remember to apply it, and generated code rarely does.

Horizon chart misuse. Horizon charts save space but require training to read. The skill describes them prominently because they are genuinely useful for expert dashboards with many concurrent series. LLMs pick up on that prominence and reach for horizon charts in contexts where the audience has never seen one. A product manager reviewing a quarterly report should not need to learn a new chart type to understand it. For audiences unfamiliar with the format, a standard line chart communicates better even if it takes more vertical space. Reserve horizon charts for contexts where the viewer will see them repeatedly and has time to learn the encoding.

Axis label collision. Time axes at different zoom levels need different tick formats. At a two-year span, "Jan 2024, Feb 2024, ..." works. At a two-day span, the same format produces 48 copies of "Jan 15, 2024" stacked on top of each other. D3's time scales handle this automatically when you use .ticks(count) with a reasonable count, but LLMs often hardcode a specific tick format or tick count that works at one zoom level and produces overlapping labels at others. The fix is to let the scale choose its own tick interval and format, overriding only when you have a specific reason.
