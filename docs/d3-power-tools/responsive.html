Responsive Means Redesigning

A chart that looks good at 800px can be unreadable at 320px. Shrinking is not adapting.

Setting width: 100% on an SVG does not make a chart responsive. It makes the chart smaller. The tick labels that were comfortably spaced at 800 pixels wide now overlap at 320. The legend that sat neatly to the right now covers half the data. The grouped bars that showed four categories side by side are now each two pixels wide. The chart still renders. It just stops communicating.

Responsive visualization means changing the design decisions as the space changes. Fewer ticks at narrow widths. Labels that rotate or disappear at breakpoints. A legend that moves from the side to below the chart. A layout that switches from grouped bars to stacked bars when horizontal space runs out. The data stays the same; the presentation adapts to serve it.

Figure 1. Quarterly revenue by product line. The same data rendered responsively: tick count, label orientation, legend placement, and bar layout all adapt to container width. A ResizeObserver on the wrapper triggers a full redraw on every change.

The key mechanism is ResizeObserver, not window.resize. A chart lives inside a container, and that container can change size for reasons that have nothing to do with the browser window: a sidebar opening, a panel collapsing, a CSS animation completing. Observing the container directly catches all of these. The one bug everyone hits is an infinite loop: the chart render changes the container height, which fires the observer, which re-renders, which changes the height again. The fix is to observe a wrapper whose dimensions come from CSS, not from its children.

The breakpoints in the demo above are not arbitrary. Each one responds to a specific legibility failure. At 600px, ten ticks start overlapping, so the count drops. At 500px, category labels no longer fit horizontally, so they rotate 45 degrees. At 400px, grouped bars become too narrow to distinguish, so the layout switches to stacked bars, which use the full width for each category. These are judgment calls, not formulas. Every dataset and every chart type will have different thresholds.

viewBox vs Redraw: The Core Tradeoff

SVG's viewBox attribute lets the browser scale your chart like an image. Set a coordinate system once, let it stretch to fit, and write no resize code at all. This works beautifully for decorative graphics, logos, and icons where there is no text to read. It fails for charts. A 14px axis label rendered at 800px becomes a 5.6px label at 320px. Tick marks that were comfortably spaced merge into a solid line. Interaction targets that were easy to click shrink below the touch threshold. The chart still technically renders. It becomes decoration.

Redraw-on-resize does the opposite: it tears down the chart and rebuilds it with new dimensions. Text stays at readable sizes because you specify it in absolute units each time. Tick count adapts because you recalculate it from the new width. The legend moves from beside the chart to below it because your breakpoint logic says to. The cost is real work on every resize event. For simple SVG charts with fewer than a hundred elements, that cost is trivial. For Canvas charts with tens of thousands of points, you may want to debounce.

There is a hybrid that sometimes makes sense: use viewBox for the data layer (lines, bars, areas), but overlay absolutely-positioned HTML for labels and annotations that need to stay at fixed pixel sizes. This splits the difference, but it introduces a coordination problem. Every resize, you need to map between the SVG coordinate system and the HTML coordinate system to keep labels aligned with the data they describe. In practice, full redraw is usually simpler and more reliable.

Aspect Ratio Management

A chart's aspect ratio is a design decision, not a constant. A wide, shallow time series works well on a desktop monitor. The same proportions on a phone screen produce a chart so short that the data collapses into a flat line. The fix is to let aspect ratio vary with width. At wide sizes, use something close to 16:9 or 2:1. As width shrinks, gradually increase the height ratio so the chart has vertical room to breathe. A common approach is to clamp the height between a minimum and a maximum while letting width flow freely.

The code in the demo above does this with Math.max(160, Math.min(280, width * 0.4)). At 800px wide, the chart is 280px tall, about 2.9:1. At 400px, it is 160px tall, a 2.5:1 ratio. The proportions shift gently rather than snapping, and the chart never gets so short that the bars disappear or so tall that it dominates the page on mobile.

For charts embedded in scrollytelling or dashboards, the container often dictates both dimensions. In that case, compute aspect ratio from the container and adapt layout accordingly. A nearly square container might warrant a different chart type entirely: a radial layout instead of a horizontal bar chart, or a compact sparkline instead of a full axis chart.

Touch Event Handling

Touch devices break two assumptions that desktop charts rely on. First, there is no hover state. Any interaction that depends on mouseover for discovery (tooltips, crosshairs, highlight-on-hover) needs a different trigger on touch. A tap-to-reveal pattern works, but it means the first tap shows information instead of performing an action, which conflicts with click-to-select or click-to-drill-down. You need to decide which interaction wins.

Second, fingers are wide. The contact area of a fingertip on glass is roughly 7mm across, which at a typical mobile pixel density covers about 40 CSS pixels. A 4px scatter plot dot that is easy to click with a mouse becomes nearly impossible to tap. WCAG 2.2 sets the minimum interactive target at 24x24 CSS pixels (and recommends 44x44), but for data visualization the practical requirement is even stricter because you need to distinguish adjacent points, not just hit a single button.

Figure 2. The same thirty-point scatter plot at two target sizes. Left: r=3, typical for mouse interaction. Right: r=10, sized for touch. The translucent gray circles on the right show the effective tap target. On mobile, the left chart's points are nearly impossible to select individually. On desktop, the right chart's spacing is generous but not wasteful.

The solution is not to make every chart touch-friendly by default. It is to detect the input type and adapt. matchMedia("(pointer: coarse)") returns true for touch-primary devices. On coarse-pointer devices, increase hit areas (transparent backing circles for SVG, expanded quadtree search radius for Canvas), replace hover interactions with tap, increase point radii, and add more padding between interactive elements. On fine-pointer devices, keep the compact, hover-rich experience.

Canvas DPI Scaling

Canvas on a retina display looks blurry unless you account for the device pixel ratio. The issue is that a <canvas> element with width="800" has 800 logical pixels. On a 2x display, those 800 pixels get stretched to fill 1600 physical pixels, producing visible fuzz on every edge and glyph. The fix requires three coordinated steps. Set the canvas element's width and height attributes to the physical pixel size (multiply by devicePixelRatio). Set the CSS width and height to the logical pixel size. Call ctx.scale(dpr, dpr) so all drawing commands operate in logical coordinates.

The trap is that setting canvas.width resets all context state: transforms, fill style, stroke style, font, globalAlpha, clip regions. If you resize by updating the canvas dimensions, you must re-apply ctx.scale(dpr, dpr) immediately afterward, before any drawing. Forgetting this after a resize produces one frame of blurry output until the next full redraw. Another subtlety: users drag windows between monitors with different pixel ratios. matchMedia(`(resolution: ${devicePixelRatio}dppx)`) fires when the DPR changes, and you should resize the canvas backing store when it does.

Iframe Embedding

Charts embedded in iframes face a unique sizing problem. The iframe is a fixed rectangle on the host page, but the chart inside it may need a different height depending on its width (responsive aspect ratio) or its data (a chart with more categories needs more vertical space). The standard solution is postMessage: the chart measures its own needed height after rendering and sends it to the parent page, which resizes the iframe to match.

This requires cooperation from both sides. The chart inside the iframe must call window.parent.postMessage({ type: "chart-resize", height }, "*") after every render. The host page must listen for that message and update the iframe's height. If the host page is on a different domain, the "*" target origin works but is a security tradeoff. In production, restrict it to the expected origin.

Two practical issues come up repeatedly. First, the iframe's initial load fires a resize before the chart has rendered, sending a zero or wrong height. Guard against this by only posting the message after the first meaningful paint. Second, the chart's ResizeObserver watches the body or a wrapper inside the iframe, and the iframe itself resizes from the host page, creating a bidirectional resize loop. Break this by having the chart only post height changes (width is controlled by the host page's layout), and having the host page only resize the iframe's height, never its width.

Breakpoint Design

Breakpoints in responsive visualization are not the same as breakpoints in responsive web design. CSS frameworks pick breakpoints at device-class boundaries: 576px for phones, 768px for tablets, 1024px for desktops. Chart breakpoints should instead be set where the design breaks. If your tick labels overlap at 550px, that is your breakpoint, even if no CSS framework recognizes it. If your legend covers data at 480px, move it there. Each breakpoint is a response to a specific legibility or usability failure, not a response to a device category.

Think of breakpoints as tiers of design simplification. At the widest tier, you have everything: full axis labels, side legend, annotations, grouped bars, hover tooltips. As width decreases, each tier removes or transforms one element. A useful progression for a typical chart component:

Above 700px: Full layout. Side legend, full tick labels, annotations visible, hover tooltips. This is the desktop experience with no compromises.

500 to 700px: Compact layout. Legend moves below the chart. Tick count drops. Long labels get abbreviated or rotated. Annotations with long text shrink to short labels or icons.

350 to 500px: Minimal layout. Axis labels rotate 45 degrees. Only every second or third tick label shows. The legend may collapse into a popover or a color key within the chart title. Grouped bars switch to stacked.

Below 350px: Survival mode. Consider whether the chart type itself should change. A grouped bar chart might become a small-multiples sparkline row. A scatter plot with hundreds of points might become a binned heatmap. A multi-series line chart might show only the most important series with a "show all" toggle. At this width, every pixel of margin and padding matters, and the chart exists primarily to convey a trend or a single comparison, not to support exploration.

The mistake most implementations make is testing only at the extremes: full-width desktop and narrow phone. The intermediate widths are where breakpoint logic most often fails. A chart at exactly 500px might be between two breakpoint tiers, producing a layout where the legend has moved below but the ticks have not yet thinned, causing overlap. Test at every breakpoint boundary and at widths 10 to 20px on either side of each boundary.

When This Skill Hurts

Failure modes to watch for

ResizeObserver infinite loops. The most common responsive charting bug: the chart render changes the container's height, which triggers the observer, which re-renders, which changes the height again. The cycle continues until the browser kills it. The fix (observe a wrapper sized by CSS, not by its children, with overflow: hidden) is mentioned in every tutorial and still gets missed. The variation that is harder to catch: a chart that changes its own width indirectly, by toggling a sibling element (like a legend) that causes a flex or grid reflow. The observer fires, the chart re-renders, the legend reappears, the container width changes, and the loop restarts.

viewBox scaling that shrinks text to illegibility. This is the default behavior when you set viewBox on an SVG and let it scale down. At half width, 12px text becomes 6px. At one-third width, it becomes 4px. The chart still renders with correct proportions and correct data, but no one can read it. LLMs are particularly prone to this because viewBox is a single attribute that "solves" responsiveness in one line, while redraw-on-resize requires restructuring the render function. The result looks correct in code review and fails in the browser.

Testing only at desktop width. LLMs generate and test charts at whatever viewport size their testing environment provides, which is usually 800 to 1200px. The chart works at that size. It was always going to work at that size. The failures are at 350px, where the axis labels overlap, and at 500px, where the legend covers the top-right bar cluster, and at exactly 600px, where the breakpoint logic transitions and produces a half-rotated, half-horizontal label set. If your test harness does not resize the container to each breakpoint boundary, responsive code is untested code.

Responsive redraws that cause layout thrashing. A naive resize handler calls render() on every ResizeObserver callback. If the render function reads layout properties (getBoundingClientRect, offsetWidth) and then writes DOM changes, and the observer fires multiple times in a single frame, you get forced synchronous layouts. The browser recalculates layout, you read, you write, it recalculates again. Debouncing with requestAnimationFrame or a short timeout solves this for most charts, but the underlying pattern (read-then-write inside a resize loop) is the actual problem.

Breakpoint logic at intermediate widths. A chart defines breakpoints at 600px and 400px. At 600px, it switches from side legend to bottom legend. At 400px, it switches from grouped bars to stacked. At 500px, both breakpoints are inactive: grouped bars with a bottom legend. This combination was never designed for and never tested. The bottom legend pushes the chart up, the grouped bars are narrow but not yet stacked, and two of the four categories are only 3px wide. The chart renders. It communicates nothing. Each breakpoint must be tested not just at its trigger point, but in combination with every other breakpoint state.
