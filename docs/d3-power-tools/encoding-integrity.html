Encoding Integrity

A chart can look polished and still lie. Honest encoding means the visual effect matches the data effect.

Edward Tufte defined the lie factor as the ratio of the size of an effect shown in the graphic to the size of the effect in the data. A lie factor of 1.0 means the visual is truthful. Anything above or below distorts perception. A truncated y-axis can make a 5% difference look like a 500% change. Radius-based circle scaling quadruples the perceived difference. These are not edge cases. They are the most common charts in dashboards and news articles.

The four figures below each start in their misleading form. Toggle each one to its honest version and watch what changes. The lie factor updates in real time. The goal is not to memorize rules but to develop an instinct: when you see a chart, your first question should be "what is this encoding doing to the numbers?"

Most of these distortions are not intentional deception. They happen because the defaults feel natural. Starting a bar chart at 90 "zooms in" on the interesting range. Using linear radius for bubbles is the obvious first choice. The damage is the same either way: the viewer walks away with a false impression of the data.

A visualization has encoding integrity when the visual effect faithfully represents the data effect, and would continue to do so with different data. Lie factor = 1.0 is the target.

Building the Instinct

These four patterns cover the most frequent encoding violations in the wild: truncated baselines, fabricated dual-axis correlations, radius-based size deception, and cherry-picked time windows. The fix in every case is structural, not cosmetic. Use yScale.domain([0, max]) for bar charts. Use d3.scaleSqrt when size encodes data. Use small multiples instead of dual axes. Show the full time range, or at minimum acknowledge what you have cut. A chart that passes these checks has encoding integrity. One that fails them is, in Tufte's precise language, lying.

5. The Lie Factor Formula

Tufte's lie factor is a single number that quantifies encoding honesty. The formula is straightforward: divide the size of the effect as shown in the graphic by the size of the effect in the data.

Tufte's Lie Factor lie factor = (size of effect shown in graphic) / (size of effect in data)

A lie factor of 1.0 means the visual is truthful. The graphic and the data agree on how big the difference is. A lie factor of 2.0 means the graphic makes a difference look twice as large as it actually is. A lie factor of 0.5 means the graphic makes a difference look half as large, which is equally dishonest in the opposite direction.

To compute it concretely, you need two percentages. First, the percentage change in the data between the two values you are comparing. Second, the percentage change in the visual measurement (pixel length, area, angle) between the corresponding marks. The lie factor is the second divided by the first.

Worked example: truncated bar chart Data: values of 92 and 97. Data effect = (97 - 92) / 92 = 5.4%
Truncated visual: axis runs 90 to 100. Bar heights = 2px and 7px. Visual effect = (7 - 2) / 2 = 250%
Lie factor = 250 / 5.4 = 46x

The formula catches both exaggeration (lie factor > 1) and minimization (lie factor < 1). In practice, exaggeration is far more common because it makes stories seem more dramatic. But minimization is arguably more dangerous: a chart that compresses a genuine crisis into a flat line can prevent action.

There are two important limitations. First, the lie factor only measures single pairwise comparisons. A chart with an honest maximum comparison but a dishonest midrange comparison will pass a spot check. Second, the formula measures encoding accuracy, not encoding appropriateness. A perfectly accurate pie chart still encodes data as angles, which humans compare poorly. The lie factor says the encoding is honest. It does not say the encoding is effective.

6. Metamorphic Relations

Static honesty checks catch a chart lying about the data it has. Metamorphic testing catches a chart that would lie about different data. The idea comes from software testing: instead of checking outputs against known answers, you check that relationships between outputs hold across input transformations. For visualizations, these transformations test whether the chart's encoding is structurally correct or just accidentally correct for one specific dataset.

Three relations cover the most important ground.

Scaling: multiply all values by a constant

If you double every value in a bar chart's dataset, the shape of the chart should be identical. The bars should have the same relative heights. The axis labels change, but the visual pattern stays the same. If it doesn't, the chart has a hardcoded domain, a magic-number maximum, or a fixed pixel value somewhere in its scale pipeline. This is the most common metamorphic failure in generated code: an LLM writes y.domain([0, 100]) instead of y.domain([0, d3.max(data, d => d.value)]), and the chart works perfectly for the synthetic data but breaks the moment you feed it real numbers.

A correct chart derives its scale domains from the data. The scaling relation tests that property without requiring you to read the code. Double the values, check the shape. If the shape changes, the encoding is fragile.

Permutation: reorder the data

Shuffle the data array. The chart should produce the same set of visual elements, possibly in a different DOM order, but with identical properties. The same bar heights, the same circle sizes, the same colors. If shuffling corrupts the visual, the chart has a missing key function. Without .data(data, d => d.id), D3's data join matches elements by index position. This works fine on the first render and breaks on every subsequent update, because entering and exiting elements get mismatched to the wrong data.

Permutation testing is especially important for animated transitions. A chart that looks correct on first render but garbles its bars when the data re-sorts has a join bug. The permutation relation catches this without requiring you to trigger a re-render manually.

Subset: remove data points

Remove 20% of the data points at random. The remaining elements should look identical to how they looked in the full dataset. Same bar heights, same positions relative to each other. If removing data points causes the remaining bars to jump or resize, the chart is recalculating its domain from the current subset. That means the chart's visual encoding depends on which data happens to be present, not on the data's actual values.

This is a real problem for dashboards with filters. A user applies a filter and the y-axis rescales, making remaining values look dramatically different even though nothing about them changed. The subset relation catches this by simulating the same mechanism: remove data, see if the survivors change appearance.

Metamorphic relations do not require expected outputs. You do not need a reference image or a golden dataset. You only need to know that a property (shape, set of marks, individual bar heights) should be preserved under a given transformation. This makes them practical for testing generated visualizations where you cannot predict the exact output.

8. Data Join Correctness

A data join is the mechanism that binds data points to visual elements. In D3, this happens through selection.data(). When the join is correct, each datum is bound to exactly one element, and that element's visual properties (position, size, color) reflect that datum's values. When the join is broken, elements display stale data, enter transitions animate from the wrong starting position, or the chart accumulates ghost elements that belong to no current datum.

The most frequent join bug is a missing key function. Without one, D3 matches data to elements by index. If the data array changes order (a sort, a filter, new items appended), index-based matching produces wrong bindings. The bar that was bound to "California" is now bound to "Texas" because Texas moved to position zero. The bar still shows California's height because it has not transitioned yet. When the transition fires, it animates from California's value to Texas's value, producing a meaningless intermediate state.

A correct data join uses .data(data, d => d.id) where the key function returns a stable identifier for each datum. This ensures that when data is reordered, filtered, or extended, each element stays bound to its original datum. The enter set contains genuinely new data. The exit set contains genuinely removed data. The update set contains data that persists.

You can verify join correctness by inspecting the bound data on DOM elements. In the browser console, d3.select("rect").datum() returns the object bound to that element. After a data update, check that each element's bound datum matches what you expect. If the first bar's datum says "California" but the bar's position and height correspond to "Texas," the join is broken.

The metamorphic permutation test automates this check. Shuffle the data, trigger an update, and verify that each element's visual properties still correspond to its bound datum. A chart that passes this test has a correct join. A chart that fails it will produce visual corruption in any scenario where data changes order, which includes sorting, filtering, streaming, and user-driven exploration.

Join correctness matters more than it might seem for encoding integrity. A chart with perfect scales, correct baselines, and honest area encoding will still lie to the viewer if it shows California's bar labeled "Texas." The encoding is mathematically correct for the wrong data. The chart is honest about numbers it has incorrectly assigned to categories.

When This Skill Hurts

Failure modes to watch for

Flagging legitimate log scales as lies. A log-scaled y-axis on a dataset spanning four orders of magnitude is not a truncated baseline. It is the only way to show the data without the bottom 99% of values collapsing into a single pixel row. But a rigid zero-baseline check flags it as a violation because bars do not start at zero. The fix is context: log scales are honest when the data distribution demands them. A bar chart on linear data with a y-axis starting at 50 is dishonest. A bar chart on exponential data with a log-scaled y-axis is the opposite. The skill needs to distinguish these cases, and mechanical application cannot.

Over-rigid zero-baseline enforcement. Not every chart needs a zero baseline. Sparklines embedded in text are meant to show trend, not magnitude. Stock charts show price movement within a narrow band. Heart rate monitors show variation around a physiological norm. Forcing these charts to include zero wastes most of the chart area on empty space, compresses the signal into a thin strip at the top, and makes the chart less honest in practice, because the reader can no longer see the variation that matters. Zero baselines are mandatory for bar charts and area charts, where length from baseline is the encoding channel. They are often counterproductive for line charts and dot plots, where position along a continuous axis is the encoding channel.

Encoding honesty is not data honesty. This skill measures whether the visual effect matches the data effect. It does not measure whether the data itself is representative. A chart that shows three cherry-picked years out of thirty can have perfect encoding integrity for those three years while giving a completely misleading picture of the trend. Selection bias, survivorship bias, and confounding variables are all invisible to encoding checks. A chart can be structurally honest and substantively deceptive. The skill catches the first kind of lie but not the second.

LLMs generating overly conservative charts. When an LLM internalizes encoding integrity rules, it sometimes overcorrects. It forces zero baselines on every chart, including sparklines and line charts where a zero baseline adds nothing. It refuses to use log scales even when the data spans multiple orders of magnitude. It adds unnecessary axis labels and baselines that consume space without informing the reader. The result is charts that are technically honest but visually wasteful: large swaths of empty whitespace, compressed data regions, and a feeling of defensive chart-making where the tool is optimizing for audit scores rather than reader comprehension. A good chart is honest and efficient. Wasting the reader's attention on empty space is its own form of disservice.
