Canvas Accessibility

Canvas is invisible to screen readers. You have to build the accessibility layer yourself, and the effort is worth it.

An SVG scatter plot is accessible almost for free. Each <circle> is a DOM element that screen readers can discover, that keyboards can reach, and that browsers know how to highlight. Canvas gives you none of that. When you draw 50 points onto a <canvas>, the browser sees one flat bitmap. A screen reader announces "image" and moves on. Keyboard users cannot tab to individual points because there are no individual points in the DOM. The visual output is rich, but the semantic output is empty.

This is the tradeoff you accept when you choose Canvas for performance. Hundreds or thousands of elements render faster on Canvas than SVG, but you inherit the responsibility of rebuilding what the browser gave you for free. That means three things: keyboard navigation so users can move between points without a mouse, ARIA live regions so screen readers announce what is focused, and a hidden data table so the underlying numbers are always reachable.

The pattern: Canvas draws the pixels, but a parallel layer of invisible HTML and ARIA attributes carries the meaning. The visual and the semantic are separate systems that you keep in sync.

ARIA Roles for Data Visualization

The first decision when making a canvas chart accessible is which ARIA role to assign. The role determines how a screen reader presents the element and which keyboard interactions the user expects. Get it wrong and the screen reader will promise an interaction model that your code does not deliver.

A static chart that the user cannot navigate should use role="img" with a descriptive aria-label. The screen reader announces it as an image, reads the label, and moves on. This is the right choice when the chart is illustrative and the underlying data is available elsewhere on the page.

An interactive scatter plot or force layout should use role="application". This tells the screen reader to pass all keystrokes through to the page instead of intercepting them for its own navigation. Without role="application", VoiceOver and NVDA will capture arrow keys for their own cursor movement and your keyboard handler will never see them. The cost is that the user loses access to screen reader shortcuts while inside the element, so you need to handle Escape to let them leave.

A bar chart with a ranked list of values maps well to role="listbox" with each bar as role="option". The screen reader announces "listbox, 12 items" on entry, and the user navigates with arrow keys expecting a linear sequence. A heatmap or matrix should use role="grid" with role="row" and role="gridcell" so arrow keys move in two dimensions. A tree layout should use role="tree" with role="treeitem" so up/down moves between siblings and left/right collapses and expands.

The role is not decoration. It is a contract. When you assign role="grid", the screen reader announces "grid" and the user expects row/column navigation. If your keyboard handler only supports left/right, the user will press down-arrow, hear nothing, and assume the chart is broken. Match the role to the navigation model you actually implement.

Screen Reader Announcements

Canvas draws pixels. Screen readers read the DOM. The bridge between them is the ARIA live region: an invisible DOM element whose text content, when changed, triggers the screen reader to speak. The live region is how you tell the screen reader what just happened on the canvas.

There are two politeness levels. aria-live="polite" waits for the screen reader to finish its current utterance before announcing new content. aria-live="assertive" interrupts immediately. For data visualization, assertive is almost always correct. When the user presses an arrow key to move to the next data point, they need to hear the new point's values now, not after the screen reader finishes reading something else. Delayed feedback makes spatial navigation disorienting.

The aria-atomic="true" attribute ensures the screen reader reads the entire content of the live region each time it changes, not just the portion that differs. Without it, some screen readers might try to diff the old and new text, leading to garbled partial announcements. For chart navigation, you always want the full announcement: "Point 7, x 42.3, y 18.9, Beta, size 6.1."

Debouncing matters. When a user holds down an arrow key to scan quickly across a scatter plot, you do not want 30 announcements piling up in the screen reader's speech queue. A 100 to 200 millisecond debounce on the live region update lets rapid key presses skip intermediate points and only announce the one the user lands on. But do not debounce too aggressively. For tree navigation where each step is deliberate, 50 milliseconds or no debounce at all is better.

Structure announcements as identity first, then values, then context. "Point 12, x 67.3, y 41.0, Alpha, size 4.2" follows the same order every time, so the user builds a mental model of where each piece of information falls. If you vary the order, the user has to parse each announcement from scratch.

Beyond point-by-point navigation, certain events deserve their own announcements. When the user enters the chart: "Scatter plot, 50 data points, use arrow keys to navigate." When they apply a filter that changes the visible data: "Filtered to 23 of 50 points." When they select a point: "Point 12, selected." When they press Escape: "Exited chart." These structural announcements give the user a mental map of the interaction state.

Focus Management Across Canvas Redraws

Canvas redraws destroy and recreate the entire image every frame. Unlike SVG, where a focused <circle> persists in the DOM across updates, Canvas has no persistent elements. The focus ring you drew on frame 1 does not exist on frame 2. You draw it again.

This means focus is entirely a JavaScript state variable. You track which data point is focused by its ID, and on every render pass you check whether the canvas has DOM focus (document.activeElement === canvas), whether a data point is selected, and if both are true, you draw the focus ring around the corresponding pixel coordinates. The focus ring is always the last thing drawn so it sits on top of all data points.

The tricky part is data updates. When the dataset changes, the point that was focused might no longer exist. If the user was focused on point 12 and a filter removes it, you need a fallback strategy. Three options: move focus to the nearest remaining point (preserves spatial context), move focus to the first point in the new set (simple and predictable), or clear focus entirely and announce "focused point removed, press arrow key to select a new point." The worst option is doing nothing, because the user will press an arrow key, nothing will happen, and they will think the chart is broken.

When the canvas participates in animated transitions, keep the focus indicator stable. If points are interpolating to new positions, update the focus ring position to track the moving target. A stationary focus ring around empty space while the point slides to a new location is confusing. If you cannot smoothly track the animation, suppress the focus ring during the transition and redraw it at the final position once the animation completes.

The Shadow DOM Approach

The hidden DOM mirror is a parallel structure of invisible, focusable HTML elements that shadow the visual elements on the canvas. For each data point drawn on the canvas, a corresponding <div> or <span> exists in a visually hidden container, carrying the ARIA attributes that describe that point. The screen reader traverses this hidden structure while the canvas provides the visual representation.

The simplest version uses a container with role="listbox" and one child per data point with role="option" and an aria-label describing the point's values. The container is positioned off-screen using the standard screen-reader-only CSS pattern (position: absolute; width: 1px; height: 1px; overflow: hidden; clip: rect(0,0,0,0)). The canvas element references the mirror with aria-owns, and tracks the focused element with aria-activedescendant.

The mirror must stay synchronized with the canvas. When data is filtered, added, or removed, the mirror needs a corresponding update. D3's data join (.selectAll().data().join()) works well here. The enter/update/exit cycle keeps the mirror's DOM nodes in sync with the data array. You run the join in the same render function that redraws the canvas, so they never drift apart.

For hierarchical data, the mirror can use role="tree" with nested role="treeitem" elements and aria-expanded attributes. For a grid or heatmap, role="grid" with role="row" and role="gridcell". The mirror structure maps directly to the ARIA role you chose for the visualization. This is more complex than the live-region-only approach, but it lets screen reader users browse the chart structure freely using their screen reader's native navigation commands, rather than being limited to whatever keyboard handler you wrote.

Testing with Screen Readers

Keyboard testing is necessary but not sufficient. A chart can be fully keyboard-navigable and still unusable with a screen reader. The only way to know what a screen reader actually announces is to turn one on and listen.

VoiceOver is built into macOS. Turn it on with Command-F5 or through System Settings. Navigate to your chart with the VoiceOver cursor (Control-Option plus arrow keys), then enter the interactive element. Listen for three things: does the screen reader announce the chart's role and label on entry? Does it read point values as you navigate with arrow keys? Does it announce state changes (selected, exited) when you press Enter or Escape? If any of these are silent, the ARIA attributes are not wired correctly.

NVDA is free and runs on Windows. It is the most widely used screen reader among blind users. NVDA handles role="application" differently from VoiceOver: it enters forms mode automatically, whereas VoiceOver may require the user to "interact" with the element first. Test with both to catch these behavioral differences. Common failure: your live region works in VoiceOver but NVDA never speaks the updates because the live region was added to the DOM after NVDA loaded and NVDA did not detect it. Adding the live region element in the initial HTML (not dynamically) avoids this.

JAWS, the other major Windows screen reader, has its own quirks with live regions and aria-activedescendant. If your audience includes enterprise users (government, finance, healthcare), JAWS testing is not optional. It handles aria-live="assertive" reliably, but its treatment of aria-activedescendant on non-standard containers has historically been inconsistent. The DOM mirror approach with actual focus movement (moving DOM focus to each mirror node rather than just updating aria-activedescendant) is more robust across all three screen readers, at the cost of additional complexity.

Automated accessibility testing tools (axe, Lighthouse) will catch missing labels and incorrect roles, but they cannot test whether the live region announcements make sense or whether the keyboard navigation flow is logical. They test the structure, not the experience. Manual testing with a screen reader is the only way to catch the difference between "technically accessible" and "actually usable."

When This Skill Hurts

Accessibility theater. Adding aria-label, role, and aria-live attributes to a canvas element does not make it accessible. It makes it look accessible in a Lighthouse audit. If the keyboard handler does not actually move focus between data points, if the live region never updates with meaningful content, if the DOM mirror exists but contains stale data, the ARIA attributes are set dressing. Screen reader users encounter an element that claims to be an "application" with "50 data points" but does not respond to any input. This is worse than no accessibility at all, because it promises an experience and then fails to deliver. A plain role="img" with a good text description is honest. A broken role="application" is a trap.

Keyboard conflicts. When you assign role="application" and intercept arrow keys, you take those keys away from the screen reader's own navigation. This is by design for chart navigation, but it can go wrong. If your handler also captures Tab, the user cannot leave the element. If it captures single-letter keys for shortcuts (S for sort, F for filter), those letters conflict with screen reader quick-navigation keys that let users jump between headings, links, and landmarks. Stick to arrow keys, Home, End, Enter, Space, and Escape. Let everything else pass through. And always, always handle Escape as "release focus and return to normal navigation."

The useless data table. A data table fallback is technically accessible. Screen readers can read tables. But a 10,000-row table with columns for x, y, category, and size provides no analytical value. The user can read individual rows, but they cannot see patterns, clusters, or outliers. The table satisfies the accessibility checklist while delivering none of the insight the visualization was built to convey. A better approach: provide summary statistics ("3 clusters centered at..."), sortable columns that surface extremes, and filtering that lets the user explore subsets. The table should be a tool for analysis, not a compliance checkbox.

LLMs generating accessibility code they cannot test. An LLM can write aria-live="assertive" and aria-activedescendant with perfect syntax. It cannot turn on VoiceOver and listen to the result. It cannot hear that the live region fires twice per keystroke because of a re-render cycle. It cannot notice that NVDA reads the label but ignores the description. It cannot feel the confusion of pressing arrow keys and hearing silence. Every accessibility pattern in this skill has been tested with actual screen readers, but when an LLM applies these patterns to a new visualization, the specific integration has not been tested. The LLM should flag this: "This chart includes ARIA attributes and keyboard navigation, but I cannot verify screen reader behavior. Test with VoiceOver (macOS) or NVDA (Windows) before shipping."
