Hierarchy Layouts

Four ways to draw the same tree, each answering a different question.

Every dataset with a parent column is a tree, but not every tree needs the same picture. A tree layout shows you who reports to whom. A treemap shows you where the headcount is concentrated. A circle-packing layout shows which groups contain which. A partition layout reveals both depth and proportion. These are four views of the same underlying structure, and the choice of layout determines which question a viewer can answer at a glance.

D3 makes this concrete. You call d3.hierarchy() once to build the tree, then pass that single root node to whichever layout function fits your question. The data stays the same. Only the geometry changes.

Different layouts reveal different things about the same tree. Treemap shows size. Tree shows topology. Pack shows nesting. Partition shows depth and proportion. Choosing the wrong layout buries the answer you are looking for.

The interactive figure below renders one company hierarchy in all four layouts. Toggle between them and notice what becomes obvious (and what becomes hidden) in each view.

The tree layout is the only one that draws edges between parent and child. It makes paths traceable but says nothing about sizes. Treemaps use rectangle area to encode a value (here, headcount), making size comparisons immediate but collapsing topology. Circle packing communicates containment through enclosure at the cost of wasted space. Partition splits each level into proportional slices, giving you both depth and relative size in a single view.

Tiling Strategies

A treemap's visual character depends on its tiling algorithm, not just its data. D3 ships four strategies that all fill the same rectangle with the same leaf areas but produce very different shapes. The choice matters because shape determines readability: long thin slivers are hard to label, hard to click, and hard to compare by area.

Squarify (the default) optimizes for aspect ratios close to 1. Cells are close to square, which makes area comparison easiest and gives labels the most room. The tradeoff is that the algorithm reorders nodes to achieve good ratios, so the spatial position of a cell is not stable across data updates. If your treemap animates between two time periods, squarify will shuffle cells around, making it impossible to track individual items.

Binary splits the data with a balanced binary partition, alternating between horizontal and vertical cuts. Aspect ratios are worse than squarify but still reasonable, and the ordering is deterministic. This is a good middle ground when you need moderate readability without animation and want to preserve some sense of the data's original order.

Slice-and-dice alternates between horizontal slices (at even depths) and vertical slices (at odd depths). It preserves the data's ordering perfectly, which makes it the right choice when position along one axis encodes something meaningful. The classic use case is a Marimekko chart: a two-level hierarchy where variable-width columns represent categories and segments within each column represent subcategories. The downside is extreme aspect ratios. Deep hierarchies produce cells so thin they are illegible without hover labels.

There is also resquarify, which caches the topology of squarify's first run and reuses it on subsequent updates. This gives you squarify's good aspect ratios with stable positions for animation. If your treemap transitions between datasets, always use resquarify.

Figure 2. The same hierarchy rendered with three tiling strategies. Squarify produces the most readable cells. Slice-and-dice preserves ordering but creates extreme aspect ratios. Binary balances between the two.

Label Placement in Treemaps

Labels are where treemaps most often fail in practice. The algorithm produces rectangles of wildly varying size, and text that fits in the largest cell will overflow or clip in smaller ones. An LLM generating treemap code will typically place a label at a fixed offset inside each cell without checking whether the text actually fits. The result is a chart with overlapping, clipped, or illegible labels.

The basic fix is to measure text against cell dimensions and hide labels that do not fit. For each cell, compare the cell width (minus padding) to the approximate text width. If the text is too long, either truncate with an ellipsis or hide the label entirely and rely on hover tooltips. The existing Figure 1 treemap does a simple version of this: labels disappear below a size threshold. A more robust approach uses getComputedTextLength() to measure the rendered text and progressively truncates until it fits or gives up.

Multi-line labels are occasionally worth the effort for larger cells. Split the name on whitespace, render each word as a <tspan> with dy="1.1em", and stop when you run out of vertical space. But this adds complexity, and for most treemaps the single-line-or-nothing approach is sufficient. The real lesson is that any treemap without label-size checking is broken by default.

Color Encoding Strategies

Color in hierarchy layouts answers one of three questions, and mixing them up produces charts that look colorful but communicate nothing.

Categorical by top-level ancestor answers "which branch does this leaf belong to?" This is the most common treemap coloring and the one used in Figure 1. Assign a color to each depth-1 node, and every descendant inherits it. Vary opacity or lightness by depth so leaves look distinct from their parent groups. The approach works well for two to three levels of hierarchy. Beyond that, leaves from different branches can look similar.

Sequential by depth answers "how deep is this node?" Map depth to a sequential color scale like d3.interpolateBlues. This makes the tree's structure visible even in space-filling layouts where edges are absent. It works best for partition and icicle layouts where depth is already spatially encoded and color reinforces it.

Diverging by value or change answers "what grew or shrank?" or "which cells are outliers?" Map a continuous variable (like year-over-year change) to a diverging scale. Red-blue works well. This is powerful for treemaps that show budget comparisons or performance dashboards where the viewer needs to spot anomalies.

Pick one color question per chart. Encoding category by hue and depth by lightness in the same treemap creates a two-dimensional color scheme that no viewer can decode without a legend they will not read.

The Stratify Pattern

Not every hierarchy arrives as nested JSON. The more common case is a flat table with id and parentId columns. d3.stratify() converts this flat structure into the nested hierarchy that all layout functions expect.

The pattern is straightforward: call d3.stratify().id(d => d.id).parentId(d => d.parentId) on an array of rows. The root row has a null or empty parentId. Every other row points to its parent. Stratify builds the tree by linking children to parents.

Where this goes wrong in practice: duplicate IDs throw a cryptic error. Orphaned nodes (where parentId refers to an ID not in the dataset) throw another. Multiple roots (more than one row with null parentId) throw a third. Trailing whitespace in CSV fields is the most common cause. Always trim IDs and validate the data before calling stratify. Check for exactly one root, no duplicate IDs, and no orphans.

For path-based data like file systems, D3 offers d3.stratify().path(d => d.filepath), which automatically creates intermediate directory nodes. This saves you from manually constructing parent IDs for every directory level.

When Pack Circles Are Worth the Waste

Circle packing wastes roughly 21% of space by geometry: circles cannot tile a plane without gaps. In practice the waste is higher because the packing algorithm does not achieve perfect tangency at every level. The result is that a circle-packed layout uses noticeably less of the viewport for actual data than a treemap of the same hierarchy.

The counterargument is containment clarity. Circles communicate "this group contains these items" more immediately than nested rectangles do. The boundary of a circle is unambiguous. In a treemap, the boundary between a parent group and its children is implicit, defined by padding and color, and easily lost when cells are small. Circle packing also handles ragged hierarchies (where some branches are deeper than others) more gracefully because the enclosing circle scales naturally regardless of the number of children.

The decision comes down to what the viewer needs. If the question is "how big is each item?" use a treemap. If the question is "which group does this item belong to, and how do group sizes compare roughly?" pack circles can be the right choice despite the space penalty. For precise size comparison, neither layout is ideal. A bar chart of leaf values will always be more accurate than either area encoding.

The sunburst above is the same d3.partition() used in Figure 1's rectangular partition view, but mapped to polar coordinates via d3.arc(). The radial scale uses d3.scaleSqrt so that outer rings do not dominate by area. Without the square root correction, a ring at radius 200 has four times the area of a ring at radius 100, even if they represent equal proportions. This is one of the most common sunburst bugs: forgetting the sqrt makes outer categories look disproportionately large.

When This Skill Hurts

Treemap for small trees. A 3-level hierarchy with 8 leaves does not need a treemap. The space-filling algorithm adds visual complexity without payoff. The viewer has to parse nested rectangles, color coding, and truncated labels to understand a structure that a bulleted list or a simple bar chart would communicate instantly. LLMs reach for treemaps because the skill describes them prominently, and the word "hierarchy" in the prompt triggers the pattern. For small trees, a tree layout or even a plain table is almost always better.

Label occlusion. The skill does not adequately address text fitting in small treemap cells. An LLM following the skill will generate labels at fixed positions inside every cell, producing text that overflows, overlaps, or clips. The result is a chart that looks broken: half-visible words layered on top of each other. Every treemap needs a label-fitting guard that hides text when the cell is too small. Without it, the chart is illegible at any nontrivial data size. Figure 1's treemap view demonstrates the minimum viable fix: labels vanish below a size threshold. But even this simple approach is missing from most LLM-generated treemaps.

Pack circle waste. Circle packing wastes roughly 21% of space by geometry, and in practice more than that due to imperfect tangency. For any task that requires precise size comparison, a bar chart of leaf values will always be more accurate. The skill presents pack as a layout option alongside treemap and partition without emphasizing this limitation. An LLM may choose pack for a dashboard where the viewer needs to compare department budgets, producing a chart where the answer is encoded in circle area, the least accurate perceptual channel for quantitative comparison. Use pack when containment and grouping matter more than exact sizes.

Depth explosion. Deep hierarchies with 5 or more levels make every layout illegible. Treemap cells at the deepest level become slivers. Sunburst arcs at the outer ring become unreadably thin. Circle packing nests so many rings that the innermost circles are invisible. Tree layouts spread across such a wide span that node labels collide. The skill should recommend flattening or aggregating before rendering, but LLMs dutifully render all levels because the data has them. If your hierarchy is deeper than 3 or 4 levels, aggregate the bottom levels into "Other" nodes or provide interactive drill-down instead of showing everything at once.
