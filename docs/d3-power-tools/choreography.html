Choreography

Sequencing transitions so the viewer processes one change at a time.

A static chart delivers all its information at once. A complex one can overwhelm: colors, positions, labels, and trend lines competing for attention. Choreography solves this by treating the visualization as a scene that unfolds over time. Each stage introduces one new idea, gives the viewer a moment to absorb it, then moves on. The result feels like a guided tour rather than a data dump.

The technique below uses the "sticky graphic" pattern. The chart stays fixed in place while explanatory text scrolls alongside it. An IntersectionObserver detects which step is visible, and each step triggers exactly one transition. Scroll down through the five stages and watch the chart build itself piece by piece.

The core judgment in choreography is sequencing. When you show everything at once, the viewer has to figure out what matters. When you control the order, you make that decision for them. One change at a time, each one motivated by the text beside it.

How It Works

The implementation follows the "stage manager" pattern from the choreography skill. An array of stage functions maps each scroll step to a specific chart state. When the IntersectionObserver fires, it calls goToStage(index), which runs the corresponding function. Each function is idempotent: it produces the correct result regardless of which stage the chart was in previously. This matters because users scroll both directions.

Transitions use D3's .transition() with explicit durations and easing. The staggered entry in Stage 2 uses a functional delay, .delay((d, i) => i * 20), so points appear one by one rather than all at once. The trend line in Stage 4 uses stroke-dashoffset animation to create the drawing effect. These are small details, but they are the difference between a chart that feels directed and one that feels accidental.

Sequencing with async/await

The scrollytelling figure above fires one transition per scroll step, so the sequencing is implicit: the reader controls the pace by scrolling. But many choreographies need to chain stages programmatically, where Stage 2 must not begin until Stage 1 has finished. D3 v6 introduced transition.end(), which returns a promise that resolves when every element in the selection completes its transition. Combined with async/await, this turns complex choreography into readable, linear code.

The pattern is straightforward. An async function calls await selection.transition().duration(800).attr(...).end() for each stage. The await pauses execution until every element in that selection has finished animating. Then the next line runs. No callbacks, no nested .on("end") handlers, no manual counting of how many elements have completed.

There is one subtlety that trips people up. If a transition is interrupted -- because the user clicks a button that starts a new transition on the same elements, or because goToStage is called before the current stage finishes -- the promise from transition.end() rejects. Your async function needs a try/catch around each await, or the entire chain breaks on the first interruption. The idiomatic approach is to catch the rejection silently and let the new transition take over. D3's default behavior of killing an active transition and starting the new one is usually correct; the try/catch just prevents the rejected promise from throwing an unhandled error.

A common mistake is calling .end() on an empty selection. If no elements match (say, no circles exist yet), the returned promise resolves immediately. This is actually convenient for idempotent stage functions: the "ensure dots exist" step costs nothing if they are already there. But if you expect the promise to wait for actual animation, an empty selection will silently skip the wait.

The Stage Manager

At the heart of any choreography is a state machine. An array of functions, one per stage, each responsible for transitioning the chart to a specific visual configuration. A single goToStage(index) dispatcher calls the right function and updates any UI indicators (progress dots, active step highlights, caption text). This is the "stage manager" pattern.

The critical design constraint is idempotency. Each stage function must produce the correct visual result regardless of which stage the chart was in previously. If the user jumps from Stage 5 directly to Stage 1 (by scrolling fast, clicking a dot, or navigating backward), Stage 1 cannot assume that Stages 2 through 4 have run first. It must clean up any elements those stages may have created and rebuild its own state from scratch. In the scrollytelling figure above, every stage function explicitly removes later-stage elements and ensures its own prerequisites exist.

This is harder than it sounds. The temptation is to write stage functions as deltas: "Stage 3 adds color to the dots that Stage 2 created." But if Stage 2 never ran (the user jumped from Stage 1 to Stage 3), there are no dots to color. An idempotent Stage 3 checks whether dots exist, creates them if needed, then applies color. The code is more verbose, but it handles every navigation path correctly.

The Sticky Graphic Pattern

The scrollytelling layout in the figure above uses the "sticky graphic" pattern: a two-column grid where the chart column has position: sticky and the text column scrolls normally. As the reader scrolls, the text steps move past while the chart stays pinned to the viewport. Each step entering the visible area triggers a transition on the chart.

The layout itself is a CSS grid with two columns. The text column contains a series of div elements (the "steps"), each tall enough that only one is prominently visible at a time. The chart column is a single sticky element. On narrow screens, the grid collapses to a single column, with the chart sticking to the top and the text scrolling beneath it.

Detection uses an IntersectionObserver. Each step element is observed, and when it crosses the visibility threshold, the observer fires. The rootMargin option controls where the trigger zone sits: "-30% 0px -30% 0px" means the middle 40% of the viewport is the active zone. A step must enter this zone to trigger. This prevents premature firing when the step is barely peeking into view at the edge.

The threshold option deserves thought. A threshold of 0.1 means the observer fires when 10% of the step is visible within the root margin. Lower thresholds fire earlier; higher thresholds fire later. For most scrollytelling, a single low threshold (0.1 to 0.3) works well because you want the transition to begin as the step enters, not when it is fully centered. If you need continuous scroll-linked animation (where a property tracks scroll position from 0 to 1), you need many thresholds or a scroll event listener with requestAnimationFrame throttling.

Scroll Direction and Reversibility

Readers scroll both ways. They read forward through the story, then scroll back to re-read an earlier stage, then scroll forward again. Every stage transition must work in both directions.

The IntersectionObserver callback receives an entry object, but it does not directly tell you the scroll direction. However, comparing the new step index to the current stage index reveals it: if the new index is lower, the user scrolled up. Some implementations (like the Scrollama library) surface a direction field explicitly. The stage manager pattern handles this naturally -- goToStage(index) does not care about direction because each stage function is idempotent -- but the visual transitions may need to differ.

Consider the trend line in Stage 4 of the figure above. When scrolling forward from Stage 3, the line draws from left to right with a smooth stroke-dashoffset animation. But when scrolling backward from Stage 5 to Stage 4, the line should not redraw from left to right -- it should already be there, because Stage 5 included it. If Stage 4 blindly replays the draw animation every time, the backward transition looks wrong: a fully drawn line disappears and redraws. The fix is to check whether the element already exists in its target state and skip the entrance animation if so.

More broadly, forward transitions tend to be additive (elements enter, color appears, annotations arrive), while backward transitions are subtractive (elements exit, color fades, annotations disappear). A well-choreographed piece handles both, so that scrolling up through the story feels like rewinding rather than watching a glitchy replay.

Designing for Re-reading

Scrollytelling is not a video. The reader can pause, go back, and re-read at will. This has design implications that are easy to overlook.

First, stage transitions should be fast enough to not frustrate a reader who is scanning. A 2-second entrance animation is pleasant the first time and annoying the third time. Keep transitions under 800 milliseconds for stages that the reader might revisit. Reserve longer animations for the single moment of highest impact.

Second, the text steps need to be self-contained. Each step should make sense on its own, without requiring the reader to remember what the previous step said. A step that says "Notice how this changes" is useless if the reader jumped directly to it. A step that says "The trend line shows a 0.6x relationship between input and response" works regardless of arrival path.

Third, the visual state at each stage should be unambiguous. If Stage 3 shows colored dots and Stage 2 shows grey dots, a reader arriving at Stage 3 should see colored dots immediately, not a flash of grey followed by a color transition. This is another argument for idempotent stage functions: they produce the final state directly, optionally animating the transition from whatever the current state happens to be.

IntersectionObserver Performance

The IntersectionObserver API is efficient because the browser batches observations and delivers callbacks asynchronously, outside the main scroll event loop. But there are still choices that affect performance.

The threshold option controls how many callbacks fire as an element crosses the viewport. A single threshold (e.g., [0.1]) fires once when 10% of the element is visible. An array like [0, 0.25, 0.5, 0.75, 1] fires five times as the element scrolls through. For step-based scrollytelling, a single threshold is almost always sufficient. Progress-based scrollytelling, where a property tracks scroll position continuously, is better served by a scroll event listener with requestAnimationFrame throttling than by hundreds of thresholds.

The rootMargin defines the trigger zone. Setting it to "-30% 0px -30% 0px" shrinks the effective viewport to the middle 40%, which prevents a step from triggering when it is barely visible at the screen edge. This is a judgment call: too narrow and the reader must scroll precisely to trigger each step; too wide and steps fire when they are not yet the reader's focus. The 30% margin used in the figure above is a reasonable default for desktop; on mobile, where the viewport is shorter, you might reduce it to 20%.

One non-obvious performance concern: observing too many elements. If you have 50 scroll steps, each observed with threshold: [0, 0.1, 0.2, ..., 1.0], you have 50 x 11 = 550 threshold crossings to track. This is unlikely to cause problems on modern browsers, but it is wasteful. Observe only the step containers, not their children. Use a single low threshold. Let the stage manager handle the rest.

When This Skill Hurts

When This Skill Hurts

  • Scroll hijacking. Sticky graphics intercept scroll behavior. If the graphic occupies most of the viewport, the user scrolls but the page barely moves -- they are "trapped" advancing the visualization instead of moving through the document. LLMs following this skill tend to make the graphic too tall, sometimes 80-90% of viewport height, leaving the reader stuck in the scrollytelling section with no sense of progress through the page. The fix is to keep the sticky graphic to roughly 40-50% of viewport height and ensure the text steps provide a visible scroll distance.
  • Non-reversible transitions. The skill emphasizes forward progression but LLMs rarely implement backward transitions. Scrolling up shows the current state frozen in place rather than reverting to the previous state. The narrative breaks: the text says "Before we add color" but the chart still shows color from three stages ahead. Idempotent stage functions (as described above) solve this, but they require deliberate engineering that LLMs tend to skip.
  • Stage count inflation. Each scroll stage is a commitment from the reader. They must scroll, wait for the transition, read the annotation, and absorb the change before continuing. LLMs following the skill tend to create 8-10 stages when 4-5 would communicate the same progression with less fatigue. More stages does not mean more insight -- it means more scrolling, more waiting, and more opportunities for the reader to disengage. If two stages communicate the same idea, merge them.
  • Mobile scroll conflict. position: sticky and IntersectionObserver behave differently on iOS Safari. The rubber-band overscroll effect can cause spurious observer callbacks. The address bar collapsing and expanding changes the viewport height, which shifts the rootMargin trigger zone mid-scroll. Touch momentum scrolling overshoots steps that would be hit precisely with a mouse wheel. The skill does not warn about these mobile quirks, and LLMs produce scrollytelling experiences that work on desktop Chrome but break on phones, where a large fraction of readers actually are.
