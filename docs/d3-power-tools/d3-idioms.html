D3 Idioms

Idiomatic D3 communicates intent through structure. Chaining indentation, margin convention, and .join() are not style preferences. They make code reviewable.

Two D3 programs can produce the exact same chart and still differ wildly in how well they communicate to the next person who reads them. Idiomatic D3 uses the library's own conventions to make the code's structure mirror the structure of the visualization: which elements belong to which group, which attributes apply to which selection, where data enters the scene and where it exits.

The pairs below demonstrate this. Each left-hand snippet and its right-hand counterpart produce an identical bar chart. The difference is legibility. Non-idiomatic code forces you to simulate execution in your head to figure out what it does. Idiomatic code shows you at a glance.

This matters most during review. When someone changes a scale domain, adds a color encoding, or wires up a brush, idiomatic structure lets you verify that the change is correct without re-reading the entire file. Non-idiomatic code makes every change a full-context-reload.

Every rule here is about the human, not the machine. The browser does not care about your indentation or whether you used .join() or a manual loop. The person reviewing your pull request does.

Reusable Chart Closures

Bostock's reusable chart pattern is a function that returns a configurable chart. The outer function declares defaults. The inner function receives a selection and renders into it. Getter-setters on the returned function let callers override defaults before rendering. The key insight: because JavaScript closures capture their enclosing scope, each call to the factory produces an independent chart instance with its own configuration state.

The pattern has three parts. First, the factory function declares local variables for every configurable property. Second, the chart function itself accepts a selection and renders using those variables. Third, getter-setter methods on the chart function allow callers to read or override each property. The getter-setter idiom uses arguments.length to distinguish reads from writes, returning the current value on read and returning the chart function on write (enabling chaining).

function barChart() {
  let width = 640, height = 400, color = "steelblue";
  let x = d => d.name, y = d => d.value;

  function chart(selection) {
    selection.each(function(data) {
      const xS = d3.scaleBand(data.map(x), [0, width]).padding(0.1);
      const yS = d3.scaleLinear([0, d3.max(data, y)], [height, 0]);
      const svg = d3.select(this)
        .selectAll("svg").data([data]).join("svg")
          .attr("viewBox", [0, 0, width, height]);
      svg.selectAll("rect").data(d => d, x).join("rect")
          .attr("x", d => xS(x(d)))
          .attr("y", d => yS(y(d)))
          .attr("height", d => height - yS(y(d)))
          .attr("width", xS.bandwidth())
          .attr("fill", color);
    });
  }

  chart.width  = _ => arguments.length ? (width = _, chart) : width;
  chart.height = _ => arguments.length ? (height = _, chart) : height;
  chart.color  = _ => arguments.length ? (color = _, chart) : color;
  chart.x = _ => arguments.length ? (x = _, chart) : x;
  chart.y = _ => arguments.length ? (y = _, chart) : y;
  return chart;
}

// Two charts, different configs, one function
d3.select("#sales").datum(salesData).call(barChart().color("#2563eb"));
d3.select("#costs").datum(costData).call(barChart().width(400).color("#dc2626"));

The .datum(data).call(chart) line is the entry point. .datum() binds data to the selected element. .call(chart) invokes the chart function with that selection. Inside, selection.each() iterates each element, pulling its bound datum via this. This is how one factory function can power a small-multiples grid: bind different data to different containers and call the same chart on all of them.

The figure below shows this in action. One barChart() factory produces two charts with different color palettes and dimensions, both rendering from the same code path.

barChart().color("#2563eb").width(320)
barChart().color("#dc2626").width(320).height(120)

Figure 2. Two charts produced by the same reusable closure, configured differently. The left chart uses the default height and a blue palette. The right chart overrides height and uses red. The underlying rendering logic is identical.

This pattern earns its weight when the same chart type appears in multiple places. Dashboard widgets, small multiples, repeated panel layouts. For a single chart that appears once, the closure adds boilerplate without benefit. The getter-setter methods double the line count, and the closure hides state from the browser console. Use it when you need it, not as a default architecture.

Naming Conventions

D3 code uses short variable names more aggressively than most JavaScript. This is intentional, not lazy. When x, y, and color appear in every .attr() call on the page, longer names like xPositionScale add noise without clarity. The convention is to name scales by what they map to (position, color, size) rather than what they are (a scale object).

ContextConventionExample
JavaScript variablescamelCaseinnerWidth, yScale, colorMap
CSS classeskebab-case.chart-axis, .bar-label, .tick-line
DOM hooksdata- attributesdata-series="revenue", data-index="3"
Datum, indexd, id => d.value, (d, i) => i * step
Positional scalesx, y.attr("cx", d => x(d.date))
Color scalecolor.attr("fill", d => color(d.group))
Root SVG, inner groupsvg, gconst g = svg.append("g")
Transitiontconst t = svg.transition().duration(750)
Shape generatorsnamed by outputline, area, arc, path

The camelCase / kebab-case split follows web platform conventions: JavaScript identifiers are camelCase, CSS classes are kebab-case. This is not a D3 rule. It is a web rule that D3 code inherits.

Use data- attributes when you need to find elements from outside D3's selection model. If a tooltip system or an external framework needs to locate the bar for "Q4", data-quarter="Q4" is a stable hook that survives re-renders. CSS classes work for styling; data- attributes work for identity.

Break the short-name convention when ambiguity appears. If a chart has two y-axes, y is ambiguous. Use yLeft and yRight, or yRevenue and yMargin. The rule is clarity, not brevity.

Common Anti-Patterns

Some patterns look reasonable but break D3's model in ways that show up later, usually when the code needs to handle updates, transitions, or composition.

  • Nested .each() for attribute setting. When every element needs the same set of attributes, chain .attr() calls directly on the joined selection. .each(function(d) { d3.select(this).attr("x", ...).attr("y", ...); }) produces identical output but breaks the declarative chain into imperative statements. Reserve .each() for cases where you need an expensive intermediate value shared across multiple attributes, or genuine side effects like appending child elements conditionally.
  • Manual DOM manipulation instead of data joins. data.forEach(d => svg.append("rect")...) bypasses the enter/update/exit cycle entirely. The code works for initial render, then breaks silently on the second call because the old elements are still in the DOM. No key identity, no exit removal, no transition hooks. If you write a forEach that calls .append(), you almost certainly want .selectAll().data().join() instead. The one legitimate exception is Canvas rendering, where there is no DOM to join against.
  • Inline styles instead of CSS classes. .style("font-size", "12px").style("fill", "#666") on every text element makes the visual language hard to change. Move repeated styling to a CSS class (.axis-label { font-size: 12px; fill: #666; }) and apply it with .attr("class", "axis-label"). Inline styles win for data-driven properties like .style("fill", d => color(d.group)) where the value depends on the datum. For everything static, prefer classes.
  • Storing selections you never reuse. const xAxis = g.append("g").call(d3.axisBottom(x)); is fine if you later call xAxis.transition().call(...). But naming a selection you never reference again just adds noise. Let the chain flow. Only break the chain to name something when you genuinely need the reference later.
  • Using .enter().append().merge() when .join() works. The old three-step pattern is four lines of code where .join() is one. It was necessary in D3 v4/v5 before .join() existed. In v7, use .join() for the simple case and pass enter/update/exit callbacks to .join() for animated transitions. The only reason to keep the old pattern is backward compatibility with pre-v6 codebases.

When This Skill Hurts

Margin convention for a 3-line sparkline. The margin convention exists to centralize spacing in charts complex enough to have axes, titles, and labels. A sparkline is 30 pixels tall with no axes. Declaring const margin = {top: 2, right: 0, bottom: 2, left: 0}, computing innerWidth and innerHeight, and appending a translated <g> is six lines of ceremony for a chart that needs none of it. The idiom was designed for a specific problem. When that problem is absent, the idiom is pure overhead. Enforcing it on every chart, regardless of complexity, teaches the convention but not the judgment behind it.

LLMs generating reusable chart closures for one-off visualizations. When an LLM is told to write "idiomatic D3," it tends to reach for the most structured pattern available: the Bostock closure with getter-setters. For a single chart that will never be instantiated twice, this adds 15-20 lines of boilerplate (getter-setters, selection.each() wrapper, the factory call) that obscure the actual rendering logic. The chart is harder to read, harder to debug from the console (closure variables are not inspectable), and no easier to maintain. A flat script with clearly named variables is perfectly idiomatic for single-use work. The reusable pattern is for reuse.

Chaining indentation rules that conflict with code formatters. D3's 2-space-new / 4-space-same indentation convention predates widespread adoption of Prettier and similar opinionated formatters. Prettier does not understand D3 selection semantics. It will reformat a carefully indented chain into its own rules, collapsing the visual distinction between new-selection and same-selection methods. Fighting the formatter with // prettier-ignore comments on every chain is noisy. In a project that uses Prettier, consistent Prettier formatting is better than inconsistent D3 indentation. The underlying principle (make selection boundaries visible) can still be communicated through blank lines and comments, even if the indent levels are flattened.

Over-abstraction via selection.call() when inline code is clearer. The .call(fn) pattern is powerful for repeated configuration, but extracting a function that is called exactly once moves code further from where it runs and names something that does not need a name. If an axis is styled in one place and that styling will not be reused, the inline .call(g => g.select(".domain").remove()) is already readable. Wrapping it in a named function (function styleAxis(g) { ... }) adds an indirection that the reader must follow without any payoff. Extract when the code repeats. Leave it inline when it does not.
