Pre-aggregated Summaries

When 800 dots slow your browser to a crawl, summarize first and render second.

If you've been following this series, you've seen earthquake data plotted as individual dots on a map. Each dot is one event at one location. That works for a few hundred points, but it falls apart when the dataset grows. Dots overlap, the browser labors to render them, and the file carrying all that coordinate data gets heavy.

The alternative: pre-aggregate the data before it reaches the browser. Instead of shipping 800 individual earthquake records and letting D3 figure out what to draw, you summarize them into 60 grid cells. Each cell carries a count, an average magnitude, and an average depth. The browser draws 60 shapes instead of 800. The payload shrinks from 25KB to 3KB. The map loads faster, and often it communicates the spatial pattern more clearly too.

This is a data strategy decision, not a visualization trick. You're choosing what to compute ahead of time and what to compute at render time.

Why hexagons? In a hex grid, every cell's neighbors are equidistant from its center. In a square grid, corner neighbors are farther away. This makes spatial aggregation more uniform. Same reason beehives use hexagons.

Three Kinds of Aggregation

Pre-aggregation isn't one technique. It's a family of strategies. The right one depends on which dimension of the data you want to compress.

Temporal aggregation

Instead of one record per earthquake with a precise timestamp, collapse events into time buckets. Daily counts become weekly or monthly counts. You lose the ability to see that an earthquake hit at 3:47 AM, but you can see seasonal patterns and long-term trends without drowning in noise.

For Japan's earthquake data, this is useful. Japan experiences thousands of small tremors per year. Plotting each one on a timeline creates an unreadable mess. Grouping by month reveals the actual patterns: are there more earthquakes in certain seasons? Is the rate increasing?

Spatial aggregation

Instead of plotting each earthquake at its exact latitude and longitude, assign events to grid cells and report one summary per cell. The hexbin approach above is one option. You could also use rectangular grids, aggregate by prefecture, or group by distance from a fault line.

Spatial aggregation helps when dots overlap so heavily that individual positions stop meaning anything. If 40 earthquakes all occurred within 50 km off the coast of Tohoku, plotting 40 overlapping circles tells you nothing more than one larger circle labeled "40 events, average M4.2."

Magnitude binning

Group earthquakes by magnitude ranges: M2-3, M3-4, M4-5, M5+. This turns a continuous variable into categories, useful for histograms and frequency distributions. The Gutenberg-Richter law says earthquake count drops tenfold for each unit increase in magnitude. Binning makes that pattern visible at a glance.

When to Pre-aggregate

Pre-aggregation is not always the right call. Here's a rough guide.

Pre-aggregate when:

  • Your dataset has more than ~1,000 points and the browser is struggling
  • Individual data points overlap so much the visualization is unreadable
  • You want an overview-first view (how many? where? when?) rather than detail-first
  • File size matters: mobile users, slow connections, embedded dashboards
  • The audience cares about patterns and distributions, not individual events

Keep raw data when:

  • Individual events matter. The 2011 Tohoku earthquake isn't interchangeable with any other M9
  • You need tooltips with specific details (time, depth, location name)
  • Aggregation would hide important patterns, like a cluster of small quakes preceding a large one
  • Your dataset is small enough that raw rendering is fast anyway
  • The audience needs to drill down to individual records
Size difference in practice: Raw data for 10,000 earthquakes runs about 300 KB as JSON. The same data aggregated into 100 spatial cells is about 3 KB. That's a 100x reduction. For a mobile user on a train in rural Hokkaido, that difference determines whether the page loads or not.

Practical Patterns in D3

D3 has built-in tools for aggregation. These patterns cover most cases.

d3.bin: magnitude histograms

When you want to group a continuous variable into ranges:

// Bin earthquakes by magnitude const bins = d3.bin() .value(d => d.mag) .domain([2, 8]) .thresholds([2, 3, 4, 5, 6, 7, 8]) (quakes); // Each bin has .x0, .x1, and .length // bins[0] → { x0: 2, x1: 3, length: 487 }

d3.rollup: monthly counts

When you want to group by a key and compute a summary:

// Count earthquakes per month const monthly = d3.rollup( quakes, v => v.length, // reducer: count d => d.date.slice(0, 7) // key: "2023-01" ); // Returns a Map: "2023-01" → 42, "2023-02" → 38, ... // For multiple summaries per group: const summary = d3.rollup( quakes, v => ({ count: v.length, avgMag: d3.mean(v, d => d.mag), maxMag: d3.max(v, d => d.mag) }), d => d.date.slice(0, 7) );

Spatial gridding: hexbin cells

When you want to aggregate lat/lon points into spatial cells:

// Simple grid-cell aggregation const cellSize = 1.5; // degrees const cells = d3.rollup( quakes, v => ({ count: v.length, avgMag: d3.mean(v, d => d.mag), avgDepth: d3.mean(v, d => d.depth), lat: d3.mean(v, d => d.lat), lon: d3.mean(v, d => d.lon) }), d => `${Math.floor(d.lat / cellSize)},${Math.floor(d.lon / cellSize)}` ); // Each cell: { count: 23, avgMag: 4.1, ... }

The raw dataset powers the "Raw Points" view and the monthly timeline. The aggregated dataset powers the "Aggregated Grid" view and the quarterly timeline. Both are in the HTML, so you can compare exactly how much space each takes.

Building the aggregated version from raw data took about six lines of JavaScript. Shipping it pre-computed instead of computing it in the browser saves a small amount of CPU time per page load. Trivial for 812 records, meaningful at 10,000 or 100,000.

A practical habit: When you ask an AI tool to build a visualization, specify raw or aggregated upfront. Say "aggregate by month" or "show individual events." That one sentence can be the difference between a fast, readable chart and a sluggish, overplotted mess.

The Bottom Line

Pre-aggregation is about where computation happens and what detail to preserve. For overviews, dashboards, and large datasets, it's almost always the right move. For detail views, event timelines, and small datasets, keep the raw data. When you aren't sure, try both, as this page does, and see which one actually communicates the pattern you care about.
