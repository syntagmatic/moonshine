Ask the Server, Right Now

How live API fetching pulls earthquake data from USGS at the moment you open this page

Every visualization needs data. So far in this series, we have looked at embedding data directly in a page (inline JSON) and loading it from separate files. Both approaches share a trait: the data is fixed at the time you publish. If an earthquake hits Osaka tonight, your visualization will not show it until you manually update the file.

There is a third strategy. Instead of shipping data with the page, you ask a server for it at runtime -- the moment a reader opens the page, your code reaches out over the network, requests fresh data, and renders whatever comes back. This is live API fetching.

This page demonstrates the technique on itself. When you loaded it a moment ago, it attempted to fetch earthquake data from the USGS. Scroll down to see whether it succeeded.

The USGS Earthquake API

The United States Geological Survey runs a free, public API for earthquake data. No registration, no API key, no authentication. You construct a URL with query parameters describing what you want, and the server returns GeoJSON.

The base URL is:

https://earthquake.usgs.gov/fdsnws/event/1/query

You append parameters to narrow the results. Here are the ones that matter for Japan:

  • format=geojson -- return GeoJSON (also supports csv, xml, text)
  • starttime / endtime -- date range, in YYYY-MM-DD format
  • minmagnitude -- ignore anything smaller than this
  • minlatitude, maxlatitude, minlongitude, maxlongitude -- bounding box

For M4.5+ earthquakes around Japan, that bounding box is roughly 24-46N latitude, 122-150E longitude. Combine everything and you get a URL like this:

https://earthquake.usgs.gov/fdsnws/event/1/query
  ?format=geojson
  &starttime=2026-03-12
  &endtime=2026-04-11
  &minmagnitude=4.5
  &minlatitude=24&maxlatitude=46
  &minlongitude=122&maxlongitude=150

Paste that into a browser and you get raw JSON back. Every programming language has tools to make that same request programmatically.

What comes back: GeoJSON

The API returns a GeoJSON FeatureCollection. If you have worked with JSON before, GeoJSON is just JSON with a geographic convention. The structure looks like this:

{
  "type": "FeatureCollection",
  "metadata": { "count": 52, ... },
  "features": [
    {
      "type": "Feature",
      "properties": {
        "mag": 6.5,
        "place": "122 km E of Yamada, Japan",
        "time": 1774534730947
      },
      "geometry": {
        "type": "Point",
        "coordinates": [143.37, 39.45, 9.5]
      }
    },
    ...
  ]
}

Each feature has properties (magnitude, place name, timestamp) and geometry (longitude, latitude, depth in km). The coordinates array is [lon, lat, depth] -- note that longitude comes first, which trips up everyone at least once.

The fetch pattern with fallback

Here is the actual pattern this page uses. The key idea: try the live API first, and if anything goes wrong, fall back to cached data embedded in the page itself.

const USGS_URL = `https://earthquake.usgs.gov/fdsnws/event/1/query`
    + `?format=geojson&starttime=...&minmagnitude=4.5`
    + `&minlatitude=24&maxlatitude=46`
    + `&minlongitude=122&maxlongitude=150`;

async function loadData() {
  try {
    // AbortController lets us set a timeout
    const controller = new AbortController();
    const timeout = setTimeout(() => controller.abort(), 8000);

    const resp = await fetch(USGS_URL, {
      signal: controller.signal
    });
    clearTimeout(timeout);

    if (!resp.ok) throw new Error(`HTTP ${resp.status}`);

    const json = await resp.json();
    return { data: json.features, source: "live" };

  } catch (err) {
    console.warn("API unavailable, using fallback:", err);
    return { data: FALLBACK_DATA, source: "cached" };
  }
}

A few things to notice:

  • AbortController timeout. Without this, a fetch to a slow or hung server could leave your page stuck indefinitely. Eight seconds is generous; five is reasonable for most uses.
  • Check resp.ok. A 404 or 500 response is still a "successful" HTTP request from fetch's perspective. You have to check the status yourself.
  • The catch block does not crash. It returns fallback data with a flag, so the rest of your code does not need to know where the data came from.
Why embed fallback data? Because an explainer that shows a blank map half the time is a bad explainer. The fallback is a snapshot from the last time you built the page -- slightly stale, but far better than nothing. Think of it as insurance.

Rate limits: be a good neighbor

The USGS API is free and public. It has no authentication. That is unusual and generous. The implicit contract is that you do not abuse it.

In practice, this means:

  • Do not poll rapidly. Fetching once on page load is fine. Fetching every 5 seconds is not.
  • Do not request everything. Use the query parameters to narrow your results. Asking for all earthquakes worldwide with no magnitude filter will return enormous payloads and stress the server.
  • Cache when you can. If your page does not need real-time data, consider fetching once and caching the result in memory or localStorage rather than hitting the API on every page navigation.
  • Respect HTTP headers. If the API sends Retry-After or returns 429 (Too Many Requests), back off.

For a visualization like this one -- fetching 30 days of M4.5+ earthquakes in a regional bounding box -- the response is small (a few dozen KB) and the load is negligible. You are fine.

Advantages of live fetching

  • Always current. If an earthquake strikes while you are reading this, refreshing the page will show it (assuming the USGS has processed it, which usually takes 5-20 minutes for significant events).
  • Smallest possible HTML file. The data is not embedded in the page, so the file itself is lightweight. Only the code and the fallback data ship with the page.
  • Single source of truth. Everyone who loads the page sees the same live data. No version drift between your local file and the server.

Disadvantages

  • Requires internet. Open this page on an airplane without Wi-Fi and the live fetch will fail. (The fallback saves you here, but the data will be stale.)
  • Slower initial load. There is a network round-trip before anything renders. On slow connections, the user stares at a loading state. Inline data renders instantly.
  • API can change or go down. The USGS could change their response format, deprecate the endpoint, or have an outage. Your page would break in ways that are invisible until someone loads it.
  • CORS restrictions. Not all APIs allow requests from a browser. The USGS does, but many APIs require a backend proxy. If you see Access-Control-Allow-Origin errors in the console, that is CORS blocking you.

When to use live fetching

Live fetching makes sense when freshness matters more than reliability. Dashboards, monitoring displays, "what is happening right now" views -- these all benefit from live data. If you are building a static analysis of historical trends, bake the data into the page instead. You will get faster loads, offline support, and no API dependency.

The best approach for an explainer or article is often what this page does: try live, fall back to cached. You get freshness when the network cooperates and reliability when it does not.

The spectrum of data strategies. Inline JSON is fully self-contained but stale. External files separate data from code but still require a build step to update. Live API fetching is always current but depends on the network. Most real projects use a combination -- live data with cached fallbacks, or periodic builds that fetch and embed fresh data.
