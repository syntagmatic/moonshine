Inline JSON

Put the data right in the HTML. No server, no fetch, no drama.

When you build a visualization, the data has to come from somewhere. The simplest approach is to put it directly inside the HTML file. No server, no separate files, no network requests. One file you can email, drop in a folder, or open from your desktop.

What "inline data" means

In most web visualizations, data lives in a separate file that the browser downloads on page load. Inline data skips that step. You write the data directly into a <script> tag as a JavaScript variable:

const quakes = [
  { lat: 38.297, lon: 142.373, mag: 9.1,
    place: "2011 Great Tohoku Earthquake" },
  { lat: 41.815, lon: 143.91, mag: 8.16,
    place: "2003 Tokachi-Oki Earthquake" },
  // ... 98 more records
];

Assign a JavaScript array to a variable. The browser never makes a network request for data.

When the page loads, the data is already there. D3 can start drawing immediately. No d3.json() call, no callback, no promise. Your code just references the variable.

Why this works well

This page itself uses inline data. Click "View Source Data" above to see the actual array embedded in this file. What you see is not a recreation. It is the real data the map reads from.

Inline data sidesteps a few problems that trip up beginners all the time:

No CORS errors. If you've ever tried to open an HTML file from your desktop that loads a local CSV, your browser probably blocked it. Browsers enforce strict rules about which files can read other files. Inline data avoids this because there is nothing to load.

Works offline. Once you have the HTML file (and assuming the D3 library is cached), the visualization works without an internet connection. Good for presentations, fieldwork, planes.

One file to share. Email it, put it on a USB drive, host it anywhere. No folder structure to maintain, no missing files.

No server required. You do not need Python's http.server or Node. Double-click the file and it opens in your browser.

The tradeoffs

Nothing is free. Inline data has real costs:

File size grows. This page is about 40 KB total. The earthquake data accounts for roughly a third of that. With a larger dataset, say all M2.5+ earthquakes for the last 25 years, the file would grow to several megabytes. Slow to open, painful to edit.

Approximate breakdown of this HTML file by content type.

Data is not reusable. If you build three visualizations from the same earthquake data, you end up with three copies. Update one and the others are stale. An external file lets multiple pages share a single source.

Harder to update. Changing inline data means editing the HTML file itself. With an external CSV, you can swap in a new file without touching the code. For data that changes often (live sensor feeds, daily reports) inline data is the wrong choice.

When to use it

Inline data is ideal for:

  • Curated datasets under about 1 MB
  • Self-contained explainers and tutorials
  • Prototypes and sketches you want to share quickly
  • Situations where CORS or server setup is a barrier
  • Archival copies that need to work years from now

For anything larger or more dynamic, you will want external files or API calls. Those are covered in the next explainers in this series.

Compression tricks

When your data is inline, every byte counts. A good trick: use compact arrays instead of objects. Compare these two encodings of the same earthquake:

// Verbose: 89 characters per record
{ "latitude": 38.297, "longitude": 142.373, "magnitude": 9.1 }

// Compact: 21 characters per record
[38.297, 142.373, 9.1]

Compact arrays drop the keys entirely. You reconstruct them in code: const [lat, lon, mag] = record;

With 100 records, the difference is around 5 KB. With 10,000 records, it saves 50 KB or more. The tradeoff is readability: someone looking at your source has to know that index 0 means latitude. A brief comment at the top of the array solves that.

This page uses the readable object format because clarity matters more here than compression. But for production work with larger datasets, compact arrays are worth considering.

External File

↓

↓

↓

↓
Visualization renders
Two requests CORS possible

The inline approach collapses the loading pipeline into a single step. For small, curated datasets, it's hard to beat.
