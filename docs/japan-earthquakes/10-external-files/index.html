External CSV/JSON Files

Loading data from companion files — the most common pattern in D3 projects

Every block and visualization you've seen so far has embedded its data directly in the HTML. That works for small datasets, but earthquake catalogs, census tables, and climate records can run to thousands of rows. At that point, you want the data in its own file.

This page demonstrates the pattern by doing it live: the earthquake map below loads 200 earthquakes from a JSON file sitting in a data/ folder right next to this HTML. Watch the loading spinner appear while the file is being fetched.

The file organization convention

D3 projects almost universally follow this layout. The HTML and the data live side by side, with data in its own folder:

my-project/
├── index.html ← your visualization
├── data/
│ ├── earthquakes.json ← 200 quakes, 33 KB
│ └── stations.csv ← could add more files
└── style.css ← optional separate styles

This convention matters because the path you write in your code — d3.json("data/earthquakes.json") — is relative to the HTML file. If you move the HTML somewhere else without bringing the data/ folder, the visualization breaks.

Loading the data

Here's the core pattern. D3 provides convenience functions that fetch a file and parse it in one step:

// Load JSON — returns a Promise d3.json("data/earthquakes.json") .then(data => { // data is now a JavaScript array of objects console.log(`Loaded ${data.length} earthquakes`); drawMap(data); }) .catch(error => { console.error("Failed to load:", error); showError(error); });

For CSV files the call is d3.csv("data/earthquakes.csv"). D3 parses each row into an object using the header row as keys. One important difference: CSV values are always strings, so you need to convert numbers yourself with a row accessor or +d.mag after loading. JSON preserves types — a number stays a number.

Loading states: never show a blank page

When data comes from an external file, there's always a moment — maybe 50 milliseconds, maybe 2 seconds on a slow connection — where your page has no data. If you do nothing, the user sees a blank white rectangle and wonders if something broke.

The fix is straightforward: show something while waiting. A spinner, a skeleton outline of the chart, or a simple "Loading..." message. The map above uses a spinner. Here's the pattern:

// 1. Show loading state (already visible in HTML) // 2. Fetch the data d3.json("data/earthquakes.json") .then(data => { // 3. Hide loading, show viz hideLoading(); drawMap(data); }) .catch(error => { // 4. Hide loading, show error hideLoading(); showError(error); });

The loading state lives in the HTML from the start — no JavaScript needed to create it. The script just hides it when data arrives (or when it fails).

Error handling and CORS

External data loading can fail. The file might be missing, the path might be wrong, or your browser might block the request entirely. That last one — the CORS issue — trips up everyone learning D3.

The local file problem. If you open an HTML file directly (the URL bar shows file:///Users/you/project/index.html), browsers block fetch() requests to other local files for security reasons. This is called the same-origin policy. Your visualization works fine on a web server but fails silently on your desktop.

The fix: run a local web server. Python ships with one built in:

# In your project folder, run: python -m http.server 8000 # Then open: http://localhost:8000

That one command solves the problem. On hosted platforms — GitHub Pages, Observable, any web server — CORS is not an issue because the files are served over HTTP already.

This page handles the error gracefully: if the external file can't be loaded, it falls back to a small inline dataset and shows a notice. That way the reader still sees something useful instead of a broken page.

Advantages and disadvantages

AdvantageDisadvantage
Separation Data and code stay independent — update one without touching the other Two things to keep in sync; broken paths break everything
Size HTML stays small and readable; data can be megabytes Extra HTTP request adds latency
Reusability Same data file can feed multiple visualizations Must bundle the data folder when sharing
Development Easier to replace data (swap in a new CSV export) Needs a local server — can't just double-click the HTML

CSV vs JSON: which format?

Both work. The choice usually comes down to what your data source gives you and how you want to use it.

CSVJSON
File size Smaller — no repeated keys, no brackets Larger — each object repeats every key name
Types Everything is a string; you parse numbers yourself Numbers, booleans, nulls preserved as-is
Structure Flat table only — rows and columns Can nest objects, arrays, hierarchies
Editing Opens in Excel/Google Sheets Needs a text editor or code
D3 loader d3.csv() — auto-parses headers d3.json() — ready to use

For this earthquake dataset, JSON is the better choice. Magnitude, depth, and coordinates are numbers — in CSV they'd arrive as strings like "4.7" and you'd need to convert each one. In JSON, 4.7 is already a number.

When to use external files

Use external files when your dataset has more than ~30 rows or when the same data feeds multiple visualizations. Use inline data (the previous pattern) when the dataset is tiny, or when you need a single self-contained file with no dependencies.

Most real D3 projects use external files. The USGS earthquake catalog, JMA station data, population tables — they all live in data/ folders, loaded with d3.csv() or d3.json(), with a spinner keeping the user company while the bytes arrive.
