Future Directions

Embeddable Microexplanations

Tiny self-contained explanations that embed in GitHub READMEs, Notion pages, blog posts, and Slack messages. A moonshine explanation reduced to its smallest useful unit.

Explanations live on web pages. Conversations happen elsewhere.

A moonshine explanation is a self-contained HTML file. It lives at a URL. You open it in a browser tab. That is fine for reading an article, but most of the moments where someone needs to understand something do not happen in a browser tab pointed at your explanation.

They happen in a GitHub pull request, where a reviewer is squinting at a performance regression and could use a sparkline that shows the trend. They happen in a Slack thread, where someone asks "what does this parameter do?" and the answer is a slider that lets you see for yourself. They happen in a Notion doc, where a project lead is summarizing last quarter's results and wants an interactive chart inline with the prose.

The conversations are happening on platforms that did not build for interactive explanations. The explanations are sitting on web pages that nobody walks over to during a conversation. The gap between where understanding is needed and where understanding is produced is a distribution problem, not a quality problem.

The core idea: A microexplanation is a moonshine explanation reduced to its smallest useful unit. One figure. One interaction. Under 50 KB. Under 200 lines of HTML. Small enough to embed anywhere a conversation is happening.

The embed format landscape

There is no universal way to embed interactive content across platforms. Each format trades off between reach, interactivity, and self-containment.

Format Interactive Self-contained Works in Catch
Screenshot + link No Yes Everywhere Not interactive
Animated GIF / video No Yes Everywhere No user control, large files
iframe (hosted) Yes No Notion, blogs, some wikis Needs a URL, blocked by CSP
data: URI iframe Yes Yes HTML pages (not GitHub, not Slack) Huge base64 string, no same-origin
Web Component Yes Partial Any page with JS Requires script tag, not universally supported
SVG with embedded JS In theory Yes Browsers (not GitHub, strips scripts) Sanitizers remove interactivity
OEmbed Yes No Slack, Notion, WordPress Needs server, allowlist registration

No single format wins everywhere. The practical path is progressive enhancement: produce multiple formats from the same source, and let the platform choose which it can render.

The progressive enhancement ladder

A microexplanation should degrade gracefully. The same content at four levels of fidelity:

Static image PNG/SVG, works everywhere, zero interactivity
Animated GIF Shows the interaction, viewer cannot control it
Hosted iframe Full interactivity, requires hosting + embed support
Native embed Web Component or data URI, self-contained + interactive
The enhancement ladder. Each step adds interactivity but reduces platform reach. A microexplanation build tool should produce all four from a single source file.

For a GitHub README, you land on rung one or two: a static screenshot or an animated GIF, with a link to the live interactive version. GitHub sanitizes SVG and strips embedded scripts, so the "SVG with JavaScript" trick does not work there. The reliable pattern is an image tag pointing to a screenshot plus a link to the hosted page.

For Notion, you get rung three: paste a URL and Notion renders an iframe embed. Same for blog platforms with oEmbed support. The microexplanation needs to be hosted somewhere, but once it is, embedding is just a URL.

For HTML pages you control (docs sites, wikis, internal tools), you can reach rung four: a data: URI iframe that requires no hosting at all. The entire microexplanation, base64-encoded, becomes a URI. This is self-contained in the strictest sense. The tradeoff is that a 15 KB HTML file becomes a 20 KB base64 string, and the resulting markup is unreadable. But it works without a server.

The 50 KB constraint and what it forces you to cut

A full moonshine explanation might be 80 KB of HTML, load D3 from a CDN (290 KB), use web fonts, and include several interactive figures. A microexplanation has a 50 KB budget for everything.

That budget forces hard decisions. D3 v7 minified is 290 KB. You cannot include it inline. You either load it from a CDN (adding a network dependency) or you do not use D3 at all, writing raw SVG manipulation with vanilla JS. For a sparkline, vanilla JS is 30 lines. For a parameter explorer, it is still manageable. D3's selection model and scale functions are convenient but not essential at this scale.

Web fonts are out. System fonts only. That means giving up Source Serif and Source Sans, accepting whatever the user's system provides. The visual identity of a moonshine explanation partly depends on its typography; a microexplanation trades that identity for size.

Multiple figures are out. One figure, one interaction. If you need two figures, you need two microexplanations. This is a feature, not a loss: it forces each micro to make exactly one point, clearly.

The size constraint is editorial. Cutting kilobytes forces you to cut concepts. A 50 KB budget does not just limit file size. It limits scope, which is the hardest thing for an explanation author to limit voluntarily.

The data-URI trick



If you control the page where the microexplanation appears, you can embed it without hosting. The data: URI scheme lets you encode an entire HTML document as a URI and set it as the src of an iframe.



<!-- A complete microexplanation, embedded with no server -->
<iframe
  src="data:text/html;base64,PCFET0NUWVBFIGh0bWw..."
  width="100%"
  height="200"
  style="border:none"
></iframe>



The trick works on any HTML page. It does not work in GitHub (which strips iframes), Slack (which does not render HTML), or email (which strips almost everything). It works in internal wikis, documentation sites, and blogs where you control the markup. The base64 encoding inflates the size by about 33%, so a 15 KB microexplanation becomes a 20 KB URI string. Unwieldy, but functional.



The deeper problem with data URIs is opacity. A base64 string is not human-readable. You cannot glance at the embed code and understand what it does. A hosted iframe URL at least tells you the domain and path. A data URI tells you nothing. For security-conscious contexts, that opacity is disqualifying.



GitHub README integration



GitHub is where many technical conversations happen, and GitHub READMEs are the most-read documentation on the planet. Interactive embeds in READMEs would be transformative. But GitHub's sanitizer is aggressive and correct to be so: it strips scripts, iframes, forms, and event handlers from rendered Markdown.



The practical path for GitHub is the progressive enhancement ladder's first rung: a static screenshot (or animated GIF) with a link to the live interactive version.



<!-- In a GitHub README -->
## Response times

[![](./docs/sparkline-preview.png)](https://example.com/sparkline.html)

<!-- Screenshot links to the live microexplanation -->



SVG files in repos are rendered inline by GitHub, and SVG can contain quite rich static graphics. An SVG sparkline with text annotations, embedded directly in the repo, renders beautifully in the README without any hosting. You lose interactivity but you gain zero-dependency, version-controlled visuals that live next to the code they describe.



The moonshine connection



If moonshine is about producing interactive explanations with LLMs, then microexplanations extend that mission to the places where people actually need explanations. A /shine workflow could offer a micro mode:



# Full explanation (article-length, multiple figures)
/shine "explain the connection pool optimization"

# Microexplanation (one figure, embeddable, <50 KB)
/shine micro "sparkline of p95 latency this week"

# Micro with specific format output
/shine micro "parameter explorer for smoothing window" --format github



The --format github flag would produce an SVG screenshot plus a self-contained HTML file. The --format notion flag would produce a hosted URL ready for iframe embedding. The --format datauri flag would produce an iframe tag with the base64-encoded page.



The LLM's job changes in micro mode. Instead of structuring a narrative across sections and figures, it produces a single figure with a single interaction that makes a single point. The constraint is the feature: it forces the LLM to be precise about what the viewer should understand, because there is room for exactly one idea.




  
Microexplanations decompose. A full moonshine explanation about connection pool optimization might contain five figures. Each one could be extracted as a microexplanation: the sparkline in the pull request, the parameter explorer in the docs, the before/after toggle in the incident report. The article is the whole; the micros are its atoms, distributed to where the conversations are.





What can go wrong




  
Risks and open problems

  
    
    
  • Interactive iframes and Content Security Policy. Many corporate environments deploy strict CSP headers that block inline scripts and iframe embeds. A microexplanation that works on a personal blog may silently fail on an internal wiki. The progressive enhancement ladder mitigates this (fall back to static image), but the failure mode is invisible: the iframe simply does not render, with no error message for the viewer.
    • 
    
  • The hosting requirement is inescapable. There is no truly hostless interactive embed that works across platforms. Data URIs come close but are blocked in many contexts. Web Components require a script tag. OEmbed requires a registered provider. Every interactive path eventually requires putting a file on a server. The dream of "paste this into any document and it works" remains unrealized for interactive content.
    • 
    
  • The temptation to cram too much. A microexplanation with two sliders, a dropdown, and three linked charts is not a microexplanation. It is a small explanation that lost its nerve halfway through being cut down. The 200-line, 50 KB constraint is not a suggestion. Exceeding it defeats the purpose: fast loading, easy embedding, one idea clearly communicated.
    • 
    
  • LLMs default to full-size. When asked to "make a chart explaining X," an LLM will typically produce a full explanation with narrative prose, multiple sections, and web fonts. Generating a good microexplanation requires a different prompt strategy: emphasize the size budget, the single-figure constraint, and the embed context. Without that guidance, the output will be too large and too complex to embed.
    • 
  





What comes next



The gap between explanations and conversations is a distribution problem. Microexplanations are the smallest bridge across that gap: one figure, one interaction, embedded where the conversation is already happening. They will not replace full explanations any more than tweets replaced essays. But they can carry a single insight to the exact moment someone needs it, in the exact place they are looking.



Building the tooling means building a compiler that takes a moonshine explanation and outputs multiple formats: self-contained HTML, SVG screenshot, animated GIF, data URI, OEmbed endpoint. The source stays the same. The output adapts to the platform. That is the work ahead.
