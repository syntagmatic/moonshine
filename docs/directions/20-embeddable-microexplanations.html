Future Directions

Embeddable Microexplanations

Tiny self-contained explanations that embed in GitHub READMEs, Notion pages, blog posts, and Slack messages. A moonshine explanation reduced to its smallest useful unit.

Explanations live on web pages. Conversations happen elsewhere.

A moonshine explanation is a self-contained HTML file. It lives at a URL. You open it in a browser tab. That is fine for reading an article, but most of the moments where someone needs to understand something do not happen in a browser tab pointed at your explanation.

They happen in a GitHub pull request, where a reviewer is squinting at a performance regression and could use a sparkline that shows the trend. They happen in a Slack thread, where someone asks "what does this parameter do?" and the answer is a slider that lets you see for yourself. They happen in a Notion doc, where a project lead is summarizing last quarter's results and wants an interactive chart inline with the prose.

The conversations are happening on platforms that did not build for interactive explanations. The explanations are sitting on web pages that nobody walks over to during a conversation. The gap between where understanding is needed and where understanding is produced is a distribution problem, not a quality problem.

The core idea: A microexplanation is a moonshine explanation reduced to its smallest useful unit. One figure. One interaction. Under 50 KB. Under 200 lines of HTML. Small enough to embed anywhere a conversation is happening.

The embed format landscape

There is no universal way to embed interactive content across platforms. Each format trades off between reach, interactivity, and self-containment.

Format Interactive Self-contained Works in Catch
Screenshot + link No Yes Everywhere Not interactive
Animated GIF / video No Yes Everywhere No user control, large files
iframe (hosted) Yes No Notion, blogs, some wikis Needs a URL, blocked by CSP
data: URI iframe Yes Yes HTML pages (not GitHub, not Slack) Huge base64 string, no same-origin
Web Component Yes Partial Any page with JS Requires script tag, not universally supported
SVG with embedded JS In theory Yes Browsers (not GitHub, strips scripts) Sanitizers remove interactivity
OEmbed Yes No Slack, Notion, WordPress Needs server, allowlist registration

No single format wins everywhere. The practical path is progressive enhancement: produce multiple formats from the same source, and let the platform choose which it can render.

The progressive enhancement ladder

A microexplanation should degrade gracefully. The same content at four levels of fidelity:

Static image PNG/SVG, works everywhere, zero interactivity
Animated GIF Shows the interaction, viewer cannot control it
Hosted iframe Full interactivity, requires hosting + embed support
Native embed Web Component or data URI, self-contained + interactive
The enhancement ladder. Each step adds interactivity but reduces platform reach. A microexplanation build tool should produce all four from a single source file.

For a GitHub README, you land on rung one or two: a static screenshot or an animated GIF, with a link to the live interactive version. GitHub sanitizes SVG and strips embedded scripts, so the "SVG with JavaScript" trick does not work there. The reliable pattern is an image tag pointing to a screenshot plus a link to the hosted page.

For Notion, you get rung three: paste a URL and Notion renders an iframe embed. Same for blog platforms with oEmbed support. The microexplanation needs to be hosted somewhere, but once it is, embedding is just a URL.

For HTML pages you control (docs sites, wikis, internal tools), you can reach rung four: a data: URI iframe that requires no hosting at all. The entire microexplanation, base64-encoded, becomes a URI. This is self-contained in the strictest sense. The tradeoff is that a 15 KB HTML file becomes a 20 KB base64 string, and the resulting markup is unreadable. But it works without a server.

The 50 KB constraint and what it forces you to cut

A full moonshine explanation might be 80 KB of HTML, load D3 from a CDN (290 KB), use web fonts, and include several interactive figures. A microexplanation has a 50 KB budget for everything.

That budget forces hard decisions. D3 v7 minified is 290 KB. You cannot include it inline. You either load it from a CDN (adding a network dependency) or you do not use D3 at all, writing raw SVG manipulation with vanilla JS. For a sparkline, vanilla JS is 30 lines. For a parameter explorer, it is still manageable. D3's selection model and scale functions are convenient but not essential at this scale.

Web fonts are out. System fonts only. That means giving up Source Serif and Source Sans, accepting whatever the user's system provides. The visual identity of a moonshine explanation partly depends on its typography; a microexplanation trades that identity for size.

Multiple figures are out. One figure, one interaction. If you need two figures, you need two microexplanations. This is a feature, not a loss: it forces each micro to make exactly one point, clearly.

The size constraint is editorial. Cutting kilobytes forces you to cut concepts. A 50 KB budget does not just limit file size. It limits scope, which is the hardest thing for an explanation author to limit voluntarily.

The data-URI trick, in depth



If you control the page where the microexplanation appears, you can embed it without hosting. The data: URI scheme lets you encode an entire HTML document as a URI and set it as the src of an iframe.



<!-- A complete microexplanation, embedded with no server -->
<iframe
  src="data:text/html;base64,PCFET0NUWVBFIGh0bWw..."
  width="100%"
  height="200"
  style="border:none"
></iframe>



The trick works on any HTML page. It does not work in GitHub (which strips iframes), Slack (which does not render HTML), or email (which strips almost everything). It works in internal wikis, documentation sites, and blogs where you control the markup.



How base64 encoding works for this



Base64 takes every three bytes of input and maps them to four ASCII characters drawn from a 64-character alphabet (A-Z, a-z, 0-9, +, /). The input is your raw HTML. The output is a long string of printable ASCII that can safely live inside a URL attribute. A 15 KB microexplanation produces roughly 20 KB of base64 text, because every 3 bytes become 4. That 33% inflation is inherent to the encoding and unavoidable.



In practice, you run the HTML through btoa() in JavaScript or base64 on the command line. The result gets pasted into the src attribute of an iframe. The browser decodes it on load and renders the HTML as if it came from a server. There is no network request. There is no server. The entire page is a string literal embedded in the host document's markup.



# On the command line
base64 -w0 sparkline.html | pbcopy

# Or in a build script
const html = fs.readFileSync('sparkline.html', 'utf-8');
const uri = 'data:text/html;base64,' + Buffer.from(html).toString('base64');



The 2 MB URL length wall



The data URI spec itself has no size limit. But browsers do. Chrome and Edge enforce a roughly 2 MB ceiling on the total length of any URL, including data URIs. Firefox is more generous, somewhere around 32 MB in recent versions. Safari sits in between, historically more restrictive than Firefox but less so than Chrome. Internet Explorer capped data URIs at 32 KB, but IE is gone now and does not matter for new work.



For microexplanations, the 2 MB Chrome limit is not usually the binding constraint. A 50 KB HTML file encodes to about 67 KB of base64, well within range. But if you tried to stuff a full moonshine explanation with inline SVGs and embedded data into a data URI, you could hit the wall. A 1.5 MB explanation (after base64 inflation, 2 MB) would work in Firefox and fail in Chrome, silently. The iframe would simply show nothing.



This is another argument for the 50 KB budget. It keeps you safely under the URL length limit on every browser, with room to spare.



The opacity problem: no postMessage, no coordination



The deeper problem with data URIs is not size. It is isolation. A data URI iframe has a null origin. It is not http://, not https://, not even file://. It is literally the string "null". This means the parent page cannot communicate with it using postMessage in any reliable cross-browser way. The iframe is a sealed box.



For a standalone sparkline, this is fine. The microexplanation does not need to talk to its host. But as soon as you want coordinated behavior (clicking a microexplanation to update a value in the parent page, or having the parent page push new data into the embed), the data URI approach breaks down. You cannot reach into it, and it cannot reach out.



Hosted iframes with a real origin do not have this problem. You can postMessage between parent and child freely, filter by origin, and build coordination protocols. Data URI iframes trade communication for self-containment. That tradeoff is acceptable only when each micro is truly independent, which is the microexplanation philosophy anyway: one figure, one idea, no coordination required.



There is also the human-readability problem. A base64 string is opaque. You cannot glance at the embed code and understand what it does. A hosted iframe URL at least tells you the domain and path. A data URI tells you nothing. Reviewing a pull request that adds a data URI iframe means either decoding the base64 by hand or trusting the author. For security-conscious contexts, that opacity is disqualifying.




  
The data URI tradeoff in one sentence: You get true self-containment (no server, no network, no CORS) at the cost of isolation (no postMessage, no origin, no human-readable source) and a hard ceiling on payload size.





GitHub README integration



GitHub is where many technical conversations happen, and GitHub READMEs are the most-read documentation on the planet. Interactive embeds in READMEs would be transformative. But GitHub's sanitizer is aggressive and correct to be so: it strips scripts, iframes, forms, and event handlers from rendered Markdown.



The practical path for GitHub is the progressive enhancement ladder's first rung: a static screenshot (or animated GIF) with a link to the live interactive version.



<!-- In a GitHub README -->
## Response times

[![](./docs/sparkline-preview.png)](https://example.com/sparkline.html)

<!-- Screenshot links to the live microexplanation -->



SVG files in repos are rendered inline by GitHub, and SVG can contain quite rich static graphics. An SVG sparkline with text annotations, embedded directly in the repo, renders beautifully in the README without any hosting. You lose interactivity but you gain zero-dependency, version-controlled visuals that live next to the code they describe.



OEmbed: auto-embedding for platforms that support it



OEmbed is a protocol from 2008 that solves a specific problem: when a user pastes a URL into a platform (Slack, Notion, WordPress, Medium), the platform needs to know how to render a rich preview of that URL. The protocol is simple. The platform sends a GET request to an OEmbed endpoint with the URL as a query parameter. The endpoint responds with JSON describing how to embed the content: an iframe URL, dimensions, a title, a thumbnail.



// OEmbed discovery: a link tag in the microexplanation's <head>
<link
  rel="alternate"
  type="application/json+oembed"
  href="https://moonshine.dev/oembed?url=https://moonshine.dev/m/sparkline-p95"
  title="P95 Latency Sparkline"
/>

// The OEmbed endpoint responds with:
{
  "type": "rich",
  "version": "1.0",
  "title": "P95 Latency Sparkline",
  "width": 400,
  "height": 200,
  "html": "<iframe src='https://moonshine.dev/m/sparkline-p95' ...>",
  "provider_name": "Moonshine",
  "thumbnail_url": "https://moonshine.dev/m/sparkline-p95/thumb.png"
}



The platform reads that JSON, sees type: "rich", and renders the provided html (typically an iframe) inline where the user pasted the URL. The user does nothing except paste a link. The platform handles the rest.



This is the most frictionless embed path for platforms that support it. Slack uses OEmbed (called "unfurling" in their API). Notion uses it. WordPress has supported it since version 2.9. Medium, Tumblr, and several others follow the same protocol or a close variant. When it works, pasting a URL is all it takes to get an interactive embed.



But OEmbed has two costs that cut against the microexplanation philosophy. First, it requires a server. Not just hosting for the HTML file, but a running endpoint that responds to OEmbed discovery requests. You need a /oembed route that takes a URL parameter and returns JSON. For a static site on GitHub Pages, that endpoint does not exist unless you add a serverless function or an external service.



Second, platforms maintain allowlists. Slack does not unfurl arbitrary OEmbed providers. You must register your domain with Slack and be approved. WordPress is more open (it supports arbitrary OEmbed discovery), but many WordPress installations lock it down. Notion is selective about which embed providers it auto-renders. Getting on these allowlists is a one-time bureaucratic cost, but it is real, and it means a new moonshine instance cannot auto-embed on day one.



The practical path: for a moonshine project hosted on a custom domain with a small API layer, OEmbed is worth implementing. The endpoint is minimal (read the URL, look up the microexplanation metadata, return a JSON object). Once it is running, embedding in Slack and Notion becomes a paste. For projects that are purely static HTML files, OEmbed is not available without a proxy service, and the progressive enhancement ladder's lower rungs (screenshot + link) are the fallback.



Responsive micro-explanations (or: pick a width and commit)



A full moonshine explanation can include media queries, fluid layouts, and breakpoints. It loads 290 KB of D3 from a CDN and can afford the overhead of a responsive design. A microexplanation at 50 KB cannot.



Media queries themselves are small. A few breakpoints add perhaps 500 bytes. The problem is not the query syntax. It is the markup and logic that diverge at each breakpoint. If a sparkline needs a different label layout on narrow screens, you are writing two layouts in one file. If a parameter explorer stacks the slider above the chart on mobile instead of beside it, that is a second flex arrangement with different padding. Each responsive branch adds HTML, CSS, and potentially JavaScript. At 50 KB total, you cannot afford many branches.



The practical answer for microexplanations is to avoid responsive design entirely and instead make the chart work at any width by construction. An SVG with a viewBox attribute scales naturally. If the sparkline's viewBox is 0 0 320 80, it renders at 320 pixels on a wide screen and shrinks proportionally in a narrow embed, with all elements scaling together. No breakpoints needed. No alternate layouts. The SVG simply gets smaller.



<!-- This scales to any container width, no media queries -->
<svg viewBox="0 0 320 80" style="width:100%;height:auto">
  <!-- All coordinates are in viewBox units, scale automatically -->
</svg>



This works perfectly for charts. It works less well for text. A label that reads clearly at 320 pixels may be illegible at 160 pixels. You could increase the font size in viewBox units, but then it looks oversized on wide screens. The tradeoff is real, and the solution is often to use very few labels. A sparkline with a single hover tooltip scales better than a bar chart with eight axis labels.



Canvas-based microexplanations lose the viewBox trick. A canvas has a fixed pixel resolution. Scaling it with CSS makes it blurry. Scaling it properly requires re-rendering at the new resolution, which means listening for resize events and redrawing. That is more code, more complexity, and more bytes. For microexplanations, SVG is almost always the right rendering path for this reason alone.



There is a harder version of this problem for embed contexts with very different aspect ratios. A Slack unfurl preview is wide and short. A Notion inline embed is roughly square. A tweet card image is about 2:1. If the same microexplanation appears in all three contexts, it needs to look reasonable in all three shapes. Designing for one aspect ratio and hoping it survives the others is the most common approach, and it mostly works when the chart is simple. The microexplanation constraint (one figure, one idea) helps here: a single sparkline or bar chart adapts to different aspect ratios more gracefully than a complex multi-panel layout.



The size budget, broken down



Fifty kilobytes sounds like plenty until you start accounting for the pieces. The figure below breaks down what actually fits in that budget and what blows through it.




  

  
Size budget breakdown. The top bar shows a microexplanation that fits: vanilla SVG, inline styles, system fonts, one interaction. The bottom bar shows what happens when you try to include D3. The library alone exceeds the entire budget. CDN loading is an option, but it breaks self-containment.





D3 v7 minified and gzipped is about 85 KB over the wire, but the raw minified file that would need to be inlined in a self-contained microexplanation is roughly 290 KB. That is nearly six times the entire budget. You cannot include D3 inline. If you load it from a CDN, the microexplanation works in contexts with network access but fails when opened from a local file, embedded as a data URI, or used in an environment that blocks third-party scripts.



The alternative is to not use D3 at all. A sparkline is a <polyline> with coordinates computed by a few lines of arithmetic. A bar chart is a loop that creates <rect> elements. Scales, axes, and transitions are the parts of D3 that save the most code, but for a single simple chart, the vanilla JavaScript replacement is often shorter than the D3 import. The microexplanation format is where you finally pay the cost of D3's generality: when you need only one chart, a general-purpose charting library is overhead.



The rest of the budget goes faster than you might expect. Inline CSS for even a minimal design is 1 to 3 KB. The HTML structure (doctype, head, body, elements, attributes) adds another 1 to 2 KB. The data itself depends on the chart, but a sparkline with 30 data points is under 200 bytes, while a dataset of 500 objects could be 10 KB. And then there is the JavaScript: event listeners, DOM manipulation, any computation. A hover interaction is about 1 KB. A slider with live update is 2 to 3 KB. That leaves room, but not much.

The pattern is clear and somewhat deflating: most platforms where conversations happen do not support inline interactivity. Notion is the exception. Everywhere else, the microexplanation degrades to a static preview plus a link. This does not make microexplanations pointless. A well-chosen static sparkline image, appearing inline in a Slack message right where someone asked the question, is still far more useful than a link to a separate page that nobody clicks. The progressive enhancement ladder exists precisely because the top rungs are rarely available. The lower rungs still carry value.



The moonshine connection



If moonshine is about producing interactive explanations with LLMs, then microexplanations extend that mission to the places where people actually need explanations. A /shine workflow could offer a micro mode:



# Full explanation (article-length, multiple figures)
/shine "explain the connection pool optimization"

# Microexplanation (one figure, embeddable, <50 KB)
/shine micro "sparkline of p95 latency this week"

# Micro with specific format output
/shine micro "parameter explorer for smoothing window" --format github



The --format github flag would produce an SVG screenshot plus a self-contained HTML file. The --format notion flag would produce a hosted URL ready for iframe embedding. The --format datauri flag would produce an iframe tag with the base64-encoded page.



The LLM's job changes in micro mode. Instead of structuring a narrative across sections and figures, it produces a single figure with a single interaction that makes a single point. The constraint is the feature: it forces the LLM to be precise about what the viewer should understand, because there is room for exactly one idea.




  
Microexplanations decompose. A full moonshine explanation about connection pool optimization might contain five figures. Each one could be extracted as a microexplanation: the sparkline in the pull request, the parameter explorer in the docs, the before/after toggle in the incident report. The article is the whole; the micros are its atoms, distributed to where the conversations are.





What can go wrong




  
Risks and open problems

  
    
    
  • Interactive iframes and Content Security Policy. Many corporate environments deploy strict CSP headers that block inline scripts and iframe embeds. A microexplanation that works on a personal blog may silently fail on an internal wiki. The progressive enhancement ladder mitigates this (fall back to static image), but the failure mode is invisible: the iframe simply does not render, with no error message for the viewer.
    • 
    
  • The hosting requirement is inescapable. There is no truly hostless interactive embed that works across platforms. Data URIs come close but are blocked in many contexts. Web Components require a script tag. OEmbed requires a registered provider. Every interactive path eventually requires putting a file on a server. The dream of "paste this into any document and it works" remains unrealized for interactive content.
    • 
    
  • The temptation to cram too much. A microexplanation with two sliders, a dropdown, and three linked charts is not a microexplanation. It is a small explanation that lost its nerve halfway through being cut down. The 200-line, 50 KB constraint is not a suggestion. Exceeding it defeats the purpose: fast loading, easy embedding, one idea clearly communicated.
    • 
    
  • LLMs default to full-size. When asked to "make a chart explaining X," an LLM will typically produce a full explanation with narrative prose, multiple sections, and web fonts. Generating a good microexplanation requires a different prompt strategy: emphasize the size budget, the single-figure constraint, and the embed context. Without that guidance, the output will be too large and too complex to embed.
    • 
  





What comes next



The gap between explanations and conversations is a distribution problem. Microexplanations are the smallest bridge across that gap: one figure, one interaction, embedded where the conversation is already happening. They will not replace full explanations any more than tweets replaced essays. But they can carry a single insight to the exact moment someone needs it, in the exact place they are looking.



Building the tooling means building a compiler that takes a moonshine explanation and outputs multiple formats: self-contained HTML, SVG screenshot, animated GIF, data URI, OEmbed endpoint. The source stays the same. The output adapts to the platform. That is the work ahead.
