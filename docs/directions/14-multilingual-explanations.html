Future Directions

Multilingual Explanations

D3 visualizations that adapt text, number formatting, reading direction, and cultural conventions for different locales. Making explanations that work beyond English.

The English assumption

Open any D3 tutorial. The axes read left to right. The decimal separator is a period. The thousands grouping is a comma. Month names are in English. The text flows from top-left to bottom-right, and nobody questions whether that orientation is natural, because for the author it is.

This is not a D3 problem specifically. It is a problem baked into how most visualization code gets written. The defaults of the programming environment (JavaScript's Number.toLocaleString, the browser's Intl API) lean Western. D3's own defaults lean harder: d3.format(",d") produces 1,234 with a comma for grouping and nothing else. No locale parameter. That comma is hardcoded into the format specifier string itself.

For moonshine explanations, this matters more than it does for internal dashboards. An explanation is meant to teach. It reaches people through hyperlinks, search engines, social shares. The reader might be in Cairo, in Tokyo, in São Paulo. If the number on the axis reads 1.234 and the reader's convention treats a period as a thousands separator, they see "one thousand two hundred thirty-four" where the author meant "one point two three four." The explanation has not just failed to communicate. It has communicated the wrong thing with complete confidence.

Localization is not translation. Translation changes the words. Localization changes the meaning of punctuation, the direction of reading, the connotation of colors, and the structure of dates. A chart can be perfectly translated and still deeply wrong.

The five adaptation layers

Making a D3 chart genuinely multilingual requires changes at five distinct levels. Most localization efforts stop at layer one and call it done.

LayerWhat changesD3 API
Text contentLabels, titles, tooltips, annotationsString lookup tables
Number formattingDecimal separator, grouping character, currency symbol, numeral systemd3.formatLocale()
Date formattingMonth names, day-of-week ordering, era systems, calendar conventionsd3.timeFormatLocale()
Layout directionAxis placement, text alignment, chart reading flowManual SVG transform + CSS direction
Cultural semanticsColor connotations, iconography, spacing for CJK, text shaping for ArabicNo D3 API. Requires design decisions.

D3 has mature locale support for layers two and three. d3.formatLocale() accepts a locale definition object that specifies the decimal separator, grouping separator, grouping pattern, currency prefix/suffix, numerals, percent sign, and minus sign. d3.timeFormatLocale() accepts month names (full and abbreviated), day names, date/time format templates, and AM/PM strings. These APIs are powerful and thoroughly documented. They are also almost never used. The vast majority of D3 code calls d3.format and d3.timeFormat directly, using the built-in en-US defaults without a second thought.

Layers four and five have no D3 API at all. You are on your own.

Number formats: more than cosmetic

Consider the number 1.234,56. In Germany, this is one thousand two hundred thirty-four and fifty-six hundredths. In the United States, this is a syntax error. In France, the same quantity would be written 1 234,56 with a non-breaking space as the thousands separator. In Switzerland, it might be 1'234.56 with an apostrophe. In the Indian numbering system, the same quantity is 1,234.56 but larger numbers group differently: ten million is 1,00,00,000, not 10,000,000.

D3's formatLocale handles most of this. You supply a locale definition object, and it returns a format function that respects those conventions. The critical fields are decimal (the decimal point character), thousands (the grouping character), grouping (an array specifying the grouping pattern, so [3] for Western conventions and [3, 2, 2, 2, 2, 2, 2, 2, 2, 2] for Indian), and currency (a two-element array with the prefix and suffix).

// German locale: comma for decimal, period for thousands
const de = d3.formatLocale({
  decimal: ",",
  thousands: ".",
  grouping: [3],
  currency: ["", " €"]
});

de.format("$,.2f")(1234.56);  // "1.234,56 €"

The $ in the format specifier is not a literal dollar sign. It is a placeholder for the locale's currency symbol. This is an important distinction. The format specifier string is locale-independent. The locale definition controls what $ expands to, what , expands to, what the decimal looks like. The same specifier "$,.2f" produces correct output for any locale, as long as the locale object is correctly defined.

Dates: the deep end

d3.timeFormatLocale() works the same way. You supply locale-specific month names, day names, and format templates. The format specifiers (%B for full month name, %A for full day name) remain the same. The locale controls what those specifiers resolve to.

The harder problems sit below the API surface. Japan uses the Gregorian calendar in most contexts but also has a traditional era system (currently Reiwa, which began in 2019). The Islamic calendar is a lunar calendar with months that do not align with Gregorian months. The Hebrew calendar is lunisolar. D3's time format locale does not handle calendar system conversions. It handles the string representation of Gregorian dates in different languages. If you need to display Hijri dates or Japanese era years, you need separate conversion logic before formatting.

Even within the Gregorian system, conventions diverge. In the United States, dates are formatted month/day/year. In most of Europe and East Asia, the order is day/month/year or year/month/day. The difference between 03/04/2025 meaning March 4th and April 3rd is not a cosmetic preference. It is a genuine ambiguity that a locale-aware chart must resolve.

RTL: the hardest problem

Arabic and Hebrew are read right to left. This is not just about text direction. It changes the entire spatial logic of a chart.

In a left-to-right bar chart, the category axis is on the left and the bars extend rightward. Time flows left to right. The eye starts at the origin (bottom-left) and scans rightward and upward. For an RTL reader, this spatial logic inverts. The category axis belongs on the right. Bars extend leftward. Time flows right to left. The origin moves to the bottom-right corner.

D3 does not have an RTL mode. The library works in SVG coordinate space, which is always left-to-right: x increases to the right, y increases downward. To produce an RTL chart, you have to manually mirror the layout. This means:

Swap the axis placement. The y-axis moves from left to right. d3.axisRight() instead of d3.axisLeft(). The x-axis domain reverses so that earlier dates or smaller values appear on the right.

Flip text alignment. Labels that were text-anchor: start become text-anchor: end and vice versa. Annotations that sat to the right of their data points now sit to the left.

Mirror the entire layout. The simplest approach is to apply a CSS transform: scaleX(-1) to the SVG element, then counter-flip all text elements so they read normally. This is a hack but an effective one. A more principled approach is to build RTL awareness into the scale construction itself, mapping the domain so that the range runs from right to left.

Reverse reading flow. If the chart has a legend, it should be right-aligned. If there are multiple panels or small multiples, they should flow right to left. Scrollytelling stages should progress from right to left.

The deepest trap with RTL is not the mirroring itself. It is that LTR-native developers cannot see the bugs. An RTL layout error looks "fine" to someone who reads left to right. The chart renders. The data is correct. The numbers are right. But the spatial narrative is backwards, like reading a comic strip from the last panel to the first. You need RTL readers to review RTL layouts.

Cultural color and typography

Red means danger in the United States and prosperity in China. Green means "go" or "safe" in the West and carries religious associations in many Muslim-majority countries. White is the color of mourning in much of East Asia. These associations do not invalidate the use of color, but they do mean that a color palette chosen with one culture in mind can send unintended signals in another.

For data visualization specifically, the higher risk is in semantic color: using red for "bad" and green for "good" in a chart that will be read across cultures. A financially negative number colored red might read as alarming to an American audience and auspicious to a Chinese one. The safest approach for cross-cultural charts is to rely on color for perceptual discrimination (can you tell the categories apart?) rather than for cultural meaning (does this color imply "good" or "bad?"). Diverging color scales that run from blue to orange tend to carry less cultural baggage than red-green scales.

Typography is a different kind of challenge. CJK (Chinese, Japanese, Korean) characters are monospaced by nature, each occupying a roughly square cell. Mixing CJK characters with Latin text in the same label creates uneven spacing unless you account for the different character metrics. Arabic script is cursive and context-dependent: the same letter looks different depending on whether it starts a word, sits in the middle, or ends it. SVG text rendering handles Arabic shaping reasonably well in modern browsers, but you cannot take it for granted. Rendering Arabic text in a <canvas> element, for instance, requires a shaping library like HarfBuzz compiled to WASM.

The i18n architecture

Making a single chart respond to locale changes, as in the demo above, requires a specific architecture. Content strings must be separated from layout code. Formatting must be parameterized, not hardcoded. Layout direction must be a variable, not an assumption.

The pattern has three pieces:

Locale definitions. Each supported locale gets a definition object containing: the d3.formatLocale spec, the d3.timeFormatLocale spec, a string table for UI labels and titles, and a direction flag ("ltr" or "rtl"). These objects are the single source of truth. Everything else reads from them.

Parameterized rendering. The chart drawing function accepts a locale object and derives all its formatting, labeling, and layout from it. No string literals. No hardcoded axis orientation. The scale domain direction, axis generator, text anchor, and label positions are all computed from the locale's direction flag.

Fallback chains. Not every string needs to be translated for every locale. A fallback chain (try the requested locale, then fall back to a related locale, then to English) allows incremental translation. You can ship a chart that is fully localized for three languages and partially localized for seven more, rather than waiting until every last tooltip is translated before supporting any additional language.

// Fallback chain: try requested locale, then region, then English
function getString(key, locale, strings) {
  return strings[locale]?.[key]
    || strings[locale.split("-")[0]]?.[key]
    || strings["en-US"][key]
    || key;
}

Moonshine connection

Moonshine explanations are self-contained HTML files that teach through interactive visualization. They are shared by URL. They are indexed by search engines. They are read by people who find them through links on Twitter, on Hacker News, on Japanese tech blogs, on Arabic-language forums.

If an explanation is only legible to English readers, its reach is capped at roughly 1.5 billion people (generously counting second-language English speakers). Adding Mandarin, Spanish, Arabic, and Japanese brings the potential audience to over 4 billion. But the numbers are less important than the principle: an explanation that claims to teach should not exclude 75% of humanity through a formatting default.

For d3-power-tools, locale awareness would mean that skills produce code with locale parameters rather than hardcoded en-US formatting. The generated code would accept a locale object and adapt. This is a deeper change than it sounds, because it means every format call, every axis label, every text string in the generated output needs to be locale-parameterized. The skill is no longer generating a visualization. It is generating a locale-aware visualization factory.

The LLM angle matters here too. When an LLM generates a D3 chart, it defaults to English and left-to-right. This is not a bug in the model. It is a reflection of the training data, which overwhelmingly contains English-language visualization code. Making the model produce locale-aware code by default would require either fine-tuning on multilingual visualization examples or a skill that explicitly injects the locale architecture into every generation pass. The skill approach is more practical. It is also an argument for why skills matter: they encode design decisions that the base model does not reliably make on its own.

Amber risks

  • RTL layout bugs are invisible to LTR developers. The chart renders, the data is accurate, the numbers are correct. But the spatial narrative reads backwards. You need native RTL readers in your review process, not just bilingual developers who happen to know Arabic or Hebrew.
  • Number format ambiguity causes silent misreading. Is 1.234 one-point-two-three-four or one-thousand-two-hundred-thirty-four? The answer depends entirely on locale. A chart without explicit locale signaling leaves this to the reader's assumptions, and assumptions diverge.
  • Cultural color misreading. A red/green color scale that means "bad/good" to a Western audience carries entirely different connotations in East Asian and Middle Eastern contexts. Perceptual color scales (viridis, cividis) are safer for cross-cultural work than semantic color scales.
  • LLMs default to en-US and treat localization as an afterthought. When asked to "make this chart work in Arabic," a model will typically translate the string labels and leave everything else untouched. The numbers stay Western, the axes stay LTR, the color semantics stay Western. Real localization requires the model to rethink the layout, not just swap the strings.
  • CJK and Arabic rendering edge cases. SVG text rendering works well for Latin scripts but can break with complex Arabic ligatures or unusual CJK line-breaking scenarios. Testing with real text in the target script, not just placeholder characters, is essential.

What it would take

Building genuine multilingual support into d3-power-tools is not a weekend project. It is a structural commitment that touches every skill and every generated visualization. The minimum viable version would include locale definition objects for the world's ten most widely spoken languages, a parameterized rendering pattern that every skill follows, RTL layout support as a first-class concern rather than an afterthought, and a testing protocol that includes native readers of RTL and CJK scripts.

The harder part is not the code. It is the cultural review. A developer in San Francisco can write locale definitions and format specifiers for Arabic and Japanese. Getting the nuances right, the things that look correct to an outsider but feel wrong to a native reader, requires collaboration with people who actually think in those languages. This is true for any localization effort, but it is especially acute for data visualization, where spatial layout carries meaning that text alone does not.

D3 already provides the low-level tools. d3.formatLocale() and d3.timeFormatLocale() are well-designed APIs that handle the formatting layer cleanly. The gap is in the layers above: layout direction, cultural semantics, and the architectural discipline to keep locale awareness threaded through every rendering decision. Closing that gap is what would make moonshine explanations genuinely global.
