Future Directions

Streaming Visualization

Real-time data pipelines with D3: from WebSocket ingestion through circular buffers and on-the-fly downsampling to smooth, continuous rendering.

D3 was designed for documents, not live feeds

The core D3 pattern is straightforward: load a dataset, bind it to DOM elements, render. d3.csv("data.csv").then(render) assumes the data is finished. It exists in its entirety before the first pixel draws. This works beautifully for the vast majority of visualizations, and it is the pattern that every D3 tutorial teaches.

But some data never finishes. A temperature sensor emits a reading every 500 milliseconds. A stock ticker pushes price updates dozens of times per second. A log viewer tails an ever-growing file. A real-time dashboard tracks active users on a website. In each case, the dataset is not a file you load. It is a stream you subscribe to.

The jump from static rendering to streaming rendering is not a small increment. It requires rethinking every layer of the visualization pipeline: how data arrives, how it is stored, how much of it is retained, how updates reach the screen, and what happens when the data arrives faster than the screen can redraw. A skill that handles streaming data well would unlock an entire category of explanations that moonshine cannot currently produce.

Ingestion. Data arrives over a WebSocket or Server-Sent Events connection. The client receives a message, parses it (typically JSON), and pushes the parsed datum into the buffer. This stage should do as little work as possible: parse, timestamp, push. No DOM work, no scale recalculation. Every millisecond spent here is a millisecond of jitter in the pipeline.

Circular buffer. A fixed-size ring buffer holds the most recent N data points. When the buffer is full, writing a new point overwrites the oldest. This guarantees O(1) memory usage regardless of how long the stream runs. It also defines the visible window: if the buffer holds 200 points at 10 points per second, the chart always shows the last 20 seconds of data. 

// A minimal circular buffer
class CircularBuffer {
  constructor(capacity) {
    this.buf = new Array(capacity);
    this.capacity = capacity;
    this.head = 0;
    this.size = 0;
  }
  push(value) {
    this.buf[this.head] = value;
    this.head = (this.head + 1) % this.capacity;
    if (this.size < this.capacity) this.size++;
  }
  toArray() {
    if (this.size < this.capacity)
      return this.buf.slice(0, this.size);
    // Unwrap the ring: oldest first
    return [...this.buf.slice(this.head),
            ...this.buf.slice(0, this.head)];
  }
}

Downsampling. When the buffer holds hundreds or thousands of points, drawing all of them wastes GPU fill rate without improving the visual result. The Largest-Triangle-Three-Buckets (LTTB) algorithm reduces the point count while preserving the visual shape of the line. It works by dividing the data into buckets and selecting the point in each bucket that forms the largest triangle with its neighbors, preserving peaks, valleys, and inflection points that matter to the eye.

RAF gating. requestAnimationFrame fires once per display refresh (typically 60 Hz). The render function should only run inside a RAF callback. If three data points arrive between two frames, the render function sees all three at once. This is backpressure: the rendering stage tells the pipeline "I can only consume at 60 updates per second, no matter how fast you produce."

Append-only rendering. The naive approach redraws the entire chart on every frame. The efficient approach translates the existing path to the left and appends a new segment on the right. Combined with an SVG <clipPath> to hide the portion that has scrolled past the left edge, this avoids recomputing the entire path string on every frame. 

// The append-only trick: translate + clip
const shift = xScale(data[1].t) - xScale(data[0].t);

// Slide the whole path group left by one data interval
pathGroup
  .attr("transform", `translate(${-shift}, 0)`)
  .transition().duration(intervalMs).ease(d3.easeLinear)
  .attr("transform", "translate(0, 0)");

// Recompute the path with the newest point included
path.attr("d", line(data));

The 1 Hz threshold

There is a natural boundary in streaming visualization design at roughly one update per second. Below 1 Hz, standard D3 transitions work well. You receive a new datum, update the scales, transition the path to its new shape over 300 or 500 milliseconds, and the result looks polished. The enter/update/exit pattern that D3 was built around handles this gracefully.

Above 1 Hz, transitions start to collide. A 500ms transition that gets interrupted by a new datum 200ms later produces jank: the line jumps to its interrupted position and starts a new transition from there. At 10 Hz, this is unwatchable. At 60 Hz, it is absurd.

The fix is to abandon transitions entirely for the streaming path and instead use requestAnimationFrame to render the current buffer state on every frame. The smoothness comes not from D3 interpolation but from the high frame rate itself. The data changes a little between frames, the line moves a little, and the eye perceives continuous motion.

Design heuristic: if your update interval is longer than your transition duration, use D3 transitions. If it is shorter, use RAF-driven immediate rendering. The crossover is around 1 Hz. This is the same distinction that separates "animated charts" from "real-time monitoring."

Backpressure and frame budgets

A 60 Hz display gives you 16.67 milliseconds per frame. If your render function takes 20ms, you drop below 60 fps. If it takes 40ms, you are at 25 fps. The data source does not know or care about this budget. It keeps pushing points at whatever rate the server sends them.

Backpressure is the mechanism that absorbs the mismatch. In a streaming visualization, the circular buffer is the backpressure absorber. Data arrives at the source rate and accumulates in the buffer. The RAF loop reads from the buffer at the display rate. If five points arrive between two frames, the render function simply reads the latest buffer state, which now includes all five. No point is "dropped" because the buffer stores them all (up to its capacity). But only one render pass occurs per frame. 

let dirty = false;

// Data arrives from WebSocket (potentially many times per frame)
ws.onmessage = (event) => {
  buffer.push(JSON.parse(event.data));
  dirty = true;  // Mark for render, but don't render here
};

// RAF loop (exactly once per frame)
function tick() {
  if (dirty) {
    render(buffer.toArray());
    dirty = false;
  }
  requestAnimationFrame(tick);
}
requestAnimationFrame(tick);

The dirty flag is the simplest form of gating. Without it, you would call render() inside the WebSocket callback, which might fire dozens of times between frames, each call doing redundant DOM work that the browser cannot even display.

LTTB: keeping the shape, losing the weight

The Largest-Triangle-Three-Buckets algorithm, introduced by Sveinn Steinarsson in 2013, is the standard for visually faithful downsampling of time series. The idea: divide N points into M buckets. For each bucket, choose the point that maximizes the triangle area formed with the selected point from the previous bucket and the average point of the next bucket. This preferentially keeps peaks, valleys, and sharp changes while discarding points on smooth slopes.

For streaming data, LTTB runs on every frame against the current buffer contents. With a 200-point buffer downsampled to 80 display points, the algorithm runs in microseconds. The visual result is nearly indistinguishable from the full dataset, but the SVG path string is 60% shorter, which matters for render performance at high update rates.

Why LTTB over uniform sampling? Uniform sampling (take every Nth point) can miss narrow peaks entirely. A 2ms spike in a 200-point buffer might land between sample points and vanish. LTTB guarantees it is preserved because it maximizes visual area.

Connection lifecycle

A production streaming visualization must handle the connection as carefully as the data. WebSocket connections drop. SSE streams stall. The server restarts.

Reconnection. An exponential backoff strategy prevents hammering a down server. But during the reconnection window, the chart must communicate "disconnected" to the viewer. A common pattern: fade the line to 30% opacity and overlay a small "Reconnecting..." badge. When the connection restores, snap back to full opacity. Do not silently show stale data as if it were live.

Gap detection. If the stream resumes after a gap, the buffer now contains a discontinuity. Drawing a straight line from the last pre-gap point to the first post-gap point produces a misleading interpolation. The correct behavior: detect gaps (timestamp delta exceeding 2x the expected interval), break the path into segments, and leave a visible gap in the chart. The viewer should see the absence of data, not a fabricated bridge.

Stale-data indicators. If the last received datum is more than a few seconds old and the connection is nominally alive, something is wrong upstream. A subtle visual cue (a dimmed chart area, a "Last update: 12s ago" timestamp) prevents the viewer from mistaking stale data for a calm signal.

Streaming and moonshine

Moonshine explanations are inherently static: an LLM generates the HTML, and the viewer opens a page with all data already embedded. Streaming visualization opens a different possibility: explanations that narrate what is happening right now, not what happened in a frozen dataset.

Imagine an explanation of TCP congestion control that does not just show a diagram of the sawtooth pattern but generates synthetic traffic, runs a simulated congestion algorithm, and renders the result live. The viewer sees the window grow, the congestion event occur, the backoff in real time. They can adjust parameters (link capacity, round-trip time, loss rate) and see the algorithm respond immediately.

Or consider an explanation of queueing theory where a simulated Poisson arrival process feeds a simulated server, and the queue depth chart updates live. The viewer raises the arrival rate until the queue explodes, and the explanation calls out the moment it happens: "The system just crossed the stability boundary. Notice how queue depth stopped fluctuating and started growing monotonically."

Streaming visualization turns moonshine explanations from illustrated essays into live laboratories. The data is not evidence of a past experiment; it is the experiment, running in the reader's browser.

What a streaming skill would provide

A streaming-line skill for d3-power-tools would encapsulate the entire pipeline described above. An LLM generating a streaming visualization would not need to reinvent circular buffers or LTTB. It would declare: 

// Hypothetical skill interface
const stream = streamingLine({
  container: "#chart",
  bufferSize: 200,
  downsample: { method: "lttb", target: 80 },
  xAccessor: d => d.timestamp,
  yAccessor: d => d.value,
  reconnect: { strategy: "exponential", maxDelay: 30000 },
  gapThreshold: 2.5,  // multiples of expected interval
});

// Push data from any source
ws.onmessage = e => stream.push(JSON.parse(e.data));

// Or from a simulation
setInterval(() => stream.push({
  timestamp: Date.now(),
  value: simulate()
}), 100);

The skill handles the buffer, the downsampling, the RAF gating, the append-only rendering, the clip path, and the gap detection internally. The LLM's job reduces to specifying the data shape and the visual encoding, which is exactly the kind of declarative, high-level work that LLMs do well.

Risks and sharp edges

  • Memory leaks from unbounded buffers. The most common LLM-generated streaming pattern is data.push(newPoint) with no eviction. After an hour at 60 pts/sec, that array holds 216,000 objects. After a day, 5.2 million. The tab quietly consumes gigabytes until the browser kills it. Circular buffers with a fixed capacity are not optional; they are the entire point.
  • DOM node accumulation in SVG mode. Appending a new <circle> per data point without removing old ones is the SVG equivalent of a memory leak. Even if the data buffer is bounded, the DOM can grow without bound. Canvas rendering or careful enter/exit management is essential at high update rates.
  • The illusion of real-time. The author sees data flowing smoothly on their fast development machine. The viewer may be on a throttled mobile connection with a slower GPU. Rendering latency is invisible to the person who writes the code but painfully visible to the person who runs it. Frame-time monitoring (as in the stats panel above) should be a default, not an afterthought.
  • LLMs generating polling loops. Given a prompt about "live data," LLMs tend to generate setInterval(() => fetch("/api/data"), 1000). This is polling, not streaming. It wastes bandwidth, adds latency, and does not scale. A streaming skill should steer the LLM toward event-driven patterns (WebSocket, SSE, or at minimum a simulation loop that pushes data on a schedule rather than pulling it).

What comes next

Building a streaming skill is a tractable project. The circular buffer is 30 lines. LTTB is another 40. The RAF gate is 10. The real work is in the API design: making the skill flexible enough for different data shapes and visual encodings while keeping the default path simple enough that an LLM can use it without deep configuration.

The harder, more interesting question is how streaming changes the nature of explanation itself. A static dataset is a closed world: the explanation can reference any point in it, compare beginning to end, highlight a specific anomaly. A stream is an open world. The explanation must react to data it has not seen yet. It must notice patterns as they emerge, call out anomalies in real time, and admit when the data is not doing anything interesting. That is a fundamentally different authoring challenge, and it is one that generative tools are only beginning to explore.
