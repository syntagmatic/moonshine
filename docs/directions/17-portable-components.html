Future Directions

Portable Components

D3 visualizations as Web Components. Custom elements with shadow DOM for framework-agnostic embedding. A chart that works anywhere HTML works.

The portability problem

You build a D3 bar chart. It looks good on your page. Then someone asks you to put it in a React app. You rewrite it with useRef and useEffect, fighting React's DOM ownership model. Then the marketing team wants it in WordPress. You build a shortcode. Then it needs to go in a Notion embed. You build an iframe wrapper. Then someone pastes the iframe in an email client that strips iframes. You make a screenshot fallback.

At every step, the chart itself has not changed. The data is the same. The visual encoding is the same. The interaction logic is the same. What changes is the glue: how the chart attaches to its host environment, how it receives data, how its styles coexist with the page around it. Every new context demands a new integration layer, and each layer accumulates its own bugs and maintenance burden.

This is not a D3 problem. It is an architectural problem. D3 renders to the DOM. The DOM belongs to whoever controls the page. If that page is a React app, React owns the DOM. If it is a CMS, the CMS owns the DOM. D3 charts are squatters, operating on DOM that some other system believes it controls. The conflicts are predictable: styles collide, lifecycle hooks fire at the wrong time, cleanup does not happen.

The browser already has a primitive for encapsulated, reusable UI components that own their own DOM subtree: Web Components. The question is whether they are a good fit for data visualization, where the requirements are more demanding than a toggle button or a date picker.

Custom elements: the basics

The Web Components spec consists of three pieces: custom elements (defining new HTML tags), shadow DOM (encapsulated DOM subtrees with style isolation), and HTML templates (inert DOM fragments for stamping out instances). For D3 visualization, the first two matter. Templates are less relevant because D3 builds its DOM programmatically rather than cloning markup.

A custom element is a JavaScript class that extends HTMLElement. You register it with customElements.define(), and from that point forward the browser treats your tag name as a first-class element. The lifecycle callbacks give you hooks for the moments that matter:

class D3Scatter extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: 'open' });
  }

  connectedCallback() {
    // Element added to the DOM. Render here.
    this.render();
    this._ro = new ResizeObserver(() => this.render());
    this._ro.observe(this);
  }

  disconnectedCallback() {
    // Element removed from DOM. Clean up.
    this._ro?.disconnect();
  }

  static get observedAttributes() {
    return ['src', 'x', 'y', 'color'];
  }

  attributeChangedCallback(name, oldVal, newVal) {
    // An observed attribute changed. Re-render.
    if (oldVal !== newVal) this.render();
  }
}

The dream API looks like this: <d3-scatter data="url.csv" x="gdp" y="life"></d3-scatter>. Drop it into any HTML page. It works. No build step, no framework adapter, no integration code. The custom element owns its rendering, its styles, and its cleanup.

Shadow DOM: isolation and its costs

Shadow DOM gives each component instance its own DOM subtree that is invisible to the outer page. Styles defined inside the shadow root do not leak out. Styles defined on the page do not leak in. This is exactly what you want for a portable chart: the host page's CSS reset cannot collapse your margins, its svg { display: none; } rule cannot hide your chart, and your chart's axis styling cannot bleed into the page's typography.

The isolation is real. But it comes with friction.

D3 relies heavily on d3.select() and d3.selectAll(), which delegate to document.querySelector() and document.querySelectorAll(). These methods search the light DOM. They do not pierce shadow boundaries. If your D3 code calls d3.select("svg"), it will not find the SVG inside your shadow root. Every D3 selection must be scoped to the shadow root explicitly: d3.select(this.shadowRoot).select("svg"). This is a small change in each line of code but a large change in habit. Most D3 examples, tutorials, and LLM-generated code assume document-level selection. Porting that code into a shadow DOM context requires touching every selection call.

Theming is the harder problem. If the host page cannot reach into the shadow DOM with CSS selectors, how does it style the component? The answer is CSS custom properties (variables). Custom properties are the one styling mechanism that crosses the shadow boundary. A component can define its internals in terms of variables:

/* Inside the shadow DOM */
.bar {
  fill: var(--viz-bar-color, #2563eb);
}
.axis text {
  fill: var(--viz-text-color, #1a1a2e);
  font-family: var(--viz-font, system-ui);
}
.axis line, .axis path {
  stroke: var(--viz-axis-color, #e2e2e8);
}

And the host page sets those variables on the element or on a parent:

/* On the host page */
d3-scatter {
  --viz-bar-color: #e74c3c;
  --viz-font: 'Georgia', serif;
}

This works, but it requires the component author to anticipate every axis of customization and expose it as a named variable. You cannot style arbitrary internals. The component's visual API is exactly the set of custom properties it documents. Nothing more.

Attributes vs. properties

HTML attributes are strings. They live in the markup and can be set declaratively. Properties are JavaScript values. They live on the element object and can hold anything: arrays, objects, functions.

For a D3 component, this creates a natural split:

MechanismGood forLimitation
AttributesData source URL, encoding channel names, simple options (max-bars="5")Strings only. Cannot pass arrays or complex objects.
PropertiesInline data arrays, custom scales, accessor functionsRequires JavaScript. Cannot be set in pure HTML.
SlotsComposing chart parts: <d3-axis slot="x">Adds complexity. Coordination between slotted parts is hard.

The pragmatic approach is to support both. Attributes handle the common case: someone drops a tag into HTML with a data URL and a couple of field names. Properties handle the power-user case: a developer in a JavaScript-heavy environment passes in data directly and configures scales programmatically. The component watches for attribute changes via attributeChangedCallback and exposes property setters that trigger re-renders.

The tricky part is keeping attributes and properties in sync. If someone sets the data property to an array, the data attribute (which is a string) is now stale. Most well-designed components treat properties as the source of truth and attributes as a convenience for initial configuration. But this means the HTML markup and the runtime state can diverge, which complicates debugging.

Lifecycle: the details that matter

A chart that renders once and never cleans up is fine for a static page. A chart inside a single-page application, where components mount and unmount as the user navigates, needs to be more careful. Web Components give you the right lifecycle hooks, but they do not enforce discipline. You have to use them.

connectedCallback fires when the element enters the DOM. This is where you create your SVG, set up scales, and do the initial render. It is also where you attach observers. A ResizeObserver watches the element's dimensions so the chart redraws when the container changes size. An IntersectionObserver can pause expensive animations when the chart scrolls out of view.

disconnectedCallback fires when the element leaves the DOM. This is where cleanup happens: disconnect observers, cancel pending fetches, remove event listeners, stop animations. If you skip this callback, every mount/unmount cycle leaks memory. In a single-page application where a user navigates back and forth, this leak compounds until the page slows to a crawl.

attributeChangedCallback fires when an observed attribute changes. The key word is "observed." Only attributes listed in the static observedAttributes getter trigger this callback. If you forget to list an attribute, changes to it happen silently with no re-render. This is a common source of bugs.

The lifecycle callbacks mirror React's useEffect cleanup pattern, but they are lower-level. React manages cleanup automatically when dependencies change. With custom elements, you manage it yourself. The upside is no framework dependency. The downside is that you are the garbage collector.

Bundle size: the elephant

D3 v7 is modular. The full package is roughly 500KB minified. But d3-scale is 18KB. d3-shape is 30KB. d3-array is 35KB. A bar chart needs d3-scale, d3-array, d3-selection, and d3-axis. That is under 60KB. For a web component distributed as a standalone package, the difference between shipping 500KB and 60KB is the difference between "fine" and "unacceptable." Tree-shaking matters. Importing import { scaleBand, scaleLinear } from 'd3-scale' instead of import * as d3 from 'd3' is not optional for this use case.

There is a deeper version of this problem. If a page uses three different D3 web components, each built independently, they might each bundle their own copy of d3-scale. Three copies of the same library, three times the download. Import maps and shared dependencies can solve this, but only if the component authors coordinate, which is exactly the kind of coordination that web components are supposed to eliminate.

Slot-based composition

Simple charts work well as monolithic components. A bar chart with a single <viz-bar> tag is clean. But complex visualizations, the kind that combine multiple chart types, custom annotations, and interactive legends, resist being crammed into a single element.

Slots offer a compositional approach: the parent component defines named insertion points, and child components fill them.

<viz-chart width="600" height="400">
  <viz-axis slot="x-axis" field="year"></viz-axis>
  <viz-axis slot="y-axis" field="revenue"></viz-axis>
  <viz-bars slot="marks" field="revenue" color="#2563eb"></viz-bars>
  <viz-line slot="marks" field="profit" color="#e74c3c"></viz-line>
  <viz-legend slot="legend"></viz-legend>
</viz-chart>

This is appealing on paper. In practice, it runs into what you might call the island problem. Each slotted component is an island with its own shadow DOM. The parent <viz-chart> cannot reach into <viz-bars> to read its data range. <viz-axis> cannot query <viz-line> to learn what scale it needs. Every piece of shared state, scales, data, dimensions, must be explicitly coordinated through the parent, either via property passing or custom events. The elegance of the markup belies the complexity of the wiring underneath.

Moonshine connection

A moonshine explanation is a self-contained HTML file. It teaches through interactive visualization. It is shared by URL, embedded in blog posts, linked from Slack threads, and opened in browser tabs that stay open for days. The "self-contained" part is important: the file works without a server, without a build step, without dependencies beyond what the CDN provides.

Web Components extend the surface area for moonshine. Instead of building a one-off explanation as a monolithic HTML file, you could build reusable explanation components. A <moonshine-scatter> that accepts a data URL and encoding channels. A <moonshine-slider> that binds to a parameter and triggers re-renders. A <moonshine-annotation> that highlights a data point and shows a callout. These components would compose into explanations the same way HTML elements compose into documents.

The distribution story gets interesting. A moonshine component, once defined, can be embedded in a WordPress post, a Notion page (via embed block), a GitHub README (via a hosted HTML page), or a React application. The component carries its rendering logic, its styles, and its interaction behavior. The host environment provides only a rectangle of screen space and, optionally, some CSS custom properties for theming.

For d3-power-tools specifically, the LLM generating D3 code could emit web components instead of raw scripts. The generated component would be a portable artifact: save it as a .js file, import it with a script tag, and use it with a custom element tag. The skill that generates the code would need to follow the custom element lifecycle pattern, scope all D3 selections to the shadow root, and include disconnectedCallback cleanup. These are constraints that the LLM needs to learn, either through skill instructions or through examples in its context.

Amber risks

  • Shadow DOM breaks D3's querySelector patterns. Nearly all D3 code assumes document-level selection. d3.select("#my-chart") will not find an element inside a shadow root. Every existing D3 example, tutorial, and LLM training example encodes this assumption. Porting D3 code into shadow DOM is not a search-and-replace. It requires rethinking every selection scope.
  • Bundle size for standalone distribution. A web component that includes D3 modules adds 60-500KB to the page, depending on how aggressively the author tree-shakes. If the host page already loads D3, the component's bundled copy is pure waste. There is no clean mechanism for a web component to say "use the host page's D3 if available, otherwise use mine."
  • The island problem. Web components are islands of encapsulation. Two components on the same page cannot coordinate without explicit wiring: custom events, shared state objects, or a mediator pattern. For complex visualizations where a brush in one chart filters another, this wiring is substantial. The encapsulation that makes components portable also makes them hard to compose.
  • SSR incompatibility. Shadow DOM is a client-side API. There is no standardized way to server-render a shadow DOM component. Declarative Shadow DOM exists as a proposal (and is shipping in Chrome), but it is not universally supported. For moonshine explanations that need to be indexed by search engines or generate social card previews, the client-only rendering is a limitation.
  • LLMs generate components with memory leaks. When an LLM writes a custom element, it reliably implements connectedCallback and often forgets disconnectedCallback. ResizeObservers, IntersectionObservers, event listeners, and pending fetches accumulate on every mount/unmount cycle. In a static page this is invisible. In a single-page application with dynamic routing, it is a slow-building crash.

What it would take

A minimal viable D3 web component library for moonshine would need a base class that handles the boilerplate: shadow DOM creation, ResizeObserver setup and teardown, attribute observation, and a render() method that receives the shadow root and the element's current dimensions. Individual chart types would extend this base class, implementing only the D3 rendering logic. The base class would enforce cleanup, so component authors cannot accidentally leak observers.

The theming system would expose a documented set of CSS custom properties: colors, fonts, spacing. The component would consume these properties with sensible defaults. A host page that does not set any variables gets a reasonable-looking chart. A host page that sets --viz-font and --viz-accent gets a chart that matches its design system.

The hard part, as always, is not the initial implementation. It is the maintenance surface. Every D3 skill that generates code would need a "web component mode" that wraps its output in the custom element lifecycle, scopes selections to the shadow root, and includes cleanup logic. This is a structural constraint on code generation that touches every skill. Getting it right once is a weekend project. Keeping it right across dozens of skills and hundreds of generated charts is an ongoing discipline.

The payoff, though, is genuine portability. Write the chart once. Embed it anywhere. The browser's component model, not a framework's adapter layer, is the integration point. For moonshine explanations that aspire to reach beyond the developer's own blog and into every context where HTML is rendered, that portability is the whole point.
