Future Directions

Skill Composition Engine

Given a dataset and a question, automatically select and layer the right combination of D3 skills. Turning 37 possibilities into the 3 that matter.

The combinatorial problem

d3-power-tools ships 37 skills. Each one is a focused piece of guidance: how to build a force layout, how to manage brushing, how to encode color accessibly, how to choreograph transitions. Any given visualization needs perhaps three to five of them. The rest are irrelevant noise.

For a human author, choosing the right skills is an act of design judgment. You look at the data, you think about the question, you consider the medium, and you pull in the skills that match. This works fine when the author already knows the territory. It breaks down in two situations: when the author is learning (they do not yet know which skills exist, let alone which ones apply) and when the task is being delegated to an LLM (the model needs to be told what to load, and the person giving the instruction may not know the answer either).

Thirty-seven skills, taken three at a time, gives 7,770 possible combinations. Taken five at a time: 435,897. Most of those combinations are nonsensical. But distinguishing the handful of coherent combinations from the noise requires exactly the design knowledge that a composition engine would need to encode.

The composition problem is not "which skill is best?" It is "which skills belong together?" A skill that is excellent in isolation can be counterproductive when layered with the wrong partner. Force layout and treemap both position nodes, but they want to position them in incompatible ways.

Selection heuristics

A composition engine cannot try all combinations. It needs heuristics that narrow the search space quickly. Three dimensions cut through most of the possibilities.

Data shape. The structure of the input data eliminates entire categories of skill. Tabular data with numeric columns does not need hierarchy-layouts or force-directed positioning. A tree structure does not need statistical distribution skills. Geographic data with coordinates activates cartography skills; data without spatial encoding does not. This first filter typically reduces 37 candidates to 10 or 12.

Question type. The analytical question further narrows the set. "How does X compare across categories?" activates comparison-oriented patterns: grouped bars, slope charts, small multiples. "How is X distributed?" activates histograms, density plots, box plots. "How does X relate to Y?" activates scatterplots and correlation views. The question type acts as a lens that focuses the remaining candidates around a purpose.

Output format. A static figure for a report has different needs than a scrollytelling explanation or a live explorer. Static output does not need brushing, linked views, or responsive interaction. Scrollytelling needs the choreography and annotation skills. An explorer needs interaction, filtering, and state management. The output format determines which interaction-layer skills to include.

These three dimensions compose as a pipeline. Data shape filters the structural skills. Question type selects the encoding skills. Output format adds or removes interaction skills. The result is typically a set of three to six candidates, close enough to a final answer that human review or a simple ranking can finish the job.

The dependency graph

Not all skills are independent. Some depend on others in ways that the composition engine must respect. Linked-views depends on coordination: you cannot synchronize two panels if the coordination infrastructure is not in place. Choreography depends on motion: sequenced transitions require the motion primitives. Hierarchy-interaction depends on hierarchy-layouts: you cannot add expand/collapse behavior to a tree that has not been laid out as a tree.

These dependencies form a directed acyclic graph. When the composition engine selects a skill, it must also pull in that skill's dependencies, recursively. This is familiar territory for anyone who has used a package manager. The skill manifest is the package.json; the dependency graph is the lock file.

Figure 2. A partial dependency graph for d3-power-tools skills. Hover over a node to highlight its dependency chain. Skills pulled in as transitive dependencies add to the token budget without directly serving the task.

Dependencies create a hidden cost. The author asks for linked-views and hierarchy-interaction. The engine resolves the dependencies and loads coordination, motion, and hierarchy-layouts as well. What looked like two skills becomes five. The token budget inflates, and three of the five loaded skills were never explicitly requested. This is the "transitive dependency problem," and it is as real for skill composition as it is for npm.

Skill conflicts

Worse than dependencies are conflicts. Some skills make assumptions that contradict each other. Both force-layout and treemap want to assign positions to nodes. If both activate, whose positions win? The chart cannot simultaneously be a force-directed graph and a space-filling treemap.

Conflicts are harder to detect than dependencies because they are often implicit. Force-layout does not declare "I conflict with treemap." Instead, both declare "I set node positions." The composition engine needs a higher-level ontology of what each skill does, not just what each skill needs. A skill that sets node positions conflicts with any other skill that also sets node positions. A skill that reads node positions (like labeling or annotation) can coexist with either one, but not with both simultaneously.

In practice, conflicts cluster around a few resource types: node positioning, color encoding, axis ownership, and animation timing. Two skills that both want to control the x-axis will fight. Two skills that both assign color semantics will produce a legend that contradicts itself. The composition engine needs a small conflict-detection layer that flags these collisions before the skills are loaded.

Composition archetypes

The jig-template skill in d3-power-tools already defines composition archetypes: dashboard, scrollytelling, detail-on-demand, explorer, spatial. Each archetype is a proven combination of skills that work together. Rather than composing from scratch every time, the engine can start from the nearest archetype and adjust.

A dashboard archetype loads coordination, linked-views, responsive, and compact-layout. A scrollytelling archetype loads choreography, annotation, motion, and narrative-positioning. An explorer loads brushing, filtering, state-management, and linked-views. These are not rigid prescriptions. They are starting points that capture the most common successful compositions.

Archetypes reduce the search space from "which of 37 skills?" to "which archetype, plus what adjustments?" This is a much friendlier question. It also makes the composition engine's reasoning transparent: "I chose the explorer archetype because you asked for interactive exploration of tabular data, then added the color-accessibility skill because your dataset has a categorical dimension with 8 levels."

Token budgets and marginal cost

Every skill loaded into context costs tokens. The 37 skills in d3-power-tools average roughly 300 tokens each, ranging from 150 for simple skills like aspect-ratio to 500 for complex ones like choreography. Loading five skills means 1,500 tokens of context. Loading ten means 3,000.

The question is not "can we afford the tokens?" Context windows are large enough. The question is whether the marginal skill improves the output enough to justify its context cost. Context is not just a budget line. It is attention. Every token of skill guidance competes for the model's attention with the task description, the data, and the conversation history. A skill that is loosely relevant dilutes the context. A skill that is tightly relevant focuses it.

Empirically, the returns diminish steeply. Going from zero skills to three produces a dramatic improvement in output quality. Going from three to five produces a moderate improvement. Going from five to eight produces almost no improvement, and occasionally makes things worse, because the model starts blending guidance from skills that pull in different directions.

The composition engine's most important job may not be selecting skills. It may be declining to select them. Saying "you only need these three" is more valuable than saying "here are eight that might be relevant." Restraint is the hard part.

The manifest.json pattern

d3-power-tools already has a form of manual composition: the block manifest. Each block in the collection specifies which skills were used to build it. This is a composition record, a mapping from (dataset, question, format) to a skill set that worked.

With 107 blocks in the collection, the manifest data is a small but real training set. The composition engine could learn from it. "When previous blocks with geographic data and a comparison question used skills X, Y, and Z, those blocks scored well. So for a new task with similar characteristics, suggest X, Y, and Z." This is collaborative filtering applied to skill selection.

The manifest pattern also enables validation. After the engine selects skills and the visualization is built, the resulting skill set is recorded in the manifest. Over time, the engine can compare its selections against the actual skills that ended up in the final block, tracking where it over-selected, under-selected, or missed a dependency.

Supporting the /shine workflow

Moonshine's /shine workflow currently asks the author to describe the explanation they want to build. The author writes something like "an interactive exploration of census data by state, showing age distributions with brushing to compare regions." From that description, the LLM generates the block.

The composition engine sits between the description and the generation. It parses the description for signals: "census data by state" activates geographic skills. "Age distributions" activates distribution skills. "Brushing to compare" activates brushing and linked-views. "Interactive exploration" selects the explorer archetype. The engine assembles the skill set, checks for conflicts, resolves dependencies, and passes the result to the generator along with the original description.

This removes a load-bearing question from the author. Instead of "which D3 patterns do I need for this?", the author just describes what they want. The engine translates intent into skill selection. The author can still override, adding or removing skills manually. But the default path requires no skill knowledge at all.

That default path is particularly valuable for the learning use case. Someone building their first scrollytelling explanation does not know that they need choreography, annotation, and motion. They know they want "a story that reveals itself as you scroll." The composition engine bridges that gap.

Where this can go wrong

  • Over-selection. The most likely failure mode. The engine loads eight skills when three would suffice. Each additional skill adds context noise. The model tries to satisfy all of them, producing a visualization that is technically comprehensive and aesthetically incoherent. A kitchen-sink chart with brushing, animation, annotation, and responsive layout that does none of them well.
  • Optimizing for the engine's preferences. If the engine learns from past blocks, it will converge on the combinations that have worked before. Unusual or creative combinations that have no precedent in the manifest will be penalized. The engine becomes conservative, recommending the same four or five skill sets for 80% of tasks. Novelty dies quietly.
  • Skill conflicts producing broken output. The conflict detection layer catches known conflicts, but implicit conflicts are hard to enumerate. Two skills that individually work fine but together produce a race condition in the animation timeline. Two skills that both set fill on the same elements. These are bugs that only appear in combination, and the engine may not catch them until the output is already broken.
  • Loss of author judgment. When the engine selects skills automatically, the author stops thinking about which patterns serve the task. Skill selection is a design decision. Delegating it to the engine is efficient, but it also removes a moment of reflection where the author considers what the visualization actually needs. Over time, authors who rely on the engine may lose the judgment to override it when it is wrong.
  • Transitive dependency bloat. A skill that looks lightweight (200 tokens) may pull in three dependencies (900 more tokens). The author sees "5 skills selected" but the actual token load is 8 skills. The budget bar fills up from invisible dependencies, and the author does not understand why their context is being consumed by skills they never asked for.

From selection to orchestration

Selecting skills is the first problem. Orchestrating them is the next one. Even when the right skills are loaded, they need to be applied in the right order. Layout skills run before encoding skills. Encoding skills run before interaction skills. Annotation runs last, after the positions and encodings are settled.

This ordering is implicit in how experienced authors write D3 code, but it is not captured in the skills themselves. Each skill assumes it will be applied at the right moment. The composition engine could make this ordering explicit, providing not just a skill set but a skill sequence: "First apply hierarchy-layouts to set positions. Then apply color-encoding to assign hues. Then apply interaction to add hover behavior. Then apply annotation to label the notable features."

That sequence is a build plan. And like any build plan, it can be cached. If the engine knows that the explorer archetype with geographic data always follows the same skill sequence, it can skip the sequencing step for that combination and emit the plan directly. This is where archetypes become more than starting points. They become pre-computed build plans.

The composition engine, then, is not just a skill selector. It is a build planner: analyzing the task, selecting the skills, resolving dependencies, detecting conflicts, sequencing the application, and handing the result to the generator as a structured plan. This is the infrastructure that turns a bag of 37 skills into a coherent system. Whether it should exist at all is a separate question. The answer depends on whether the selection problem is common enough and hard enough to justify the machinery. With 107 blocks and growing, the evidence suggests it is.
