Future Directions

Notebook Integration

Bridging Jupyter and Observable notebooks with moonshine explanations. Explore in a notebook, publish as an interactive article with the exploratory mess stripped away.

The gap where insights die

Data scientists explore in notebooks. Writers explain in articles. Between those two activities sits a gap that swallows most of the world's analytical work. A researcher finds something genuinely interesting in cell 47 of a Jupyter notebook. They share the notebook with a colleague. The colleague opens it, sees 80 cells of import statements, data wrangling, dead-end plots, and debugging output, then closes the tab.

The insight survived the analysis. It did not survive the communication.

Notebooks are built for exploration: run a cell, see a result, change your mind, run another. They accumulate the full history of a thought process, including the parts that went nowhere. That accumulation is valuable to the person thinking. It is a burden to the person reading. The notebook is a lab bench covered in half-finished experiments. An explanation is a clean paper describing the one experiment that worked.

The conventional solution is to start over. Copy the final chart into a slide deck. Rewrite the findings in a Google Doc. Rebuild the visualization in a different tool. This works, but it discards the connection between the analysis and its explanation. Change the underlying data and the explanation is stale. Want interactivity? Rebuild from scratch.

The real problem is not that notebooks are messy. The real problem is that there is no pipeline from "I found something" to "let me show you what I found" that preserves the data, the encoding decisions, and the interactivity while shedding the exploratory scaffolding.

Two notebook worlds

The bridge looks different depending on which notebook you are coming from, because the two dominant notebook environments have fundamentally different architectures.

Observable notebooks

Observable notebooks are the closer cousin to moonshine. They run JavaScript in the browser. They render D3 natively. Their cells are reactive: change a value upstream and everything downstream updates automatically. In many ways, an Observable notebook already is an interactive explanation.

But it is an interactive explanation that requires Observable's runtime. Each cell is a node in a dependency graph managed by Observable's dataflow engine. Extract a single cell and you lose the reactivity. Extract three cells and you need to reconstruct their dependency relationships. The notebook is self-contained only within Observable's platform.

The bridge from Observable to moonshine is conceptually straightforward: resolve the dependency graph for the cells you want, inline the data, flatten the reactive references into plain JavaScript variables, and emit a single HTML file. The chart code is already D3. The data is already JSON. The gap is the runtime, not the language.

// Observable cell: reactive, depends on upstream 'filtered'
chart = {
  const svg = d3.create("svg").attr("viewBox", [0, 0, 640, 400]);
  svg.selectAll("circle").data(filtered)  // 'filtered' is another cell
    .join("circle")
    .attr("cx", d => x(d.income))
    .attr("cy", d => y(d.life_exp));
  return svg.node();
}

Jupyter notebooks

Jupyter is a harder problem. The analysis happens in Python. The charts are rendered by Altair, Plotly, matplotlib, or seaborn. The data lives in pandas DataFrames. Nothing is natively D3, nothing is natively HTML, and nothing is natively reactive.

The bridge from Jupyter has to cross a language boundary. There are three plausible strategies:

Strategy 1: Extract the Vega-Lite spec. Altair charts compile to Vega-Lite JSON. Vega-Lite is a declarative grammar that can be rendered client-side by the Vega runtime or translated to D3 code. Export the spec, embed it in an HTML file with the Vega-Lite embed library, and you have a working interactive chart. The limitation is that you are locked into what Vega-Lite can express.

Strategy 2: Export data, rebuild in D3. Serialize the DataFrame subset to JSON. Write (or generate) D3 code that re-renders the chart client-side. This gives you full control over the output, but the translation from a Python chart to a D3 chart is non-trivial. An LLM can help here, reading the Altair or matplotlib spec and generating equivalent D3 code.

Strategy 3: Screenshot and narrate. The fallback. Render the chart server-side, embed it as a static image, wrap it in explanatory prose. No interactivity, no data binding, but it works for every chart library. This is what most people actually do today.

What transfers, what gets dropped

The notebook contains everything. The explanation contains the parts that matter. The hard question is what falls on which side of that line.

Data transfers, but curated. The notebook might load a 200 MB CSV. The explanation needs the 43 rows that illustrate the point. Inline data in a self-contained HTML file means the dataset must be small enough to embed. That forces a healthy discipline: if you cannot reduce the data to the essential subset, you may not yet understand your own finding well enough to explain it.

Encoding decisions transfer. The choice to put income on the x-axis and life expectancy on the y-axis. The color mapping for continents. The log scale. These are analytical judgments worth preserving. An export tool that respects them (rather than rebuilding from defaults) saves the author from re-making decisions they already made carefully.

Annotations transfer and grow. A notebook annotation might be a comment: "interesting cluster here." The explanation turns that into a visible callout with a line pointing to the cluster and a paragraph explaining why it matters. The seed travels; the explanation tool helps it bloom.

Narrative order transfers, but inverted. Notebooks proceed chronologically: load data, clean, explore, discover. Explanations often proceed from conclusion backward: here is the finding, here is the evidence, here is the method. The export must reorder, not just copy.

Dead ends get dropped. The cell where you tried a bar chart and decided it did not work. The cell where you printed df.describe() to check for nulls. The cell where you accidentally plotted the wrong column. All of that is valuable during exploration and worthless in explanation. The export discards it.

The cell selection problem

A finished notebook is a tangled graph of intent. Some cells carry the insight. Some cells exist only because the analyst needed to think out loud. Some cells did useful work but in a form that has to be rewritten before it makes sense outside the notebook. Telling these apart is the first real editorial decision in the export pipeline, and it is harder than it looks.

Consider a notebook with 30 cells. Cell 1 imports libraries. Cells 2 through 5 load and clean the data. Cell 6 prints the first five rows. Cell 7 computes summary statistics. Cells 8 through 14 are a series of scatter plots where the analyst tried different variable pairings. Cell 15 is the scatter plot that revealed the pattern. Cells 16 and 17 add a regression line and compute the correlation coefficient. Cell 18 prints a debugging trace because the regression blew up on a null. Cell 19 filters the nulls. Cell 20 reruns the regression. Cells 21 through 25 explore whether the pattern holds across subgroups. Cells 26 and 27 build a small multiples display that compares three subgroups. Cell 28 adds annotations. Cell 29 tries an alternative color scheme and reverts. Cell 30 is a markdown cell summarizing the finding.

Of those 30 cells, maybe five carry the insight: the data load (with filters applied), the main scatter, the regression computation, the small multiples, and the summary annotation. Another three or four did real work that the explanation needs but in a form that has to change: the data cleaning has to collapse from four cells into an inline filter; the subgroup definitions have to become parameters rather than separate DataFrames. The remaining twenty-odd cells are scaffolding. They were necessary for the analyst to think clearly, but they have no place in the explanation.

Manual tagging is the simplest approach. A comment convention like # @export or # @keep lets the author flag cells during exploration or in a review pass after the analysis is done. This requires discipline. Most people will not tag consistently while they are in the middle of an analytical flow, and a post-hoc review pass means rereading your own notebook with fresh eyes, which is its own skill.

Automated classification is more ambitious. An LLM that reads the full notebook can propose which cells matter, using heuristics: cells that produce visible output and are referenced downstream are likely keepers; cells that print raw DataFrames or contain only print() statements are likely scaffolding; cells that were followed by a near-identical cell (suggesting the first one failed) are candidates for discard. The LLM can also trace the dependency graph: if the final chart references a filtered DataFrame that was defined in cell 19, then cell 19 is needed even if the analyst never explicitly marked it.

The hard cases are the cells that sit between keep and discard. A cell that computes an intermediate aggregation might be needed as a data source for the explanation, but the code itself (five lines of pandas groupby) is not something you want to show the reader. That cell is a "transform": its result transfers, but its implementation gets rewritten or hidden. Recognizing this category, and handling it correctly, is what separates a naive cell-copy export from a real editorial pipeline.

The data handoff problem

Notebooks operate on datasets that can be arbitrarily large. Pandas DataFrames in Jupyter routinely hold millions of rows. Observable notebooks fetch data from APIs and databases. A moonshine explanation, by contrast, must be a single self-contained HTML file. The data goes inline, embedded as a JavaScript array or a JSON blob in a script tag.

This creates a hard constraint. A 10,000-row dataset serialized as JSON is roughly 500 KB. That is acceptable. A 100,000-row dataset is 5 MB. That is slow to load. A million rows is 50 MB. That is unusable.

The solution is not compression or lazy loading (both of which break self-containment). The solution is curation. The export step must reduce the dataset to the subset that the explanation actually needs. This is not a technical problem. It is an editorial one. It forces the author to answer: which rows matter for this story?

Sometimes the answer is all of them, and you need to aggregate. A scatter plot of 50,000 points can become a hexbin of 200 cells, or a sampled subset of 500 representative points, or a density contour with no individual data at all. The aggregation changes the chart type, which changes the encoding, which sometimes changes the story. The export pipeline has to support these transformations, not just pass through raw data.

Data extraction strategies

There are three distinct strategies for getting notebook data into a self-contained explanation, and each makes a different tradeoff between fidelity, portability, and file size.

Strategy A: Inline the subset. Filter the DataFrame down to the rows and columns the chart actually references, serialize to JSON, and embed directly in a <script> tag. This is the simplest approach and the most portable. The explanation has zero external dependencies. The tradeoff is that any row or column the author forgot to include is gone forever. The inline subset also freezes the data at the moment of export. If the upstream dataset updates daily, the explanation is permanently a snapshot. For datasets under a few hundred rows with a handful of columns, inlining is almost always the right call. The JSON overhead per row is small, the browser parses it instantly, and the file stays under 100 KB.

Strategy B: Host a Parquet file. For larger datasets that still need row-level detail in the explanation, the data can ship as a companion Parquet file loaded at runtime via fetch() and parsed with a library like Apache Arrow JS. Parquet files are columnar and compressed: a 50,000-row dataset that would be 5 MB as JSON might be 300 KB as Parquet. The tradeoff is that the explanation is no longer a single file. It requires two artifacts (the HTML and the Parquet) to be kept together. This breaks the one-file portability guarantee that makes moonshine artifacts easy to share, easy to archive, and easy to open a decade from now. It also adds a runtime dependency on the Arrow library. For internal documentation or dashboards where the files live in a known directory, this can work. For explanations that get emailed, dropped into Slack, or posted on a blog, the companion file is a liability.

Strategy C: Generate synthetic data from summary statistics. Compute the summary statistics of the real data (means, standard deviations, correlations, quantile boundaries, cluster centroids) and use those to generate a synthetic dataset that preserves the statistical shape without carrying any real records. A scatter plot of 10,000 GPS coordinates showing taxi pickup density becomes 500 synthetic points drawn from a kernel density estimate. The synthetic data tells the same visual story at a fraction of the size, and it sidesteps privacy concerns entirely.

The cost is fidelity. Synthetic data preserves distributions and correlations, but it loses individual records and outliers. If the explanation depends on pointing to a specific data point ("this city is an anomaly"), the synthetic version cannot reproduce that specificity. If the explanation is about a general pattern ("cities with more tree cover tend to be cooler"), synthetic data works well. The author has to decide whether the explanation is about the population or about individual cases, and choose the data strategy accordingly.

In practice, most explanations will combine strategies. The main chart data is inlined as a curated subset. A contextual background dataset (all countries, not just the highlighted ones) is synthesized from summary statistics to provide reference marks without bloating the file. Annotations reference specific real data points that are included by name in the inline subset.

An export tool that makes this choice transparent (showing what was kept, what was dropped, and why) is more honest than one that does it silently.

The draft/final model

Writing has drafts and final copies. Code has development branches and releases. Notebooks and explanations fit the same pattern: the notebook is the draft, the explanation is the final.

This analogy is useful but imperfect. A writing draft and its final form share the same medium (text). A notebook and an explanation do not. The notebook is code cells with side effects, execution state, and a kernel. The explanation is static HTML with embedded data and self-contained rendering. The "revision" from draft to final crosses a format boundary, not just an editorial one.

Still, the model clarifies roles. In the draft (notebook), you are allowed to make mistakes, pursue tangents, and leave loose ends. In the final (explanation), everything must serve the reader. The notebook asks "what is true?" The explanation asks "what matters?"

This also means the relationship is one-directional. You can export from notebook to explanation, but you cannot import an explanation back into a notebook and continue exploring. The explanation is a snapshot, not a live connection. If the data changes, you return to the notebook, re-explore, and re-export. This is a limitation, but it is also a feature: the explanation is stable. It does not break when the notebook changes. It captures a moment of understanding, not a live query.

Observable's different bet

Observable took a different approach: the notebook is the published form. There is no separate export step because the notebook itself is designed to be read. Cells are arranged for narrative flow. Prose and code interleave. The reactive runtime means inputs and charts stay connected without manual wiring.

This is elegant, but it couples the explanation to Observable's platform. An Observable notebook requires Observable's runtime to function. Fork it, and you need the runtime. Embed it, and you need an iframe pointing to observablehq.com. The explanation is not self-contained in the way a moonshine HTML file is self-contained (one file, no external dependencies beyond a CDN).

Observable Framework addresses this partially: it compiles notebooks into static sites. But the compiled output is still a multi-file application with JavaScript modules, not a single portable HTML file. The tradeoff is clear. Observable optimized for the authoring experience (reactive cells, live data connections, collaborative editing). Moonshine optimized for the output artifact (self-contained, portable, permanent).

The bridge between them is not about replacing one with the other. It is about letting someone who explored in Observable's excellent authoring environment publish as a moonshine artifact when portability and permanence matter more than reactivity and live data.

The Observable-to-HTML compiler

Observable cells look like JavaScript, but they are not plain JavaScript. Each cell is a generator function managed by Observable's dataflow runtime. A cell can reference other cells by name, and the runtime resolves those references through topological sorting, re-executing downstream cells when upstream values change. The viewof keyword creates two-way bindings between inputs and their values. The mutable keyword allows controlled state mutation. None of this exists in standard browser JavaScript.

Compiling Observable cells into standalone JS that runs without the runtime requires solving several linked problems. First, the dependency graph has to be resolved statically. For a given set of selected cells, the compiler walks backward through references to collect every upstream cell that contributes a value. If the chart cell references filtered, which references data, which references url, all three must be included even if the author only selected the chart cell for export.

Second, the reactive references have to become plain variable assignments. In Observable, writing filtered = data.filter(d => d.pop > 5e6) creates a reactive binding: if data changes, filtered recomputes. In a static export, there is no recomputation. The binding becomes a const declaration, evaluated once in dependency order. The compiler emits the cells as a sequence of const statements, topologically sorted so that each variable is defined before it is used.

Third, viewof cells need special handling. A viewof cell like viewof threshold = Inputs.range([0, 100]) creates both an HTML input element and a reactive value. In the compiled output, the input element still needs to exist, but the reactive binding has to become an event listener that updates a plain variable and re-renders the chart. This is where the translation gets genuinely difficult: the compiled code has to mimic the runtime's update semantics using vanilla DOM events.

Fourth, data cells that fetch from external APIs have to be resolved at compile time. A cell like data = d3.csv("https://example.com/data.csv") becomes an inline const data = [...] with the fetched result embedded. The compiler runs the fetch during the build step, serializes the result, and embeds it. If the data is too large to inline, the compiler falls back to one of the data extraction strategies described above.

The output of the compiler is a single <script> block: a sequence of const declarations (data, scales, encodings) followed by imperative D3 code that builds the DOM. No runtime, no module loader, no dependency graph. The chart code that was once reactive is now a static rendering function that runs once on page load. Interactive elements (sliders, dropdowns) work through explicit event listeners rather than reactive bindings.

// Observable source: 4 reactive cells
url = "https://api.example.com/cities.json"
data = d3.json(url)
filtered = data.filter(d => d.pop > threshold)
viewof threshold = Inputs.range([0, 1e6], {step: 1e4})

// Compiled output: static JS with event listener
const data = [/* 312 rows, fetched at build time */];
let threshold = 500000;
const slider = d3.select("#threshold");
slider.on("input", function() {
  threshold = +this.value;
  renderChart(data.filter(d => d.pop > threshold));
});

This compilation is lossy. Observable cells that rely on async generators, Promises as values, or the runtime's built-in invalidation signal have no clean static equivalent. A pragmatic compiler handles the common 80% (data fetches, simple reactivity, standard inputs) and flags the remaining 20% for manual rewriting. The goal is not a perfect Observable emulator. The goal is a good-enough translation that produces clean, readable D3 code the author can then refine.

Jupyter widget export

Jupyter has its own interactivity story, and it is architecturally different from Observable's. The ipywidgets library lets Python code create sliders, dropdowns, and other controls that live in the notebook and communicate with the kernel. When the user moves a slider, a message travels from the browser to the Python kernel, the kernel recomputes, and the result travels back. The interactivity depends on a live kernel connection.

This creates an obvious problem for export. A moonshine HTML file has no kernel. There is no Python process listening for widget messages. Any interactivity in the exported explanation has to run entirely client-side, in JavaScript, without a server.

The nbconvert ecosystem offers partial solutions. The widgetsnbextension package can serialize widget state to JSON and embed it in the HTML output. For simple widgets (a static slider that filters a pre-rendered table), this works: the widget's JavaScript model runs in the browser, updating the display without a kernel. For widgets that depend on Python callbacks (a slider that triggers a pandas query or a scikit-learn prediction), the serialized state is a snapshot with no way to recompute.

The practical path for moonshine is to distinguish between two kinds of Jupyter interactivity. Display-only interactivity (tooltips, zoom, pan, highlight on hover) can be recreated in D3 with no kernel dependency. The export pipeline translates the chart to client-side D3 and adds standard interaction patterns. Compute-dependent interactivity (refit the model with different parameters, requery the database with a new filter) cannot be recreated without the kernel. For these, the export has two options: precompute a set of results for a discrete set of parameter values and embed them as a lookup table, or drop the interactivity and present a static view with a note explaining what was interactive in the notebook.

The precomputation approach works surprisingly well for many common cases. A slider that controls a smoothing parameter can be precomputed at, say, 20 discrete values. The exported HTML embeds all 20 results and swaps between them as the user moves the slider. The experience is not truly continuous, but at 20 steps the visual difference from continuous is negligible. The cost is data size: 20 precomputed states means 20 times the data. The data budget strategies above apply with even more force here.

Libraries like jupyter-widgets-html-manager can embed widget state and run the JavaScript side of ipywidgets without a kernel. The voila project goes further, serving notebooks as standalone web applications, though it still requires a running kernel on the server. For moonshine's goals (single file, no server, permanent), the most robust path is to avoid trying to replicate Jupyter's widget protocol and instead rewrite the interactive elements as native D3 interactions that operate on pre-embedded data.

How this connects to moonshine

Moonshine's /shine process has two phases: discover the story, then build the explanation. Notebook integration maps directly onto this. The notebook is the discovery phase. The author loads data, makes exploratory charts, follows hunches, and eventually finds the insight worth explaining. The export is the transition to the building phase: take what you found and construct an explanation around it.

An LLM can help at both stages. During exploration, it can suggest alternative views, flag anomalies, and propose hypotheses. During export, it can generate the D3 code that re-renders the chart, write the explanatory prose, and structure the narrative. The notebook gives the LLM context that a blank prompt does not: here is the data, here are the encodings the human chose, here is the finding they flagged. That context makes the generated explanation far more grounded than starting from nothing.

The pipeline might look like this: the author works in Jupyter, exploring a dataset about urban heat islands. They produce a scatter plot showing the correlation between tree canopy coverage and summer temperature anomalies across 50 cities. They mark three outlier cities that are cooler than expected. They run a cell that says /shine export. The tool captures the data subset (50 cities, 4 columns), the chart specification (scatter, x = canopy_pct, y = temp_anomaly, color = region), and the annotation (three outlier cities). It feeds these to an LLM with the instruction: generate a moonshine explanation. The LLM produces a self-contained HTML file with D3 code rendering the scatter, callout annotations on the three outliers, and explanatory prose about why those cities beat the trend.

Where this can go wrong

  • The exploratory chart is not the explanatory chart. The scatter plot that helped you discover the pattern may not be the best chart for communicating it. Discovery is about the analyst's question; explanation is about the reader's understanding. A chart optimized for one audience rarely serves the other without redesign. An export tool that blindly copies the notebook's chart into the explanation skips the most important editorial step.
  • Data size explosion. Inlining a full dataset into HTML is tempting and dangerous. A DataFrame that loads instantly in Jupyter becomes a 20 MB page that takes 8 seconds to parse in a browser. The export pipeline must enforce size budgets: hard limits on inline data, with clear error messages when exceeded. Aggregation and sampling should be explicit, not hidden.
  • Notebook idioms that break in static HTML. Observable's reactive cells depend on the runtime's topological execution order. Jupyter's cells depend on kernel execution state and global variables. Neither translates cleanly to a self-contained script tag. Observable cells that reference viewof inputs need those inputs recreated. Jupyter cells that call plt.show() need a completely different rendering strategy. The translation is never mechanical; it always requires understanding what the code is actually doing.
  • LLMs generating notebook-style code. An LLM trained on notebook code will produce notebook-style code: global variables, mutable state, side effects, implicit execution order. This code works in a notebook where cells run in sequence with a shared kernel. It breaks in a self-contained HTML file where everything must be explicitly scoped. The export tool or the LLM prompt must enforce module-style hygiene: IIFEs, explicit data flow, no reliance on global state.

What a bridge tool would need

Building this for real (not just as a thought experiment) would require several capabilities that do not yet exist as a cohesive tool.

A cell selector. The author marks which cells contribute to the explanation and which are scaffolding. This could be as simple as a tag (# @export in Jupyter, a cell annotation in Observable) or as sophisticated as an LLM that reads the notebook and proposes which cells matter.

A data slicer. Given the full dataset and the exported chart specification, extract the minimal data subset. If the chart uses a filtered view, apply the filter and export only the result. If the chart aggregates, export the aggregated values. The slicer must understand the data flow from raw DataFrame to chart input.

A spec translator. Convert the chart specification from its notebook representation (Altair JSON, matplotlib axes object, Observable cell code) into D3 rendering code. This is the hardest piece. For Altair, the Vega-Lite spec provides a reasonable intermediate representation. For matplotlib, there is no clean spec to extract, only the rendered output. An LLM may be the most practical translator here: given the original code and a screenshot of the chart, generate D3 that reproduces it.

A narrative wrapper. Take the chart, the data, and the author's annotations, and generate the HTML explanation: prose, headings, figure captions, annotation callouts, the works. The LLM is well suited to this part, especially if the author's notebook comments provide seeds for the narrative.

A size budget enforcer. Hard limits on the total HTML file size, with clear feedback. If the inline data exceeds 200 KB, warn. If it exceeds 1 MB, error. Suggest aggregation strategies or sampling approaches when the budget is exceeded.

From thought experiment to practice

This direction is speculative, but the pieces exist independently. Altair already emits Vega-Lite specs. Observable already compiles to static JavaScript. LLMs already translate between chart libraries with reasonable accuracy. Moonshine already generates self-contained HTML. The missing piece is the glue: a pipeline that connects these steps into a smooth workflow where the author stays in their notebook and the explanation appears at the other end.

The most pragmatic starting point is probably the narrowest bridge: Observable notebook to moonshine HTML. Both are JavaScript. Both use D3. The translation is mostly about inlining data and flattening reactive references. It sidesteps the hard problem (cross-language translation) and proves the concept on the easy case before tackling Jupyter.

For Jupyter, the Altair path is the most tractable. Altair's Vega-Lite output is a declarative spec that can drive client-side rendering. The LLM's job becomes translating a Vega-Lite spec into idiomatic D3, which is a constrained enough task to be reliable. The matplotlib path is harder and may not be worth automating until the others are proven.

The deeper question is whether the bridge metaphor is even right. Maybe the notebook and the explanation should not be separate artifacts connected by a pipeline. Maybe the authoring environment should support both modes: exploratory cells for the analyst, explanatory output for the reader, in the same tool. Observable gestured toward this. Moonshine could go further by making the explanation a first-class output mode of the exploration process, not a separate step.

That is the ambitious version. The practical version is simpler: help people get their findings out of notebooks and into forms that anyone can read, without requiring the reader to install Python or understand cell execution order. That alone would save a remarkable number of good insights from dying in notebooks that nobody else will ever open.
