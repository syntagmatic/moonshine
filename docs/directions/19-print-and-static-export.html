Future Directions

Print & Static Export

Generating publication-quality static figures from interactive D3. Every chart that works on screen should also work on paper, on a slide, and in a tweet.

The screen-to-print gap

You build an interactive D3 chart that works beautifully. The axes are clean, the colors are distinct, the tooltip responds crisply to hover. Your collaborator says: can you put that in the paper? Can you add it to the slide deck? Can you post it on social media?

You right-click, "Save image as," and get a blurry PNG at 96 DPI with half a tooltip frozen in the corner. The axis labels are illegible at print size. The light gray gridlines vanish when printed on a department laser printer. The thin 1-pixel strokes that looked elegant on a retina screen become invisible on paper.

The chart was designed for one medium and one viewing context. Every other medium degrades it. This is not a minor inconvenience. Researchers, journalists, and analysts routinely produce interactive work that cannot travel beyond the browser tab where it was born. The screen-to-print gap is where good visualization work loses its reach.

The problem is not that exporting is hard. The problem is that exporting well requires rethinking the chart for the destination medium. A static figure is not a screenshot of an interactive chart. It is a different artifact, optimized for a different context, that happens to show the same data.

Three export formats, three tradeoffs

There are three paths from a D3 chart in the browser to a static file on disk. Each has a different fidelity ceiling, a different failure mode, and a different level of control.

FormatMethodStrengthsWeaknesses
SVG Serialize the DOM SVG node, inline computed styles, embed fonts Vector, infinitely scalable, editable in Illustrator Font embedding fragile, no raster effects, large file if many elements
PNG Draw SVG to Canvas via drawImage(), then canvas.toDataURL() Universal, works everywhere, predictable rendering Raster (fixed DPI), large file at high resolution, no editability
PDF Playwright page.pdf() or client-side via svg2pdf.js Vector, embeds fonts properly, standard for print submission Requires server-side tooling (Playwright) or a complex client library

SVG export

The most natural path for D3, because D3 already renders SVG. The chart is already a tree of SVG elements in the DOM. Serializing it should be as simple as svgNode.outerHTML. And often it is exactly that simple, until it is not.

The first trap is styles. D3 charts typically apply styles through CSS classes or inline attributes. When you serialize the SVG node, class-based styles do not come along. The exported SVG references .axis-label, but the stylesheet that defines .axis-label lives in a <style> tag in the HTML document, not inside the SVG. The fix is to walk every SVG element, call getComputedStyle(), and inline the relevant properties. This works but produces verbose output and sometimes captures styles you did not intend (inherited font families, browser-default line heights).

The second trap is fonts. If your chart uses a web font loaded via Google Fonts, the exported SVG references that font by name. Open the SVG on a machine without that font and the text renders in a fallback. The robust solution is to embed the font as a base64 data URI inside the SVG's <defs> block. This makes the SVG self-contained but significantly larger.

The embedding process works like this. You fetch the font file (the actual .woff2 binary), convert it to a base64 string, and inject a @font-face rule into a <style> element inside the SVG's <defs>. The src in that rule points not to a URL but to data:font/woff2;base64,... followed by the encoded bytes. Every font weight and style you use in the chart needs its own @font-face block with its own embedded binary. A single weight of Source Sans 3 runs about 40 KB as woff2, which balloons to roughly 55 KB when base64-encoded. Use two weights (regular and bold) across two families (sans for labels, serif for the caption) and the embedded fonts alone can add 200 KB or more to the SVG file.

The alternative is to not embed and accept font substitution. This is the default when you call svgNode.outerHTML without any font work. The SVG will specify font-family: 'Source Sans 3' and the rendering environment will try to find that font. On a machine without it, the browser or PDF viewer picks a fallback, usually a generic sans-serif or, worse, a serif face with completely different metrics. The labels shift horizontally because the glyph widths differ. A label that fit neatly below an axis tick now overlaps its neighbor. Centered text drifts left or right. The chart looks broken, not because the data is wrong but because the text is set in a font the designer never intended.

// Embed a web font as a base64 data URI in the SVG
async function embedFont(svgNode, fontUrl, family, weight) {
  const response = await fetch(fontUrl);
  const buffer = await response.arrayBuffer();
  const base64 = btoa(String.fromCharCode(...new Uint8Array(buffer)));
  const rule = `
    @font-face {
      font-family: '${family}';
      font-weight: ${weight};
      src: url(data:font/woff2;base64,${base64}) format('woff2');
    }`;

  let defs = svgNode.querySelector("defs")
    || svgNode.insertBefore(document.createElementNS(
         "http://www.w3.org/2000/svg", "defs"
       ), svgNode.firstChild);
  let style = defs.querySelector("style")
    || defs.appendChild(document.createElement("style"));
  style.textContent += rule;
}

// Inline computed styles onto SVG elements before export
function inlineStyles(svgNode) {
  const elements = svgNode.querySelectorAll("*");
  const props = ["fill", "stroke", "stroke-width", "font-family",
                 "font-size", "font-weight", "opacity", "text-anchor"];
  elements.forEach(el => {
    const computed = getComputedStyle(el);
    props.forEach(p => {
      el.style[p] = computed.getPropertyValue(p);
    });
  });
}

PNG export

For PNG, you rasterize the SVG onto a Canvas element at the desired resolution, then extract the pixels. The DPI question matters here. A chart that is 700 pixels wide on screen needs to be 2800 pixels wide for a 300-DPI print at journal column width (roughly 3.5 inches). That means creating a canvas at 4x the screen dimensions and scaling the SVG to fill it.

The process: serialize the SVG (with inlined styles), create an Image object with the SVG as a data URI source, draw it onto a scaled canvas, and call canvas.toDataURL('image/png'). The result is a high-resolution raster image that looks sharp in print.

The failure mode is CORS. If the SVG references external resources (images, fonts loaded from a CDN), the canvas becomes tainted and toDataURL() throws a security error. Everything must be inlined or same-origin.

PDF export

PDF is the gold standard for print submission. Journals want it. Conference proceedings require it. The challenge is that browsers do not natively export PDF from arbitrary DOM content (only via window.print(), which gives limited control).

The server-side approach uses Playwright or Puppeteer: load the page in a headless browser, call page.pdf(), and get a vector PDF. This produces excellent results but requires a Node.js process. The client-side approach uses libraries like svg2pdf.js or jsPDF to convert SVG elements to PDF drawing commands directly. This avoids the server but struggles with complex CSS and text layout.

For moonshine explanations, the Playwright approach is more natural. The explanation is already a self-contained HTML file. Load it in headless Chrome with a ?export=pdf query parameter that triggers print-specific styles, render, export. The pipeline is simple because the input artifact (the HTML file) is already well-defined.

What each medium needs

Exporting to a static file is only half the problem. The other half is adapting the chart for the destination. A chart designed for a 1440-pixel-wide browser window viewed at arm's length is not the same artifact as one printed at 3.5 inches wide in a two-column journal layout, or projected on a conference screen at 40 feet, or cropped to 1200 by 630 pixels in a social media card.

Print adaptations

Paper is unforgiving. Thin strokes disappear. Low-contrast colors merge. Small text becomes illegible. Interactive affordances (cursor changes, hover highlights, tooltip anchors) become visible artifacts that confuse the reader.

The specific adjustments: increase stroke widths by 1.5 to 2x. Boost contrast by darkening text and deepening color saturation. Switch body text to a serif face with proper leading for the target point size. Remove all cursor and pointer-event styling. Remove tooltip elements entirely. Add figure numbers and captions below the chart. Expand axis labels to full words ("Population (millions)" instead of "Pop (M)"). Increase tick label font size to at least 8pt at the final printed size.

Slide adaptations

Slides are viewed at distance on a projector with washed-out contrast. The chart competes with ambient light, a speaker, and the audience's phones. Every element must be bolder and simpler.

The specific adjustments: increase all text by 1.5x or more. Use bold weight for axis labels. Reduce the number of tick marks (5 ticks, not 10). Use fewer but more saturated colors. Make the title prominent and positioned inside the chart area. Strip secondary annotations. Increase point sizes and stroke widths. Drop gridlines if they are not load-bearing.

Social media adaptations

Social cards are tiny, viewed on phones, scrolled past in half a second. The chart has to communicate one message at a glance. The 1200 by 630 pixel Open Graph format leaves no room for nuance.

The specific adjustments: crop to the most interesting region of the data. Overlay a bold title and a one-sentence takeaway. Add branding (source attribution, logo). Simplify the chart to its single most important pattern. Remove axes entirely if they distract from the shape. Use a dark gradient overlay at the bottom so white text is readable against any chart background.

    The @media print shortcut and its limits

    CSS provides @media print for exactly this purpose. You write styles that activate when the page is printed or exported to PDF via the browser's print dialog. For simple cases, this works: hide the navigation, remove hover underlines, switch to a white background.

    For D3 charts, @media print is necessary but insufficient. It can hide tooltip containers and remove cursor styles. It can change background colors and adjust font sizes. It can swap a sans-serif body font for a serif face. It can set display: none on any DOM element you want to suppress. These are real and useful capabilities. A well-written @media print block can handle dozens of small adaptations that would be tedious to implement in JavaScript: hiding navigation bars, stripping link underlines, changing background colors to white so the printer does not waste ink, bumping up font-size on axis text to a legible point size.

    But CSS operates on CSS properties. It can change fill, stroke, opacity, font-size, display, and visibility. It cannot change SVG attributes set via .attr() in D3. The radius of a circle (r), the coordinates of a line (x1, y1, x2, y2), the d path string of an area or line generator, the transform attribute on a <g> element: these live in the DOM as attributes, not as style properties, and media queries cannot reach them. You cannot use @media print to reduce the number of axis ticks from 8 to 5. You cannot use it to add a figure number that was never in the DOM. You cannot use it to reposition a legend from the upper right to below the chart. You cannot use it to expand abbreviated axis labels ("Pop (M)" to "Population (millions)"). Anything that requires calling a D3 scale, a tick generator, or an axis component is beyond what CSS can do alone.

    There is a subtler limitation too. CSS print styles are applied at the moment the browser invokes the print dialog or Playwright calls page.pdf(). They are not previewed in the normal page flow. The author cannot see the print version of the chart without opening the print dialog or using Chrome DevTools to emulate the print media type. This makes iterating on print styles slow and error-prone. You tweak a rule, open the print dialog, squint at the preview, close it, tweak again. Compared to a mode toggle that re-renders the chart live in the browser, it is a painful workflow.

    The practical approach is a two-layer strategy. Use @media print for what it can reach: hiding interactive chrome, adjusting colors, switching fonts. Use JavaScript for what CSS cannot reach: a re-render function that takes a mode parameter and draws the chart with medium-appropriate settings.

    // Two-layer export strategy

// Layer 1: CSS handles the outer page
@media print {
  .tooltip, .controls, .hover-hint { display: none; }
  body { background: #fff; }
  .axis text { font-size: 10pt; }
  /* Can change fill, stroke, opacity, font properties */
  .dot { stroke-width: 1.5px; }
  .axis line, .axis path { stroke: #333; }
  /* Cannot change r, cx, cy, d, transform, tick count */
}

// Layer 2: JS re-renders the chart for the medium
function renderChart(mode) {
  const config = {
    screen: { strokeW: 1,   r: 4,  fontSize: 11, ticks: 8  },
    print:  { strokeW: 1.8, r: 5,  fontSize: 10, ticks: 6  },
    slide:  { strokeW: 2.5, r: 7,  fontSize: 16, ticks: 4  },
    social: { strokeW: 2,   r: 6,  fontSize: 14, ticks: 3  },
  }[mode];
  // ... re-render with config values
}

    Typography across media

    Typography is where the screen-to-print gap is most visible and least forgivable. A chart with 11px sans-serif axis labels looks fine on a retina screen at 100% zoom. Print that same chart in a journal column and the labels render at roughly 6pt, which is below the legibility threshold for most readers.

    The fix is not simply "make the text bigger." Each medium has its own typographic conventions. Academic papers use serif faces (Times, Palatino, Charter) with specific point sizes mandated by the style guide. Presentations use sans-serif faces (Helvetica, Gill Sans) at large sizes with generous letter-spacing. Social media cards use display faces with heavy weights for the headline and nothing else.

    Beyond face and size, leading (line spacing) and kerning (letter spacing) matter more in print than on screen. Screen rendering engines apply subpixel hinting that smooths over tight letter spacing. Print exposes every kerning error. A label that reads "1,000,000" on screen might show uneven digit spacing in a PDF because the font's tabular figures were not activated.

    The practical rule: specify font sizes in the export config as points at the final printed size, not as pixels at the screen size. If the chart will be 3.5 inches wide in a two-column layout, and axis labels should be 8pt, then the SVG text elements need to be sized so they render at 8pt when the SVG is scaled to 3.5 inches. Working backward from the physical output size is the only way to get this right.

    Cleaning interactive artifacts

    Interactive D3 charts accumulate invisible affordances that become visible artifacts in static export. A few common ones:

    Hover states frozen in place. If the user exports while hovering over a bar, the bar stays highlighted. If a tooltip is visible, it appears in the export. The fix: reset all hover states before serializing. Call a resetInteractions() function that removes active classes, hides tooltips, clears brush selections, and returns every element to its default state.

    Cursor styles baked into SVG. Elements with cursor: pointer are meaningless in print. Worse, some PDF renderers interpret cursor metadata and produce unexpected artifacts. Strip all cursor properties during export.

    Transparent hit targets. D3 developers often add invisible rectangles or expanded stroke widths to make small elements easier to click. These invisible targets sometimes become visible in print when the PDF renderer draws "transparent" as white, creating unexpected white rectangles over the chart.

    Brush handles and selection outlines. If the chart uses d3.brush(), the brush overlay and its drag handles are SVG elements in the DOM. In a static export, they appear as a mysterious gray rectangle with handles that the reader cannot interact with. Remove the entire brush group before export.

    A good export function is a cleanup function. Before serializing, it must undo every interactive enhancement: hide tooltips, clear highlights, remove brush overlays, strip cursor styles, disable transitions. The chart should look like it was never interactive in the first place.

    Color across media

    Screens emit light. Paper reflects it. This physical difference means that colors that are distinct on screen can become indistinguishable in print, especially in the blue-purple range and in low-saturation palettes.

    The specific risks: light grays (#ccc and above) vanish on cheap paper. Blues and purples merge under fluorescent office lighting. Red and green become indistinguishable for colorblind readers (a problem on screen too, but more acute when you cannot hover for a tooltip that clarifies the category). Gradients that look smooth on screen show banding when printed at lower color depth.

    A print-safe palette uses higher saturation, avoids the blue-purple ambiguity zone, and ensures that every color pair has sufficient luminance contrast to be distinguishable in grayscale. The chart in the interactive demo above shifts to a higher-contrast palette in print mode for exactly this reason.

    RGB vs. CMYK: the gamut problem

    For CMYK output (offset printing, not laser), there is a deeper issue than contrast. The RGB color space and the CMYK color space do not overlap. They are different shapes in perceptual color space, and each contains colors the other cannot reproduce. Vivid blues, saturated greens, and neon cyans in RGB sit outside the CMYK gamut entirely. A neon blue (#0066ff) on screen prints as a duller, slightly purple version of itself. Bright orange (#ff8800) loses its punch and shifts toward brown. These are not subtle differences. Side by side, the printed version of a chart can look like a faded photocopy of the screen version, even on a well-calibrated offset press.

    Professional print workflows handle this with ICC profiles: mathematical descriptions of how a specific device maps color values to physical ink densities. The designer works in a proofing color space (usually Adobe RGB or ProPhoto RGB), assigns an output profile for the target printer, and the color management system handles the conversion. But browser-based export tools know nothing about ICC profiles. The browser works in sRGB. The exported SVG or PNG contains sRGB values. When those values reach a CMYK press, the conversion is handled by whatever RIP (Raster Image Processor) the print shop uses, and the designer has no control over the mapping.

    Some palettes survive this translation better than others. Perceptually uniform palettes designed with lightness as the primary varying dimension tend to hold up, because CMYK can reproduce lightness differences more faithfully than hue differences. The viridis palette, which varies primarily along a yellow-to-purple lightness ramp, prints remarkably well. Its colors stay distinguishable in CMYK because the perceptual distances between steps are encoded in lightness, not in saturated hue. Inferno and magma share this property. Cividis, designed specifically for colorblind-safe visualization, also translates cleanly.

    Categorical palettes fare worse. The classic red-green-blue-orange-purple approach to distinguishing five categories relies on hue differences at roughly equal saturation. In CMYK, reds shift toward brick, greens shift toward olive, and the perceptual distance between them shrinks. A red-green categorical pair that is easily distinguishable on screen can become ambiguous on paper, both for colorblind and for normally-sighted readers. The orange and red categories merge. The purple and blue categories merge. What was a clear five-way distinction on screen becomes a muddled three-way distinction on paper.

    The pragmatic answer is to choose colors that are safely inside both gamuts: warm reds, earthy greens, dark blues, and grays. Avoid relying on saturated hue alone to distinguish categories. Use lightness, shape, and label as redundant channels. And if the chart will be printed, test it on real paper at least once before submission. No amount of software proofing substitutes for holding the printed page under office fluorescent lighting and checking whether the legend still works.

    Automating the pipeline

    The ideal workflow is zero-friction: the author builds an interactive explanation, and the export artifacts appear automatically. Each moonshine explanation could ship with a companion export/ folder containing pre-rendered assets:

    explanation.html
export/
  figure-1-screen.svg     # editable vector, screen colors
  figure-1-print.svg      # vector with print adjustments
  figure-1-print.pdf      # embeddable in LaTeX
  figure-1-print@2x.png   # 300 DPI for journal submission
  figure-1-slide.png      # 1920x1080, bold styling
  figure-1-social.png     # 1200x630 Open Graph card

    The pipeline would be a Playwright script that loads the explanation, iterates through each figure, switches to each export mode, and captures the output. For SVG, serialize the DOM. For PNG, use Playwright's screenshot() with a device scale factor of 4 for print DPI. For PDF, use page.pdf() with a custom viewport sized to the figure dimensions.

    The key design decision is where the medium-specific logic lives. It could live in the export script (Playwright manipulates the DOM to apply print styles). Or it could live in the explanation itself (the HTML file accepts a ?mode=print parameter and re-renders accordingly). The second approach is better because the author controls the adaptation. They know which annotations to emphasize for print, which detail to crop for social, which title to use for slides. The export script just captures what the explanation already knows how to render.

    How this connects to moonshine

    Moonshine explanations are self-contained HTML files. That self-containment is the key enabler for automated export. Each file can be loaded in a headless browser without external dependencies, rendered in any mode, and captured as a static asset. No build step, no framework, no server. Just a URL and a headless Chrome instance.

    This matters because every moonshine explanation is also a potential paper figure, a slide illustration, and a social media asset. The interactive version lives on the web. The print version goes into the LaTeX document. The slide version goes into the keynote. The social version gets tweeted with the key finding as a bold overlay. One source of truth, four output artifacts, all automatically generated.

    For an LLM generating moonshine explanations, this means the rendering function should always accept a mode parameter. The chart is not one thing. It is a family of renderings parameterized by medium. The screen version is the default, but the print, slide, and social versions are first-class outputs that deserve the same attention to detail.

    Where this can go wrong

    • Font embedding failures. Web fonts loaded from Google Fonts or a CDN are available when the page renders in a browser. They are not available when you serialize the SVG to a standalone file. The exported SVG references "Source Sans 3" but the reader's system does not have it. Text falls back to Arial or, worse, a default serif, and the layout breaks. The only reliable fix is base64-encoding the font into the SVG, which can add 100-400 KB per font weight.
    • Color drift between screen and paper. An RGB value has no fixed physical appearance. It depends on the display's color profile, the paper's whiteness, the ink's absorption, and the ambient lighting. A careful palette that distinguishes five categories on your calibrated monitor may collapse to three distinguishable groups on a reader's office laser printer. There is no purely technical solution; the author must test on real paper.
    • Frozen interaction artifacts. Half-visible tooltips, active brush handles, hovered bars stuck in their highlight state. These are the most common visual defects in static exports of interactive charts, and they are embarrassing in a publication. The export pipeline must actively clean the DOM before capture, not just serialize whatever is currently visible.
    • LLMs generating screen-only charts. LLMs trained on web-based D3 examples produce charts optimized for browser viewing. They use thin strokes, small fonts, hover-dependent labeling, and CSS transitions that become meaningless in static output. Without explicit prompting for export-aware rendering, LLM-generated charts will produce ugly static figures that require manual post-processing. The fix is to make export-awareness a first-class requirement in the generation prompt, not an afterthought.

    From one medium to every medium

    The ambition here is not just "make export work." It is to change the default assumption about what a D3 chart is. Today, a D3 chart is a thing that lives in a browser. Export is an afterthought, a lossy conversion performed reluctantly when someone needs a figure for a paper.

    The alternative is to treat the chart as a media-independent specification. The data, the encodings, and the analytical intent are the chart. The rendering in any particular medium (screen, paper, projector, phone) is a projection of that specification into a specific physical context. The screen rendering is the most interactive projection. The print rendering is the most constrained. But neither is more "real" than the other.

    This is how good graphic design already works. A brand identity manual specifies colors, typography, and spacing rules, then provides rendering guidelines for different media: web, print, signage, merchandise. Each medium gets its own treatment, but the underlying design is one thing. D3 charts deserve the same treatment. Build once, render everywhere, with the rendering adapted to what each medium demands and each audience needs.

    That is the direction. The tooling to get there is not fully built yet. But the pieces are clear: a mode-parameterized render function, a Playwright capture pipeline, and the discipline to treat export as a first-class output rather than a screenshot you take at the last minute before a deadline.
