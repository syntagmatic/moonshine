Observable Plot Bridge

When to use Observable Plot's concise grammar vs D3's full control, and how to mix them in a single explanation.

Most D3 code is scaffolding. Open any D3 scatter plot tutorial and count the lines before the first data point appears on screen. You will find a margin convention (4 lines), an SVG container (3 lines), an x-scale and y-scale (4 lines), two axes with labels (12 lines), and a color scale (2 lines). That is 25 lines before you binddata to circles. The binddata step itself is another 8 lines. The total is 35 to 45 lines for a scatter plot that any charting library renders from a single function call.

This is not a flaw. D3 gives you those 40 lines because every one of them is a decision you might want to make differently. The margin, the scale type, the axis format, the transition behavior, the tooltip implementation. D3 trusts you with the details because sometimes the details are the whole point.

But sometimes the details are not the point. Sometimes you need a scatter plot and the default decisions are fine. You do not need a custom axis format. You do not need a tooltip that follows the cursor with a cubic-bezier easing function. You need a picture of your data, quickly, so you can decide what to build next.

Observable Plot exists for those moments.

A Grammar of Graphics in JavaScript

Plot is a JavaScript library by the same people who made D3. Mike Bostock and Philippe Riviere designed it as a higher-level layer that handles the common cases, so that D3 can focus on the uncommon ones. It draws on the grammar-of-graphics tradition: a chart is a composition of marks (geometric objects like dots, bars, lines, areas), channels (visual properties like x, y, fill, r mapped to data fields), and transforms (data operations like bin, group, stack, window applied before rendering).

A scatter plot in Plot looks like this:

Plot.dot(data, {x: "gdp", y: "life_expectancy", fill: "region"}).plot()

That single expression produces a complete SVG with scaled axes, a legend, properly formatted tick labels, and colored dots. Plot inferred the scale types from the data (linear for numbers, ordinal for strings). It chose reasonable margins. It generated axis labels from the field names. All the decisions D3 forces you to make explicitly, Plot makes implicitly, guided by the data.

A histogram with binning:

Plot.rectY(data, Plot.binX({y: "count"}, {x: "income"})).plot()

A faceted small-multiples grid:

Plot.plot({
  facet: {data: data, x: "region"},
  marks: [Plot.dot(data, {x: "gdp", y: "life_expectancy"})]
})

Small multiples from a single configuration object. In D3, faceting requires manually partitioning the data, creating nested SVGs or groups, and synchronizing scales across panels. It is 80 to 100 lines of careful coordinate math. In Plot, it is one property.

Plot is to D3 roughly what Vega-Lite is to Vega. A constrained grammar that trades control for conciseness. Both Plot and Vega-Lite implement a grammar of graphics. Both generate visualization specifications that compile down to lower-level rendering primitives. The key difference: Plot outputs a DOM node (an SVG element) that you can inspect, style, and attach D3 behaviors to. Vega-Lite outputs a Vega spec that compiles to a Canvas or SVG managed by the Vega runtime. Plot's DOM output is what makes the hybrid pattern possible.

When Each Tool Wins

The choice between Plot and D3 is not a preference. It is a function of what you need the chart to do.

Task Best Tool Why
Quick scatter, bar, or line chart Plot 5 lines vs 40. Default scales and axes are good enough.
Faceted small multiples Plot Plot's facet system handles layout, shared scales, and labels. D3 requires manual grid math.
Statistical transforms (bin, group, window) Plot Built-in transforms compose with marks. D3 requires pre-processing the data.
Exploratory data analysis Plot Speed of iteration matters more than polish.
Custom brush/drag/zoom interaction D3 Plot has no interaction system. D3's behaviors are battle-tested.
Animated transitions between states D3 Plot re-renders from scratch. D3 interpolates smoothly.
Force-directed layouts D3 Simulation requires per-tick rendering and mutable node positions.
Canvas/WebGL rendering D3 Plot produces SVG. High-performance rendering needs Canvas.
Novel visual encodings D3 If the chart type does not exist as a Plot mark, you build it from D3 primitives.
Static base chart + interactive overlay Hybrid Plot for the chart, D3 for the interaction layer. Best of both.

The pattern that emerges: Plot wins when the chart type is standard and the interaction is minimal. D3 wins when the interaction is the point or the visual form is novel. The hybrid approach wins when you want a standard chart with a non-standard interaction bolted on top.

The Same Data, Three Ways

The comparison below renders a scatter plot of 60 countries, plotting GDP per capita against life expectancy, colored by region. Toggle between three approaches: pure D3 (all 42 lines), pure Plot (5 lines), and a hybrid where Plot renders the base chart and D3 adds a brush interaction on top. The code panel on the left shows the source. The chart panel on the right shows the result.

Pure D3
// 1. Setup
const w = 500, h = 340;
const m = {top: 20, right: 20,
             bottom: 40, left: 50};
const iw = w - m.left - m.right;
const ih = h - m.top - m.bottom;

// 2. SVG container
const svg = d3.select("#target")
  .append("svg")
  .attr("width", w)
  .attr("height", h);
const g = svg.append("g")
  .attr("transform",
    `translate(${m.left},${m.top})`);

// 3. Scales
const x = d3.scaleLog()
  .domain(d3.extent(data, d => d.gdp))
  .range([0, iw]);
const y = d3.scaleLinear()
  .domain(d3.extent(data, d => d.life))
  .range([ih, 0]);
const color = d3.scaleOrdinal()
  .range(d3.schemeTableau10);

// 4. Axes
g.append("g")
  .attr("transform",
    `translate(0,${ih})`)
  .call(d3.axisBottom(x)
    .ticks(5, "$,.0f"));
g.append("g")
  .call(d3.axisLeft(y));

// 5. Axis labels
g.append("text")
  .attr("x", iw / 2)
  .attr("y", ih + 35)
  .attr("text-anchor", "middle")
  .text("GDP per capita");
g.append("text")
  .attr("transform", "rotate(-90)")
  .attr("x", -ih / 2)
  .attr("y", -35)
  .attr("text-anchor", "middle")
  .text("Life expectancy");

// 6. Binddata to circles
g.selectAll("circle")
  .data(data)
  .join("circle")
  .attr("cx", d => x(d.gdp))
  .attr("cy", d => y(d.life))
  .attr("r", 4)
  .attr("fill", d => color(d.region))
  .attr("opacity", 0.75);
Pure Observable Plot
Plot.dot(data, {
  x: "gdp",
  y: "life",
  fill: "region"
}).plot({x: {type: "log"}})

That is the entire program. Plot infers the scale domains from the data, creates axes with formatted labels, builds a color legend, and returns an SVG element you can append to the DOM. Five lines replace forty-two.

Hybrid: Plot + D3 Brush
// 1. Plot renders the base chart
const chart = Plot.dot(data, {
  x: "gdp",
  y: "life",
  fill: "region"
}).plot({x: {type: "log"}});
document.body.append(chart);

// 2. Extract Plot's scales
const sx = chart.scale("x");
const sy = chart.scale("y");

// 3. D3 adds a brush on top
const svg = d3.select(chart);
const brush = d3.brush()
  .on("brush end", ({selection}) => {
    if (!selection) {
      dots.attr("opacity", 0.75);
      return;
    }
    const [[x0,y0],[x1,y1]] = selection;
    dots.attr("opacity", d =>
      sx(d.gdp) >= x0 &&
      sx(d.gdp) <= x1 &&
      sy(d.life) >= y0 &&
      sy(d.life) <= y1 ? 1 : 0.1);
  });
svg.append("g").call(brush);

Plot handles the tedious parts (scales, axes, legends). D3 handles the interactive part (brush selection and opacity updates). 18 lines total, with brush interaction that Plot alone cannot provide.

The Hybrid Pattern in Detail

The key insight that makes hybrid work is architectural: Plot returns a plain SVG element. Not a managed canvas, not an opaque widget, not a virtual DOM node. A real SVG element in the real DOM, with <circle> and <line> elements you can select, style, and manipulate with D3.

Observable Plot marks, scales, axes, legend
→
SVG Element real DOM, inspectable
→
D3 Selection brush, zoom, drag, transitions

Figure 2. The hybrid pipeline. Plot produces the SVG. D3 operates on it. The SVG element is the bridge between the two libraries.

The steps are concrete:

Step 1: Plot renders the chart. Call Plot.dot(data, options).plot() or Plot.plot({marks: [...]}). This returns an SVG element. Append it to your container. The chart is now visible but static.

Step 2: Extract Plot's scales. Plot exposes the scales it created via chart.scale("x") and chart.scale("y"). These are functions that map data values to pixel coordinates, just like the D3 scales you would have built manually. This is the bridge: you can use Plot's scales inside D3 event handlers to convert between data space and screen space.

Step 3: Wrap in a D3 selection. d3.select(chart) turns the Plot output into a D3 selection. From here, you have full access to D3's API: append groups, attach behaviors, select elements, bind data, run transitions.

Step 4: Attach D3 behaviors. Add a d3.brush(), d3.zoom(), or d3.drag() behavior to the SVG. In the event handler, use the extracted scales to translate screen coordinates back to data coordinates. Update the mark elements (the circles, rects, or paths that Plot created) using D3's .attr() or .style() methods.

// The general pattern
const chart = Plot.plot({marks: [...]});
container.append(chart);

// Plot's scales, available for D3 to use
const sx = chart.scale("x");  // {type, domain, range, ...}
const sy = chart.scale("y");

// Reconstruct D3 scales from Plot's scale descriptors
const x = d3.scaleLinear()
  .domain(sx.domain).range(sx.range);
const y = d3.scaleLinear()
  .domain(sy.domain).range(sy.range);

// Now use D3 behaviors with Plot's coordinate system
const brush = d3.brush().on("brush", ({selection}) => {
  const [[px0, py0], [px1, py1]] = selection;
  const xMin = x.invert(px0), xMax = x.invert(px1);
  const yMin = y.invert(py1), yMax = y.invert(py0);
  // Filter, highlight, update...
});
d3.select(chart).append("g").call(brush);

The extracted scale descriptors are the contract between Plot and D3. Plot says "I put the x-axis at these pixel positions, covering this data range." D3 says "Good, I will use those positions to interpret brush coordinates." Neither library needs to know about the other's internals. The SVG and the scale descriptors are the entire interface.

The Code-Length Tradeoff

The figure below shows line counts for common chart types across the three approaches. For standard charts with no interaction, Plot is dramatically shorter. As interaction requirements grow, D3 code becomes necessary. The hybrid approach sits in between, spending Plot lines on the chart and D3 lines on the interaction.

Figure 3. Approximate lines of code for six chart types, across three approaches. Plot dominates for static charts. D3 is required for interaction-heavy or novel chart types. The hybrid approach splits the difference.

The pattern is consistent: Plot saves 70-90% of the code for the chart part, but adds zero for the interaction part. If your explanation is 80% chart and 20% interaction, the hybrid approach saves roughly 60% of total code. If it is 20% chart and 80% interaction, the savings are marginal and the cost of mixing two mental models may not be worth it.

Why This Matters for Explanations

A moonshine explanation is not a dashboard. It is not an exploratory tool. It is a narrative that uses interactive graphics to make an argument or illuminate a concept. The interactive elements serve the narrative, not the other way around.

This means most of the graphics in an explanation are in service of the text. They illustrate a point. They show a dataset. They provide a reference figure. Many of them do not need custom interaction at all. A scatter plot that shows the relationship between two variables, a bar chart that compares categories, a line chart that shows a trend over time. These are figures, not applications.

For these figures, Plot is the right tool. An author writing a 2000-line moonshine explanation should not spend 200 lines on a scatter plot that exists to show a correlation. Five lines of Plot produce the same scatter plot. The author reclaims 195 lines of attention, which they can spend on the parts of the explanation that actually require custom D3: the interactive elements where the reader manipulates the visualization to build understanding.

Consider the time allocation for a typical moonshine explanation with four figures:

// Figure 1: Static scatter (context-setting)
//   Pure D3:   42 lines, 20 minutes
//   Plot:       5 lines,  2 minutes

// Figure 2: Interactive brush-linked pair (the main event)
//   Pure D3:  140 lines, 90 minutes
//   Hybrid:    60 lines, 40 minutes

// Figure 3: Faceted small multiples (supporting evidence)
//   Pure D3:   95 lines, 45 minutes
//   Plot:       8 lines,  3 minutes

// Figure 4: Animated state transitions (the payoff)
//   Pure D3:  120 lines, 75 minutes   (no alternative)

// Total: pure D3 = 397 lines, 230 min
//   Plot+hybrid = 193 lines, 120 min

Half the code. Half the time. Not because Plot replaces D3, but because it handles the parts where D3's power goes unused.

The skill-system implication: a "d3-power-tools" skill for Plot would help an LLM author decide, per figure, whether to use Plot, D3, or the hybrid pattern. The skill would encode the decision matrix above: standard chart type with no interaction? Use Plot. Custom interaction required? Use D3 or hybrid. Novel visual form? D3 only. The LLM would stop generating 42 lines of D3 for a chart that Plot produces in 5.

Embedding Plot in Self-Contained HTML

Moonshine explanations are self-contained HTML files. Loading Plot from npm requires a bundler. Loading it from a CDN requires a network request and a <script> tag. Since Plot is an ES module, the script tag needs type="module" and an import statement:

<!-- Loading Observable Plot from CDN -->
<script type="module">
  import * as Plot from
    "https://cdn.jsdelivr.net/npm/@observablehq/plot/+esm";

  const chart = Plot.dot(data, {
    x: "gdp", y: "life", fill: "region"
  }).plot();
  document.getElementById("chart").append(chart);
</script>

This works and is the recommended approach. Plot's bundle is roughly 90KB gzipped, which is modest compared to D3's 80KB. The combined weight is about 170KB of JavaScript, well within budget for a self-contained explanation. Both libraries cache after the first load.

There is a subtlety with ES modules and the existing D3 script tag. D3 is typically loaded as a global script (<script src="d3.v7.min.js">), making d3 available as a global variable. Plot, loaded as an ES module, lives in module scope. The two can coexist: D3 is global, Plot is imported. But if the explanation's main code is inside a regular <script> tag (not a module), it cannot access the Plot import. The cleanest solution is to put all code in a single <script type="module"> block that imports Plot and accesses D3 from the global scope.

Risks and Open Questions

  • API stability. Plot is younger than D3 and its API is still evolving. Between Plot 0.4 and 0.6, the scale accessor syntax changed, the facet API was restructured, and several mark options were renamed. D3's API, by contrast, has been stable for years. An explanation written against Plot 0.5 may break when the CDN serves Plot 0.7. Pinning a specific version in the CDN URL mitigates this, but it means the explanation does not automatically benefit from bug fixes.
  • Two mental models. Mixing Plot and D3 requires the author to think in two frameworks simultaneously. Plot uses a declarative, specification-driven approach (describe the chart, let Plot handle the rendering). D3 uses an imperative, selection-driven approach (select elements, bind data, set attributes). Switching between these models within a single file is manageable for experienced authors but confusing for beginners and for LLMs that have not been trained on the hybrid pattern.
  • LLM confusion. Current LLMs frequently generate 40 lines of D3 for a chart that Plot produces in 5. They also sometimes try to use Plot for tasks that require D3 (generating Plot.brush, which does not exist, or trying to animate a Plot chart, which requires re-rendering from scratch). A skill that encodes the decision matrix would help, but the training data is overwhelmingly D3 examples, which biases generation toward D3 even when Plot is the better choice.
  • Scale extraction fragility. The hybrid pattern depends on chart.scale("x") returning a usable scale descriptor. This API exists in current Plot versions, but the exact shape of the returned object (domain, range, type, interpolation) is an implementation detail that could change. A defensive wrapper that normalizes the descriptor into a D3 scale would add resilience but also complexity.
  • Re-rendering vs. updating. Plot has no update mechanism. To change the data, you discard the old SVG and call .plot() again. This means any D3 behaviors attached to the old SVG (brushes, zoom states, drag handles) are destroyed and must be re-attached. For explanations where the base chart is static and only the interaction layer changes, this is not a problem. For explanations where the base data changes (streaming, user-driven filtering), the constant tear-down and rebuild can cause visual flicker and lost interaction state.
  • Scope creep. The convenience of Plot can encourage authors to add "one more chart" because it only takes five lines. An explanation with twelve figures is not twice as good as one with six. The constraint of D3's verbosity was, paradoxically, a natural governor on chart proliferation. With Plot, the governing constraint must come from editorial discipline, not from the cost of writing code.
