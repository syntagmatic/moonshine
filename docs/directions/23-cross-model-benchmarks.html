Future Directions

Cross-Model Benchmarks

Generating the same visualization with different LLMs and comparing quality systematically. Moving model selection from tribal knowledge to evidence.

The question nobody answers with data

Ask a team which LLM they use for code generation and you will get a confident answer. Ask them why and the confidence evaporates. "Claude writes better D3." "Gemini is faster." "GPT handles edge cases." These are impressions formed from a handful of memorable successes and failures, generalized into policy. The plural of anecdote is not data, but it is how most model selection happens.

This is strange, because the tools to do better already exist. The d3-power-tools pipeline has generate-blocks-claude.py and generate-blocks-gemini.py. It has a 4-dimension audit that scores every block on visual_critic, encoding_integrity, stress_test, and cognitive_load. It has best-blocks.json tracking which blocks pass. Direction 21 proposed perceptual_accuracy as a fifth dimension. Direction 22 proposed explanation_effectiveness as a sixth. The infrastructure for systematic comparison is already there. It just needs to be generalized and made rigorous.

The key realization: d3-power-tools is already a cross-model benchmark. Two model-specific generation scripts, a shared audit pipeline, a shared scoring rubric. The gap is not infrastructure. It is protocol: running both generators on the same prompts, under the same conditions, and comparing the results systematically instead of impressionistically.

The benchmarking protocol

A meaningful cross-model comparison requires controlling everything except the model. The prompt must be identical. The skills loaded must be the same set. The timeout must match. The evaluation pipeline must be blind to which model produced the output. Every other variable is noise.

# Benchmark run: one prompt, multiple models, shared evaluation
for model in claude-sonnet gemini-pro gpt-4o llama-70b:
  generate \
    --prompt "prompts/wildfire-risk-map.md" \
    --skills "d3-bindJoin,choropleth,tooltip" \
    --model "$model" \
    --timeout 300 \
    --output "runs/$model/wildfire-risk-map/"

  audit \
    --block "runs/$model/wildfire-risk-map/index.html" \
    --dimensions "visual_critic,encoding_integrity,stress_test,cognitive_load,perceptual_accuracy,explanation_effectiveness" \
    --output "runs/$model/wildfire-risk-map/audit.json"
done

# Collect results
benchmark-report --runs "runs/*/wildfire-risk-map/audit.json"

The block manifest provides the benchmark suite. With 107 blocks covering the full skill range, each with a defined prompt and skill set, the suite spans bar charts, choropleths, force layouts, scroll-driven narratives, interaction-heavy explorers, and everything in between. Running all 107 prompts through all models is expensive but informative. A representative subset of 15 blocks covering distinct visualization types provides a useful signal at lower cost.

What the data reveals

The radar chart above is synthetic, but it reflects a real pattern observed across code generation benchmarks: no model dominates every dimension. Each has a signature profile.

Claude tends to write structurally correct D3 code. Its bindings are clean, its data joins are idiomatic, its event handlers work. It also tends to add thorough annotations and explanatory text, which boosts explanation_effectiveness. But it sometimes generates code that is correct on small datasets and collapses on large ones, because it optimizes for readability over performance. The stress_test dimension catches this.

Gemini tends toward spatial precision. Its geographic projections land in the right place. Its color scales respect perceptual uniformity. Its layouts handle dense data without overlap. These are the qualities that perceptual_accuracy and stress_test measure. But its annotations are terse, sometimes absent. The explanation sits in the code comments rather than on the page, which hurts explanation_effectiveness.

GPT-4o produces the most visually polished output on first glance. Its typography choices are solid, its whitespace is generous, its color palettes are harmonious. But look at the axes and you find mismatches: a label that says "millions" on data that is in thousands, a reversed scale that puts the highest value at the bottom without signaling that the reversal is intentional. Visual_critic scores well. Encoding_integrity scores poorly. The chart looks good but lies.

The implication for model selection: "Which model is best?" is the wrong question. "Which model is best for this type of visualization on this dimension?" is the right one. A choropleth might route to Gemini. An annotated scatter to Claude. A dashboard layout to GPT. The benchmark data makes this routing possible.

Cost-quality tradeoffs

Quality is not the only axis. Models differ in cost per token, latency, and rate limits. A model that scores 10% higher but costs 5x more may not be worth it for a batch run of 100 blocks. The cost-efficiency scatter (toggle to "Cost vs Quality" above) plots composite quality against cost per block, revealing the Pareto frontier: the set of models where no other model is both cheaper and better.

Open models introduce another tradeoff. They cost near-zero per token when self-hosted, but their quality on complex D3 tasks is currently well below the frontier models. The gap is closing with each release. A benchmark run every quarter tracks whether the gap has narrowed enough to justify switching batch generation to a self-hosted model.

The block manifest as benchmark suite

The 107 blocks in d3-power-tools are not just a portfolio. They are a benchmark suite, each block defined by a prompt and a skill set. The prompts span the full difficulty range: a simple bar chart with five values, a multi-layered geographic visualization with scroll-driven transitions, a force-directed graph with search and filter. The skill sets record exactly which capabilities were loaded.

A benchmark run generates each block with each model and audits the results. The output is a matrix: 107 blocks by N models by 6 dimensions. This matrix is too large to inspect manually, which is why the heatmap view (toggle to "Block Heatmap" above) exists. Color intensity encodes the score. Patterns emerge at a glance: vertical stripes reveal blocks that are hard for all models (perhaps the prompt is ambiguous). Horizontal stripes reveal models that are consistently strong or weak. Isolated dark cells reveal model-specific failures that merit investigation.

Temporal tracking

A benchmark run is a snapshot. Model quality changes with every release. Claude 3.5 Sonnet produced different D3 than Claude 3 Opus. Gemini 1.5 Pro handles geographic projections differently than Gemini 1.0. The benchmark framework needs a time axis.

Each run gets a timestamp and a model version string. Plotting dimension scores over time produces trend lines that answer questions analytics dashboards cannot: Is Claude getting better at stress_test? Is Gemini's explanation_effectiveness improving? Has GPT's encoding_integrity regressed since the last update?

Model-aware generation for moonshine

The twenty-two prior directions in this series propose capabilities that stress different aspects of code generation. Collaborative visualization (direction 1) requires WebSocket code. Sonification (direction 13) requires Web Audio API. Tactile export (direction 15) requires SVG path manipulation. Print export (direction 19) requires precise CSS for print media. Each of these has a different model affinity.

With benchmark data, the generation pipeline can route prompts to the model most likely to succeed. A prompt tagged with interaction:high routes to the model with the best encoding_integrity on interaction-heavy blocks. A prompt tagged with geographic routes to the model with the best perceptual_accuracy on choropleth blocks. The routing table is derived from the benchmark matrix, not from intuition.

# Model routing based on benchmark data
def select_model(prompt_tags, benchmark_matrix):
    relevant_blocks = benchmark_matrix[
        benchmark_matrix.tags.apply(lambda t: any(tag in t for tag in prompt_tags))
    ]
    # Weighted score: prioritize dimensions relevant to these tags
    weights = dimension_weights_for(prompt_tags)
    model_scores = relevant_blocks.groupby("model").apply(
        lambda g: (g[dimensions] * weights).mean().sum()
    )
    return model_scores.idxmax()

This is model selection as a data problem, not a preference problem. The benchmark suite provides the evidence. The routing function is a lookup, not a debate.

The evaluator problem

There is a circularity lurking here. The audit pipeline uses an LLM to evaluate the output of other LLMs. If the evaluator is Claude, does it favor Claude-generated code? If it is GPT, does it favor GPT conventions?

This is not hypothetical. Research on LLM-as-judge shows measurable self-preference bias: models rate their own outputs higher than equivalent outputs from other models. The fix is to use a fixed evaluator for all models in a benchmark run, and to cross-validate by running the same evaluation with a different evaluator model. If the rankings change substantially, the benchmark is measuring evaluator bias, not generation quality.

The strongest signal comes from dimensions that are objectively verifiable. Encoding_integrity can be partially checked by running the code and comparing rendered output to the data. Stress_test can be checked by feeding 10x data and measuring whether the visualization still renders. Perceptual_accuracy can be checked against Cleveland-McGill baselines. These dimensions resist evaluator bias because the ground truth is external to the evaluator's preferences.

From benchmark to ecosystem

Direction 12 proposed a skill composition engine that selects and layers skills for a given task. Direction 9 proposed AI-assisted annotation. Direction 10 proposed adaptive complexity. All of these are capabilities that different models will implement with different quality. The benchmark framework does not just select a model; it maps the capability surface of the entire model ecosystem.

A team maintaining a d3-power-tools deployment could run the benchmark suite weekly as a CI job. When a model provider releases an update, the next benchmark run reveals the impact. Regressions are caught before they reach production. Improvements are quantified before they are celebrated. The benchmark becomes infrastructure, not a one-off experiment.

# Weekly CI benchmark job
name: model-benchmark
schedule: "0 3 * * 0"  # Sunday 3 AM
steps:
  - run: benchmark --suite representative-15 --models all
  - run: benchmark-report --compare-previous --alert-on-regression
  - run: benchmark-publish --dashboard s3://benchmarks/latest/

Risks and tensions

  • Benchmark gaming. Model providers read benchmarks and tune for them. If the d3-power-tools benchmark suite becomes public, models may be specifically optimized to score well on its prompts without improving on general D3 generation. The suite needs rotation: a stable core of blocks for trend tracking, plus a rotating set that changes each quarter to resist overfitting.
  • The snapshot problem. Benchmark results are stale within months. A model that won in January may have been superseded by March. Teams that set model policy based on a single benchmark run and never re-evaluate are making decisions on outdated evidence. The temporal tracking described above mitigates this, but only if the benchmarks actually run regularly.
  • Composite score myopia. A single composite number hides everything interesting. A model with scores of [9, 9, 9, 9, 2, 9] and a model with scores of [7, 7, 7, 7, 7, 7] have the same composite but radically different profiles. The dashboard must always show dimension-level breakdowns. The composite is a sorting key, not a quality measure.
  • Cost of comprehensive benchmarks. Running 107 blocks through 4 models with 6 evaluation dimensions means 2,568 generation calls and 2,568 audit calls. At $0.10 per generation and $0.05 per audit, that is roughly $385 per full benchmark run. Weekly runs cost $20,000 per year. The representative-15 subset brings this to roughly $54 per run, which is sustainable.
  • Skill fragmentation. If different models work best with different skill versions, the skill ecosystem fragments. Claude might need verbose skills with detailed examples. Gemini might perform better with terse, specification-style skills. Maintaining model-specific skill variants doubles or triples the maintenance burden and undermines the portability that skills are supposed to provide.

What this means in practice

The path from tribal knowledge to evidence-based model selection is shorter than it looks. The audit pipeline exists. The generation scripts exist. The block manifest provides the test suite. The missing piece is a thin orchestration layer that runs the same prompt through multiple models, collects the audit results, and produces a comparison dashboard.

The first useful benchmark is small: 15 blocks, 3 models, 4 dimensions. It runs in under an hour and costs less than a team lunch. The results will not be definitive, but they will be better than the current state of the art, which is "I tried it once and Claude seemed better." That low bar is the opportunity. Any systematic measurement at all is a step change from where most teams are today.

The deeper value is cultural. Once a team sees benchmark data, they stop arguing about models and start asking about dimensions. "Claude is better" becomes "Claude scores 1.4 points higher on encoding_integrity but 0.8 points lower on stress_test, and the cost difference is 50%." That level of specificity changes the conversation from preference to engineering.
