Perceptual Testing

Automated Cleveland-McGill accuracy tests for generated charts. Does the encoding actually let viewers decode the data?

Twenty directions into this series, every one of them has assumed the same thing: that the chart communicates what the data says. Directions 1 through 4 considered how to author visualizations. Directions 5 through 8 considered how to handle data at scale. Directions 9 through 12 considered how AI could automate chart creation. Directions 13 through 16 considered how to make charts accessible. Directions 17 through 20 considered where charts live. None of them asked: does the viewer actually read the chart correctly?

That question has an empirical answer, and it goes back to 1984.

The Cleveland-McGill hierarchy

William Cleveland and Robert McGill ran experiments. They showed people charts that encoded the same data using different visual channels, then asked the viewers to estimate values. The results were consistent and replicable: some encodings produce accurate estimates, others produce systematic errors.

Their ranking, from most accurate to least:

The encoding accuracy hierarchy. Bars show relative accuracy from Cleveland and McGill's experiments. Position on a common scale (bar charts) lets viewers estimate values within a few percent. Color saturation (heatmaps) introduces errors of 50% or more.

This ranking is not a matter of taste. It is an empirical measurement of the human visual system. People can compare positions along a common axis with high precision because the comparison reduces to judging a single spatial offset. Length comparisons are nearly as good when the bars share a baseline, but degrade when they do not (stacked bar segments, for instance). Angle judgments are worse: the classic finding is that people systematically underestimate acute angles and overestimate obtuse ones, which is why pie chart slices near 90 degrees look larger than they are.

Area judgments are worse still. Stevens' power law predicts this: perceived area scales as the physical area raised to approximately 0.7, meaning people consistently underestimate the area of large circles relative to small ones. A bubble that is 4x the area of another looks roughly 3x the size. Volume is even more compressed. And color saturation sits at the bottom because the visual system adapts to its surroundings: the same blue looks lighter on a dark background and darker on a light one, making absolute comparisons unreliable.

The hierarchy is about decoding, not encoding. A designer's job is to encode data visually. The viewer's job is to decode it back into numbers. Cleveland-McGill measured the decoding side. A chart that is easy to create can be hard to read, and a chart that is hard to read is failing at its only job.

Automating the test

A human doing this test is the gold standard, but you cannot put a human in a CI pipeline. The goal for d3-power-tools is to approximate Cleveland-McGill's results computationally. Not by running experiments on every generated chart, but by inspecting the chart's encoding choices and predicting the expected accuracy.

The automated pipeline has four steps:

1
Render the chart. Produce the SVG or Canvas output from the D3 code.
↓
2
Identify encoding channels. Parse the SVG to detect what visual variables carry data: position, length, angle, area, color.
↓
3
Map channels to the hierarchy. Score each encoding by its Cleveland-McGill rank. Weight by how central the encoding is to the chart's primary comparison.
↓
4
Produce an accuracy score. Output a perceptual accuracy estimate (0 to 100) plus specific suggestions for encoding improvements.
The automated perceptual testing pipeline. Each step maps to an existing capability: SVG rendering (direction 8), DOM inspection (standard tooling), ranking lookup (a static table), and scoring (arithmetic).

Step 2 is the hard part. Detecting encoding channels from raw SVG means recognizing that a set of <rect> elements with varying heights sharing a y-baseline is a bar chart (encoding: position/length), while a set of <path> elements forming arc segments is a pie chart (encoding: angle). This is pattern recognition over the DOM, and it is where an LLM can help: feed the SVG to a vision model, ask "what encoding channels does this chart use for its primary comparison?", and parse the structured response.

Step 3 is a lookup table. Given that the chart uses angle encoding, the expected accuracy is rank 4 out of 7. Given that it uses position on a common scale, rank 1. The interesting case is a chart that uses multiple encodings: a bubble chart uses position (rank 1) for placement and area (rank 5) for magnitude. The score depends on what question the viewer is trying to answer. If the question is "which has the largest value?", the answer depends on area, so the score should reflect rank 5, not rank 1.

// Perceptual accuracy scoring (conceptual)
const hierarchy = {
  "position_common_scale": { rank: 1, accuracy: 0.95 },
  "length_aligned":         { rank: 2, accuracy: 0.90 },
  "length_unaligned":       { rank: 3, accuracy: 0.78 },
  "angle_slope":            { rank: 4, accuracy: 0.65 },
  "area":                   { rank: 5, accuracy: 0.55 },
  "volume":                 { rank: 6, accuracy: 0.40 },
  "color_saturation":       { rank: 7, accuracy: 0.30 },
};

function scoreChart(encodings, primaryQuestion) {
  // Find which encoding answers the primary question
  const primary = encodings.find(e => e.answersQuestion(primaryQuestion));
  return {
    score: Math.round(hierarchy[primary.channel].accuracy * 100),
    rank: hierarchy[primary.channel].rank,
    suggestion: primary.channel === "angle_slope"
      ? "Consider a bar chart for more accurate comparison"
      : null
  };
}

Integrity vs. accuracy: two different failures

The d3-power-tools project already has an encoding-integrity skill. Integrity asks: does the chart lie? A truncated y-axis, a dual axis with mismatched scales, an area encoding where the baseline is not zero: these are integrity failures. The chart misrepresents the data.

Perceptual accuracy asks a different question: does the chart communicate? A pie chart with seven slices does not lie. It faithfully represents the proportions. But viewers will struggle to compare the fourth-largest slice to the fifth-largest, because angle is a low-accuracy encoding for fine comparisons. The data is honest. The communication is poor.

These two dimensions are complementary. A chart can have perfect integrity and terrible accuracy (an honest pie chart with twelve slices). It can have good accuracy and terrible integrity (a bar chart with a truncated axis). The audit needs both.

Perceptual accuracy is the fifth audit dimension. The autoresearch audit pipeline currently runs four checks: visual_critic (does it look polished?), encoding_integrity (does it lie?), stress_test (does it break with edge-case data?), and cognitive_load (is it overwhelming?). Perceptual accuracy would be the fifth: does the encoding let viewers decode the data with precision?

The right encoding for the question

The hierarchy alone is not enough. A bar chart is not universally better than a pie chart. The encoding must match the question.

If the question is "compare these five values precisely," a bar chart on a common scale is the clear winner. Position encoding, rank 1. But if the question is "does category A represent roughly half the total?", a pie chart might communicate that more directly. The half-circle visual is immediately recognizable. A bar chart answering the same question would require the viewer to mentally sum all bars and compare one bar to that sum. The pie chart's angle encoding (rank 4) communicates the part-to-whole relationship faster than the bar chart's position encoding (rank 1) for this specific question.

This means the automated test cannot just look at the chart. It must know the question the chart is trying to answer. In the moonshine context, that question is usually the key insight of the explanation: the thing the author wants the reader to understand. The perceptual test would check whether the encoding the chart uses is the most accurate one available for communicating that specific insight.

// Question-aware scoring
const questionTypes = {
  "compare_values":     ["position_common_scale", "length_aligned"],
  "part_to_whole":      ["angle_slope", "length_aligned"],
  "distribution":       ["position_common_scale"],
  "correlation":        ["position_common_scale"],  // scatterplot
  "magnitude_relative": ["area", "length_aligned"],
  "spatial_pattern":    ["color_saturation"],     // heatmap is best here
};

// A heatmap scores poorly for "compare values" but well
// for "spatial pattern" -- context changes the answer

Notice the last entry. Color saturation sits at the bottom of the hierarchy for value comparison, but a geographic heatmap answering "where are the hotspots?" is using color saturation for exactly the task it is suited for: detecting spatial patterns, not reading precise values. The hierarchy is about precision of decoding. Some questions do not require precision.

What this looks like for moonshine

A moonshine explanation typically makes one core argument, supported by two to five interactive figures. Each figure exists to communicate a specific point. The perceptual test would run over each figure in an explanation and ask: is the encoding well-matched to the point this figure needs to make?

Consider an explanation about income inequality. Figure 1 is a line chart showing the Gini coefficient over time (position encoding, rank 1, question: trend comparison). That scores high. Figure 2 is a treemap showing budget allocation (area encoding, rank 5, question: compare proportions). That scores low. The automated test would flag Figure 2: "This treemap uses area encoding (rank 5) for a proportion comparison task. A stacked bar chart (rank 2) or a simple sorted bar chart (rank 1) would let readers compare these proportions more accurately."

The author can then make an informed choice. Maybe the treemap is the right call because it also shows the hierarchical nesting of budget categories, and that structure is the real insight. The perceptual test is a flag, not a veto. It tells you the cost you are paying in decoding accuracy and lets you decide if the tradeoff is worth it.

Connecting the audit dimensions

With perceptual accuracy as a fifth dimension, the autoresearch audit pipeline would cover five distinct failure modes:

Dimension Question Failure mode
visual_critic Does it look polished? Clipped labels, overlapping text, color clashes
encoding_integrity Does it lie? Truncated axis, misleading scale, broken area baseline
stress_test Does it break? Overflow with 100 categories, NaN values, empty data
cognitive_load Does it overwhelm? Too many series, cluttered legend, excessive annotation
perceptual_accuracy Does it communicate? Low-accuracy encoding for the chart's primary comparison
The five audit dimensions. Each catches a different way a chart can fail. A chart can pass four and fail one: a polished, honest, robust, simple pie chart with twelve nearly-equal slices still fails the perceptual accuracy check.

What can go wrong

Risks and open problems

  • The hierarchy is an average, not a law. Cleveland-McGill's results are population means. Individual variation is substantial. Some people read pie charts more accurately than others. Expertise matters: cartographers read color scales better than novices. The hierarchy predicts aggregate accuracy, not any single viewer's experience. Using it as a hard constraint would overfit to the average viewer at the expense of specialized audiences.
  • Context overrides theory. Pie charts are better for "roughly half." Heatmaps are better for spatial patterns. A geographic choropleth using color saturation (rank 7) is the right encoding for a map. Replacing it with a bar chart (rank 1) would destroy the spatial context. The hierarchy applies to isolated comparisons, not to charts embedded in a spatial or narrative framework. An automated test that ignores context will produce bad recommendations.
  • Perceptual accuracy is not communication effectiveness. A bare bar chart with no annotations scores high on perceptual accuracy: position encoding, rank 1. But a pie chart with a clear annotation saying "Marketing is nearly half the budget" might communicate that insight faster and more memorably than the bar chart. Annotation, labeling, and narrative framing can compensate for lower-accuracy encodings. Optimizing purely for perceptual accuracy penalizes explanatory design.
  • LLMs may over-optimize for the metric. If perceptual accuracy becomes an audit dimension, an LLM generating charts will learn to always pick bar charts and scatterplots, since those use position encoding (rank 1). The result: a monotonous visual vocabulary where every chart looks the same. Treemaps, bubble charts, heatmaps, and other rich encodings would disappear from the generated output, reducing the visual diversity that makes explanations engaging. The metric must be balanced against variety.
  • Detecting encodings from SVG is fragile. The DOM structure of a D3 chart does not label its encoding channels. Distinguishing a bar chart from a histogram from a Gantt chart requires heuristic pattern matching. A stacked bar and a grouped bar look similar in the DOM but use different encoding channels (unaligned length vs. aligned length). Misclassifying the encoding leads to wrong scores. This is where LLM vision models can help, but they introduce their own error rates.

What comes next

The first twenty directions in this series built the machinery of chart creation, delivery, and accessibility. None of them measured the output. Perceptual testing introduces measurement. Not subjective preference ("I like this chart"), not integrity checking ("this chart does not lie"), but empirical accuracy prediction: given what we know about the human visual system, how precisely can a viewer decode this chart's data?

That measurement is incomplete. It cannot account for context, annotation, familiarity, or narrative framing. It reduces a rich act of communication to a decoding accuracy estimate. But it is a number, and numbers can be tracked, compared across chart versions, and flagged when they drop below a threshold. For an automated system generating hundreds of charts, having any perceptual quality signal is better than having none.

Moonshine explanations exist to make data understandable. An explanation that picks the wrong encoding for its key insight is undermining its own purpose. The Cleveland-McGill hierarchy, forty years old and still replicable, tells us which encodings let people read data accurately. The work ahead is to turn that knowledge into an automated check that runs every time a chart is generated.
