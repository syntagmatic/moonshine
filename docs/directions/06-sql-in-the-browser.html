SQL in the Browser

DuckDB-WASM as a data backend for D3 visualizations. Query millions of rows with SQL, render only the aggregated results.

Call d3.rollup on ten thousand rows and it returns instantly. Call it on one million rows and the tab freezes for 200 milliseconds. During those 200 milliseconds the browser cannot respond to clicks, brushes, or scroll events. The frame budget is 16ms. You have overshot it by an order of magnitude. The user perceives the chart as broken.

This is the scaling wall. D3's data manipulation functions (d3.group, d3.rollup, d3.bin, d3.cross) are beautifully designed for in-memory JavaScript arrays of modest size. They compose well, read clearly, and integrate directly with D3's data-join pattern. But they operate on row-major JavaScript objects, which means every aggregation touches every field of every row, even the fields you do not need. At 100K rows this is merely slow. At 1M rows it stalls the main thread. At 10M rows it exhausts memory.

The direction explored here is: what if the data manipulation happened in a SQL engine compiled to WebAssembly, running in the browser, operating on columnar data? The chart would send a SQL query and receive back a small result set. D3 would render that result set. The heavy lifting would happen off the main thread, on data stored in a format optimized for aggregation.

That engine exists. It is called DuckDB-WASM.

The Query-Driven Rendering Pattern

The architecture has four stages, and each one does exactly one thing.

Parquet / CSV columnar, compressed
→
DuckDB-WASM SQL engine in WebAssembly
→
Result Set tens to hundreds of rows
→
D3 Render binddata, bindDOM, bindpixels

Figure 1. The query-driven rendering pipeline. Raw data stays in DuckDB. Only aggregated results cross into JavaScript.

Stage 1: Load. A Parquet file is fetched (or a CSV is loaded) into DuckDB-WASM's in-memory store. Parquet is ideal because it is columnar (the engine reads only the columns referenced by the query), compressed (a 500MB CSV becomes a 50MB Parquet file), and supports HTTP range requests (you can load a subset without downloading the whole file). The data never becomes a JavaScript array. It lives in WASM memory, in DuckDB's internal columnar format.

Stage 2: Query. When the user interacts with the chart (drags a brush, changes a filter, selects a category), the application constructs a SQL query and sends it to DuckDB. The query does all the grouping, aggregation, filtering, and sorting. A query like SELECT region, AVG(value) FROM data WHERE year BETWEEN 2010 AND 2020 GROUP BY region produces maybe 8 rows. It ran on 5 million.

Stage 3: Transfer. The result set crosses from WASM memory into JavaScript. Because it is small (tens to hundreds of rows), serialization is negligible. This is the critical bottleneck inversion: instead of transferring millions of rows into JavaScript and aggregating there, you aggregate in WASM and transfer only the result.

Stage 4: Render. D3 receives a small, flat array of objects and renders it the usual way. selection.data(results).join("rect") binds 8 objects to 8 bars. Scales, axes, transitions, tooltips, all of the things D3 excels at, work on this small result set with zero performance concern.

DuckDB-WASM does not replace D3. It replaces the data layer that feeds D3. Everything downstream of the result set (scales, joins, rendering, interaction) remains pure D3. The two libraries occupy different layers of the stack and do not compete.

// The pattern in code
const db = await duckdb.instantiate();
await db.registerFileURL("data.parquet", url);

async function queryAndRender(yearMin, yearMax) {
  const result = await db.query(`
    SELECT region, AVG(value) as avg_val, COUNT(*) as n
    FROM data
    WHERE year BETWEEN ${yearMin} AND ${yearMax}
    GROUP BY region
    ORDER BY avg_val DESC
  `);

  // result is ~8 rows. D3 binds and renders trivially.
  bars.data(result, d => d.region)
    .join("rect")
    .transition().duration(300)
    .attr("width", d => x(d.avg_val));
}

Cross-Filtering via SQL

Cross-filtering is the interaction pattern where brushing in one chart filters all the others. If you drag a time range in the line chart, the bar chart and the scatter plot update to show only data from that period. It is the signature interaction of exploratory dashboards.

The classic tool for this is crossfilter, written by Mike Bostock and Jason Davies in 2012. Crossfilter builds bitmap indexes over each dimension at load time. When a brush moves, it performs bitwise AND operations across those indexes to identify which rows pass all active filters. It is astonishingly fast: filtering 1M records takes under 10ms. But it pays for that speed with constraints. It supports only simple filter predicates (range on a single dimension). It cannot do joins, window functions, or arbitrary aggregations. And the bitmap indexes consume memory proportional to the dataset size.

DuckDB-WASM offers a different tradeoff. Cross-filtering becomes a set of SQL queries, each with a WHERE clause reflecting the active filters.

// Crossfilter approach: pre-indexed bitmaps
const cf = crossfilter(data);
const byYear = cf.dimension(d => d.year);
const byRegion = cf.dimension(d => d.region);
byYear.filterRange([2010, 2020]);
const results = byRegion.group().reduceSum(d => d.value).all();

// DuckDB approach: SQL queries with WHERE clauses
const results = await db.query(`
  SELECT region, SUM(value) as total
  FROM data
  WHERE year BETWEEN 2010 AND 2020
  GROUP BY region
`);

The SQL approach is 2-5x slower for simple range filters on pre-indexed data. Crossfilter's bitmap operations are hard to beat when the query fits their model. But the SQL approach handles cases that crossfilter cannot: joining two tables, computing a rolling average before filtering, using CASE expressions for conditional aggregation, or filtering on a subquery.

Crossfilter is a specialized engine for a specific interaction pattern. DuckDB is a general-purpose analytical engine that happens to support that pattern. The choice depends on whether your queries will stay simple or grow complex. For a moonshine explanation that starts with basic filters and evolves toward multi-table analysis, DuckDB's generality is the safer bet.

The Query Composition Pattern

The real power of SQL-backed cross-filtering is not any single WHERE clause. It is how multiple clauses compose. When the user brushes a time range in chart A, that interaction produces a clause: WHERE year BETWEEN 2012 AND 2018. When the user clicks a category in chart B, that produces a second clause: AND region = 'Northeast'. When the user adjusts a threshold slider in chart C, that produces a third: AND population > 50000. All three clauses are ANDed together into a single query. Each chart's interaction contributes one fragment. The query engine receives the composed whole.

This composition is algebraic. Each interaction maps to exactly one clause. Adding an interaction appends a clause. Removing it (clearing the brush, deselecting the category) removes the clause. The interaction state of the entire dashboard is fully described by the set of active clauses. There is no hidden state, no bitmap index that accumulates side effects across filter operations. The query is the state.

In code, this means maintaining a dictionary of active clauses keyed by the originating chart. When any chart's interaction changes, the system rebuilds the WHERE clause by joining all active entries with AND, constructs the full query, and sends it to DuckDB. Each chart then re-queries with the shared WHERE clause but its own GROUP BY and SELECT. The brush chart groups by time. The bar chart groups by region. The scatter plot groups by nothing (it wants raw rows from the filtered subset). Same WHERE, different projections.

// Clause dictionary: each chart owns one entry
const clauses = new Map();

// Brush in time chart sets a range clause
clauses.set("timeChart", `year BETWEEN ${y0} AND ${y1}`);

// Click in category chart sets an equality clause
clauses.set("categoryChart", `region = '${region}'`);

// Compose: join all active clauses with AND
const where = clauses.size
  ? `WHERE ` + [...clauses.values()].join(` AND `)
  : ``;

// Each chart queries with the shared WHERE but its own projection
const barResult = await db.query(
  `SELECT region, AVG(value) as avg FROM data ${where} GROUP BY region`
);
const lineResult = await db.query(
  `SELECT year, SUM(value) as total FROM data ${where} GROUP BY year`
);

This pattern scales cleanly to any number of linked charts. Adding a tenth chart means adding one more clause source. The composition logic does not change. And because each query is independent, they can be sent to DuckDB in parallel. On a multi-core machine, DuckDB will execute them concurrently, which means the wall-clock time for updating ten charts is not ten times the single-query time. It is closer to one or two times, depending on how many cores are available and how much data each query touches.

In practice, the latency difference matters most during brush-drag, when the user is continuously moving a handle and expects the other charts to update in real time. At 60fps the frame budget is 16ms. Crossfilter can comfortably filter and re-aggregate within that budget for datasets up to several million rows. DuckDB-WASM typically takes 10-50ms for the same operation on 1M rows, which means you feel a slight lag during fast dragging. The mitigation is straightforward: debounce the query (fire at most every 50ms during drag, then fire a final query on brush-end) and use progressive refinement (show a fast approximate result immediately, replace it with the exact result when the query completes).

Progressive Refinement

Progressive refinement is the strategy of running two queries in response to a single interaction: a fast approximate query that returns immediately, and a precise query that runs in the background and replaces the approximation when it finishes. The user sees an instant response (the approximation) and a refined response (the exact result) a moment later. If the approximation is close to the exact answer, the refinement is nearly invisible. If the two differ, the chart smoothly transitions from approximate to exact.

DuckDB supports this natively through its SAMPLE clause. A query like SELECT region, AVG(value) FROM data TABLESAMPLE 10 PERCENT WHERE year BETWEEN 2010 AND 2020 GROUP BY region touches one tenth of the rows and returns in one tenth of the time. The result is statistically noisy but structurally correct: the same columns, the same grouping, the same sort order. D3 can render it with the same code that renders the exact result. When the exact query completes, the bars shift slightly to their true positions. The user perceives responsiveness, not inaccuracy.

The implementation pattern has three moving parts. First, a query queue that tracks which query is in flight and which is pending. Second, a version counter that prevents stale results from overwriting fresh ones (if the user moves the brush again before the precise query returns, the stale precise result should be discarded). Third, a visual indicator that distinguishes approximate from exact results, something subtle like a slight desaturation or a thin pulsing border that resolves when the exact result arrives.

// Progressive refinement pattern
let queryVersion = 0;

async function progressiveQuery(where) {
  const version = ++queryVersion;

  // Phase 1: fast approximate result (10% sample)
  const approx = await db.query(`
    SELECT region, AVG(value) as avg_val
    FROM data TABLESAMPLE 10 PERCENT
    ${where}
    GROUP BY region ORDER BY avg_val DESC
  `);

  if (queryVersion !== version) return; // stale
  renderBars(approx, { approximate: true });

  // Phase 2: precise result (full scan, background)
  const exact = await db.query(`
    SELECT region, AVG(value) as avg_val
    FROM data
    ${where}
    GROUP BY region ORDER BY avg_val DESC
  `);

  if (queryVersion !== version) return; // stale
  renderBars(exact, { approximate: false });
}

The key subtlety is the version check. Between the approximate and exact queries, the user may have moved the brush again, which increments the version. If the exact result arrives after the version has changed, it belongs to an outdated interaction and must be thrown away. Without this guard, results from overlapping queries arrive out of order and the chart flickers between states that no longer correspond to the current brush position.

For datasets under 500K rows, progressive refinement is unnecessary. DuckDB returns the exact result fast enough. For datasets above 5M rows, it is essential: the exact query may take 100ms or more, and the user should not wait that long for visual feedback during a brush drag. The 500K-to-5M range is where the strategy transitions from "nice to have" to "required," and the crossover depends on query complexity. A simple GROUP BY on one column is faster than a multi-table JOIN with window functions.

Progressive refinement turns latency from a binary problem (fast enough or not fast enough) into a gradient. The approximate result sets a floor on perceived responsiveness. The exact result sets a ceiling on accuracy. As long as the floor is under 16ms and the ceiling arrives within a few hundred milliseconds, the user experiences the chart as both responsive and precise.

Choosing a Format: Parquet, CSV, and Arrow

Three formats compete for the role of "data delivery layer" in browser-based SQL analysis. Each makes a different tradeoff between file size, parse overhead, and runtime memory. The right choice depends on dataset size, query patterns, and how much complexity the author is willing to take on.

CSV: Simple and Universal

CSV is the lingua franca of tabular data, but it is a terrible format for large-scale browser analysis. It is row-major (you must parse every column even if you only need two), uncompressed (a 100-column dataset with 1M rows can easily be 500MB), and untyped (every value is a string that must be parsed on load). For small datasets, none of this matters. Fetching a 200KB CSV, parsing it with d3.csv, and passing the array to d3.rollup is the simplest possible pipeline. No WASM decoder, no special tooling, no build step. For moonshine explanations with under 10K rows, CSV remains the right default.

The trouble starts at scale. Parsing a 50MB CSV in the browser takes seconds. The parser must allocate a JavaScript string for every cell, then convert numeric strings to numbers. This creates millions of small objects that pressure the garbage collector. And because CSV is row-major, DuckDB-WASM cannot skip columns during ingest. It must read the entire file before it can execute the first query. There is no partial loading, no range-request trick. You download the whole thing or nothing.

Parquet: Small and Columnar

Parquet is the opposite of CSV. It is columnar: each column is stored separately, so a query that touches 3 of 100 columns reads only 3% of the data. It is compressed: Snappy or Zstandard compression typically achieves 3-10x reduction. It is typed: integers are stored as integers, not as strings that must be parsed. And it supports range requests: an HTTP server can serve individual column chunks without sending the whole file, which means DuckDB-WASM can begin querying before the full file has downloaded.

A 500MB CSV becomes a 50MB Parquet file. DuckDB-WASM can begin executing queries against it after loading just the metadata footer (a few kilobytes) and the relevant column chunks (a few megabytes). The user sees results while the rest of the file is still downloading.

The cost is complexity. Parquet is a binary format. You cannot open it in a text editor, cannot debug it with console.log, cannot generate it with a shell script. Producing a Parquet file requires a library (Python's pyarrow, DuckDB's COPY ... TO ... (FORMAT PARQUET), or a Node.js tool). And reading it in the browser requires a WASM decoder, which means the 10MB DuckDB-WASM binary must be loaded before the first byte of data can be interpreted. For large datasets this overhead is trivial compared to the savings. For small datasets it is the dominant cost.

// Loading a remote Parquet file
await db.registerFileURL(
  "census.parquet",
  "https://data.example.com/census-2020.parquet",
  duckdb.DuckDBDataProtocol.HTTP
);

// This query reads only the 'state' and 'population' columns
// from the Parquet file. The other 48 columns are never fetched.
const result = await db.query(`
  SELECT state, SUM(population) as pop
  FROM census
  GROUP BY state
  ORDER BY pop DESC
  LIMIT 10
`);

Parquet's columnar layout means the network transfer is proportional to query complexity, not dataset size. A query that aggregates two columns from a 100-column, 10M-row file transfers the same amount as a query on the same two columns from a 10K-row file (plus metadata overhead). This makes "big data in the browser" feasible without big bandwidth.

Arrow IPC: Zero-Copy but Bulky

Apache Arrow IPC (the .arrow file format) sits between CSV and Parquet. Like Parquet, it is columnar and typed. Unlike Parquet, it is not compressed. The data is stored in Arrow's in-memory columnar layout, which means there is no decoding step. DuckDB-WASM can register an Arrow buffer and begin querying immediately, with zero deserialization cost. The bytes on disk are the bytes in memory.

This zero-copy property makes Arrow the fastest format for time-to-first-query after the file has arrived. There is no parse step and no decompression step. The engine just maps the buffer and starts scanning. For workflows where the data is fetched once and queried many times (a dashboard that loads at startup and runs queries for the rest of the session), Arrow's front-loaded network cost is repaid by its zero per-query overhead.

But the tradeoff is file size. Because Arrow stores uncompressed columnar data, it is typically 2-5x larger than Parquet and only modestly smaller than CSV (sometimes larger, if the CSV has high string entropy). A dataset that is 50MB as Parquet might be 150MB as Arrow and 500MB as CSV. For browser delivery over a network, this size difference is often disqualifying. The user waits longer for the download, and the browser must hold the entire uncompressed buffer in memory.

Arrow makes the most sense when the data is already local (loaded from IndexedDB, generated by a previous computation, or transferred from a Web Worker), when network transfer is not the bottleneck, or when the dataset is small enough that the size difference is negligible. For remote data delivery to a browser, Parquet almost always wins.

The Decision Matrix

The choice reduces to three questions. How large is the dataset? How complex are the queries? And how much tooling is the author willing to set up?

Under 10K rows, use CSV. The simplicity is unbeatable and the performance is identical. From 10K to 100K rows, CSV still works but starts to feel slow on load. Parquet saves network time but requires a WASM decoder. The break-even point depends on column count: a 5-column dataset gains little from Parquet's columnar skipping, while a 100-column dataset gains enormously. Above 100K rows, Parquet is the clear winner for network delivery. Arrow enters the picture only for local-first workflows or when zero deserialization latency matters more than transfer size.

What D3 Still Does

Moving the data layer to DuckDB does not diminish D3. It sharpens D3's role. D3 stops being asked to do work it was not designed for (aggregating millions of rows) and focuses on what it does better than anything else: turning data into visual marks.

Scales and axes. d3.scaleLinear, d3.scaleBand, d3.scaleTime. These take the domain from the query result and map it to pixel coordinates. No change needed.

Data joins. selection.data(results).join("rect"). The join operates on 8 rows instead of 8 million. Enter, update, exit work the same way. Key functions work the same way.

Transitions. When the brush moves and a new query fires, the old bars transition smoothly to the new values. D3's transition system interpolates between the previous and current result sets. This is where the 10-50ms query latency helps, paradoxically: the transition can start immediately with the stale data and snap to the new data when the query completes, creating a naturally progressive update.

Interaction. Brushes, zooms, drag behaviors, voronoi overlays. These are all DOM-side concerns. DuckDB knows nothing about them. The brush fires an event; the event handler constructs a query; the query result feeds back into the render. D3 owns the full interaction loop.

Layouts. Force simulations, tree layouts, Sankey diagrams. These operate on the result set, not the raw data. A Sankey diagram of trade flows between 20 countries does not need 10M individual transaction records. It needs the aggregated flow between each pair. DuckDB computes the aggregation; D3 computes the layout.

Why This Matters for Explanations

Moonshine explanations are self-contained HTML files. They carry their data with them or fetch it on load. This self-containment is a core design principle: no server, no API, no build step. Just a file.

This principle has, until now, imposed a practical ceiling on dataset size. An explanation about U.S. census demographics cannot include 330 million rows of individual-level data in an inline JSON array. So the explanation author pre-aggregates: they compute state-level summaries, age-group distributions, and income brackets offline, embed those summaries as small JSON objects, and the visualization renders the summaries. The reader sees the author's pre-chosen aggregations. They cannot re-aggregate, re-slice, or ask questions the author did not anticipate.

DuckDB-WASM changes this. The explanation can reference a Parquet file (hosted on a CDN, fetched via range requests) and run arbitrary SQL queries against it in the browser. The reader is no longer limited to the author's pre-computed views. They can filter, group, and aggregate on any dimension. The explanation becomes a guided exploration, not a fixed presentation.

Consider a moonshine explanation of climate data. The author provides a narrative about global temperature trends. But the Parquet file contains station-level data for 50,000 weather stations across 100 years. The reader can brush a region on the map and see temperature trends for just those stations. They can group by elevation. They can compare coastal and inland stations. All of this happens in the browser, with no server, because DuckDB-WASM is doing the heavy lifting.

This is the leap from "explanation as presentation" to "explanation as instrument." The author provides the dataset, the initial view, and the narrative scaffolding. The reader provides the questions.

The 10K-Row Threshold

Not every explanation needs DuckDB. Most do not. If your dataset has 500 rows, d3.rollup is the right tool. If it has 5,000 rows, d3.rollup is still the right tool. The overhead of loading a 10MB WASM binary, initializing it (2-3 seconds of cold-start time), and writing SQL instead of JavaScript is not justified by a dataset that JavaScript can process in under a millisecond.

The threshold is roughly 10,000 rows for simple aggregations and lower for complex ones (joins, window functions, multi-pass aggregations). Below this threshold, the DuckDB overhead dominates. Above it, the DuckDB advantage compounds with every additional row.

This creates a clean decision rule for moonshine authors: if your data fits comfortably in a JSON array and d3.rollup runs in under 10ms, stay with pure D3. If your data is large enough that you are pre-aggregating it offline to avoid shipping millions of rows, consider shipping the raw Parquet file and letting DuckDB-WASM aggregate on demand.

Risks and Open Questions

  • Bundle size. DuckDB-WASM is roughly 10MB compressed. For an explanation that otherwise weighs 50KB, this is a 200x increase in payload. The WASM file can be cached aggressively (it is the same binary for every explanation), but the first load is heavy. A CDN-hosted shared copy helps. So does lazy-loading: do not fetch the WASM until the user interacts with a component that needs it.
  • Cold-start time. Instantiating the WASM module takes 2-3 seconds on a typical laptop. During this time, the explanation should show a loading state or render an initial view from pre-embedded summary data. The transition from static summary to live queryable data should be seamless, not a blank screen followed by a pop.
  • SQL injection. If the explanation exposes a query builder or text input that incorporates user text into SQL strings, it must use parameterized queries. Template-literal string concatenation of user input into SQL is the same vulnerability as in server-side code, even though DuckDB-WASM runs locally and cannot reach a production database. A malicious query could still crash the tab or exfiltrate data loaded in the same DuckDB instance.
  • Impedance mismatch. SQL returns flat tabular results. D3 often wants hierarchical or nested data (for treemaps, nested groups, force graphs). Bridging this gap requires post-processing the SQL result on the JavaScript side, which partly defeats the purpose of offloading computation. Window functions and nested JSON aggregation in DuckDB help, but they add SQL complexity.
  • LLM confusion about layers. When an LLM generates code for a DuckDB+D3 pipeline, it may conflate the layers: writing d3.rollup on the raw data (bypassing DuckDB), or trying to use SQL for rendering logic. The skill system needs clear guidance about which layer owns which responsibility. Data transformation belongs in SQL. DOM manipulation belongs in D3. The boundary is the result set.
  • Offline and mobile. WASM performance varies across devices. A query that takes 20ms on a laptop may take 200ms on a low-end Android phone. Mobile browsers also have tighter memory limits. An explanation that loads a 500MB Parquet file into DuckDB on a phone with 2GB of RAM will crash. Progressive loading and dataset-size caps are essential for mobile-friendly explanations.
