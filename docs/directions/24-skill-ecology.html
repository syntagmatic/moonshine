Future Directions

Skill Ecology

The 37 skills in d3-power-tools are not independent modules. They form an ecology of dependencies, compositions, and conflicts. Understanding their relationships is necessary for building anything that uses more than one.

Skills are not independent

It is tempting to think of the 37 skills in d3-power-tools the way you think of tools in a drawer: self-contained, independent, each doing one thing. You reach in, grab the one you need, put it back when you are done. But the drawer metaphor breaks down as soon as you try to use two tools at once.

Some skills depend on others. Hierarchy-interaction cannot function without hierarchy-layouts, because you cannot add expand/collapse behavior to a tree that has not been laid out as a tree. Choreography depends on motion, because sequenced transitions require motion primitives. Linked-views depends on coordination, because synchronized panels need an event bus to talk to each other. These are explicit dependencies, the kind a package manager would resolve.

Other dependencies are implicit. Any interactive visualization needs responsive, because an interactive chart on a narrow screen is not really interactive. Any chart with categorical encoding needs color, because the default palette will be unreadable. These are not declared anywhere. They are learned through experience, through building things that break on phones or confuse colorblind readers.

And then there are conflicts. Force-layout and hierarchy-layouts both want to position nodes. Color and distributions both want to own the color channel. Motion and choreography both want to control animation timing. When two skills fight for the same resource, one of them wins and the other produces artifacts. The visualization may render, but it will not communicate.

The skills form an ecology, not a toolkit. They have relationships: mutualism (brushing and linked-views make each other more useful), competition (force and treemap fight for the same territory), and parasitism (a poorly calibrated skill can degrade the output while consuming context). Understanding these relationships is prerequisite to composition.

Three kinds of relationships

Dependencies are directed. Skill A requires skill B. The arrow points from the dependent to its requirement. When you load A, you must also load B, and B's dependencies in turn. This is the familiar model from package managers. The dependency graph for d3-power-tools is shallow (no dependency chain is longer than two steps) but it is not trivial. Loading hierarchy-interaction pulls in hierarchy-layouts. Loading choreography pulls in motion. Loading linked-views pulls in coordination. Each transitive dependency adds tokens to the context without directly serving the author's intent.

Compositions are bidirectional. Skills A and B compose well together, each making the other more effective. Brushing and linked-views compose naturally: brushing selects a region in one view, and linked-views propagates that selection to other views. Choreography and annotation compose into narrative: choreography sequences the transitions, annotation explains what each transition reveals. These compositions are the productive heart of the ecology. They are where emergent quality comes from.

Conflicts are mutual. Skills A and B both claim the same resource. Force-layout and hierarchy-layouts both set node positions. If both activate, the positions are incoherent. Color and distributions both assign meaning to the color channel. If both activate, the legend contradicts itself. Conflicts are harder to detect than dependencies because they are implicit: neither skill declares "I own the color channel." Instead, both quietly assign fills, and the second one overwrites the first.

Dependency graphs

The explicit dependency graph for d3-power-tools is sparse. Of the 37 skills, most have no dependencies at all. The dependency chains that do exist are short:

  • linked-views requires coordination
  • hierarchy-interaction requires hierarchy-layouts
  • choreography requires motion
  • canvas-accessibility requires canvas
  • edge-bundling requires network

The implicit dependencies are more numerous and more subtle. The responsive skill is a dependency of virtually every interactive visualization, but it is never declared as one. D3-idioms is a dependency of every skill that generates code, but no skill lists it. Scales underpins every quantitative encoding but is never explicitly required. These "invisible" dependencies mean that the declared graph underrepresents the actual structure by a significant margin.

Implicit dependencies matter for the composition engine (direction 12). When the engine selects skills based on declared dependencies, it misses the invisible ones. A chart generated without responsive will work on the author's monitor and break on everyone else's phone. A chart generated without d3-idioms will work but produce code that is hard to read and modify. The ecology model must capture both explicit and implicit dependencies to be useful.

Conflict detection

Conflicts cluster around a small number of resource types. In practice, four resources account for most conflicts:

Node positioning. Force-layout, hierarchy-layouts, and network all assign x/y coordinates to nodes. Only one can win. Treemap is a special case: it depends on hierarchy-layouts for its input structure but overrides the positions. Two positioning skills activated simultaneously produce a chart where nodes flicker between two layouts or settle at an incoherent average.

Color encoding. Color and distributions both want to assign semantic meaning to the fill channel. Color uses hue to encode categories. Distributions uses color to encode density or quantile bands. If both activate, the same blue might mean "category A" in one region and "high density" in another. The legend cannot explain both meanings.

Animation timing. Motion provides primitive transitions. Choreography sequences those transitions into stages. If both are invoked independently (rather than choreography delegating to motion), their timelines collide. Elements start moving before the previous stage completes, or two transitions overwrite each other on the same element.

Axis ownership. Time-series and distributions both want to define the x-axis. Time-series uses time. Distributions uses a quantitative measure. On the same chart, both cannot own x. In linked-views they can coexist (one per panel), but within a single panel they conflict.

// A conflict detection sketch
const resources = {
  "node-position": ["force", "hierarchy-layouts", "network"],
  "color-channel":  ["color", "distributions"],
  "animation":      ["motion", "choreography"],
  "x-axis":         ["time-series", "distributions"]
};

function detectConflicts(selectedSkills) {
  const found = [];
  for (const [resource, claimants] of Object.entries(resources)) {
    const active = claimants.filter(s => selectedSkills.includes(s));
    if (active.length > 1)
      found.push({ resource, skills: active });
  }
  return found;
}

Composition patterns

If conflicts are the destructive interactions, compositions are the constructive ones. Certain skill combinations produce results that exceed what either skill delivers alone. These are emergent properties: the combination has qualities that neither component has independently.

The most reliable composition patterns map to the archetypes in the jig-template skill:

Dashboard. Brushing + linked-views + coordination + responsive. The coordination skill provides the event bus. Linked-views wires the panels. Brushing gives the user a way to ask questions. Responsive ensures the layout works at any width. Together, they produce a multi-panel interactive explorer. None of the four skills produces this result alone.

Narrative. Choreography + annotation + scales + motion. Motion provides the transitions. Choreography sequences them into stages. Annotation explains each stage. Scales ensure the quantitative mappings are correct. The result is a scrollytelling piece that reveals a story through sequenced visual states. This composition has a dependency chain (choreography requires motion), which means the composition engine must resolve that chain when assembling the skill set.

Explorer. Brushing + parallel-coordinates + distributions + coordination. Parallel coordinates shows all dimensions. Brushing lets the viewer select ranges on any axis. Distributions shows the shape of the selected subset. Coordination keeps them synchronized. This combination turns a wide dataset into an interactive investigation tool.

Structural. Hierarchy-layouts + hierarchy-interaction + annotation + cognitive-load. The layout positions nodes. Interaction adds expand/collapse. Annotation labels the important subtrees. Cognitive-load prevents the tree from overwhelming the viewer. The last skill is the one most often omitted and most often needed.

The best compositions share a structural feature: each skill in the set has a distinct role. One handles layout, one handles interaction, one handles explanation, one handles resilience. When two skills in a composition have the same role, they are more likely to conflict than to compose. The composition engine's job is to assemble sets where every skill occupies a unique niche.

Keystone skills

In ecology, a keystone species is one whose impact on the ecosystem is disproportionate to its abundance. Remove it and the whole system reorganizes. D3-power-tools has keystone skills with the same property.

d3-idioms improves the quality of every block it touches without being domain-specific. It does not encode data or position elements. It shapes how code is written: idiomatic selections, proper enter/update/exit patterns, clean margin conventions. Its effect is diffuse but pervasive. Blocks built with d3-idioms active are more maintainable, more readable, and more likely to work correctly on the first attempt. It is the least visible skill and arguably the most important one.

responsive is the other keystone. Like d3-idioms, it is domain-agnostic. It does not care whether the chart is a treemap or a scatter plot. It ensures the chart fits its container, responds to resize events, and does not break on narrow viewports. Without it, every other skill's output is fragile: beautiful on the author's screen, broken everywhere else.

scales occupies a similar position for quantitative encoding. Every chart that maps data to visual properties goes through a scale. Getting the domain, range, and type right is the foundation that annotation, color, and interaction build on. A misconfigured scale propagates errors through every downstream skill.

Keystone skills share a trait: they are not exciting. Nobody picks up d3-power-tools because they want to learn about responsive design or code conventions. They come for force-directed graphs and choreographed transitions. But the keystones are what make the exciting skills work reliably. The ecology depends on them.

Skill health metrics

If the ecology is a system, it needs health metrics. Five measurements capture most of what matters:

Token cost. Each skill consumes context tokens when loaded. The 37 skills range from roughly 150 tokens (sparkcharts) to 500 tokens (choreography). Token cost is not just a budget constraint. It is an attention constraint. Every token of skill guidance competes with the task description and the data for the model's attention. A skill that costs 400 tokens and contributes little is not just expensive. It is actively harmful because it dilutes the context.

Activation frequency. How often does the skill appear in the 107-block collection? Annotation and scales activate in over half the blocks. Edge-bundling and WebGL activate in fewer than five. Activation frequency is not a quality signal (rare skills may be excellent for their niche), but it does indicate which skills get the most real-world testing and calibration.

Keep rate. In the autoresearch iterate pipeline, skills are proposed for each generation. Some get kept in the final output; others get dropped during iteration because they did not improve quality or introduced problems. Keep rate measures the fraction of proposals where the skill survives to the final block. A skill with a high keep rate is reliably useful. A skill with a low keep rate may be poorly calibrated, too expensive, or too niche for the tasks it is being proposed for.

Composite impact. When a skill is present, how does it affect the composite quality score compared to when it is absent? This requires controlled comparison: two otherwise-identical blocks, one with the skill and one without. The difference in composite score is the skill's marginal impact. Keystone skills have high marginal impact. Niche skills have high marginal impact within their domain and near-zero impact outside it.

Conflict frequency. How often does the skill participate in a detected conflict? Skills with high conflict frequency are not bad skills. They are skills that occupy contested territory. Force-layout has high conflict frequency because node positioning is a contested resource. That does not mean force-layout is poorly designed. It means the ecology around node positioning is competitive.

The feedback loop

The autoresearch pipeline described in directions 21 through 23 measures quality, identifies which skills contribute, and informs which skills to compose next. The ecology is not a static map. It is the feedback loop itself.

The cycle works like this. The iterate pipeline proposes a set of skills for a new block. The block is generated and scored on the composite dimensions (visual-critic, encoding-integrity, cognitive-load, stress-test). The scores are attributed back to the active skills: which ones correlated with high scores, which with low? Over many iterations, each skill accumulates a performance profile. Skills that consistently correlate with high scores in their domain get proposed more often. Skills that consistently correlate with low scores get flagged for recalibration or sharpen-tool compression.

This loop connects four of the prior directions. Perceptual testing (direction 21) provides one of the quality signals. Explanation effectiveness (direction 22) provides another. Cross-model benchmarks (direction 23) test whether the ecology's performance generalizes across models or is overfit to one. And the composition engine (direction 12) uses the ecology's structure to assemble skill sets for each new block.

The ecology is what makes the pipeline adaptive rather than static. Without it, the pipeline would propose skills based on keyword matching ("the prompt mentions hierarchy, load hierarchy-layouts"). With it, the pipeline reasons about relationships: "hierarchy-layouts composes well with annotation but conflicts with force. The task involves a hierarchy. Load hierarchy-layouts and annotation. Do not load force."

Evolution

Ecologies change over time. Species evolve, go extinct, invade from neighboring systems. The skill ecology evolves too.

When a skill is sharpened (compressed via the sharpen-tool), its token cost drops, which changes its cost-benefit ratio and may change which compositions it participates in. A skill that was too expensive to include in a five-skill set may become affordable after sharpening, opening up compositions that were previously over budget.

When a skill is calibrated (via the calibrate-tool), its guidance changes, which may resolve conflicts that existed before calibration. A conflict between motion and choreography might arise because the two skills gave contradictory advice about easing functions. Calibrating them to use consistent defaults resolves the conflict without changing the ecology's structure.

New skills may emerge. If the block collection reveals a pattern that no existing skill covers well, a new skill may be written to capture it. That new skill enters the ecology with its own dependencies, compositions, and potential conflicts. It changes the landscape for every existing skill it interacts with.

Old skills may become obsolete. If two skills that used to serve different purposes converge after repeated sharpening, one may be merged into the other. The 37 skills are not a fixed number. The ecology is a living system.

Connection to moonshine

Moonshine's premise is that explanations built from real data stories communicate better than charts built from technique showcases. The skill ecology determines what moonshine can produce. It is, in a concrete sense, moonshine's capability surface.

When moonshine builds an explanation, it navigates the ecology. It selects skills that serve the story, resolves their dependencies, checks for conflicts, and assembles a composition. The quality of that navigation determines the quality of the output. A well-understood ecology produces clean compositions. A poorly understood one produces charts where skills fight each other and the result is incoherent.

This is why the ecology is the capstone direction. The other 23 directions are proposals: things d3-power-tools could do, ways moonshine could expand. This direction is about the infrastructure that holds all of them together. The skill ecology is the map of moonshine's current capabilities and the guide to extending them. Every new direction that proposes a new technique (collaborative visualization, sonification, tactile output) is, at bottom, a proposal to add a new node to this graph. Understanding the graph is understanding what moonshine is.

Risks and tensions

  • The ecology is descriptive, not prescriptive. Knowing that brushing and linked-views compose well does not tell you whether to use them for a given task. The ecology captures relationships but not judgment. The danger is treating the ecology as a decision procedure ("the graph says these compose, so use them") rather than as a map that informs judgment. Maps show terrain. They do not choose the route.
  • Map-territory confusion. The force-directed graph above is a model of skill behavior. It is not skill behavior itself. The node positions are determined by a force simulation, not by any ground truth about how skills relate. Two nodes that happen to land near each other may not have a meaningful relationship. The visual proximity is an artifact of the layout algorithm. Treating it as a finding is a classic visualization trap.
  • Maintenance burden. Every time a skill is added, removed, sharpened, or recalibrated, the ecology model must be updated. Dependency and conflict data must be kept current. If the model falls out of date, it becomes worse than useless: it becomes misleading. The cost of maintaining 37 nodes and their relationships is manageable. The cost of maintaining 70 or 100 nodes may not be.
  • The temptation to formalize. A graph of skill relationships invites ontology-building: formal type systems for resources, category hierarchies, composition algebras. These are intellectually satisfying and practically counterproductive. Heuristic composition ("these usually work together") is faster, cheaper, and often more accurate than formal composition ("these satisfy the type constraints for joint activation"). Over-engineering the ecology into a formal ontology costs effort that would be better spent building blocks.

Tying the landscape together

This is direction 24 of 24. The series opened with collaborative visualization and narrative grammars, proposing new forms of explanation. It moved through data infrastructure (streaming, SQL, serverless), intelligence (AI annotation, adaptive complexity, composition engines), accessibility (sonification, tactile, cognitive), platforms (components, notebooks, embeddable), and evaluation (perceptual testing, effectiveness, benchmarks). Each direction proposed something new. This direction proposes understanding what already exists.

The 37 skills are the atoms. The ecology is the chemistry. You can list the elements without understanding how they combine, but you cannot build molecules that way. The skill ecology is the periodic table for d3-power-tools: not a new capability, but a framework for understanding all the existing ones and predicting which new combinations are possible.

The practical implication is straightforward. Before adding direction 25 (whatever it may be), check the ecology. What skills does it depend on? What does it conflict with? What does it compose into? If those questions have good answers, the direction has a place in the system. If they do not, the direction may be interesting in isolation but difficult to integrate. The ecology is the integration test.
