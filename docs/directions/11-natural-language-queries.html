Future Directions

Natural Language Queries

Instead of building controls for every possible interaction, let the viewer type a question. The LLM translates natural language into visualization operations.

Pre-scripted answers to anticipated questions

Every interactive chart is a set of answers to questions the author anticipated. The dropdown that filters by region answers "What does this look like for Europe?" The brush that selects a date range answers "What happened between 2015 and 2020?" The toggle that switches encodings answers "What if I see this as a bar chart instead?"

The author thought of these questions, built the controls, wired the interactions. If the viewer has a question the author did not anticipate, the chart has no answer. It just sits there, static and unresponsive, holding data that could address the question if only someone would ask it the right way.

This is the fundamental limitation of pre-built interactivity. Every slider, checkbox, and dropdown is a specific answer to a specific question. The viewer can only explore along the axes the author provided. Anything off that grid is invisible.

Natural language changes the interface contract. Instead of clicking a control that the author built, the viewer types a question in their own words. An LLM interprets the intent, maps it to a visualization operation, and D3 executes it. The viewer's vocabulary, not the author's control palette, defines the space of possible interactions.

The translation pipeline

The path from a typed sentence to a chart update has four stages. Each one introduces constraints, and the design of the system depends on how those constraints are managed.

Stage 1: The query. The viewer types "Show me the outliers" or "Compare Alpha and Beta" or "What changed after 2020?" This is unconstrained natural language. It might be precise ("Filter to category Alpha") or vague ("Show me the interesting ones"). It might reference data dimensions that exist in the chart, or dimensions that do not.

Stage 2: Intent recognition. The LLM receives the query along with chart context: what dimensions exist, what encodings are active, what the data range is, what the current view state is. This context is critical. "Show me the big ones" means nothing without knowing that "big" could refer to the x-axis value, the y-axis value, the point size, or the category count. The LLM uses chart metadata to disambiguate.

Stage 3: Operation mapping. The LLM returns a structured operation object, not a natural language response. This is the key architectural decision. The model does not generate D3 code. It does not write JavaScript. It returns a JSON spec like { "op": "highlight", "filter": "zscore > 2" } that the chart's operation handler knows how to execute. The operation vocabulary is finite and pre-defined. The LLM selects from it; it does not invent new operations.

Stage 4: Execution. D3 receives the operation and animates the transition. Points fade, a trend line appears, the layout shifts. This step is ordinary D3 work, no LLM involved. The animation provides feedback that the query was understood.

The vocabulary of operations

A natural language chart interface does not need to understand everything. It needs a finite set of operations that cover the most useful analytical questions. The vocabulary is smaller than you might expect.

OperationWhat it doesExample query
filterShow only matching points, dim or hide the rest"Show only Alpha and Beta"
highlightVisually emphasize matching points without hiding others"Highlight the outliers"
sortRearrange points by a dimension"Sort by value"
groupCluster points by a categorical dimension"Group by category"
aggregateCompute and display a summary statistic"What is the average?"
annotateAdd a visual element (trend line, reference line, label)"Draw a trend line"
zoomFocus on a data subset by adjusting axes"Zoom into the top right"
change-encodingSwitch the visual mapping (color, size, position)"Color by category"
resetReturn to the original state"Reset" or "Start over"

Nine operations. That is not a lot. But consider how many analytical questions they cover. "Which category has the highest average?" is an aggregate plus a sort. "How do Alpha and Beta differ?" is a filter. "Is there a trend?" is an annotate. The power comes not from the number of operations but from the natural language layer that maps unbounded human questions onto this bounded set.

Ambiguity and context

The hardest problem in this pipeline is not parsing English. LLMs can parse English just fine. The hard problem is resolving ambiguity when the query is underspecified relative to the chart's data.

"Show me the big ones." Big by which dimension? If the chart has x, y, and size encodings, there are at least three plausible interpretations. The LLM needs to know which dimension the viewer likely means, and that requires context about the chart's purpose, the current view state, and the conventions of the domain.

The context payload sent to the LLM alongside the query should include:

Schema. The field names, types, and ranges. The model needs to know that revenue is a number from 0 to 500 and region is a category with four values.

Encodings. Which fields map to which visual channels. If revenue is on the y-axis and headcount is on the x-axis, "the big ones" probably refers to high revenue (the y-axis is the more salient dimension in a scatter plot).

Current state. What filters are active, what is highlighted, whether a trend line is showing. "Remove that" only makes sense if the model knows what "that" refers to, which means it needs to know the most recent operation.

Conversation history. If the previous query was "Show me Alpha," then "Now compare it to Beta" means "filter to Alpha and Beta." Without history, "it" has no referent.

This context window is not large. A typical chart might need 200 to 500 tokens of metadata. But it must be accurate. Stale metadata (claiming a filter is active when it has been reset) will cause the model to misinterpret follow-up queries. The chart must update its context payload after every operation.

Latency and the feel of conversation

An LLM round-trip takes one to three seconds. For a chat conversation, that is acceptable. For a chart interaction, it is a different matter.

Click a dropdown and the chart updates in 16 milliseconds. Type a query and wait two seconds for the response. The interaction model shifts from direct manipulation (I move, it moves) to conversation (I ask, it thinks, it answers). This is not inherently worse, but it is categorically different, and mixing the two models in one interface creates dissonance.

The practical solution is to separate the two modes. Direct manipulation (hover, brush, click) stays instantaneous and does not involve the LLM. Natural language queries go through the pipeline and accept the latency. The viewer understands the difference because they chose to type instead of click. Typing implies a question; questions take time to answer.

Pre-computation helps at the margins. If the chart author knows that "Show me the outliers" is a likely query, they can pre-compute the outlier set and cache the operation mapping. When the viewer types that exact query (or something close enough), the response is instant. This hybrid approach, cached answers for common queries and LLM fallback for uncommon ones, keeps the most frequent interactions fast while preserving the open-endedness of natural language.

A dashboard pre-builds ten interactions and makes them instant. An NL interface pre-caches ten operations for speed and sends the rest to the LLM. The dashboard has hard limits on what the viewer can ask. The NL interface has soft limits: cached queries are fast, novel queries are slower but still possible.

How this differs from a dashboard

A dashboard with dropdowns, sliders, and radio buttons is a carefully designed set of pathways through the data. The author decides which dimensions are filterable, which values can be compared, which views are available. The viewer navigates these pre-built pathways. The dashboard is an authored experience.

An NL query interface is something else. The viewer can ask a question the author never considered. "Are there any points where x is high but y is low?" is a perfectly valid analytical question that might not correspond to any pre-built control. The NL layer can parse it, construct a filter, and show the result. The viewer is no longer limited to the author's imagination.

This is powerful but also dangerous. The authored experience has been tested. The author knows what every combination of controls produces and has verified that the resulting views are sensible. An NL query can produce views that nobody has tested: a filter that returns zero points, a comparison between categories that have nothing in common, a trend line fitted to three data points. The open-endedness that makes NL queries valuable also makes them unpredictable.

The moonshine connection

Moonshine explanations are linear narratives with interactive figures that support the argument. The author controls the pace: first you see this, then you see that, then you interact with this to build intuition for this claim. The explanation has a direction.

NL queries add a sideways dimension. At any point in the narrative, the reader can pause and interrogate the data. "Wait, what about the outliers? I want to see those before I accept your claim about the trend." The explanation answers back, not with a pre-scripted response but with the data itself, filtered and highlighted to address the reader's specific doubt.

The best use of NL queries in an explanation is not to replace the narrative but to let the reader stress-test it. The author says "the trend is clear." The reader types "Show me the outliers" and sees whether they undermine or support the claim. The explanation becomes a conversation rather than a lecture.

For LLM-generated moonshine explanations, NL queries create an interesting feedback loop. The same model that generated the explanation can field questions about it. It knows the data, the narrative structure, and the claims being made. A question like "Why did you choose this encoding?" could be answered not with a chart operation but with a paragraph of reasoning. The explanation becomes interrogable.

The operation spec

The structured object returned by the LLM looks like this:

// Example: "Show me the outliers"
{
  "op": "highlight",
  "filter": {
    "type": "zscore",
    "field": "y",
    "threshold": 2
  },
  "description": "Highlighting points with y-values > 2 standard deviations from the mean"
}

// Example: "Compare Alpha and Beta"
{
  "op": "filter",
  "categories": ["Alpha", "Beta"],
  "dim_others": true,
  "description": "Filtering to Alpha and Beta, dimming other categories"
}

The description field is for the viewer, not the chart. It appears in the translation panel so the viewer understands what the model decided to do. This transparency is essential: if the model misinterprets the query, the viewer can see the mistake in the description and rephrase.

The operation vocabulary is closed. The LLM must select from a pre-defined set of operation types. It cannot invent new operations. It cannot return arbitrary JavaScript. This constraint is what makes the system safe. The worst the model can do is select the wrong operation from a set of operations that are all individually correct and tested.

Risks and open questions

This direction has real problems

  • Hallucinated operations. The LLM might construct a filter referencing a field that does not exist, or claim to highlight "outliers in the time dimension" when there is no time dimension. Validation at the operation layer catches some of these, but semantically plausible but factually wrong operations are harder to detect. "Filtering to category Epsilon" sounds reasonable but fails silently if the data only has Alpha through Delta.
  • The uncanny valley of almost-understanding. A system that handles "Show me the outliers" perfectly but fails on "Which points are unusual?" feels broken in a way that a system with no NL interface does not. Partial understanding is worse than no understanding, because the viewer cannot predict which queries will work. The expectation set is impossible to manage.
  • Latency breaking flow. Two seconds of latency is tolerable for a single query. But analytical exploration involves rapid-fire questions: "Show me outliers. Now filter to Alpha. What is the trend? Reset." If each step costs two seconds, the eight-second cumulative delay shatters the feeling of direct exploration. Pre-caching helps for common queries but cannot cover the combinatorial space of follow-up chains.
  • Replacing designed interactions with a chat box. For guided explanations, a well-designed set of controls is better than an open text field. The controls communicate what is possible. They constrain the viewer toward productive interactions. An NL interface says "ask me anything" and then fails on most things. For explanations with a specific pedagogical goal, pre-built interactions are more reliable than open-ended ones.
  • Privacy. If the NL query goes to a remote API, the viewer's analytical questions are visible to the API provider. "Show me the fraud cases" or "Which employees are underperforming?" are queries that reveal intent. A local model avoids this, but local models are currently too large for browser deployment. The privacy implications of sending analytical queries to a third-party API are serious and underexplored.

These risks do not invalidate the direction, but they constrain where it is appropriate. NL queries work best as a supplement to designed interactions, not a replacement for them. The controls stay. The NL bar sits beside them. The viewer can use either. The controls handle the common, pre-tested pathways. The NL bar handles the questions the author did not anticipate. Together, they cover more of the viewer's curiosity than either could alone.
