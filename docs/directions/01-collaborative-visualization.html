Collaborative Visualization

What if multiple people could co-author a D3 explanation at the same time? Shared state, cursor presence, and conflict resolution for interactive documents.

Building an interactive explanation is solitary work. You choose the dataset, bind it to shapes, layer in interactions, write the prose. When it is finished, you share a link. Your reader can scroll, hover, drag a slider. But they are alone in it. They cannot point at a cluster of points and say "what about these?" and have you see exactly what they see. The artifact is a finished thing. It does not invite joint exploration.

This is a loss. The best conversations about data happen when two people are looking at the same chart, pointing at the same feature, and adjusting parameters together. In a meeting room, you get this for free: you both see the same screen. Remotely, you do not. You share a screenshot, annotate it in Slack, and the other person opens the link and tries to recreate what you were looking at. Half the context evaporates.

The direction explored here is: what would it take to make D3 explanations collaborative? Not collaborative in the "edit the code together" sense of a shared notebook, but collaborative in the "explore the finished explanation together" sense. Two people see the same chart, with the same parameters, with real-time awareness of where the other person is looking and what they are adjusting.

The goal is not Google Docs for writing code. It is Google Maps for looking at data together. The explanation already exists. Collaboration means sharing the act of reading and exploring it.

What State Is Shareable?

A D3 visualization has layers of state. Some are inherently shareable. Others are inherently local. Getting this boundary right is the central design problem of collaborative visualization.

Shareable state is anything you can serialize to JSON and have the other side reconstruct. The data array. Scale domains and ranges. Parameter values (slider positions, toggle states, selected categories). The set of active filters. Annotation positions and text. Brush extents. These are plain values. They travel over a wire trivially.

Local state is anything tied to the rendering frame or the viewer's physical context. The current hover target. The animation frame's interpolation progress (sending 60 partial values per second across a network is wasteful and visually jarring). Scroll position, unless the explanation is specifically designed for synchronized scrolling. Tooltip positioning, because it depends on the local viewport dimensions.

// Shareable: plain values, easy to serialize
const sharedState = {
  data: [...],
  pointSize: 5,
  colorScheme: "viridis",
  brushExtent: [[120, 40], [380, 260]],
  filters: { category: ["A", "B"] }
};

// Local: tied to rendering or viewport
const localState = {
  hoveredPoint: null,          // flickers too fast to share
  animationT: 0.73,           // mid-transition interpolation
  scrollY: 1240,               // viewport-dependent
  tooltipPos: { x: 312, y: 87 } // screen coordinates
};

D3 selections themselves cannot be shared. A selection is a reference to live DOM elements. It has no serializable form. But you can share the specification that produced the selection: the data, the join key function, the filter predicate. The remote side replays the specification against its own DOM to arrive at the equivalent selection.

Think of shared state as the recipe, not the meal. You send the ingredients and instructions (data, parameters, filter predicates). Each client cooks locally (binds data, updates DOM, runs transitions). The result looks the same, but each side owns its own DOM.

Authoring State vs Viewer State

There are two fundamentally different collaboration modes, and they require different architectures.

Viewer collaboration is the simpler one. The explanation is finished. Multiple readers explore it together. The shared state is small: parameter values, brush extents, selected items, perhaps scroll position. The state space is bounded because the author defined every possible parameter. This is achievable today with modest engineering.

Authoring collaboration is the hard one. Multiple people are building the visualization simultaneously. One person is editing the data transform. Another is adjusting scale parameters. A third is writing the prose annotation. The shared state is the code and configuration itself. This is the territory of collaborative editors (VS Code Live Share, Google Docs), and it is orders of magnitude harder because the state space is unbounded. Any line of code might conflict with any other.

The 90/10 rule applies here. Viewer collaboration covers 90% of what people actually want ("look at this together, explore together, point at things together") for perhaps 10% of the implementation effort. Authoring collaboration is the remaining 10% of value for 90% of the work. For interactive explanations, viewer collaboration is the right target.

The Network Layer: Extending d3.dispatch

D3 already has the right abstraction for event routing. d3.dispatch is a named event bus with namespaced listeners. Charts communicate through it without coupling. The leap to collaborative visualization is small in concept: make dispatch.call() send the event both locally and over a WebSocket channel.

// A collaborative dispatch wrapper
function collaborativeDispatch(ws, ...types) {
  const local = d3.dispatch(...types);
  const myId = crypto.randomUUID();

  // Incoming messages from collaborators
  ws.addEventListener("message", e => {
    const { type, args, source } = JSON.parse(e.data);
    if (source === myId) return; // ignore own echoes
    local.call(type, null, ...args);
  });

  return {
    on: (type, fn) => local.on(type, fn),
    call: (type, that, ...args) => {
      local.call(type, that, ...args); // fire locally
      ws.send(JSON.stringify({        // broadcast remotely
        type, args, source: myId
      }));
    }
  };
}

The source guard (if (source === myId) return) prevents feedback loops: when the server echoes your own message back, you skip it because you already applied it locally. This is the same pattern used for preventing dispatch feedback loops between charts, elevated to a network context.

The harder question is what to do about latency. D3 transitions are frame-synchronized. If Alice starts a 500ms transition at time T, and Bob receives the message at T+80ms, Bob's transition starts 80ms late. For slow transitions this is invisible. For snappy 150ms interactions, the visual mismatch is noticeable. The solution is to send the transition's start time and duration, then have each client calculate how far along the transition should be when the message arrives. This is clock synchronization, and it gets complicated fast.

Conflict Resolution: CRDT vs OT

When two users change the same parameter simultaneously, you have a conflict. The resolution strategy depends on the nature of the state being modified.

Last-writer-wins (LWW) is the simplest CRDT strategy. Each state update carries a timestamp. When two updates arrive for the same key, the one with the later timestamp wins. This is surprisingly adequate for visualization parameters. If Alice sets the point size to 8 and Bob sets it to 4, one of them wins. There is no meaningful "merge" of two point sizes. LWW is the right tool because the state is a single scalar, not a composable structure.

Operational Transformation (OT) is needed when the state is ordered and composable. Consider two users editing an SVG path's d attribute simultaneously, or reordering items in a legend. OT tracks the position-aware operations (insert at index 3, delete at index 7) and transforms them against each other so both sides converge. This is what Google Docs uses for text editing. For visualization authoring it is necessary; for viewer collaboration it is almost never needed.

Under LWW, one change wins based on timestamp. Under a merge strategy, the system must decide how to combine two incompatible color palettes.

Cursor Presence

Of all the features in collaborative tools, cursor presence has the highest impact-to-effort ratio. Seeing another person's cursor on the chart changes the entire experience. You know where they are looking. You know they are paying attention. When they hover over a data point, you can see it highlighted on your screen too. This is the foundation of shared attention, and it requires almost no state: just an (x, y) position, a user name, and a color.

The position should be sent in data coordinates, not pixel coordinates. If Alice is on a 4K monitor and Bob is on a laptop, pixel coordinates will be wrong. But if Alice's cursor is at data point (income: 45000, life_expectancy: 72.3), Bob's client can project that through his own scales and place the cursor correctly regardless of his viewport size.

// Send cursor position in data space, not pixel space
svg.on("mousemove", event => {
  const [px, py] = d3.pointer(event);
  dispatch.call("cursor", null, {
    user: myName,
    x: xScale.invert(px),  // data coordinates
    y: yScale.invert(py),
    color: myColor
  });
});

Cursor updates are high-frequency (every mouse move) but low-priority. Dropping frames is fine. Throttling to 15-20 updates per second is sufficient for smooth awareness without saturating the channel. And unlike parameter changes, cursor positions never conflict: each user owns their own cursor. No CRDT needed.

Why This Matters for Explanations

A moonshine explanation is designed to be read actively. The reader adjusts parameters, brushes data, triggers transitions. But active reading is still a private act. You and I can both read the same explanation, but we are reading separate copies. My brush extent is invisible to you. My parameter settings vanish when I close the tab.

Collaborative exploration changes the nature of the artifact. The explanation stops being a document and starts being a workspace. A teacher walks a student through a scatter plot: "drag this slider to the right. See how the cluster splits? That is the effect of controlling for income." The student does it, and the teacher sees them do it, in real time, on the same chart. This is qualitatively different from writing instructions in prose and hoping the reader follows them.

It also opens a path to a different kind of authoring. Instead of one person deciding every parameter range and annotation, a group could explore a dataset together and build the explanation incrementally. One person notices an outlier and drops an annotation. Another adjusts the color scale to reveal a pattern. The explanation emerges from the collective exploration. This is the collaborative equivalent of a whiteboard session, except the whiteboard is an interactive D3 document.

Tradeoffs

Collaborative visualization adds infrastructure. A WebSocket server, state synchronization logic, cursor broadcast, conflict resolution. For an explanation that will be read by one person at a time, this is pure overhead. The honest question is: how often does anyone actually explore an explanation with someone else in real time?

The answer depends on context. For published explanations on the open web, almost never. Readers arrive asynchronously. They do not coordinate to read at the same time. For classroom teaching, frequently. The teacher presents the explanation and wants students to see the same thing simultaneously. For data analysis teams reviewing a visualization together, very frequently. The "let me share my screen" ritual exists precisely because there is no native sharing for interactive tools.

The pragmatic path is to build the parameter-sharing layer and nothing else. A shared URL with real-time parameter synchronization covers the vast majority of "look at this together" use cases. Cursor presence is a nice addition. Full authoring collaboration is a separate, much larger project that should not block the simpler version.

The 90/10 rule: sharing parameter values and brush extents covers 90% of the collaborative value for roughly 10% of the implementation complexity. Do that first. Do the rest only if demand proves it necessary.

Risks and Open Questions

  • Operational complexity. A WebSocket server must be provisioned, scaled, and kept alive. For self-contained HTML explanations that work as static files, this is a category change in deployment requirements.
  • Latency sensitivity. D3 transitions are tuned for local responsiveness. A 100ms network round-trip can make coordinated transitions feel sluggish or visually out of sync, particularly for brush-linked charts where both views should update simultaneously.
  • State explosion. An explanation with 15 parameters, each independently adjustable by N users, creates a combinatorial state space. Debugging "why does Bob see something different from Alice" becomes a synchronization puzzle.
  • Cold start problem. When a new user joins mid-session, they need the full current state, not just the stream of changes. This requires either a server-side state snapshot or a peer-to-peer state transfer protocol.
  • Privacy. Cursor position reveals reading behavior. Users may not want collaborators to know they have been staring at the same data point for thirty seconds. Presence should be opt-in.

The Latency Spectrum

There are two architectural extremes for distributed state. Server-authoritative means the server holds the single source of truth. Every change goes to the server first, the server validates and orders it, then broadcasts the canonical state to all clients. This is simple to reason about (one source of truth) but adds a full round-trip to every interaction. If the server is 50ms away, every slider drag has 50ms of latency before the change is reflected.

Local-first with sync means each client applies changes immediately to its own state, then sends the change to peers. This is optimistic: you see your own changes instantly, and conflicts are resolved after the fact. CRDTs make this work because they guarantee convergence regardless of message ordering. The cost is occasional visual "jumps" when a remote change arrives that conflicts with a local one.

For visualization parameters, local-first is the right choice. The user drags a slider and sees the chart update immediately. The remote user sees it 80ms later. Nobody notices. The parameter is a single scalar, so LWW convergence is trivially correct. Server-authoritative makes more sense for authoring state (code edits, annotation text) where conflict resolution needs to be precise and ordered.
