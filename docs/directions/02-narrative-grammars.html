Narrative Grammars

What if scroll-driven stories had a spec, not just a script?

Every scrollytelling project reinvents the same machinery. You wire up an IntersectionObserver. You build a state machine for stages. You write enter and exit transitions by hand. You wrestle with position: sticky on mobile Safari. You debug backward-scroll edge cases at 11pm. Then the next project starts, and you do it all again.

The code differs each time, but the structure rarely does. Most scroll-driven narratives follow the same pattern: a sequence of named stages, each with elements that enter on arrival and exit on departure. The chart stays pinned while text scrolls past. Stages advance forward and reverse backward. Occasionally a parameter interpolates continuously with scroll position, but usually the transitions are discrete.

This is a specification problem, not a code problem. The narrative structure is simple enough to declare. What if you could write a small JSON document describing your stages, transitions, and bindings, and a runtime would handle the scroll mechanics, the sticky positioning, the bidirectional state management, and the mobile quirks?

The idea is not new. Vega-Lite proved that constraining a grammar to the common case unlocks enormous leverage: you trade full generality for a tenfold reduction in specification complexity. A narrative grammar would apply the same bet to scrollytelling. Nail the 90% of narratives that are linear stage sequences. Provide escape hatches for the rest.

The grammar

A narrative grammar needs four primitives. Stages are the named states of the visualization. Transitions describe how visual elements change when entering or exiting a stage. Bindings map scroll position to parameters, either discretely (stage triggers) or continuously (interpolation). Constraints express rules like reversibility, duration limits, and dependency ordering.

A minimal spec looks like this:

{
  "stages": [
    { "id": "empty", "enter": { "axes": "draw" } },
    { "id": "data",  "enter": { "points": "stagger-in" },
                     "exit":  { "points": "fade-out" } },
    { "id": "color", "enter": { "points.fill": "category" },
                     "exit":  { "points.fill": "gray" } },
    { "id": "trend", "enter": { "trendline": "draw" },
                     "exit":  { "trendline": "erase" } },
    { "id": "annotate", "enter": { "callouts": "fade-in" },
                        "exit":  { "callouts": "fade-out" } }
  ]
}

Listing 1. A five-stage narrative spec. Each stage names the visual elements that change and the transition type for each. The runtime resolves "stagger-in", "draw", "fade-in" to concrete D3 transition implementations.

The spec says nothing about IntersectionObservers, sticky positioning, timing functions, or SVG structure. Those are runtime concerns. The spec says only: here are my stages, and here is what changes at each one. This separation is the whole point. The same spec could be rendered as a scroll-driven narrative, a click-through slideshow, an auto-playing animation, or a static PDF with one panel per stage.

Stages as first-class objects

A stage is a named snapshot of the visualization's state. It has an id (used for navigation and URL fragments), an enter object describing transitions to run when arriving at this stage, and an optional exit object describing transitions to run when leaving. The stage does not contain the full visual specification; it contains only the delta from the previous stage.

This delta model is natural for narratives because each stage builds on what came before. But it creates a problem for random access: if the reader jumps from Stage 1 to Stage 4, the runtime must replay Stages 2 and 3 to reach the correct state. A smarter approach is to compile the deltas into absolute states at parse time. The runtime precomputes what every element should look like at every stage, so jumping to any stage is instant. Enter/exit transitions are visual sugar applied on top of the resolved state, not the source of truth for what the state is.

Transition verbs

The grammar needs a vocabulary of transition types. Some candidates: draw (stroke-dashoffset animation for paths), fade-in/fade-out (opacity), stagger-in/stagger-out (sequential entrance of collection elements), morph (shape interpolation), slide-x/slide-y (translate from offscreen). Each verb maps to a concrete D3 transition implementation in the runtime. Custom verbs can be registered for project-specific effects.

The tricky part is pairing enter and exit transitions. If entering a stage means "stagger-in the points," exiting should mean "stagger-out the points" or at least "fade-out the points." If the author forgets to specify an exit transition, the runtime needs a sensible default. One option: the default exit for any enter transition is its visual inverse. fade-in defaults to fade-out. draw defaults to erase. stagger-in defaults to stagger-out. This makes specs terser without losing reversibility.

Bindings: discrete vs. continuous

Most narrative stages are discrete: you are either at the stage or you are not. The IntersectionObserver fires, and the runtime transitions to the new state. But some narratives need continuous bindings, where a parameter changes smoothly as the reader scrolls. A map zoom level that tracks scroll position. A histogram bin count that increases as the reader scrolls through a methods section. A rotation angle for a 3D object.

The grammar handles this with an optional bind key on a stage:

{
  "id": "zoom-in",
  "bind": {
    "scrollProgress": {
      "target": "map.zoom",
      "range": [1, 12],
      "ease": "cubicInOut"
    }
  }
}

Listing 2. A continuous binding. As the reader scrolls through this stage's region, the map zoom interpolates from 1 to 12 with cubic-in-out easing.

The runtime maps the scroll position within the stage's viewport region to a 0-to-1 progress value, applies the easing function, and interpolates the target parameter across the specified range. This is the same math that d3.interpolate does for transitions, but driven by scroll position instead of time.

How existing tools compare

This is not a vacuum. Several tools already address parts of the scrollytelling problem. Understanding where they stop is useful for understanding where a grammar would start.

Scrollama is the most widely used scrollytelling library, and it does one thing well: detecting when elements scroll into view. It wraps IntersectionObserver with sensible defaults and emits "stepEnter" and "stepExit" events. But it is purely a detection layer. You still write every transition, every state machine, every sticky layout by hand. It is the scroll equivalent of an event emitter with no framework on top.

Idyll goes further. It is a markup language that compiles to a React application, with built-in scroll detection, reactive variables, and a component library. You can write a scrollytelling piece in Idyll with far less code than raw D3. The cost is that you are now in Idyll's world: React as a dependency, custom compilation, limited escape hatches when you need something Idyll does not support. The abstraction is thick.

Reveal.js solves a related but different problem: sequenced presentations. Its stage model (slides with fragments) is clean, and reversibility is built in. But it assumes a presentation context, not a reading context. There is no scroll; there are arrow keys. The transition vocabulary is CSS-oriented rather than data-visualization-oriented.

Observable notebooks offer reactivity through viewof and cell dependencies, which is powerful for exploration but lacks narrative structure. There is no concept of "the reader is at stage 3" in a notebook. Cells react to inputs, not to reading position. You can build scrollytelling in Observable, but you are back to wiring IntersectionObservers manually.

The gap is clear: no existing tool provides a declarative stage specification with built-in transition vocabulary, first-class reversibility, and portability across rendering targets. That is the space a narrative grammar would fill.

Reversibility as a first-class concept

Most scrollytelling implementations treat backward scrolling as an afterthought. The forward path gets all the attention: elements enter with carefully tuned stagger delays and easing curves. The backward path, if it works at all, is usually the forward transitions played in reverse, which looks wrong. A fade-in played backward is a fade-out, but a stagger-in played backward is not a stagger-out. It is a stagger-in from the last element to the first, which reads as the data un-entering from the wrong end.

A grammar makes reversibility explicit by requiring both enter and exit transitions. The author specifies how things appear and how they disappear. The runtime does not guess. When the spec says "exit": {"points": "fade-out"}, the runtime knows exactly what to do when scrolling backward past the "data" stage.

Figure 3. The stage machine as a state diagram. Forward transitions (blue arrows) run enter handlers. Backward transitions (amber arrows) run exit handlers. Each stage is a node with declared forward and backward paths. No state is unreachable.

The state machine view reveals something important: the forward and backward paths through a linear narrative are mirror images. Stage N's enter is Stage N's backward-exit. Stage N's exit is Stage N-1's forward-enter precondition. This symmetry means the spec is complete: every possible scroll direction at every stage has a defined behavior. There are no undefined transitions.

A more ambitious grammar could support non-linear narratives, where stages branch. A reader who scrolls past a fork might enter Branch A or Branch B depending on a previous choice. The state machine becomes a directed graph instead of a sequence. The spec would need a when clause: enter this stage only when a condition is met. This is expressible, but it pushes the grammar toward a full programming language, which is exactly the trap to avoid. The 90% case is linear. Branch support is an escape hatch, not the core.

The 90% case

Vega-Lite succeeded because it chose its constraints wisely. It does not try to express every possible visualization. It expresses the visualizations that data analysts actually build: bar charts, scatter plots, line charts, heatmaps, histograms, layered and faceted compositions. If your chart fits within those idioms, Vega-Lite is vastly more productive than writing D3 by hand. If it does not, you drop down to Vega or D3.

A narrative grammar should make the same bet. The 90% of scroll-driven narratives that actually get built share this structure:

A linear sequence of 3 to 8 stages. One sticky graphic that changes at each stage. Discrete stage transitions triggered by scroll position. Forward enter and backward exit animations. Responsive layout that collapses to single-column on mobile. Accessible: works with keyboard navigation, respects prefers-reduced-motion.

If the grammar nails this case, it covers most of the New York Times graphics desk, most of The Pudding, most data journalism scrollytelling, and nearly all of the scrollytelling that moonshine's /shine workflow produces. The remaining 10% involves continuous scroll bindings, branching narratives, synchronized multi-panel layouts, and bespoke interaction patterns. Those are the escape hatch cases where the author drops down to imperative D3.

Portability

Because the spec is declarative, the same narrative can render in multiple modes. A scroll-driven renderer uses IntersectionObserver and sticky positioning. A click-through renderer uses Back/Next buttons and a stage counter. An auto-play renderer steps through stages on a timer. A static renderer produces one image per stage for print or PDF. A screencast renderer records the auto-play as a video.

This is not hypothetical flexibility. It solves real problems. A scroll-driven narrative that works on a 1440px desktop monitor breaks on a 375px phone screen where there is not enough vertical space for the scroll regions. The same spec rendered as click-through works on any screen size. A conference presenter wants to step through the narrative with arrow keys, not by scrolling. Same spec, different renderer.

The portability argument extends to tooling. An editor could visualize the spec as a timeline or a storyboard. A linter could check for missing exit transitions, unreachable stages, or excessively long transition chains. A diff tool could compare two versions of a spec structurally, not as text. None of this is possible when the narrative is expressed as imperative JavaScript.

The moonshine connection

This is where the direction connects to d3-power-tools specifically. Today, when the /shine workflow generates an interactive explanation, it outputs a self-contained HTML file with inline JavaScript. The narrative structure is embedded in the code: stage functions, an IntersectionObserver setup, a goToStage dispatcher. The code is correct and readable, but it is coupled to a specific rendering approach and difficult to modify structurally.

If /shine instead generated a narrative spec, the output would be a JSON document describing stages, transitions, and bindings. A runtime (included as a small library or inlined) would interpret the spec and produce the scroll-driven narrative. The author could then modify the narrative structure by editing JSON, not by refactoring JavaScript. Add a stage: add an object to the array. Reorder stages: reorder the array. Change a transition: change a string. The imperative code that implements each transition verb lives in the runtime, not in the output.

The generated spec is the explanation's source of truth. The runtime is a commodity. This inverts the current model, where the generated JavaScript is both the specification and the implementation, and editing one risks breaking the other.

There is a deeper benefit for the evaluate-iterate loop. The evaluator could inspect the narrative spec directly: check that stages are sequenced logically, that enter and exit transitions are paired, that the narrative arc has a clear buildup and payoff. Today, evaluating narrative structure requires parsing the generated JavaScript, which is fragile. A spec makes it trivial.

Tradeoffs

Every abstraction has a cost. A narrative grammar is no exception.

Abstraction vs. control. The grammar constrains what you can express. If your narrative needs a transition that is not in the vocabulary, you either extend the runtime (adding a custom verb) or abandon the grammar for that section. The Vega-Lite lesson is instructive: the grammar works beautifully when your intent aligns with its idioms, and it fights you when it does not. The "uncanny valley" of almost-right generated output is real and frustrating. A narrative that is 95% correct but has one transition that feels slightly off is worse than a hand-coded narrative that is 100% right.

Runtime size. A grammar interpreter adds weight. The transition vocabulary, the scroll detection, the responsive layout engine, the state machine, the binding resolver: these add up. A raw Scrollama setup is 1.5 KB gzipped. A narrative grammar runtime might be 8-15 KB gzipped. For a single explanation, this is negligible. For embedding tiny microexplanations, it might be too much. The grammar would need a tree-shaking strategy: include only the transition verbs that the spec actually uses.

Debugging. When a hand-coded transition misbehaves, you set a breakpoint in the stage function and step through. When a spec-driven transition misbehaves, you are debugging the interpreter, which is a layer removed from your intent. The grammar needs excellent error messages ("Stage 'color' references element 'points.fill' but no element named 'points' exists") and a visual debugger that shows the compiled state at each stage.

Amber risk: premature standardization

  • Premature grammar. Standardizing a narrative grammar before enough scrollytelling pieces exist to reveal the true design space risks encoding the wrong abstractions. The 90% case is a hypothesis. It might be 70%. It might be 95%. Building a dozen diverse narratives by hand first, then extracting the common structure, is safer than designing the grammar speculatively.
  • The abstraction tax. Custom transitions that do not fit the vocabulary require escape hatches, and escape hatches accumulate. If 30% of transitions in practice are custom, the grammar is not saving much over imperative code, and you pay the complexity cost of both the grammar and the escape hatch mechanism.
  • The uncanny valley. AI-generated narrative specs that are almost right but subtly wrong in their sequencing, pacing, or transition choices may be harder to fix than AI-generated imperative code. With code, you can edit any line. With a spec, you can only express what the grammar allows. If the fix requires a transition the grammar does not support, you are stuck.
  • Adoption risk. Vega-Lite succeeded partly because academic and data-journalism communities adopted it. A narrative grammar without an equivalent community would be a private language, useful only within moonshine. That limits feedback, testing, and the evolutionary pressure that makes grammars good.

What would it take

Building a real narrative grammar is a multi-year project. The steps, roughly: (1) Build 20 more scrollytelling explanations by hand, cataloging every stage pattern, transition type, and scroll binding that appears. (2) Extract the common vocabulary. (3) Design a spec format and write a reference interpreter. (4) Port 5 existing hand-coded narratives to the spec and see what breaks. (5) Iterate on the grammar until the ported versions are indistinguishable from the originals. (6) Build the /shine integration: train the proposer to output specs instead of code. (7) Build the tooling: visual editor, linter, differ, multi-target renderers.

The risk at step 2 is that the vocabulary is larger than expected. Scrollytelling authors are creative, and "creative" is the enemy of "standardizable." The risk at step 6 is that LLMs are worse at generating constrained specs than at generating freeform JavaScript, because the spec requires understanding the grammar's constraints. The risk at step 7 is that the tooling never gets built, and the spec format becomes write-only.

But the upside is real. A working narrative grammar would make scroll-driven explanations as composable and portable as charts became after Vega-Lite. It would make the moonshine evaluate-iterate loop faster, because the evaluator could reason about narrative structure declaratively. And it would let authors who are not JavaScript developers build interactive explanations by editing a spec, which is the entire point of tools like d3-power-tools.

The interpreter above is a toy. Five stages, one chart, a handful of transition verbs. But it demonstrates the core idea: a spec drives a state machine, the state machine drives transitions, and the transitions run forward and backward without any imperative wiring. That idea scales. The question is whether the vocabulary scales with it.
