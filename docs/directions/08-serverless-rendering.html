Serverless Rendering

Generating D3 visualizations server-side for thumbnails, social cards, PDF export, and email embeds. The chart that nobody ever sees in a browser.

The invisible chart problem

You spend hours building an interactive D3 visualization. Transitions ease in. Tooltips follow the cursor. A brush lets the reader filter by date range. You deploy it. Then someone shares the link on Twitter, and the preview card shows... nothing. A blank rectangle, or worse, your site's generic favicon stretched to 1200x630 pixels.

Social media crawlers do not run JavaScript. Neither do email clients. Neither does a PDF renderer. Neither does the Slack link unfurler, or the search engine generating a thumbnail for its results page, or the screen reader that falls back to an <img> tag when JavaScript is unavailable. All of these consumers need a static image of your chart. The chart they need is one that was never rendered in a browser window that any human looked at.

This is the serverless rendering problem: how do you take a D3 visualization that was built for the browser and produce a static image or PDF from it, automatically, on a server or in a CI pipeline, without a human clicking "Export"?

The answer matters for moonshine directly. Every explanation in this collection is an interactive HTML file. If we want a gallery page with thumbnails, social sharing cards, printable PDF versions, or email-friendly digests, we need a way to render each explanation to an image. The d3-power-tools testing pipeline already uses Playwright for screenshot-based visual regression tests (see scripts/test-viz.py). The rendering infrastructure is half-built. The question is how to extend it from testing into production asset generation.

Three ways to render a chart outside a browser

Every approach to server-side rendering involves the same fundamental task: execute the JavaScript that builds the SVG (or Canvas), then capture the result as an image or a serialized string. They differ in how much of the browser they simulate, and that difference determines their speed, accuracy, and complexity.

1. Playwright / Puppeteer: the real browser

The most straightforward approach: spin up a headless Chromium instance, navigate to the HTML file, wait for the chart to render, take a screenshot. This is what d3-power-tools already does for visual regression tests. Playwright launches a real browser with a real layout engine, real CSS cascade, real font rendering, and real Canvas 2D context. Every pixel matches what the user would see.

// Playwright screenshot pipeline
const browser = await chromium.launch();
const page = await browser.newPage();
await page.setViewportSize({ width: 1200, height: 630 });
await page.goto(`file://${htmlPath}`);
await page.waitForSelector('svg .rendered');
await page.screenshot({ path: 'card.png' });

The cost is weight. Chromium is a 200+ MB binary. Launching a browser process takes 1-3 seconds. Navigating and rendering a page takes another 0.5-2 seconds depending on complexity. You also need to solve the wait-for problem: how does the script know that the chart has finished rendering? Network idle is unreliable (what if the chart animates after the data loads?). A fixed timeout wastes time on fast charts and truncates slow ones. The most robust approach is a done signal: the chart sets a flag or adds a CSS class when it has finished its initial render, and the screenshot script waits for that specific selector.

2. JSDOM + D3: the fake DOM

JSDOM is a pure-JavaScript implementation of the web platform. It provides document.createElement, querySelector, attribute getters and setters, and enough of the DOM API that D3's selection machinery works. You run D3 in Node.js with JSDOM as the document, and D3 happily creates SVG elements without knowing it is not in a browser. Then you serialize the resulting SVG to a string and convert it to PNG with a rasterizer like sharp or resvg.

// JSDOM + D3 in Node.js
const { JSDOM } = require('jsdom');
const d3 = require('d3');
const dom = new JSDOM('<body></body>');
const svg = d3.select(dom.window.document.body)
  .append('svg')
  .attr('width', 800).attr('height', 400);
// ... binddata, bindscales, bindaxes ...
const svgString = dom.window.document.body.innerHTML;
// rasterize with resvg or sharp

This is fast. There is no browser to launch, no Chromium binary to install, no process overhead. A simple bar chart can render and serialize in under 200ms. But the fake DOM is a lossy simulation. JSDOM does not compute CSS layouts. There is no getComputedStyle that returns real values. There is no Canvas 2D context (you need a separate native addon like canvas for that, and font rendering will differ from the browser). Web fonts do not load. getBBox() returns zeros. Any D3 code that relies on computed geometry, such as measuring text width to position labels, will produce wrong results in JSDOM.

3. Satori + resvg: the JSX shortcut

Vercel's Satori takes a JSX tree (or a compatible object tree) and converts it directly to an SVG string, using its own layout engine inspired by the CSS flexbox spec. You do not write D3 code at all. Instead you describe the image as nested components, and Satori handles the layout and text measurement. Resvg then rasterizes the SVG to PNG.

// Satori approach (conceptual)
const svg = await satori(
  <div style={{ display: 'flex', flexDirection: 'column' }}>
    <h1>Monthly Revenue</h1>
    <Chart data={data} />
  </div>,
  { width: 1200, height: 630 }
);

The speed is remarkable: under 50ms for a typical card. But Satori supports only a subset of CSS (no grid, no position:absolute relative to arbitrary ancestors, limited text wrapping). It is designed for Open Graph cards and similar constrained layouts, not for the full generality of a D3 visualization. You would need to translate your D3 chart into Satori's idiom, which in practice means rewriting the rendering logic. For simple bar charts and summary cards this is viable. For complex interactive explanations with axes, annotations, and data-driven paths, it is not.

The practical split. Use Playwright when accuracy is non-negotiable (visual regression tests, production social cards, PDF export of complex layouts). Use JSDOM + D3 when the chart is SVG-only and does not depend on computed styles (simple generated charts, CI thumbnails, bulk batch rendering). Use Satori when you are building a branded card template that wraps a simplified version of the chart, not the chart itself.

The wait-for problem

The hardest part of screenshotting a D3 chart is not launching the browser or framing the viewport. It is knowing when the chart is ready. D3 visualizations do not fire a "done" event. They render through a chain of asynchronous operations: data loads, transitions animate, fonts download, images decode. The screenshot script must wait for all of these to complete, but "all of these" is not a well-defined set.

There are four common strategies, listed from least to most reliable:

Fixed timeout. Wait 3 seconds, then screenshot. This works for demos but wastes time on fast charts and truncates slow ones. In a CI pipeline rendering 100 charts, a 3-second timeout per chart adds 5 minutes of dead wait time.

Network idle. Playwright's waitUntil: 'networkidle' waits until no network requests have fired for 500ms. This catches data fetches and font downloads, but misses post-load transitions and requestAnimationFrame-driven animations. A chart that loads data instantly but animates bars into position over 1 second will be screenshotted mid-animation.

Selector-based. Wait for a specific DOM element to appear: page.waitForSelector('.chart-ready'). The chart code adds this class after its initial render completes. This is reliable but requires the chart author to add the signal. It is opt-in, and charts that forget the signal hang the pipeline.

Custom signal. The chart calls window.__renderComplete = true or dispatches a custom event, and the screenshot script polls for it. This is the most flexible: it works across transitions, async data loads, and conditional rendering. The d3-power-tools test pipeline uses a variant of this, where the test harness injects a small script that listens for a specific CSS class on the root SVG element.

// In the chart code: signal completion
function render(data) {
  svg.selectAll('rect')
    .data(data)
    .join('rect')
    /* bindattributes... */
    .transition().duration(600)
    .attr('height', d => yScale(d.value))
    .on('end', function() {
      // Signal after the last bar finishes transitioning
      svg.classed('render-complete', true);
    });
}

// In the screenshot script: wait for the signal
await page.waitForSelector('svg.render-complete', { timeout: 10000 });

Responsive rendering: one chart, many sizes

A social card is 1200x630. A thumbnail is 300x200. A PDF page is 595x842 points (A4). A mobile screenshot is 375x667. Each requires different information density, different font sizes, and sometimes entirely different chart types. A scatter plot with 500 labeled points works at 1200 pixels wide. At 300 pixels wide, it is an illegible blob.

The cleanest way to handle this is size-aware rendering: the chart code reads its container dimensions and adapts its output. D3 charts that use viewBox and percentage-based sizing already scale geometrically, but geometric scaling is not the same as responsive design. Scaling a 1200-pixel chart down to 300 pixels makes the text unreadably small. A responsive chart would instead reduce the number of axis ticks, hide secondary annotations, increase relative font sizes, and simplify the geometry.

// Size-aware rendering decisions
const width = container.getBoundingClientRect().width;
const isThumbnail = width < 400;
const isSocial = width >= 1000 && width <= 1300;

const tickCount = isThumbnail ? 3 : isSocial ? 6 : 10;
const showLabels = !isThumbnail;
const fontSize = isThumbnail ? 14 : 11;
const titleSize = isSocial ? 28 : 16;

For the screenshot pipeline, this means setting the viewport to the target dimensions and letting the chart respond. The same HTML file generates different outputs at different viewport sizes, without any server-side templating or separate code paths. The pipeline renders each size in sequence (or in parallel, if you launch multiple browser contexts).

PDF generation

Playwright provides page.pdf(), which calls Chromium's built-in print-to-PDF. This produces high-quality vector PDFs from any web page, including SVG charts. The text remains selectable. The vectors remain sharp at any zoom. This is the simplest path from interactive D3 chart to printable document.

await page.emulateMedia({ media: 'print' });
await page.pdf({
  path: 'report.pdf',
  format: 'A4',
  printBackground: true,
  margin: { top: '2cm', bottom: '2cm' }
});

The limitation is control. Chromium's PDF output respects @media print CSS rules, but it does not understand pagination hints like "keep this chart and its caption on the same page" without explicit page-break-inside: avoid rules. For multi-page reports with running headers, footers, and cross-references, dedicated tools like Paged.js or Prince offer much finer control. Paged.js runs in the browser (or headless Chromium) and implements the CSS Paged Media specification, giving you named pages, margin boxes, and automatic page numbering. Prince is a commercial tool that produces the highest-quality PDF output from HTML+CSS, at the cost of a license fee and a separate binary.

For moonshine's purposes, Playwright's page.pdf() is sufficient. Each explanation is a single-page document. The print stylesheet hides interactive controls, reduces margins, and adjusts colors for paper. The same CI pipeline that generates thumbnails can generate PDF versions as a second pass.

What this means for moonshine

Every explanation in this collection is a self-contained HTML file. Each one contains its own data, its own D3 code, its own styles. This self-containment is what makes server-side rendering tractable: there are no external API calls to mock, no shared state to initialize, no build step to run. You point Playwright at the file, set the viewport, wait for the render signal, and screenshot.

A rendering pipeline for moonshine would produce, for each explanation:

  • A social card (1200x630 PNG) for Open Graph and Twitter Card meta tags
  • A gallery thumbnail (300x200 PNG) for the index page
  • A print PDF (A4, high-contrast, no interactive controls)
  • An email-safe image (600px wide PNG, inline-friendly)

The d3-power-tools test infrastructure is already most of the way there. scripts/test-viz.py uses Playwright to load each generated block, wait for it to render, and take a screenshot for visual comparison. Extending this from "take a screenshot for testing" to "take a screenshot for production" requires three additions: multiple viewport sizes, output format metadata (title, description, branding overlay), and a manifest that maps each explanation to its generated assets.

Self-generating documentation. If each explanation includes a <meta> tag with its title and a render-complete signal, a single CI script can crawl the directory, render every file at every target size, and produce a complete set of assets. The explanations document themselves. No separate design tool, no manual screenshot curation.

Risks and Open Questions

  • Font rendering drift. Fonts render differently across operating systems, browser versions, and rasterization engines. A chart screenshotted on macOS with subpixel antialiasing looks subtly different from the same chart on Linux with grayscale hinting. In CI (typically Linux), the rendered text will not match what a Mac user sees in their browser. This causes visual regression test flakiness and means your social card may not be pixel-identical to the live chart. Pinning a specific Chromium version and installing identical fonts in CI helps, but perfect cross-platform consistency is not achievable.
  • Canvas requires a real browser. JSDOM cannot render Canvas 2D content. If your D3 chart uses <canvas> (common for heatmaps, particle systems, and large scatter plots), JSDOM will produce an empty image. You are forced onto the Playwright path for Canvas charts, with all its weight and latency. There is no lightweight middle ground.
  • CI pipeline complexity. Running Playwright in CI means installing Chromium (or a compatible browser) in your Docker image. This adds ~300MB to the image, increases build time, and introduces a new failure mode (browser crashes, display server issues, sandbox permissions). On platforms like GitHub Actions the Playwright team provides pre-built browser binaries, but the setup is still more involved than a pure Node.js pipeline.
  • LLM-generated charts that break under JSDOM. When an LLM generates D3 code, it often uses CSS features that assume a real browser: getComputedStyle, getBBox for text measurement, viewport-relative units, @media queries. These features return zeros or fail silently in JSDOM. A chart that looks correct in Chrome may produce garbage under JSDOM, with overlapping labels, zero-width elements, or missing text. The skill system would need to flag or avoid these patterns when the target is server-side rendering.
  • Viewport-dependent layout. Charts that use window.innerWidth or resize observers produce different output depending on the viewport set by the screenshot script. If the script does not precisely match the intended target dimensions, the chart may render at the wrong aspect ratio or trigger an unintended responsive breakpoint. Every target size needs its own screenshot pass.
  • Transition timing. If the screenshot fires mid-transition, the output captures a partially animated state. The wait-for signal must fire after all entry transitions complete, not just after the data binds. Charts that use staggered transitions (.delay((d, i) => i * 50)) are particularly tricky: the done signal must wait for the last element, not the first.
