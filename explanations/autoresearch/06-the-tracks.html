The Tracks

Same loop, different decision functions, different behaviors

Any artifact that can be automatically modified and automatically evaluated can be iterated. That is the core idea behind the autoresearch loop. Read the artifact. Propose a change. Render the result. Audit the output. Decide whether to keep or discard. Repeat.

The loop structure never changes. What changes is the decision function: what you optimize, what you constrain, and how you determine keep vs. discard. Swap out the decision function and the same loop produces very different behavior.

The decision function is the soul of each track. It encodes what you value. Everything else in the loop is shared infrastructure.

Three Tracks

The autoresearch system runs three distinct tracks. Each targets a different optimization objective. The figure below lays them side by side. Click a track to see its decision logic and a sample experiment.

The Decision Function

Each track's character comes from what it asks at the Decide stage. The block track asks: did the code get shorter without hurting quality? The prompt track asks: did the generation get faster without losing features? The redesign track asks: did quality improve without bloating the code?

These are simple predicates. There is no weighting, no multi-objective Pareto frontier, no learned reward model. Just a boolean: keep or discard. The simplicity is the point. A clear decision rule makes the loop predictable and easy to debug.

Block track: conservative compression

The block track targets a single HTML file and tries to make it shorter. It keeps a revision only when the line count drops and the composite quality score does not fall by more than 0.3. This is a conservative rule. It would rather discard a good compression than risk degrading the visualization.

Prompt track: aggressive speed

The prompt track targets the prompt text that generates a block. It rewrites the prompt to produce faster generation while preserving required D3 features. The bar is high: the rewritten prompt must generate at least 15% faster. Features are checked by grep, so they are either present or not. No partial credit.

Redesign track: creative improvement

The redesign track targets the same HTML file as the block track, but with the opposite priority. It optimizes for quality and tolerates modest code growth. Up to 20 additional lines are acceptable if they buy at least 0.1 composite score improvement. This makes it the most adventurous track. The proposer can add features, refine layouts, and improve interactions.

The Deferred Track

There is a fourth track not yet built: the skill track. It would iterate on SKILL.md files to improve the quality of blocks generated using those skills. This is the hardest variant because each experiment requires generating multiple blocks (four to six) to get a stable quality signal. A single block is too noisy. The cycle time balloons to 15 or 20 minutes per experiment.

The decision function would look something like: keep if the median composite across generated blocks improves by at least 0.2 and no individual block drops below 6.0. But that is speculative. The signal-to-noise ratio at that scale is still an open question.

Beyond D3

Nothing about this pattern is specific to D3 visualizations. The same loop could iterate on CSS stylesheets (optimize for file size, constrain visual regression tests). It could iterate on SQL queries (optimize for execution time, constrain result correctness). It could iterate on prose (optimize for readability score, constrain factual accuracy).

The requirements are modest. You need an artifact that a machine can modify. You need a quality signal that a machine can compute. And you need a decision function that encodes your values. The loop handles the rest.
