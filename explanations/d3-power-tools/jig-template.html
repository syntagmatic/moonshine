Jig Template

How Canvas, SVG, and HTML layers stack into one interactive visualization.

A rich interactive chart is not one thing. It is three rendering technologies layered on top of each other, each doing what it does best. Canvas draws thousands of data marks at 60fps. SVG renders crisp axes, grid lines, and interaction handles like brush rectangles. HTML provides controls (sliders, buttons) and tooltips that reflow with text. The jig template defines how these layers share a coordinate system, communicate through state, and redraw only what changed.

The visualization below renders 600 colored dots on a Canvas layer, draws axes and a brush rectangle on an SVG layer, and places a dot-size slider and tooltip in an HTML layer. Toggle each layer on and off to see what it contributes. Click "Separate layers" to pull them apart vertically so the stacking order becomes visible.

The key architectural insight: all three layers share the same container (position: relative) and the same coordinate system. Canvas gets ctx.translate(margin.left, margin.top) once so its pixel grid matches the SVG <g> transform. They look like one chart because they literally occupy the same rectangle.

When the slider moves, only the Canvas layer redraws. When the brush moves, the Canvas redraws to highlight selected dots and the SVG updates the brush rectangle, but the HTML layer stays untouched. This selective redrawing is managed by a dirty-flag bitmask: each event marks which layers need a repaint, and a single requestAnimationFrame call flushes them all at once.

State Flow

Three kinds of state live in a jig. Data state is the raw dataset, frozen and never mutated by interaction. View state holds scales and layout geometry, recomputed on resize or data change. SVG vs Canvas: Beyond the Element Count

The layer stack table in the skill makes element count the primary decision variable, and that is a reasonable starting heuristic. But element count alone is misleading. SVG performance depends on how much work the browser does per frame, which depends on what kind of elements they are, how they change, and how the browser composites them.

SVG is a retained-mode scene graph. The browser keeps every element in the DOM, lays it out, paints it, and composites the result. When you change an attribute, the browser walks the invalidation chain: style recalculation, layout (if geometry changed), paint, composite. For static or rarely-updated charts, this overhead is invisible. For continuous updates (drag, brush, zoom), each frame pays the full cost. Two thousand SVG circles with a drag-to-reorder interaction stutter because every frame triggers reflow on the moved element and its siblings. The same two thousand circles sitting still are perfectly fine.

Canvas is immediate-mode. You issue draw commands, the browser rasterizes them into a pixel buffer, and that is the end of its responsibility. No DOM nodes, no style recalculation, no layout. The cost per element is a few microseconds of JavaScript plus the rasterization time. The downside is that you lose everything the browser provides for free: hit testing, accessibility, focus management, CSS styling, and the ability to inspect elements in DevTools.

The hybrid pattern exists because these tradeoffs are complementary. Canvas handles the data layer where element count and update frequency are high. SVG handles axes, labels, brush rectangles, and focus rings where the element count is low (dozens, not thousands) and where the browser's retained-mode features are valuable. The SVG layer gives you accessible axis tick labels, inspectable brush handles, and CSS-driven styling. The Canvas layer gives you 60fps rendering of 10,000 data marks. Neither alone does both jobs well.

There are also cases where SVG is the right choice regardless of element count. A chart with 200 bars that the viewer needs to click, tab through, inspect in a screen reader, or style with external CSS is better served by SVG than by Canvas with a parallel hidden-DOM accessibility tree. The accessibility tree approach works, but it is a second system that must stay synchronized with the Canvas draw state. If you do not need Canvas performance, do not pay the accessibility synchronization cost.

The handoff pattern deserves special mention. Some visualizations need Canvas performance during interaction (smooth animation during a layout transition or a force simulation settling) but need SVG features at rest (clickable nodes, accessible labels, CSS theming). The technique is to render in Canvas while things are moving and, once the animation completes, render the final state in SVG and hide the Canvas. The viewer never notices the switch. D3's force layout is a natural fit: run the simulation on Canvas, then draw the settled positions in SVG for interaction.

Composition Archetypes

Different analytical goals produce different composition structures. The five archetypes below are not exhaustive, but they cover the overwhelming majority of multi-view visualizations. Each creates a different relationship between the views, and each has a characteristic failure mode when the architecture is wrong.

Explorer

Multiple linked views where brushing or selecting in one view highlights corresponding data in all others. The insight comes from cross-view correlation: brush the high-revenue cluster in the scatterplot, and the histogram reshapes to show they are concentrated in two industries. The viewer holds the brush position in working memory and watches the other views respond. For this to work, the response must be perceptually instantaneous. Empirical research (Liu and Heer, 2014) found that latencies above 500ms break the cognitive link between views. The viewer stops comparing and starts waiting. The dirty-flag architecture exists precisely to keep linked updates within the 16.6ms frame budget: one brush event marks all affected layers dirty, one requestAnimationFrame flushes them all.

Scrollytelling (Narrative)

A scroll-driven or step-driven sequence where the visualization transforms as the reader progresses. Each step changes the data, the encoding, or the annotation to build an argument. The transitions between steps show the data transforming under a new lens, and object constancy lets the reader track individual data points across changes. The architectural challenge is choreography. A step transition might need to (1) fade out the old annotations, (2) rescale the axes, (3) animate the data marks to new positions, and (4) fade in new annotations. These must happen in sequence or the intermediate state is confusing. The jig's initialization sequence helps: each step runs a subset of steps 4 through 10, and the ordering prevents cascading redraws mid-transition.

Dashboard

A CSS Grid of charts sharing a dataset and a color scale, each answering a different question about the same data. Dashboards look simple to build, and they are, until resize. Each chart panel gets its own ResizeObserver callback, and a window resize fires all of them simultaneously. If each callback independently triggers a full redraw with scale recomputation, you get a cascade of layout thrashing. The fix is to batch: collect all the new dimensions in one pass, recompute shared scales once, then redraw all panels in a single animation frame. A shared color scale must use the full-data domain, not the filtered subset's domain. Otherwise, filtering changes the color mapping and the viewer cannot compare colors across filter states.

Detail-on-Demand

An overview visualization with a mechanism (click, hover, brush) to reveal a detail view for a selected subset. The canonical example is Shneiderman's visual information-seeking mantra: "overview first, zoom and filter, then details on demand." The detail view might be a data table, an enlarged sub-chart, or a tooltip with a sparkline. The key architectural constraint is that the detail view must not interfere with the overview. A detail panel that overlaps the chart and blocks the brush, or a tooltip that triggers a reflow that shifts the chart, breaks the exploration flow. The jig's z-index layer order (Canvas, SVG, HTML) naturally supports this: the HTML layer floats above everything and does not affect Canvas or SVG layout.

Spatial Explorer

A map with overlaid data layers, zoom-dependent level of detail, and linked side panels. This is the most architecturally demanding archetype because four coordinate systems must agree: the geographic projection, the Canvas pixel grid (with DPR scaling), the SVG viewBox, and the zoom transform. A bug in any one produces offset markers, blurry tiles, or data points that drift during zoom. The resize contract is especially strict: a map resize must recompute the projection's fitSize, re-render all Canvas layers at the new DPR-scaled resolution, and update the SVG viewBox, all without visible flicker.

The Resize Contract

Resize is the most common source of bugs in composed visualizations. It looks simple (recompute scales, redraw) but the devil is in the interaction between layers and the order of operations.

When a container changes size, every layer that uses pixel coordinates is stale. Canvas backing store dimensions must match the new CSS size times device pixel ratio, or the output is blurry or clipped. Scale ranges must update before any layer redraws, or axes and data marks disagree. Brush extents are stored in pixel coordinates and become meaningless after resize. They must be converted to data coordinates before the resize, then back to pixel coordinates after the new scales are ready.

The initialization sequence from the skill (steps 3 through 10) is the resize handler. Resize literally re-runs the same code as initial setup, starting from layer creation. This is not a coincidence. It is a deliberate design: if initial render and resize follow different code paths, they will diverge, and one of them will have bugs you only discover when the other path does not trigger.

Debounce timing depends on the layer composition. A Canvas-only visualization can resize on every ResizeObserver callback with no debounce because Canvas redraws are cheap. A Canvas-plus-SVG hybrid should debounce at around 100ms to avoid churning SVG axis redraws that the viewer cannot perceive anyway. A full dashboard with six or more panels should debounce at 150ms because each panel's resize triggers scale recomputation, and the cumulative JavaScript cost can drop frames. Test on a low-powered device. A 2024 MacBook Pro hides resize performance problems that a Chromebook or mobile device will expose.

Zoom transform state requires special care. If the viewer is zoomed into a region and the container resizes, you must decide whether to preserve the viewport center (so the same data stays in view) or reset the zoom (so the full dataset is visible at the new size). The skill recommends preserving the viewport center and recomputing translateExtent to match the new dimensions. Resetting the zoom on resize is disorienting: the viewer loses their place in the data.

Performance Budgets

A 60fps target gives each frame 16.6 milliseconds from the moment the browser starts JavaScript to the moment pixels appear on screen. In practice, the browser takes about 2ms for its own bookkeeping (style recalculation, composite, and present), leaving roughly 14ms for your code. That 14ms must cover everything that happens in response to an interaction: updating state, filtering data, computing layouts, drawing Canvas pixels, updating SVG elements, and modifying DOM nodes.

The figure below breaks a single frame into its constituent costs. Drag the complexity slider to add more data points and watch how each phase grows. When the total exceeds the 16.6ms deadline, the frame drops and the interaction feels sluggish. The relative proportions matter as much as the total: if Canvas draw dominates, you need progressive rendering or typed arrays. If DOM update dominates, you need to batch updates or virtualize the element list. If data filtering dominates, you need a bitmap index or a Web Worker.

When the budget is exceeded, the responses are ordered by effort-to-impact ratio. First, split cheap and expensive work: highlight the brushed subset immediately and debounce the histogram rebinning 16ms later. The viewer sees instant feedback on the cheap operation and barely notices the one-frame delay on the expensive one. Second, progressive rendering for the Canvas data layer, using requestAnimationFrame to draw in chunks across multiple frames. Third, skip transitions during continuous interaction and only apply them on brush end or zoom end. Fourth, offload filtering to a Web Worker so the main thread stays free for drawing. Fifth, for 100K+ rows, use a bitmap index (BitFilter from the linked-views skill) that can intersect filter predicates in microseconds.

Memory is the other budget. Each 2x DPR Canvas at 1200 by 800 pixels consumes about 7.7 megabytes of GPU memory. A five-layer stack burns 38MB before you render a single data point. On mobile devices with aggressive memory limits, this matters. The fix is to merge layers that always redraw together. If the highlight layer always redraws when the data layer does, they can share a single Canvas with the highlight drawn on top. Two layers instead of three saves 7.7MB and one clearRect call per frame.

State Architecture Patterns

The skill describes three kinds of state (data, view, interaction) and this separation is correct, but it leaves open the question of how interaction state flows between views. There are two dominant patterns, and they suit different composition sizes.

Centralized Store

A single state object holds all interaction state: brush extents, selected keys, zoom transforms, filter predicates. Every view reads from the store and subscribes to changes. When the brush in view A updates the store, views B and C receive a notification and redraw. This is the pattern used in the demo above (the state object) and in libraries like Crossfilter.

The centralized store works well for compositions with up to about six views. The state shape is predictable, debugging is straightforward (log the state object and you see everything), and there is a single place to add persistence (save the state to URL params or localStorage). The weakness is that the store becomes a coordination bottleneck. If view A's brush update triggers a redraw of view C, which triggers a derived-state recomputation that triggers another redraw of view B, you have a cascade that is hard to reason about. Strict unidirectional flow (interaction writes state, state triggers renders, renders never write state) prevents this.

Distributed Dispatch

Each view publishes events via d3.dispatch and other views subscribe. There is no shared state object. View A dispatches "brush" with the extent, view B listens and filters its data, view C listens and updates its highlight. The state is distributed across the views themselves.

This pattern scales to larger compositions where a centralized store's shape would become unwieldy. It also works well when views are developed independently (by different team members or as reusable components) because each view only needs to know the event contract, not the entire state shape. The weakness is that distributed state is harder to debug. When something is wrong, you cannot inspect a single object to see the full picture. You need to trace event flows across multiple subscribers. It is also harder to persist, since there is no single object to serialize.

The hybrid approach, common in practice, uses a centralized store for shared state (selected keys, global filter) and local state within each view for view-specific concerns (zoom transform, tooltip position). The store handles cross-view coordination and each view manages its own internal state. The skill's d3.dispatch pattern and the createStore pattern from linked-views are not competing options. They are complementary tools for different scopes of state.

When This Skill Hurts

Infrastructure for a bar chart. The jig template is infrastructure. It solves coordination problems that only exist when multiple layers, multiple views, or continuous interaction are present. For a single static bar chart, or even a single interactive scatterplot with a tooltip, the full jig is overhead that produces no benefit. LLMs applying this skill to simple single-chart explanations will generate a Canvas layer, an SVG layer, an HTML layer, a state object, dirty flags, and a requestAnimationFrame flush loop for a visualization that could be 40 lines of SVG code. The architecture is not wrong, but it is a cost with no corresponding payoff. If the visualization has one view and fewer than 500 elements, skip the jig entirely and use plain SVG.

Dirty flags more complex than the redraw. The dirty-flag bitmask and requestAnimationFrame coalescing pattern pays for itself when multiple interaction events fire within a single frame and trigger overlapping layer redraws. For a visualization where the only interaction is a slider that redraws one Canvas layer, the dirty-flag machinery is more lines of code than the draw function it is gating. LLMs tend to apply the pattern mechanically because the skill presents it as standard architecture. But if each interaction maps to exactly one layer and interactions do not overlap, you can call the draw function directly from the event handler. The frame coalescing buys you nothing, and the indirection makes the code harder to read.

Layer soup. Canvas, SVG, and HTML layers composited with position: absolute create a z-index debugging surface that grows with each added layer. The hit Canvas is invisible. The SVG overlay intercepts pointer events unless you carefully manage pointer-events: none on the right elements. The HTML tooltip layer must float above everything but pass clicks through to the SVG brush below it. When something goes wrong (a tooltip does not appear, a brush does not start, a hover target is off by the margin offset), you are debugging the interaction between three coordinate systems and three pointer-event models simultaneously. DevTools inspection is limited: you cannot inspect Canvas pixels, and the SVG overlay obscures the Canvas layer in the Elements panel. For compositions with only one or two data layers, the multi-layer architecture creates more z-index debugging than it prevents performance problems.

Resize state loss. The resize contract requires converting pixel-coordinate state (brush extents, zoom transforms) to data coordinates before resize and back afterward. LLMs routinely forget this conversion. The brush visually resets on resize, the zoom snaps to the origin, or the selected set silently empties because the pixel-space filter predicate no longer matches any points at the new scale. The skill describes the remapBrush pattern explicitly, but the failure only appears during resize, which is not tested during single-shot code generation. In practice, this means that any jig-template visualization generated by an LLM will have broken resize behavior for stateful interactions until someone manually tests a window resize while a brush is active.
