Coordination

How d3.dispatch lets independent charts talk to each other without knowing about each other.

A single chart is self-contained. It owns its data, its scales, its DOM. But put two charts on a page and suddenly you need a decision: when the user brushes one chart, how does the other one find out? The naive answer is to import Chart B into Chart A and call its update method directly. That works for two charts. It falls apart at three, because now every chart must know about every other chart.

d3.dispatch solves this with indirection. Instead of charts calling each other, they all talk to a shared event bus. Chart A says "a brush happened" and the dispatch relays that message to whoever is listening. Charts B and C subscribe independently. They never import each other. They do not even know the other exists.

The key insight: d3.dispatch is decoupling. The source of an event does not choose its recipients. The recipients choose which events they care about. Adding a fourth chart means adding one more subscriber, not rewiring every existing chart.

Namespaced listeners make this safe. dispatch.on("brush.chartA", fn) registers a listener tagged with chartA. A second call with the same namespace replaces the old listener instead of stacking another one. When a chart is removed, dispatch.on("brush.chartA", null) tears down its listener cleanly. No memory leaks, no stale closures.

Namespace Conventions

The .on("event.namespace") syntax is not decoration. It solves a real problem: if two separate pieces of code both call dispatch.on("brush", fn), the second one silently replaces the first. The first listener vanishes without warning. Namespacing gives each subscriber its own slot: dispatch.on("brush.scatter", fn) and dispatch.on("brush.histogram", fn) coexist because D3 treats them as separate registrations on the same event type.

The convention is eventType.subscriberId. Keep it flat. "brush.scatter" and "filter.timeline" are good. "brush.chart1.filterPanel.leftAxis" is a comprehension burden that helps no one. The namespace exists so you can tear the listener down later, not to encode a component hierarchy.

Teardown Patterns

Every .on("event.namespace", fn) call creates a reference that the dispatch object holds indefinitely. If the chart that registered it is removed from the page, the listener keeps firing into a detached DOM. That is a memory leak, and in a single-page application that creates and destroys dashboard views, it accumulates until the page slows to a crawl.

The fix is to pass null as the handler: dispatch.on("brush.scatter", null). In a framework context, this goes in the cleanup function. React's useEffect return function, Vue's onUnmounted, or a plain destroy() method on a chart object. The pattern is always the same: register with a namespace, tear down by namespace.

// React cleanup pattern
useEffect(() => {
  dispatch.on("brush.myChart", range => updateChart(range));
  return () => dispatch.on("brush.myChart", null);
}, [dispatch]);

Framework Bridges

D3's dispatch is imperative: you call dispatch.call() and listeners fire immediately. React and Vue are declarative: you describe what the UI should look like given some state, and the framework decides when to render. Bridging these two models is where most coordination bugs live.

The solution in React is to create the dispatch once (via useMemo or useRef) and share it through context. Each chart component subscribes in a useEffect and cleans up on unmount. The critical rule: never let a dispatch listener call setState synchronously during a render. That triggers a re-render inside a render, which React will either batch (hiding the bug) or throw on (in strict mode).

In Vue, provide/inject replaces context. The dispatch lives in the dashboard parent and each chart injects it. Cleanup goes in onUnmounted. Angular's challenge is different: D3 events fire outside Angular's zone, so change detection does not run. Use NgZone.runOutsideAngular for high-frequency events like brush, then call NgZone.run only for the final state change.

Dispatch vs Store vs Direct Calls

Three architectures, three sweet spots. For two charts that need to talk, a direct function call is simplest. Chart A imports Chart B's update method and calls it. No indirection, no string-based event names, easy to follow in a debugger. The cost is coupling: if you add Chart C, you rewire Chart A to also call Chart C, then Chart B to call Chart C, and so on. This is O(n²) in the number of connections.

d3.dispatch is O(n). Each chart registers with the bus and emits to it. Adding a new chart is one more subscriber. This is the right choice for 3-8 charts with simple, event-like communication: "a brush happened," "a filter changed," "a point was hovered."

A shared state store (a plain object with a setState/getState API and subscriber notification) is better when the state is deep. If you have zoom level + active filters + sort order + selected rows + undo history, dispatch events for each become a combinatorial mess. The store holds the single source of truth. Charts read from it and subscribe to changes. This is Redux in miniature, and for complex dashboards it is worth the overhead.

If you find yourself passing more than two arguments through dispatch.call(), that is a signal the state has outgrown dispatch. Wrap it in a store.

Bitmap Indexing for Fast Filtering

When coordinated filtering hits 100K+ rows, the bottleneck shifts from rendering to filtering. Calling Array.filter() across multiple dimensions rebuilds the entire result set from scratch on every interaction. At 60fps, that is 16ms per frame, and a three-dimension filter over 200K rows will miss the budget.

The crossfilter pattern replaces per-row predicate checks with bitmask operations. Each filter dimension maintains a bitmask: bit i is 1 if row i passes that dimension's filter, 0 otherwise. To combine dimensions, AND the masks together word by word. Bitwise AND on 32-bit integers tests 32 rows per CPU instruction, roughly 32x faster than iterating an array of booleans. The result is a single combined mask that you scan once to collect visible row indices.

// Combine dimension bitmasks — one AND per 32 rows
const words = Math.ceil(N / 32);
const result = new Int32Array(words).fill(~0);
for (const mask of dimensionMasks) {
  for (let w = 0; w < words; w++) result[w] &= mask[w];
}

This matters for coordination because a brush on one chart updates one dimension mask, then the coordinator ANDs all masks and pushes the combined result to every chart. Each chart only needs to re-render the rows whose visibility changed, not re-filter the entire dataset.

Feedback Loops

The most dangerous coordination bug is a feedback loop. Chart A dispatches an event. Chart B receives it, updates itself, and dispatches its own event. Chart A receives that, updates, dispatches again. The browser locks up in an infinite cascade of synchronous calls.

The standard fix is a source guard: include the originator's identity in every dispatch call, and skip the update if the event came from yourself. dispatch.call("brush", this, {range, source: "chartA"}). In your listener, check if (d.source === myId) return. The figure below demonstrates this. Without the guard, click "Fire event" and watch the counter spiral. Toggle the guard on and the loop stops after one round-trip.

When This Skill Hurts

Costs of over-applying coordination patterns

  • Premature abstraction. For two charts, direct function calls are simpler than dispatch. The skill pushes dispatch universally, and LLMs follow it even for trivial cases, adding ceremony that obscures the logic. If Chart A is the only thing that ever calls Chart B, a function call is clearer and easier to trace than wiring both through a bus.
  • Namespace proliferation. The skill encourages detailed namespacing, but LLMs generate names like "update.chart1.filterPanel.brushHandler" that become their own comprehension burden. A namespace should be one identifier: the subscriber's name. Anything deeper is encoding architecture into a string, which is worse than no structure at all.
  • Hidden control flow. Dispatch makes event flow invisible. When debugging, you cannot grep for what calls a handler because it is wired through a string name at runtime. LLMs rarely add the logging infrastructure that makes dispatch debuggable. In any non-trivial dashboard, add an event log (like Figure 2 above) during development, or you will spend hours tracing phantom updates.
  • Memory leaks. Listeners not cleaned up on teardown accumulate. The skill mentions cleanup, but LLMs often omit it, especially in single-page explanations that "never" unmount. In a real application, every .on() call without a corresponding .on(name, null) on destroy is a leak. Review generated code for teardown before shipping.
