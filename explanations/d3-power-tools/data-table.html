Data Tables as Visualization

A well-designed table is a visualization, not a fallback. It shows the exact values that charts can only approximate.

Charts encode values as position, length, or color. Those encodings are efficient for spotting trends, clusters, and outliers, but they are lossy. A reader looking at a scatter plot can tell that two points are close, not that one is 47.3 and the other is 48.1. When the viewer's task requires exact numbers, comparison across several attributes, or mixed units in a single view, a table outperforms any chart.

The interactive below shows the same 30-row dataset in two forms side by side. The scatter plot reveals the shape of the relationship between revenue and employees. The table reveals the specific values. Hover a row to highlight the corresponding point, or hover a point to highlight the row. Click a column header to sort. Type in the filter box to narrow both views at once.

Tables and charts answer different questions about the same data. "Is there a correlation?" is a chart question. "What is Acme Corp's exact revenue?" is a table question. Offering both lets the reader switch tasks without switching tools.

Accessible Table Markup

A table that looks right but lacks semantic structure is invisible to screen readers. The interactive above uses role="grid" on the <table> element and role="columnheader" on each <th>. Those are not decorative. They tell assistive technology that this is an interactive data grid, not just a layout table, and that clicking or pressing Enter on a header does something.

Three attributes carry most of the accessibility weight. role="grid" on the table signals that cells are navigable with arrow keys, not just tab stops. scope="col" on header cells binds each <th> to the column below it, so a screen reader can announce "Revenue, $47.3 million" rather than just "$47.3 million" with no context. And aria-sort on the active sort column tells the user which column is sorted and in which direction, without requiring them to visually spot a tiny arrow glyph.

The sort interaction deserves special attention. Each <th> needs tabindex="0" so keyboard users can reach it, plus an event handler for both Enter and Space. When the sort changes, a live region (a div with aria-live="polite") should announce the new state: "Sorted by Revenue, descending." Without that announcement, a keyboard user presses Enter on a header and has no confirmation that anything happened. The table in Figure 1 sets aria-sort to "ascending", "descending", or "none" on every header after each sort action.

For tables with row headers (the first column identifies each row), add scope="row" to those cells. This lets a screen reader announce "Apex Corp, Revenue, 247.3" when navigating across a row, instead of just "247.3."

Right-Aligned Numbers and Tabular Figures

Numbers in a table column need to be right-aligned so the decimal points (or the ones digits, for integers) stack vertically. Left-aligned numbers are nearly impossible to compare because the digits do not line up. The reader has to mentally right-justify each value before they can see that 1,234 is larger than 987.

Right alignment alone is not enough. Proportional fonts give different widths to different digits: a "1" is narrower than a "0," so even right-aligned numbers wobble from row to row. The fix is font-variant-numeric: tabular-nums, which forces every digit to occupy the same horizontal space. The table above uses it on every .num cell. The visual effect is subtle but important: columns of numbers look like they were typeset in a ledger, with clean vertical alignment.

The CSS is two lines. Put text-align: right and font-variant-numeric: tabular-nums on numeric <td> elements. Mirror the text-align: right on the corresponding <th> so the header sits above its column of numbers rather than floating off to the left. Pair these with a consistent d3.format per column: every value in the Revenue column gets ,.1f, every value in Employees gets ,. Mixing decimal places within a column defeats the purpose of alignment.

Chart-Table Toggle Patterns

The toggle in Figure 1 offers three modes: chart only, table only, and both side by side. This is the simplest form of the pattern. The chart container and the table container are siblings in the DOM, and the toggle just sets display: none on whichever view is hidden. Both views bind to the same data array, so filter and selection state transfer automatically when switching.

Two implementation details matter. First, toggling to "chart only" should not destroy the table DOM. The table rows are expensive to rebuild from data if the dataset is large, and preserving them means the table reappears instantly when the user switches back. Just hide with display: none. Second, toggling to "table only" should not keep the chart's SVG or Canvas in memory if performance is a concern. Charts are cheap to redraw from data, and the pixel buffer of a Canvas element consumes real memory. For most tables the difference is negligible, but for dashboards with many chart-table pairs, lazy-destroying the chart on toggle and rebuilding on switch-back keeps memory bounded.

The toggle buttons use aria-pressed to communicate which view is active. A screen reader user pressing the "Table view" button hears "Table view, pressed" and knows the chart is now hidden. The hidden container should also carry aria-hidden="true" so assistive technology does not try to navigate invisible content.

Linked Highlighting Between Chart and Table

Cross-highlighting is the interaction that makes the chart-table pair feel like a single tool rather than two unrelated views. Hover a scatter point and the corresponding table row lights up. Hover a table row and the dot in the scatter plot grows and gets a stroke. The shared state is a single variable, hoveredId, that both views read from.

The update path is deliberately asymmetric. When the hover comes from the chart, the table row gets a CSS class (.highlighted) that sets a background color. The table does not scroll to the highlighted row. Scrolling on hover fights the user: if they are reading the chart, the table jumping around in the corner is distracting rather than helpful. When the hover comes from the table, the chart dot updates its radius and stroke, a small visual change that does not disrupt the chart's spatial layout.

Scroll-to-row is appropriate only for click or selection events, not for hover. If the user clicks a dot (or selects it with the keyboard), that is an intentional focus change, and scrolling the table to show the selected row is helpful. The distinction between hover (ephemeral attention) and click (intentional selection) should drive the scroll decision.

For Voronoi-based hover on the chart side, build the Delaunay triangulation from the filtered dataset, not the full dataset. If a point is filtered out, hovering near its former position should not highlight a ghost. The scatter in Figure 1 rebuilds the Delaunay on every filter change for this reason.

Virtual Scrolling for Large Datasets

A table with 50 rows renders instantly. A table with 50,000 rows creates 50,000 <tr> elements, each containing several <td> elements, and the browser spends hundreds of milliseconds building that DOM. Scrolling becomes sluggish. Sort operations freeze the page while D3 removes and re-adds thousands of nodes.

Virtual scrolling solves this by rendering only the rows that are visible in the scroll viewport, plus a small buffer above and below to prevent white flash during fast scrolling. The technique requires a fixed row height (so the total scrollable height can be computed as data.length * rowHeight) and a scroll listener that computes which rows are in view.

The implementation has five parts. A scroll container with max-height and overflow-y: auto provides the scrollbar. A spacer element inside it is sized to data.length * rowHeight so the scrollbar thumb reflects the full dataset. On scroll, compute startIdx = Math.floor(scrollTop / rowHeight) - buffer and endIdx = startIdx + visibleCount + 2 * buffer. Slice the sorted/filtered data to that range. Use D3's keyed data join to bind the slice, and offset the <tbody> with transform: translateY(startIdx * rowHeight) so the visible rows appear at the correct vertical position.

D3's data join handles the recycling: tbody.selectAll('tr').data(slice, d => d.id).join('tr') reuses existing row elements and creates or removes only the difference. This is the same pattern as a regular table, just applied to a moving window of data.

Two gotchas. First, after sorting, you must re-render the visible window even though the scroll position has not changed: the data order changed beneath the scroll position. Second, the browser's Ctrl+F find-in-page cannot search rows that do not exist in the DOM. Compensate with a visible search input that uses the filter pipeline, as the skill recommends. Set aria-rowcount on the <table> to the full dataset length so screen readers know there are more rows than the DOM contains.

Inline Sparklines in Table Cells

A sparkline in a table cell adds a sixth sense to tabular data. The viewer reads across a row and sees not just the current value but the trajectory that led to it. Edward Tufte called sparklines "data words," and embedding them in table cells is the most literal version of that idea: the chart lives inside the sentence of numbers.

The technique appends a small inline <svg> (typically 60 to 80 pixels wide, 16 to 20 pixels tall) inside a <td>. D3's d3.line() generator draws the path from a d.series array stored on each row. The x scale maps array index to pixel position. The y scale maps the value range. Because each sparkline may have a different y range (per-row normalization) or share a global range (for cross-row comparison), the scale choice is a design decision, not just an implementation detail.

Per-row scaling makes each sparkline fill its cell vertically, emphasizing the shape of each individual trend. Shared scaling lets the viewer compare magnitudes across rows, but rows with small values flatten to near-invisible lines. The table in Figure 2 uses shared scaling so the viewer can see which companies had the most volatile quarters alongside which had the highest absolute revenue.

For accessibility, each sparkline SVG carries an aria-label that summarizes the trend in text: "Trending up 12.3%" or "Down 5.1% over 8 quarters." A screen reader user gets the story without needing to interpret a tiny path element.

A company showing $120M in the latest column might be recovering from a dip or declining from a peak. The sparkline disambiguates without requiring the reader to open a separate time-series chart.

When This Skill Hurts

Tables as fallback rather than designed artifact. The skill's central argument is that a table is a visualization, not a consolation prize. But LLMs frequently treat tables as the "accessible alternative" they tack on after the chart is finished, rather than a view that deserves its own design work. The result is a bare <table> with default browser styling, no alignment, no sort, no hover, no filter. The skill tries to counter this by leading with the table-as-visualization framing, but the pattern persists. If you find yourself generating a plain HTML table dump beneath a polished D3 chart, stop and ask whether the table deserves the same care as the chart.

Sort interaction that resets scroll position. When the user clicks a column header to re-sort, the browser re-renders the table body. If the implementation clears and rebuilds the <tbody> (as the naive D3 join does by removing all rows and re-appending), the scroll container jumps back to the top. The user was looking at row 35 of 200, clicked "Sort by Growth," and now they are staring at row 1. They have to scroll back down to find what they were looking at. The fix is to use a keyed data join (d => d.id) that reorders existing DOM nodes instead of destroying and recreating them, or to save and restore scrollTop around the re-render.

Filter that produces empty state with no feedback. A text filter that narrows to zero results and leaves the table body simply blank is a usability failure. The viewer types a query, the rows vanish, and they cannot tell whether the filter is broken or the data genuinely has no matches. Always show an explicit "No matching rows" message and keep the status bar updated with the count. For screen reader users, the live region must announce the zero-result state: "Showing 0 of 200 rows." Silence after a filter action is indistinguishable from a bug.

Too many columns forcing horizontal scroll. A table with twelve columns on a 740-pixel-wide article layout forces horizontal scrolling, which breaks the viewer's spatial model. They can no longer see the row label and the value they are reading at the same time. If horizontal scroll is unavoidable, freeze the first column with position: sticky; left: 0 so the row identifier stays visible. Better yet, rethink the column set. A table trying to show every attribute of every row often means the designer has not decided which attributes matter for the viewer's task. Pare the columns to the five or six that serve the primary question, and offer a "show more columns" control for the rest.
