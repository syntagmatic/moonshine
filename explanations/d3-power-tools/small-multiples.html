Small Multiples

Repeating the same chart across categories lets the eye compare by position, the most accurate visual channel we have.

When you overlay six or more lines on a single chart, the result is spaghetti. The viewer spends all their effort untangling instead of comparing. Small multiples solve this by giving each category its own panel with identical axes. The eye scans from panel to panel, comparing shapes by their position against a shared grid. Position is the one visual channel where humans detect even tiny differences, which is why this layout is so effective for the question "do these series follow the same pattern?"

The grid below shows monthly average temperature for 12 cities. Each panel repeats the same chart structure: months on x, temperature on y. The shared layout turns comparison into scanning instead of decoding. Try toggling between shared and independent Y scales to see the tradeoff. With shared scales, you can compare magnitudes across cities directly: Singapore is clearly hotter than Reykjavik. With independent scales, each panel stretches to fill its own range, and you can see that even low-amplitude cities like Singapore have a subtle seasonal shape that the shared view crushes flat.

The scale decision is the most important design choice in small multiples. Shared scales answer "which is bigger?" Independent scales answer "what shape does each one have?" Pick the question you want the viewer to answer, then pick the scale.

Brush across any panel and every other panel highlights the same months. This synchronized interaction is the key UX pattern for small multiples. It works because every panel shares the same X scale, so a pixel range in one panel maps to the same data range in all the others. The implementation needs one guard: when your code programmatically moves the brush in the other panels, those brush events fire too. Without checking event.sourceEvent, you get an infinite loop. The fix is a single line: if the event was not triggered by user input, return early.

The grid layout adapts to the container width. At full width you see 4 columns. Narrow your browser and it drops to 3, then 2, then 1. This is not CSS Grid doing the work. The code uses a ResizeObserver to measure available width, picks the column count that keeps each panel above a minimum readable size, and re-renders the SVGs into the new grid. Debouncing the resize callback by 150ms prevents expensive re-renders on every frame while dragging the browser edge.

Grid Layout Math

The geometry of a small multiples grid is determined by three values: the container width, the number of panels, and a minimum panel width below which the chart becomes illegible. From those three inputs, everything else follows.

Start with columns. Given container width W, gap size g, and minimum panel width m, the maximum number of columns is floor((W + g) / (m + g)). In practice, you iterate down from a desired maximum (often 4 or 5) and pick the first column count that keeps each panel above the threshold. Each panel then gets width (W - (cols - 1) * g) / cols. Rows follow from the panel count: rows = ceil(n / cols).

Cell size is the main lever. Each cell has an outer size (what the grid sees) and a plot area inside it, reduced by margins for axis labels, tick marks, and the panel title. The plot area is where the data lives: plotWidth = cellWidth - margin.left - margin.right, and the same for height. When panels are small, margins consume a larger fraction of the cell. This is the fundamental tension: more panels means smaller cells, which means margins eat into data space, which means the chart loses resolution. Below about 120px of plot width, line charts lose their shape and become blobs.

The aspect ratio of each cell matters more than people realize. For time series, wider cells (roughly 2:1 or 3:1 width to height) let the eye track slopes. For scatter plots, square cells preserve the geometric relationships in the data. The formula cols = ceil(sqrt(n * containerAspect)) produces roughly square grids adjusted for rectangular containers, which is a good default when you have no strong reason to prefer a particular shape.

Shared vs. Independent Scales

The scale choice is not a toggle between two equally valid options. It is a claim about what matters in the data, and picking the wrong one actively misleads the reader.

Shared scales answer magnitude questions: which series is biggest, which grew fastest, which has the highest peak. They work when the data ranges overlap substantially. If your twelve cities all have temperatures between -5 and 35 degrees, a shared y-axis from -10 to 40 gives every panel enough room to show its pattern while preserving direct comparability. The reader can glance at two panels and see that one line sits higher than another.

Independent scales answer shape questions: does this series have the same seasonal pattern as that one, where are the local peaks and troughs, is the trend accelerating or decelerating. They work when the data ranges differ enough that a shared scale would crush low-variance series into flat lines. In Figure 1 above, Singapore has an annual temperature range of about 3 degrees. On the shared scale, its line is essentially flat. On independent scales, you can see the subtle dip in December and January.

The dangerous middle ground is when ranges differ by a factor of roughly 3 to 10. In this zone, shared scales are not so extreme that they obviously crush small-range panels, but the small-range panels still lose most of their visible variation. The reader sees what looks like a pattern in the high-range panels and a flat line in the low-range panels, and concludes (incorrectly) that the low-range series have no interesting behavior. Independent scales fix this, but they introduce the opposite problem: the reader sees similar-looking curves in every panel and assumes they represent similar magnitudes, when in fact the y-ranges differ by an order of magnitude.

The hybrid approach, sharing the x-axis while giving each panel its own y-axis, is the standard choice for time series where the categories have different units or wildly different magnitudes (temperature vs. precipitation, revenue vs. click-through rate). The shared time axis keeps temporal comparison intact. But even here, clearly label each panel's y-range. Do not assume the reader will check the axis ticks.

When ranges differ by more than about 10x, consider normalizing the data (z-scores, percent of maximum, index to a base period) before plotting. This lets you use shared scales for shape comparison while making the normalization explicit rather than hidden inside per-panel scale domains.

Panel Ordering

The order of panels in a small multiples grid is an editorial choice, the same way that paragraph order in a written argument is an editorial choice. It determines what patterns the reader notices first, which comparisons are easy (adjacent panels) and which are hard (panels far apart), and whether the grid tells a story or just presents a catalog.

Alphabetical is the default for most tools, and it is almost never the best choice for analysis. Its one advantage is lookup speed: if the reader knows they want to find "Mumbai," they can scan to the M section. It is appropriate for reference displays where the reader arrives with a specific panel in mind, like a dashboard of named metrics. It is inappropriate for comparison, because adjacent panels share nothing except similar first letters.

By a summary value (mean, peak, total, slope, variance) creates a visual gradient across the grid. The eye moves from the highest panel to the lowest, or from the most volatile to the most stable, and the ranking itself becomes part of the message. Sorting by peak temperature in Figure 1 would place Moscow and New York at the top (highest summer peaks) and Reykjavik at the bottom. The reader perceives the gradient without reading any numbers.

By similarity groups panels that share the same shape. Two cities with identical seasonal patterns end up adjacent, making it trivial to confirm they match. Two cities with opposite patterns (one peaks when the other troughs, like northern vs. southern hemisphere) end up far apart, making the contrast visible. Computing similarity requires a distance metric: Pearson correlation for shape-only comparison, Euclidean distance for shape-plus-level, or dynamic time warping when the shapes are similar but offset in time. This ordering is the most analytically powerful but also the most opaque to the reader, who cannot easily reconstruct why the panels are in their particular order without a label or explanation.

The figure below shows the same 16 panels reordered three ways. Toggle between orderings and watch how the grid tells a different story each time. Alphabetical is a phone book. By peak value, the gradient from hot to cool is immediately visible. By similarity, the northern-hemisphere cities cluster together and the southern-hemisphere cities cluster together, revealing a pattern that neither of the other orderings surfaces.

The 25-Panel Limit

Small multiples work because the reader can hold one panel's shape in visual short-term memory and compare it to the next panel. This comparison loop requires that both panels be visible at the same time, at a size large enough to perceive the shape. On a typical desktop screen, that budget is roughly 20 to 25 panels in a 5x5 or 4x6 grid.

Beyond 25, three things break simultaneously. First, each panel shrinks below the threshold where shape is perceptible: a 120px-wide line chart becomes a 70px smudge. Second, the reader can no longer hold a panel's shape in memory long enough to scan across the grid: by the time they reach the panel they want to compare against, they have forgotten the shape they started with. Third, on most screens, some panels scroll offscreen, and scrolling destroys the spatial map the reader was building. You cannot compare what you cannot see at the same time.

LLMs do not feel this constraint. Given a dataset with 50 categories, an LLM will happily generate a 50-panel grid because the code to do it is trivially a loop over the data. The LLM does not experience the viewport, does not feel the panels shrinking, does not notice that panels 35 through 50 are below the fold. The result is technically correct and visually useless.

When the data has more than 25 categories, the right move is to aggregate, filter, or tier. Show the top 12 and an "all others" aggregate. Let the reader pick a subset with a dropdown or search box. Or use a summary chart (a bar chart of the key statistic, sorted) alongside a smaller grid of highlighted panels. This overview-plus-detail approach preserves the small-multiples comparison for the categories that matter while giving the full picture through a simpler chart form.

Overview + Detail at the Facet Level

A common problem with small multiples: the reader sees 12 panels but has no way to see the aggregate. Each panel shows one category's story, but what does the whole dataset look like? Where does any individual category sit relative to the average? The panels answer "what shape does each one have?" but not "how does each one compare to the whole?"

The solution is to pair the faceted grid with a summary chart. The summary chart shows all series overlaid (as a spaghetti plot, or as a single aggregate line with a confidence band) and serves as an orientation device. The reader looks at the summary to identify which series are interesting, then looks at the faceted grid to examine those series in detail. Highlighting a panel in the grid can highlight the corresponding line in the summary, and vice versa.

This is the facet-level version of the overview-plus-detail pattern from linked views. The overview is a single chart that shows everything at once but sacrifices individual legibility. The detail is the small multiples grid that gives each series room to breathe but sacrifices the aggregate picture. Together they answer both the "what is the overall pattern?" and "how does this category differ from it?" questions that neither can answer alone.

Practically, place the summary chart above or to the left of the grid. It should be noticeably larger than any individual panel (at least 2x the panel height) to visually signal that it is not a peer but a parent. Use the same x-scale as the panels so that vertical alignment works. If the summary uses a shared y-scale and the panels use independent scales, label this clearly: the viewer will otherwise try to compare magnitudes between the summary and the panels, which is comparing two different scale domains.

When This Skill Hurts

Failure modes to watch for

Too many panels. An LLM given a dataset with 50 US states will produce a 50-panel grid because the code is a loop and the data has 50 values. It will not check whether 50 panels fit on screen, whether each panel is large enough to read, or whether the reader can meaningfully compare panel 7 to panel 43. The result is a wall of tiny, identical-looking rectangles that communicates nothing except "there are 50 states." The fix is never mechanical (just make the panels smaller). It requires editorial judgment: which states matter for this story, and which can be aggregated or omitted?

Independent scales that prevent comparison while appearing comparable. This is the subtlest failure mode. When each panel has its own y-range, two panels can show curves with identical visual shapes but completely different magnitudes. A panel that peaks at 5 and a panel that peaks at 500 look the same if both stretch to fill their vertical space. The reader, trained by every other chart they have ever seen, assumes that similar-looking curves mean similar values. They do not check the axis ticks on all 12 panels. The result is a chart that actively misinforms: it looks like it supports comparison while silently defeating it. Mitigate by labeling the y-range prominently in each panel, using a different background shade when scales differ, or pairing independent panels with a shared-scale summary chart that anchors the reader's sense of magnitude.

Shared scales that waste space when ranges differ by 10x. The mirror-image problem: when one series ranges from 0 to 1000 and another from 0 to 10, a shared y-axis allocates the full vertical span to the high-range series and crushes the low-range series into a thin sliver at the bottom. The low-range panel has a flatline that uses 1% of the available pixels. The reader concludes "nothing interesting here," when in fact the low-range series has rich structure that a zoomed-in view would reveal. The shared scale is not wrong, but it is hiding data in pursuit of comparability. If the ranges differ by more than about 10x, shared scales require normalization to be honest.

Responsive reflow that breaks spatial memory. When the browser resizes and the grid reflows from 4 columns to 3, every panel after the third one moves to a new position. The reader who had built a mental map ("Tokyo is top-left, Moscow is second row center") loses that map entirely. If the reader was mid-comparison, they have to find both panels again and restart. Frequent reflows during a resize drag make this worse: the panels jump repeatedly, and the reader gives up. Debouncing helps (reflow only after the resize settles) and keeping column count stable across common breakpoints helps more. But the fundamental issue is that reflow changes the grid's topology, and topology is what the reader's spatial memory is built on. For critical comparison tasks, consider a fixed layout that scrolls horizontally rather than reflowing.
