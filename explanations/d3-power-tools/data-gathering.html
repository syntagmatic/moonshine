Data Gathering

80% of visualization work is getting the data into the right shape. D3's power comes from its transformation functions, not just its rendering.

Most D3 tutorials start with the pretty part: bindings, scales, axes, transitions. But the work that determines whether a chart tells the truth happens before any of that. CSV cells arrive as strings. Dates look like text. Missing values hide as empty strings that silently coerce to zero. The difference between a correct visualization and a misleading one almost always lives in the data preparation code, not the rendering code.

D3 ships a small but precise toolkit for this: type coercion in row accessors, d3.group and d3.rollup for reshaping, d3.bin for histograms, d3.sort for ordering. Each function takes flat tabular data and transforms it into the structure a particular chart type needs. The table below contains 20 rows of city weather data, exactly as it would arrive from a CSV file. Step through the pipeline to watch the data change shape, and notice how each transformation unlocks a different kind of chart.

Bad data does not throw errors. It draws wrong charts. A bar at zero that should be absent, a line that zigzags because dates are unsorted, a scale that blows out because of a sentinel value. Every transformation below is a visual bug you will never find by reading code alone.

What Each Step Does

Type coercion is the most critical step and the easiest to skip. In CSV, every value is a string. The expression "3" + "5" evaluates to "35", not 8. The unary plus operator (+d.temp) converts to a number, and d3.timeParse converts date strings into Date objects. But the real trap is empty cells: +"" evaluates to 0, so a missing temperature reading silently becomes a freezing day. The safe pattern is to check for empty strings before coercing: d.temp === "" ? null : +d.temp.

Grouping with d3.group reorganizes flat rows into a nested InternMap keyed by category. This is the structure that small multiples, stacked charts, and color-coded series need. Rollup with d3.rollup goes further: it collapses each group into a single summary value (mean, sum, count), producing the data shape that bar charts and summary tables consume. Binning with d3.bin partitions a continuous numeric range into intervals, which is exactly what a histogram needs. And d3.sort returns a new sorted array (unlike Array.sort, which mutates in place), ensuring lines connect in the right order and ranked bar charts read correctly.

autoType: Convenient Until It Isn't

d3.autoType inspects every cell in a CSV row and guesses its type. Numbers become numbers. ISO-8601 date strings become Dates. Empty strings become null. For quick prototyping this is a gift: one argument to d3.csv and your data has real types. But autoType has no idea what your data means. It only knows what the strings look like.

The classic failure is identifiers that happen to be numeric. US ZIP codes, FIPS county codes, NAICS industry codes, and phone numbers all start with meaningful leading zeros. "07030" becomes 7030. A choropleth that joins county geometries by FIPS code silently drops every county whose code starts with zero, because 7030 !== "07030". The map has holes and no error message explains why. The same problem hits boolean-looking strings: "true" and "false" become actual booleans, which breaks any scale domain that expects strings.

The fix is straightforward. Use an explicit row accessor for any column where the type matters, and reserve d3.autoType for the columns where a wrong guess is harmless. In practice, that means you almost always write your own accessor once the dataset moves past the exploratory phase.

The Three Faces of Missing Data

Missing data in JavaScript takes at least three forms, and they behave differently. An empty CSV cell arrives as "" (empty string). JSON can give you null (the key exists but has no value) or undefined (the key is absent entirely). These three values coerce differently under the unary plus operator: +"" is 0, +null is 0, and +undefined is NaN. The first two are the dangerous ones, because they produce a real number where there should be nothing. A bar chart that maps 0 to a bar height draws a bar at zero, which a viewer reads as "the measured value was zero," not "no measurement was taken."

D3's aggregation functions handle null and undefined gracefully. d3.mean, d3.sum, and d3.extent all skip NaN, null, and undefined values. But they do not skip 0. So the correct missing-data pattern is to convert empty strings and sentinel values to null in your row accessor, then let D3's aggregation functions do the right thing downstream. Never rely on the empty-string-to-zero coercion and hope the chart will look right.

Cross-Join and Lookup Patterns

Real analysis rarely works with a single table. You join population data to county geometries, merge temperature readings with station metadata, or attach category labels to numeric codes. D3 provides d3.index for building O(1) lookup maps: pass it an array and one or more key accessors, and it returns an InternMap you can query with .get(). For multi-key lookups, nest the accessors: d3.index(data, d => d.state, d => d.year) gives you lookup.get("CA").get(2024).

Cross-joins are less common but occasionally necessary. If you have an array of regions and an array of years, and you need every combination, use d3.cross(regions, years). This returns an array of [region, year] pairs. The typical use case is filling in a matrix where some combinations have no observations: generate the full cross-product, then left-join your actual data, so the gaps become explicit null values rather than missing rows. A heatmap built from incomplete data without this step will have invisible holes where the viewer assumes the cell is empty, not absent.

When joining datasets, always check for orphans: keys that exist in one table but not the other. For choropleths this is especially critical because a FIPS code in the data with no matching geometry means a data point disappears, and a geometry with no matching data means an unfilled polygon. Both are silent failures.

Normalization Strategies

When comparing variables that live on different scales, raw values mislead. Income in dollars and life expectancy in years cannot share an axis meaningfully. Normalization remaps each variable into a common range. The choice of normalization method changes what the chart emphasizes.

Min-max normalization rescales to [0, 1]: (x - min) / (max - min). Simple and intuitive, but a single extreme outlier compresses all other values into a narrow band. Best for bounded data where the range is stable. In D3, this is just a scaleLinear with domain set to d3.extent and range set to [0, 1].

Z-score normalization (standardization) centers each variable at zero with unit variance: (x - mean) / stddev. Outliers keep their relative distance rather than compressing everything else. This is the standard choice for parallel coordinates, where axes must be comparable. Compute it manually: const mu = d3.mean(values); const sd = d3.deviation(values); const z = values.map(v => (v - mu) / sd);

Rank normalization replaces each value with its position in sorted order: first, second, third. This eliminates the effect of outliers entirely, because the gap between the largest and second-largest value becomes the same one-rank step as any other adjacent pair. The tradeoff is that you lose all information about the magnitude of differences. In D3, use d3.rank(values) to get the rank array, then normalize to [0, 1] by dividing by values.length - 1.

Date Parsing Pitfalls

Date handling in JavaScript is notoriously fragile, and D3's date parsing inherits that fragility while trying to control it. The core issue: the Date constructor itself is inconsistent. new Date("2024-03-15") parses as UTC midnight, but new Date("2024-03-15T00:00:00") parses as local midnight. This one-line difference can shift your data by a full calendar day depending on the user's timezone.

D3 offers two parallel tracks: d3.timeParse / d3.timeFormat for local time, and d3.utcParse / d3.utcFormat for UTC. The rule of thumb is straightforward. If your dates are calendar dates with no time component (birth dates, election days, monthly reports), use UTC everywhere: d3.utcParse to parse, d3.scaleUtc for the axis, d3.utcDay for interval calculations. This avoids the midnight-shift bug where March 15 in UTC displays as March 14 in UTC-5.

The most common D3 date bug is a format mismatch that returns null silently. d3.timeParse("%Y-%m-%d")("15/03/2024") returns null because the format string expects dashes and the data has slashes. No error, no warning. Your entire date column becomes an array of null values, and the chart renders empty. Always validate at least one parsed date immediately: const d = parse(raw); if (!d) throw new Error("Date parse failed: " + raw);

ISO 8601 format (YYYY-MM-DD) is the safest choice for data exchange because it is unambiguous. Locale-dependent formats like MM/DD/YYYY versus DD/MM/YYYY are a persistent source of bugs. The string "04/05/2024" is April 5 in the US and May 4 in most of Europe. If your data comes from a spreadsheet, check the actual format before assuming the American convention.

Daylight saving time transitions create another class of subtle bug. Around the spring-forward transition, d3.timeDay bins are 23 hours long. Around the fall-back, they are 25 hours. For most visualizations this does not matter. But if you are computing rates (events per hour within each day bin), the uneven bin widths introduce artifacts. Use d3.utcDay for uniform 24-hour bins when rates matter.

When This Skill Hurts

Failure modes to watch for

autoType guessing wrong types. ZIP codes, FIPS codes, NAICS codes, phone numbers, and product SKUs all look numeric but are identifiers where leading zeros carry meaning. d3.autoType strips the zeros, and downstream joins fail silently. The same problem hits columns that contain only "true" and "false": they become booleans, which breaks any ordinal scale or string comparison. The fix is simple but requires discipline. Every column that you will use as a join key or categorical label should be parsed explicitly as a string, not trusted to autoType's heuristics.

LLMs generating synthetic data. When a language model creates sample data to demonstrate a visualization, the numbers tend to be suspiciously clean. Uniform distributions where real data is skewed. No missing values. No duplicates. No inconsistent category spellings. Perfectly formatted dates. The chart looks great, but it never encounters the edge cases that real data produces: the empty-string-to-zero coercion, the sentinel value that blows out the scale, the date format that varies between rows. When the same code runs on actual data, it breaks in ways the prototype never predicted.

Reshaping that loses provenance. Once you d3.rollup a dataset, you have summary statistics with no way to trace back to the source rows. If a bar in your aggregated chart looks wrong, you cannot click through to see which rows contributed to it. For exploratory analysis, keep the pre-aggregated data available for drill-down. For published visualizations, document the aggregation in the caption so readers understand what was collapsed. The worst case is a pipeline of multiple reshaping steps where a bug in the second step is invisible because you cannot inspect the intermediate form.

Binning that hides or fabricates patterns. d3.bin with too few bins merges distinct clusters into one bar, hiding bimodality or subpopulations. Too many bins fragmentize continuous distributions into noisy spikes that suggest patterns where none exist. The default threshold algorithm in D3 (Sturges' formula) works reasonably well for roughly normal data, but it underestimates for skewed distributions and overestimates for very small samples. Always try two or three different bin counts before committing to one, and compare the resulting histograms to see whether the story changes.
