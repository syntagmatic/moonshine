Annotation

A chart without annotation shows data. A chart with annotation tells a story.

Most charts stop too early. They show the data accurately, the axes are labeled, the colors are correct, and the viewer is left to figure out what they are looking at. Annotation bridges the gap between "here is what happened" and "here is why it matters." A reference line gives the data a baseline for comparison. A callout points to the one moment that justifies the chart's existence. A shaded region groups time into before and after. Direct labels eliminate the back-and-forth between a legend and the data.

The key insight: annotation is editorial judgment. You are choosing what the reader should notice first. Without annotation, every data point has equal weight. With it, the chart makes an argument.

Good annotation follows a hierarchy. Callouts with leader lines carry the headline claim. Reference lines provide the "compared to what?" context. Direct labels answer "which one?" And tooltips handle everything else on demand. The skill file calls this the 3-annotation rule: more than three callouts and you have a list, not a story.

Force-Based Label Placement

Scatter plots with labels face a combinatorial problem: labels are rectangular, data points are arbitrary, and any fixed offset strategy breaks down once two points sit close together. The brute-force solution is manual nudging, but that falls apart whenever the data changes. A better approach is to run a secondary force simulation whose sole job is finding non-overlapping positions for labels.

The idea is to create a parallel set of nodes, one per label, each tethered to its data point by a short link force. A rectangular collision force pushes overlapping labels apart. A bounding-box force keeps labels inside the chart area. After enough ticks, every label has settled into a position that is close to its point and does not collide with its neighbors.

The critical detail is that this simulation should not animate. Run it synchronously before the first paint: call simulation.stop(), then loop simulation.tick() 120 or more times. The viewer sees only the final result. If you let the simulation animate, labels visibly jitter into place on load. That jitter signals that the chart is still computing, and it draws attention to the labels themselves rather than the data they describe.

Rectangular collision is the hard part. D3's built-in forceCollide treats nodes as circles, which wastes space around short labels and underestimates overlap on long ones. The skill file provides a custom rectCollideForce that checks axis-aligned bounding boxes. It runs O(n^2) per tick, which is fine for the 10-50 labels typical in annotated charts. Beyond that, a quadtree spatial index brings it back to O(n log n).

Estimating label width without the DOM is useful for pre-computation. A rough heuristic: label.length * fontSize * 0.6 works for proportional sans-serif fonts. For precise results, append the text to a hidden SVG, call getComputedTextLength(), then remove it. The hidden-SVG approach is slower but handles variable-width characters and font ligatures correctly.

Two tuning parameters matter most. Link distance controls how far a label can drift from its point. Too short and the simulation cannot resolve tight clusters. Too long and the leader lines become confusing, with labels appearing to belong to the wrong data point. A good starting range is 30 to 60 pixels. The second parameter is iteration count. 120 ticks is usually enough for 20 labels. If labels still overlap after that, increase to 200 or add more charge repulsion rather than more ticks.

Responsive Labels

Labels that work at 900 pixels wide do not work at 360 pixels. The text is the same size, but the chart has shrunk by more than half, so labels overlap, overflow the bounds, or crowd the data into illegibility. Responsive annotation requires more than reflowing the layout. It requires deciding which annotations to keep and which to cut.

The simplest strategy is thinning. At narrow widths, show every second or third label. Keep the ones that are annotated as important, the ones the viewer must see, and drop the rest. The skill file gives a pattern for this: check the container width at breakpoints, set display: none on non-essential labels, and keep annotated outliers visible regardless of width.

For callout annotations, store positions in data coordinates rather than pixel coordinates. On resize, recompute pixel positions from the scales. Then run overflow detection: if a callout's note box exceeds the right edge, flip dx to negative. If it exceeds the bottom, flip dy. This prevents callouts from being clipped or rendered outside the viewport on small screens.

At the narrowest breakpoints (under 400 pixels), consider replacing callout annotations with tooltips entirely. A callout that works at desktop width may consume a quarter of the chart area on a phone. The annotation's content is still available on tap, but it no longer competes with the data for space. This is a legitimate tradeoff, not a compromise. Mobile readers expect tap-to-reveal interactions in a way desktop readers do not.

Force-based label placement interacts with responsiveness in a subtle way. If you re-run the label simulation on every resize, the labels will jump to new positions. That is usually fine for a one-time reflow, but if the user is dragging the window edge, it creates a distracting flickering effect. Debounce the simulation re-run by 150-250ms so it only fires once the resize has settled.

The d3-annotation Library

Susie Lu's d3-annotation library formalizes annotations into a three-part architecture: subject (the data element being annotated), connector (the leader line), and note (the text). Each part is independently configurable. The library provides built-in types for callouts, circles, rectangles, and badges, along with connector styles including curve, elbow, and line.

The library's strongest feature is editMode(true), which makes annotations draggable in the browser. You position them visually, read the final coordinates from the console, and paste them back into your code. This is far faster than the guess-and-reload cycle of hand-positioning callouts. For charts with three or more richly styled callouts, d3-annotation saves real time.

The tradeoff is maintenance. The library was last updated in 2019 and targets D3 v5. It still works with D3 v7 in most cases because the selection API has remained stable, but edge cases around transitions and event handling can surface. It also bundles its own opinions about styling, which may conflict with your chart's typographic hierarchy. For Canvas rendering, custom connector geometries, or projects that need to control the full annotation lifecycle, hand-rolling is the better path.

A middle ground is to treat annotations as structured data (the pattern the skill file calls "annotation as data") without using the library. Define each annotation as a JSON object with coordinates, offset, title, body, priority, and connector type. Render them with a standard .data().join() pattern. You get the separation of concerns and priority filtering that d3-annotation provides, but with full control over the rendering and no external dependency.

Tooltip Design

The first decision is HTML div vs. SVG <title>. The SVG title element is accessible by default and requires no positioning code, but it renders as a browser-native tooltip with no styling control, a delay before appearance, and no rich content. For anything beyond a single line of plain text, use a positioned HTML div. Every serious D3 tooltip in production uses this approach.

Position the tooltip relative to the data point, not the mouse cursor. Mouse-relative tooltips wobble as the user moves, creating a jittery reading experience. Point-relative tooltips are stable: compute the screen position of the data element, offset by a fixed amount, and clamp to the viewport edges. The skill file's edge-clamping pattern handles this: measure the tooltip's own width and height after setting its content, then cap the left and top values so the tooltip never overflows the window.

For scatter plots and dense line charts, a Voronoi overlay dramatically improves the hover experience. Instead of requiring the cursor to land directly on a 3-pixel dot, the Voronoi tessellation gives every point a large hover region. Use d3.Delaunay.from() to build the tessellation, then attach pointer events to an invisible overlay rect. On pointermove, call delaunay.find(mx, my) to locate the nearest point. This approach also anchors the tooltip to the data point rather than the cursor, so the tooltip stays locked in place while the user's mouse moves within that point's Voronoi cell.

Keep tooltip content minimal. A tooltip that displays eight fields in a tiny box is a sign that the chart's visual encoding is not doing enough work. If the viewer needs the tooltip to understand the chart, the chart has a design problem. Tooltips should confirm what the viewer already suspects from the visual pattern, or provide a secondary detail (an exact value, a name, a date) that the encoding cannot carry. Format numbers, use the same units as the axis, and never repeat information that is already visually encoded.

Transition opacity rather than display. Snapping a tooltip in and out feels jarring. A fast fade (100-150ms) smooths the appearance without adding perceptible delay. But do not animate the tooltip's position. A tooltip that slides across the screen as the user moves between data points is distracting and slows down the scanning pattern that tooltips are meant to support.

Direct Labeling for Multi-Series Charts

Legends force the viewer to look away from the data. They match a color swatch to a name, then carry that association back to the chart, then repeat for the next series. This round trip is slow. Direct labels eliminate it by placing the series name right next to the data it describes.

The most common placement is at the endpoint of each line. Extend the right margin to make room, then place each label at the last data point's y-coordinate, offset a few pixels to the right. Color the label to match its line. If two lines end at similar y-values, nudge one up and the other down by half the font height. For more than four or five series, a simple pairwise nudge breaks down and you need the force-based approach described above.

An alternative is inline labeling: placing the series name along the line itself at a point of maximum separation from other series. Find the x-position where the y-gap between the target series and its nearest neighbor is largest, then place the label there. This avoids the right-margin crowding problem entirely, but it requires enough whitespace within the chart area that the label does not obscure other data.

For area charts and stacked layouts, label placement inside the area is natural. Center the label vertically within the band at the widest point. If the band is too thin for text, fall back to a leader line pointing into the band from outside. The perceptual threshold for readable text inside a colored band is roughly 1.5 times the font height. Below that, the text feels cramped even if it technically fits.

Direct labels interact with color in an important way. If the chart uses a categorical palette, the label color should match the series color exactly. But if the label sits on a colored background (inside an area, for instance), it needs sufficient contrast against that background, not against the white chart background. Check contrast ratios. A blue label on a light blue area is unreadable even though both are "blue."

When This Skill Hurts

Failure modes to watch for

Over-annotation. The skill encourages callouts, reference lines, and direct labels. LLMs take this as a mandate to annotate everything. A chart with a callout on every data point is worse than a chart with no callouts at all, because the visual hierarchy collapses: nothing is emphasized when everything is emphasized. The 3-annotation rule exists precisely to prevent this, but it gets ignored in practice. If you find yourself writing a fourth callout, step back and ask whether the chart is trying to make too many claims at once.

Labels that obscure the data. A label placed directly on top of the data point it describes defeats its own purpose. The viewer cannot see the value that the label is calling attention to. This happens most often with force-based placement when the link distance is too short, with direct labels on dense scatter plots, and with callout notes that cover nearby data points. Labels should sit adjacent to the data, not on top of it. Background rectangles with slight opacity help, but they are a patch for a positioning problem.

Tooltips as a crutch for poor visual design. When a chart requires a tooltip to be understood at all, the chart's visual encoding is not working. Tooltips should provide supplementary detail, not carry the primary message. A common pattern: a scatter plot where every point is the same color and size, with the only distinguishing information in the tooltip. That is a data table with extra steps. If removing all tooltips makes the chart incomprehensible, the chart needs a redesign, not more tooltips.

Force-label simulation that visibly jitters on load. Running the label simulation with animation turned on makes every label slide into position over the first second of page load. This looks broken. It draws the viewer's eye to the mechanics of the layout rather than the data. The fix is simple: pre-compute the simulation synchronously before rendering. Call simulation.stop() and then simulation.tick(120) in a loop. The viewer sees only the converged result. If the simulation is still producing visible movement after pre-computation, increase velocityDecay to 0.6 or higher, or add more ticks.
