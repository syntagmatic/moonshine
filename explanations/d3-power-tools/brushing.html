Brushing

Asking questions with your hand instead of writing queries.

A database query is precise but slow. You write a filter, run it, look at the result, adjust, run again. Brushing inverts that loop. You drag a rectangle across a scatter plot and the answer appears while your hand is still moving. The question and the answer happen in the same gesture.

The interaction below shows this directly. Drag a rectangle over any region of the scatter plot. The points inside the rectangle light up; everything outside fades to a ghost. A linked histogram on the right updates in real time to show how the selected subset is distributed. Click anywhere outside the brush to clear it and return to the full dataset.

Brushing works because it turns a precise question ("show me points where 30 < x < 60 and y > 40") into a spatial one ("show me that cluster over there"). You do not need to know the coordinates. You just point.

What the Code Does

The scatter plot uses d3.brush(), which handles all the pointer event plumbing: drag to create the rectangle, drag edges to resize, drag interior to move, click outside to dismiss. When the brush extent changes, a single callback receives the pixel coordinates of the rectangle. The callback converts those pixels back to data coordinates using scale.invert(), tests each point for containment, and splits the data into two layers: active (full opacity) and ghost (low opacity). The histogram re-bins just the active subset on every brush event.

The linked histogram is the key idea from cross-filtering. Both views share the same data array. When one view changes its selection, the other view redraws to reflect it. The selection manager pattern from the d3-power-tools brushing skill generalizes this: any number of views can listen to the same selection, and each view skips its own events to avoid feedback loops.

Brush Types

D3 provides three built-in brush constructors. d3.brush() creates a two-dimensional rectangular selection, the one used in Figure 1 above. d3.brushX() creates a one-dimensional horizontal brush, constraining the selection to a range along the x-axis. d3.brushY() does the same vertically. The 1D variants are more common in practice because most filtering tasks involve a single dimension: selecting a date range on a timeline, narrowing a price range on a histogram, picking a score band on a parallel coordinate axis.

The choice matters for the viewer. A 2D brush asks "show me the cluster in this region." A 1D brush asks "show me everything in this range." When your question is really about one axis, a 2D brush forces unnecessary precision on the other axis. People struggle to draw rectangles that span the full height of the chart. A brushX solves this by constraining the selection to a horizontal band automatically.

A good heuristic: if the interaction maps to a WHERE clause on a single column, use a 1D brush. If it maps to a WHERE clause on two columns, use a 2D brush. If the selection shape is irregular (selecting a cluster that is not axis-aligned), use a lasso.

Beyond rectangles, there is the lasso: a freeform closed path drawn by the user. The viewer clicks and drags to trace an arbitrary polygon around the points of interest, and on release, a point-in-polygon test determines which data falls inside. This is powerful for selecting non-rectangular clusters in scatter plots. The d3-power-tools skill provides a ray-casting pointInPolygon function, and D3 itself offers d3.polygonContains from the d3-polygon module.

One-Dimensional Brushing

The histogram below shows the same 200 data points, binned by their x-values. Drag horizontally to select a range. The bars inside the range stay solid; bars outside fade to ghosts. The count label updates to show how many points fall in the selected range. This is how d3.brushX() works: the brush extent is a pair of pixel values [x0, x1] rather than a rectangle, and scale.invert() converts them back to data coordinates for filtering.

Notice that the brush handle spans the full height of the chart. You cannot accidentally miss data by dragging too narrow a rectangle. It also communicates intent clearly: the viewer sees a vertical band sweeping across the axis and understands immediately that this is a range filter.

Lasso Selection

Sometimes the data forms clusters that no rectangle can capture cleanly. A tight group of points in a scatter plot might be diagonal, crescent-shaped, or separated from its neighbors by a gap that does not align with either axis. For these cases, lasso selection lets the viewer draw a freeform boundary around exactly the points they want.

The scatter plot below supports lasso selection. Click and drag to draw a closed path around a group of points. When you release, the path closes and the containment test runs: points inside light up, points outside dim. Click anywhere to clear. The lasso uses getCoalescedEvents() when available to capture smooth pointer paths, and falls back to regular events on browsers that do not support it.

Lasso selection is powerful but harder to discover than rectangular brushing. If you use it, add a visible hint or prompt. Without instructions, most viewers will try to click individual points or drag a rectangle, and never realize they can draw a freeform path.

The Ghost/Active Layer Pattern

Every figure on this page uses the same visual strategy: when a selection is active, unselected data dims to a ghost rather than disappearing. This solves a real problem. If you simply remove unselected points, the viewer loses all spatial context. Where did the rest of the data go? How does my selection relate to the whole? The ghost layer answers both questions. The full dataset remains visible as a faint silhouette, and the selected subset draws on top in full color.

Implementation is two SVG groups (or two Canvas passes). The ghost layer draws all data at low opacity, typically 0.08 to 0.15. The active layer draws only selected data at full opacity, on top. When the selection changes, update both layers: the ghost layer reduces opacity for everything, and the active layer redraws with just the selected points. When the selection clears, remove the active layer and restore the ghost layer to full opacity.

This pattern also works with Canvas. For high-volume data (10K+ points), use two Canvas layers stacked with CSS. The background canvas draws all points with globalAlpha: 0.05. The foreground canvas draws selected points with globalAlpha: 0.8. Clear and redraw both on every selection change. The visual effect is the same as the SVG version, but without the DOM overhead of thousands of elements.

Fisheye Distortion

Brushing is a binary operation: data is either in the selection or out. Fisheye distortion offers a softer alternative. Instead of splitting data into two groups, it warps the coordinate space around the pointer so that nearby data magnifies and distant data compresses. Nothing is hidden. Nothing is dimmed. The viewer explores by moving the mouse, and the data rearranges continuously around the focus point.

The core formula maps each point's distance from the focus through a distortion function: d * (k + 1) / (d * k + 1), where k controls the distortion strength and d is the normalized distance within a given radius. Points near the focus spread apart; points far from the focus squeeze together. Cartesian fisheye applies this to one axis at a time (useful on parallel coordinate axes). Radial fisheye applies it to the 2D distance, which works on scatter plots and node-link diagrams.

Fisheye works best when the goal is local inspection of dense data, not selection. It is a focus+context technique: the viewer sees both the neighborhood in detail and the broader dataset in compressed form. The drawback is that it distorts coordinates, which can mislead if the viewer forgets the distortion is active. Labels and axes should move with the distortion or be suppressed entirely during fisheye mode.

Cross-Chart Linking

The most powerful application of brushing is linking multiple charts together. Brush in chart A, and charts B and C update to show only the selected subset. This is the principle behind Shneiderman's dynamic queries (1994) and the crossfilter library. The viewer asks a question in one view and sees the answer propagated across all views simultaneously.

The implementation requires a shared selection manager: an object that tracks which data indices are currently selected and notifies all views when the selection changes. Each view registers a listener. When one view brushes, it updates the manager with the selected indices and a source identifier. The manager dispatches the new selection to all listeners. Each listener skips the event if it originated from itself, preventing feedback loops.

The "source" parameter on selection events is critical. Without it, view A's brush triggers a redraw of view A, which triggers another selection event, which triggers another redraw, and the browser locks up in an infinite loop. Every cross-chart linking system needs this guard.

Figure 1 at the top of this page demonstrates this pattern in miniature: the scatter plot and histogram are linked through a shared data array and a single brush callback that updates both. A full cross-filtering dashboard extends this to three, four, or more views, each with its own brush, all connected through a SelectionManager.

SVG vs Canvas Selection State

In SVG, selection state lives in the DOM. Each element is a node you can query, style, and animate individually. To highlight selected points, set their opacity attribute. To dim ghosts, reduce their opacity. D3's data join handles enter/update/exit transitions naturally. The cost is DOM overhead: beyond a few thousand elements, the browser slows down updating that many attributes on every brush event.

In Canvas, there is no DOM for data elements. Selection state must live in a JavaScript data structure, typically a Set of indices or a boolean array. On every brush event, you clear the canvas and redraw everything from scratch, using the set to decide each point's opacity and color. This is faster for large datasets because there is no DOM diffing, but it requires careful bookkeeping. You cannot "select" a Canvas pixel. You must track which data each pixel represents and test containment yourself, often using a quadtree for efficient spatial lookup.

The hybrid approach is the most practical for complex interactions: use Canvas for the data layer (points, lines, filled regions) and SVG for the interaction layer (brush handles, lasso path, axes, labels). The Canvas handles the volume; the SVG handles the manipulation. This is the pattern used by d3.parcoords and recommended by the d3-power-tools skill.

Snap-to-Data Brushing

By default, a brush can be positioned anywhere within its extent. The selection rectangle snaps to pixels, not to data boundaries. This is fine for continuous data, but for categorical or discrete data it creates ambiguity: does the brush include the bin it partially overlaps?

Snap-to-data brushing constrains the brush extent to align with data boundaries. On a histogram, the brush snaps to bin edges. On a bar chart, it snaps to bar boundaries. The implementation uses the brush's end event: when the user releases, round the brush extent to the nearest data boundaries and call brush.move() to snap the selection into place. The snap should happen after the user finishes dragging, not during, to avoid fighting the pointer.

The code pattern is straightforward: in the end handler, read the current selection, compute the snapped values using Math.round or a quantize scale, and if the snapped values differ from the raw values, call brushBehavior.move(group, [snappedX0, snappedX1]). The brush transitions smoothly to the snapped position.

When This Skill Hurts

Discoverability. Brushing is invisible UI. There is no button, no icon, no visual affordance that says "drag here to select." The viewer sees a static chart and has no reason to try dragging. LLMs following this skill often build powerful brush interactions that no one ever finds because the chart looks the same as a non-interactive one. Always include a visible text hint ("Drag to select a region") or an animated tutorial pulse on first load.

Touch incompatibility. A rectangular brush captures the drag gesture, which on touch devices conflicts with scrolling and panning. The viewer tries to scroll past the chart and instead creates a brush selection. The skill does not adequately warn about this, and LLMs rarely add touch alternatives. On mobile, consider replacing the brush with tap-to-select, a range slider below the chart, or at minimum a toggle that switches between scroll and brush modes.

Over-linking. Connecting every chart through a shared selection manager creates a cognitive maze. When brushing in chart A simultaneously updates charts B, C, and D, the viewer cannot tell what changed or why. Each linked view adds a channel of information the viewer must track. Two linked views are easy to follow. Three is manageable. Four or more requires careful visual design: consistent highlight colors, smooth transitions, and spatial proximity so the eye can track changes across views without hunting.

Empty selection confusion. Brushing to a region with no data can leave every chart blank with no feedback. The histogram shows zero-height bars. The scatter plot shows nothing. The viewer wonders if the visualization is broken. The skill should encourage "no matches" messaging, but LLMs often skip it. Always check for empty selections and show a clear message: "No data in this range" or "0 of 200 points selected." The figures on this page handle this by updating the status bar count, but more explicit messaging is better for dashboards.
