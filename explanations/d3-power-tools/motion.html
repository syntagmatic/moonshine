Motion

A transition that connects two states helps the viewer track what changed. A cut forces them to rebuild context.

Click "Re-sort" below with the default "No transition" mode selected. The bars jump to new positions instantly. Unless the chart is tiny, it is nearly impossible to follow which bar went where. Now switch to "Position" and click again. The bars slide, and suddenly your eye can track each one. The same data, the same operation, but motion turns a disorienting jump into a comprehensible change.

This is the core purpose of animation in data visualization: maintaining object constancy. When a bar slides from one slot to another, your visual system treats it as the same object in a new location. Without the transition, each bar is destroyed and a new one appears somewhere else. You have to re-read every label to figure out what happened.

Animation should answer one question: "What changed?" If a transition does not help the viewer track changes, remove it. Gratuitous motion is worse than none at all, because it wastes time and suggests significance where there is none.

Why Staggering Matters

The staggered mode does something that a simultaneous transition cannot: it reveals the process of reordering, not just the result. When all bars move at once, you see the starting state and the ending state with a blur of motion in between. When they move one at a time, you can actually watch each bar find its place. For a small number of elements this is the clearest way to communicate a sort. For more than about 20 elements, staggering becomes slow, and a simultaneous transition with a good easing curve works better.

The enter and exit animations use the same principle. A bar that grows from zero height is clearly new. A bar that shrinks to zero is clearly leaving. Without these transitions, the chart population changes silently and the viewer has to scan every label to figure out what is different.

The Personality of Easing

Every easing function encodes a physical metaphor. d3.easeCubicOut decelerates into its final position, like a drawer sliding shut on a soft-close rail. It is the right default for most UI transitions: the fast start feels responsive, and the gentle stop draws the eye to the final state. d3.easeLinear moves at constant speed, which looks mechanical and slightly unnatural for interface motion, but is exactly right for data playback, timelines scrubbing forward, or any case where the viewer expects uniform progress. d3.easeElasticOut overshoots and bounces, demanding attention. Use it once to call out a critical change; use it everywhere and the chart feels like a toy.

The less common easings have real uses too. d3.easeBackOut pulls back slightly before settling, creating anticipation that works for elements arriving on screen. d3.easeCubicInOut is the safest general-purpose choice: it accelerates into and decelerates out of the motion, making the intermediate state feel smooth rather than abrupt. The rule of thumb from animation practice holds in data visualization: out-easing for entrances (things arriving), in-easing for exits (things departing), in-out for position changes (things moving from here to there).

Interrupting In-Flight Transitions

When a user clicks while a transition is still running, D3's default behavior is to interrupt the old transition and start the new one from whatever intermediate state the element has reached. This is usually what you want. But if you do not handle it explicitly, you can end up with transitions that stack, compound, or restart from unexpected positions.

The method is selection.interrupt(). Call it before starting a new transition on the same element to cancel any in-flight transition cleanly. Without it, rapid clicks queue up transitions that play out in sequence, creating a sluggish, laggy feel where the element is always chasing the user's last input. The figure below demonstrates this. Click the left panel rapidly and watch the circle chase your clicks with a growing delay. Then do the same on the right panel, which calls .interrupt() before each transition, and notice how the circle responds immediately.

A subtlety: when you interrupt a transition and start a new one, you need the element's current visual state as the starting point. Do not read DOM attributes mid-transition to get this, because the values returned are the interpolated intermediate values and can jitter. Instead, cache the intended target in a variable or a Map, and when interrupting, read the element's current computed position via getBBox() or the attribute you last set. The skill recommends a target-cache pattern for this reason.

Canvas Animation

D3's .transition() works on DOM elements. Canvas has no DOM elements to transition, so you animate by redrawing the entire frame in a loop. The standard approach is requestAnimationFrame, which gives you a callback before each screen repaint, typically 60 times per second. Each frame, you clear the canvas, interpolate every object's position one step closer to its target, and redraw.

A simple lerp (linear interpolation) loop works well for most cases: on each frame, move each object 5% to 10% of the remaining distance toward its target. This produces an exponential ease-out that feels natural without requiring an easing library. When the remaining distance drops below half a pixel, snap to the target and stop updating that object. When all objects have settled, cancel the animation frame to stop burning CPU.

For more controlled timing, use d3.timer, which integrates with D3's internal clock and pauses automatically when the browser tab is hidden. You compute an elapsed fraction, apply an easing function, and interpolate between start and end states. This gives you the same easing vocabulary as SVG transitions, just in a manual loop. The tradeoff is that you must manage start states, end states, and the transition lifecycle yourself.

Object Constancy through Key Functions

The bar chart above works because each bar has an id field, and the data join uses .data(data, d => d.id) to match elements to data points across renders. This is the key function, and it is the single most important detail in D3 transitions. Without it, D3 matches by index: the first data point binds to the first element, the second to the second, and so on. When you sort, add, or remove items, index-based matching produces nonsensical transitions where bars slide to positions that have nothing to do with where their data moved.

The key function must return a value that is stable across data updates and unique within the dataset. Database IDs, names, ISO country codes, and timestamps all work. Array indices do not, because they change when the array is sorted or filtered. This is the most common transition bug in LLM-generated D3 code: the transition is set up correctly, the easing is chosen, the duration is tuned, but the data join has no key function, and the animation tells a false story about which element went where.

Respecting prefers-reduced-motion

Some viewers experience motion sickness, vertigo, or seizures from screen animation. The prefers-reduced-motion media query lets you detect when a user has asked their operating system to minimize motion. The responsible approach is to check this query and, when it matches, skip the interpolation and jump directly to the final state.

This page already does this: the getDuration() function returns 0 when reduced motion is preferred, which makes every transition instantaneous. The CSS at the top of the page also sets animation-duration and transition-duration to 0 globally. This two-layer approach catches both D3 transitions (which run in JavaScript) and CSS transitions (which do not).

A common mistake in reduced-motion implementations is setting the duration to 0 but keeping transition.end() callbacks that depend on the transition actually running. A zero-duration transition in D3 still fires end events, so this usually works. But if your code uses setTimeout to coordinate stages based on expected durations, those timings break when everything finishes instantly. The safe pattern is to use D3's .transition().end() promise or coordinated delays (where the delay becomes 0 alongside the duration), not independent timers.

Transition as Cognitive Aid

The deepest question about animation in visualization is not "how" but "whether." A transition earns its place when it helps the viewer maintain a mental model across a state change. Sorting a bar chart is the canonical example: without motion, the viewer has to rebuild their mental map of which bar is which. With motion, the map updates continuously and effortlessly. The transition does cognitive work that the viewer's eye would otherwise have to do by scanning back and forth.

But a transition that does not connect two states the viewer cares about is pure decoration. A fade-in on page load does not help anyone understand data. A bounce on hover does not reveal structure. A slide transition between two unrelated views does not maintain constancy because there is no constancy to maintain. These are the "PowerPoint effects" of data visualization: they signal effort but communicate nothing.

A useful heuristic: if you removed the transition and replaced it with an instant cut, would the viewer lose any understanding? If yes, the transition is doing real work. If no, it is decoration, and you should remove it. Most transitions fail this test.

The research literature supports this intuition. Heer and Robertson (2007) found that animated transitions improved accuracy for tracking object identity and understanding data changes, but only when the animation was congruent with the data operation. Incongruent animation, where the visual motion does not match the logical change, actually hurt performance compared to no animation at all. The lesson: animation is not a neutral embellishment that can only help. It is a cognitive signal that can mislead.

When This Skill Hurts

Failure modes to watch for

Gratuitous animation. The skill makes transitions easy to add, so LLMs add them everywhere. Fade-in on page load, bounce on hover, slide on every update. Most of this does not serve understanding and creates a "PowerPoint effect" where the visualization feels busy and self-important. The cost is not just aesthetic: every transition takes time, and cumulative animation delay makes the interface feel sluggish even when each individual transition is within the recommended 300-500ms range. A good rule is that most elements on the page should not animate at all. Reserve motion for the one or two state changes where object constancy matters.

Duration blindness. The skill suggests 300-500ms for simple position changes, and this feels right when you test it once. But when a viewer is actively exploring data, clicking a sort button every second or two, that 500ms transition plays fifty times in a session. What felt smooth in a demo becomes agonizing during actual use. LLMs pick a duration from the skill's table and never revisit it. The fix is to scale duration down with interaction frequency: 500ms for a first sort, 200ms for subsequent rapid sorts, or even instant if the user is clicking faster than the transition can finish.

Transition stacking. Without .interrupt(), rapid interactions queue transitions. Each click adds another 500ms to the queue. After five quick clicks, the chart is two and a half seconds behind the user's intent, swimming drunkenly toward a position the user has already moved past. The skill warns about this and provides the interrupt pattern, but LLMs often omit interrupt handling because the basic transition works fine in a single-click test. The bug only appears under rapid interaction, which is exactly how real users behave when exploring data.

Accessibility neglect. Motion can trigger vestibular disorders, causing dizziness, nausea, or disorientation. The skill mentions prefers-reduced-motion, but LLMs rarely implement it, and when they do, they often just set the duration to 0. This works for simple attribute transitions but breaks layout transitions that depend on transition.end() callbacks, staged delays, or coordinated multi-phase animations. A zero-duration transition that triggers a layout recalculation at the wrong time can produce a flash of mispositioned elements before the final state resolves. The correct approach is to skip the transition entirely: set final attributes directly, fire any post-transition logic immediately, and ensure the end state is identical to what the animated path would have produced.
