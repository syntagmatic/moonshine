Canvas Rendering

SVG creates a DOM element for every data point. Canvas draws pixels directly. Past about 2,000 elements, that distinction becomes the whole ballgame.

Every SVG circle is a real DOM node. The browser tracks its position, its style, its event listeners, its place in the accessibility tree. That is powerful when you have hundreds of elements. It becomes ruinous when you have tens of thousands. The browser chokes on layout recalculations, paint operations slow to a crawl, and interactions become unresponsive. Canvas sidesteps all of this: it gives you a bitmap and a drawing API. You issue commands like "draw a circle here," and pixels appear. No DOM nodes, no layout, no overhead per element.

The tradeoff is real, though. With SVG, the browser handles hit detection, accessibility, and event dispatch for free. With Canvas, you get a flat image. Hovering over a point? You need spatial indexing. Screen reader support? You need a hidden DOM mirror. But for large-scale data visualization, the performance gain is not marginal. It is the difference between a working tool and an unusable one.

D3 handles data: scales, layouts, binning. Canvas handles pixels. There is no data join. You compute positions into arrays, then iterate and draw. One beginPath/fill pair for all points of the same style, not one per point.

Somewhere around 2,000 points, SVG starts to labor. By 10,000, it struggles visibly. Canvas barely notices. On the Canvas side, hover over the plot to see quadtree hit detection at work: a d3.quadtree finds the nearest point in O(log n) time, regardless of how many points are on screen.

Quadtree Hit Detection

Without the DOM, Canvas has no built-in way to know what the cursor is over. The standard solution for point data is d3.quadtree, a spatial index that recursively subdivides the plane into quadrants. Given a mouse position, it walks the tree to find the nearest point in O(log n) time. In the figure above, when you hover in Canvas mode, the quadtree finds the closest point within a 20-pixel search radius and highlights it with a ring. The lookup is fast enough to run on every pointer move, even at 50,000 points, without any perceptible delay.

Multi-Layer Canvas Architecture

A serious Canvas visualization is not one canvas. It is a stack of canvases, each responsible for a different update frequency. The background layer holds the full dataset and redraws only when data changes. The interaction layer sits on top and redraws on every pointer event, just to paint a highlight ring or a selection rectangle. An SVG overlay handles axes, labels, and tooltips, because text and ticks are exactly what SVG is good at.

┌──────────────────────────────────┐ │ SVG overlay (pointer-events) │ axes, labels, tooltips │ Canvas: interaction (top) │ hover highlight, selection ring │ Canvas: foreground │ selected/filtered subset │ Canvas: background │ full dataset, dimmed │ Container div (position:rel) │ └──────────────────────────────────┘

The separation pays off immediately. When the user hovers over a point, you clear and redraw only the interaction canvas. The background layer, which may contain 50,000 circles, stays untouched. That single optimization can take hover response from 30ms down to under 1ms. Each layer is a <canvas> element positioned absolutely inside a relative container, stacked by DOM order. The SVG overlay sits on top with pointer-events: none so it does not intercept mouse events.

A common mistake is putting axes on the same Canvas as the data. When you zoom, you redraw the data, the axes, the grid lines, and the labels every frame. With the SVG overlay pattern, the data canvas draws circles and nothing else. The SVG axes use d3.axisBottom and update only when the scale domain changes. You can even transition axis labels smoothly without touching the Canvas at all.

DPR-Aware Setup

Every Canvas visualization must account for the device pixel ratio. On a 2x Retina display, a 600px-wide canvas that is actually 600 physical pixels looks blurry. The fix: set the canvas element's width and height attributes to the CSS dimensions multiplied by devicePixelRatio, style the element back down to CSS dimensions, and call ctx.scale(dpr, dpr) so all subsequent drawing uses CSS-pixel coordinates.

function addCanvas(container, width, height) {
  const dpr = devicePixelRatio;
  const canvas = document.createElement("canvas");
  canvas.width  = width * dpr;
  canvas.height = height * dpr;
  canvas.style.width  = width + "px";
  canvas.style.height = height + "px";
  canvas.style.position = "absolute";
  canvas.style.top = "0";
  canvas.style.left = "0";
  container.appendChild(canvas);
  const ctx = canvas.getContext("2d");
  ctx.scale(dpr, dpr);
  return ctx;
}

After the ctx.scale call, all drawing commands use CSS pixels. ctx.arc(100, 50, 3, 0, Math.PI*2) draws a crisp circle at CSS coordinate (100, 50) on any display density. You never think about physical pixels again, unless you are doing direct ImageData manipulation, which always works in physical pixel space.

One subtlety: the DPR can change at runtime. A user drags the browser window from a Retina display to an external 1x monitor. Listen for the change with matchMedia and rebuild canvases when it fires.

Typed Arrays for Performance

Standard JavaScript arrays store each number as a 64-bit heap-allocated object. For 100,000 points, that means 100,000 objects the garbage collector must track, scattered across memory with no locality guarantees. Typed arrays (Float32Array, Float64Array) store numbers in a contiguous block of memory with a fixed byte width per element. The CPU cache can prefetch them efficiently, and the garbage collector ignores the contents entirely.

The practical pattern: compute scaled positions from D3 scales into typed arrays once, then iterate the arrays in your draw loop. The draw loop itself touches only flat numeric buffers. No property lookups, no object dereferences, no GC pauses mid-frame.

const xs = new Float32Array(n);
const ys = new Float32Array(n);
for (let i = 0; i < n; i++) {
  xs[i] = xScale(data[i].x);
  ys[i] = yScale(data[i].y);
}

// Draw loop: iterate flat arrays
ctx.beginPath();
for (let i = 0; i < n; i++) {
  ctx.moveTo(xs[i] + r, ys[i]);
  ctx.arc(xs[i], ys[i], r, 0, Math.PI * 2);
}
ctx.fill();

Typed arrays also transfer to Web Workers at zero copy cost via postMessage(buffer, [buffer]). The original array is neutered after transfer. This makes it practical to do heavy computation (force layout ticks, binning, clustering) off the main thread and ship the result back without serialization overhead.

Batched Rendering by Color Bucket

Canvas state changes are expensive. Every time you set ctx.fillStyle, the browser must flush its internal path buffer and potentially reconfigure the rasterizer. If you set a new fill color for each of 50,000 points, you issue 50,000 state changes, each one followed by its own beginPath/fill cycle. This is the single most common Canvas performance mistake, and LLMs reproduce it reliably because the naive per-point pattern looks correct.

The fix: group points by visual style, then issue one beginPath/fill pair per group. For a continuous color scale, quantize into buckets. Twenty buckets means twenty fill calls instead of fifty thousand, with color differences smaller than most viewers can perceive.

// Group points into 20 color buckets
const buckets = 20;
const groups = Array.from({length: buckets}, () => []);
for (let i = 0; i < data.length; i++) {
  const b = Math.min(buckets - 1, Math.floor(data[i].v * buckets));
  groups[b].push(data[i]);
}

// One beginPath/fill per bucket, not per point
for (let b = 0; b < buckets; b++) {
  if (!groups[b].length) continue;
  ctx.fillStyle = colorScale((b + 0.5) / buckets);
  ctx.beginPath();
  for (const d of groups[b]) {
    const px = xScale(d.x), py = yScale(d.y);
    ctx.moveTo(px + r, py);
    ctx.arc(px, py, r, 0, Math.PI * 2);
  }
  ctx.fill();
}

Frame Budgets

At 60 frames per second, each frame gets 16.6 milliseconds. Everything that happens between one requestAnimationFrame callback and the next must fit inside that window: JavaScript execution, layout, paint, compositing. If your Canvas draw loop takes 25ms, the browser drops frames. The user sees jank.

The first rule: coalesce events into a single render. Pointer events, resize events, and data updates can all fire multiple times before the next frame. A dirty flag ensures you draw at most once per frame, no matter how many events arrive.

let dirty = false;
function markDirty() {
  if (!dirty) {
    dirty = true;
    requestAnimationFrame(render);
  }
}
function render() {
  dirty = false;
  drawBackground(bgCtx, data);
  drawForeground(fgCtx, selected);
}

If a full redraw exceeds the frame budget, progressive rendering splits the work across multiple frames. Shuffle the data first so every partial frame shows a representative sample. Draw 2,000 points per frame until the queue is empty. The user sees the chart fill in over several frames, but the first frame appears within 16ms and the interface stays responsive throughout.

Measure before optimizing. Use performance.now() around your draw loop. The stats bar in Figure 1 does exactly this. If the number stays under 10ms, you have headroom. If it crosses 16ms, start with batching, then LOD, then progressive rendering.

Zoom with Level of Detail

Zoom and level of detail are natural partners. At the full extent, showing 50,000 points as 1-pixel dots makes sense: the viewer sees density and clusters. Zoomed in to a neighborhood of 200 visible points, the same 1-pixel treatment wastes the opportunity. Now you have space for 4-pixel circles, colored borders, and text labels that identify individual points. The zoom level determines how many points are visible, which determines what detail is affordable.

The technique is straightforward. On each zoom event, compute the visible viewport by inverting the transform, cull points outside it, and choose a rendering tier based on how many points survive culling. Below about 500 visible points, draw full labels. Between 500 and 5,000, draw sized circles with borders. Above 5,000, drop to 1-pixel dots. The thresholds are not fixed rules; they depend on your layout and what your users need. The principle is: use the screen space you have.

Figure 2. Level-of-detail demo with 8,000 points. At the default zoom, all points render as small colored dots. Zoom in and the visible subset shrinks. When fewer than 500 points remain on screen, labels appear and circles gain borders. The transition is automatic: the renderer picks the detail tier based on how many points are visible.

Combining zoom with LOD solves a fundamental problem in large-data visualization: the overview and the detail cannot coexist at the same scale. Instead of choosing one or the other, you let the viewer navigate between them. The zoom transform provides both the viewport (which points to draw) and the density signal (which detail tier to use). The Canvas just draws whatever the current tier demands.

When This Skill Hurts

Canvas removes DOM accessibility. An SVG scatter plot is visible to screen readers: each circle can carry an ARIA label, and the browser exposes it in the accessibility tree. A Canvas scatter plot is a flat bitmap. Screen readers see a single <canvas> element and nothing inside it. The skill mentions hidden DOM mirrors, but building one correctly is significant work: you need off-screen elements that match the visual state, keyboard focus management, and live-region announcements for data changes. Most Canvas visualizations shipped without this effort are completely inaccessible. If your audience includes screen reader users, this is not a minor gap to paper over later.

Quadtree hit detection is approximate. A quadtree finds the nearest point center within a search radius. It does not know about the visual size of each point. If you have variable-radius circles, the nearest center may belong to a small dot behind a large one the user clearly intended to click. For overlapping shapes, the quadtree has no concept of z-order. For non-circular shapes (bars, paths, polygons), quadtree distance from center is simply wrong. These edge cases do not show up in demos with uniform 3-pixel dots, but they show up fast in production scatter plots with size encoding.

LLMs draw every element individually instead of batching. The naive Canvas pattern looks like: loop over data, set fillStyle, beginPath, arc, fill. This is syntactically correct and visually correct at 500 points. At 50,000 points it is 10x slower than batched rendering because every iteration triggers a state change and a new path. LLMs reproduce this pattern because it maps cleanly to "for each datum, draw a circle," which is how SVG works. Batching requires grouping by style first and then drawing all same-style elements in one path, which is a different mental model.

Canvas text rendering ignores subpixel antialiasing. Browser-rendered SVG text benefits from the operating system's font rasterizer, which uses subpixel rendering (exploiting the RGB substructure of LCD pixels) to make text look sharp at small sizes. Canvas fillText uses grayscale antialiasing by default. At 11px or 12px, the difference is visible: Canvas text looks slightly blurry next to SVG text on the same page. For labels that the viewer reads (axis ticks, tooltips, annotations), SVG text in an overlay is sharper. Reserve fillText for labels that need to move with the data during zoom or animation, and keep them at 13px or larger where the difference fades.

Losing the ability to style elements with CSS. SVG elements inherit from stylesheets. You can change the color of every circle with a CSS rule, toggle dark mode with a class, apply hover states with :hover, and inspect individual elements in dev tools. Canvas has none of this. Colors are baked into draw calls. Dark mode means rewriting fill values in JavaScript. Hover requires spatial indexing. Dev tools show you a rectangle of pixels. For visualizations that need frequent style iteration, the inability to tweak a CSS rule and see the change instantly slows development significantly.
