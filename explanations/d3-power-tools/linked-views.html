Linked Views

Different representations reveal different aspects of the same data. Linking them lets the viewer see connections they would miss in any single chart.

A scatter plot shows how two continuous variables relate. A bar chart shows how categories compare. A histogram shows how values distribute. Each answers its own question and ignores the others. Linked views connect them so that touching one chart ripples through the rest, turning three isolated pictures into a single instrument for exploration.

The coordination pattern below uses d3.dispatch to broadcast two kinds of events: hover (a single data point is under the cursor) and filter (a category has been selected). Every chart listens for both. When a filter is active, each chart draws the full dataset as a faint ghost layer and the filtered subset in full color, so the viewer never loses the overall context.

The ghost/active layering pattern is key. Without the ghost, filtering feels like data is disappearing. With it, the viewer sees the subset in relation to the whole, which is where most of the insight lives.

Try it: hover any point in the scatter to see its category light up in the bar chart and its bin glow in the histogram. Click a bar to filter everything down to that category. Click the same bar again (or press the Clear button) to reset.

Overview + Detail

The most common linked-views pattern pairs a compressed "overview" with a high-resolution "detail." The overview shows the full dataset at a scale too coarse to read individual values; the detail shows a slice at a scale where every point is legible. A brush on the overview controls which slice the detail displays.

This is focus+context in its purest form. The viewer's spatial memory stays anchored to the overview: "the spike is in the middle, the dip is on the right." The detail view lets them zoom into any region without losing that anchor. The key constraint is that the overview must remain readable at its compressed scale. If the overview is so dense that the brush covers an indistinguishable blob, the pattern fails -- the viewer cannot orient themselves.

Brush the gray overview below to control the detail view's domain. The detail redraws on every brush event, showing only the points inside the selected window.

Cognitive Load Limits

Cowan's research on working memory suggests humans can track about four independent chunks simultaneously. Each linked view is at least one chunk. A three-view dashboard (scatter + bar + histogram, like Figure 1) sits comfortably within that budget. A five- or six-view dashboard pushes past it: the viewer starts losing track of which interaction caused which change, and the "aha" moments that linked views are supposed to produce get buried in confusion.

The practical ceiling is three to four simultaneously visible views. Beyond that, cluster views into a hierarchy: one "driver" panel that the viewer interacts with, plus two or three "target" panels that respond. If you need more perspectives, put them behind tabs or scrolling -- visible on demand rather than all at once.

The complementarity test: for every pair of linked views, ask "what does a selection in view A reveal in view B that I could not see in view A alone?" If the answer is nothing, remove the link or the view.

Ghost / Active Layering in Depth

The ghost layer is the single most important pattern in linked views. Without it, filtering feels destructive: the viewer clicks a category and most of the data vanishes. They lose the reference frame. With the ghost, the full dataset stays visible as a faint gray backdrop, and the filtered subset appears in full color on top. The viewer sees the subset in relation to the whole, which is almost always where the insight lives.

There are two strategies for what happens to non-selected data, and they serve different purposes:

Filter mode hides non-selected data entirely. The ghost layer provides the only context. This works when the viewer wants to isolate a subset and study it without visual clutter -- when the question is "what does this group look like?"

Highlight mode keeps all data visible but dims the non-selected points. No ghost layer is needed because the full dataset is already there, just at reduced opacity. This works when the viewer wants to see where a subset falls within the whole -- when the question is "where does this group sit relative to the rest?"

State Serialization for Shareable URLs

A linked-views dashboard has state: which filters are active, which brush extents are set, which zoom transforms apply. If that state lives only in JavaScript variables, the viewer cannot share what they found. They cannot bookmark it. They cannot paste a link in a message that says "look at this."

The fix is to serialize the dashboard state into the URL. Every time the viewer changes a filter or moves a brush, encode the current state as query parameters or a hash fragment and push it with history.replaceState. When the page loads, read the URL, parse the state, and replay it. The viewer's exploration becomes a first-class artifact: copy the URL, paste it anywhere, and the recipient sees the exact same slice.

What to serialize: filter keys (category names, brush extents as [x0, x1] pairs), zoom transforms (k, x, y), and any toggle states (like filter-vs-highlight mode). What not to serialize: hover state (transient), animation progress (ephemeral), or anything derived from the data itself (recompute it).

Scaling: Crossfilter Bitmap Indexing

The naive approach to cross-filtering is: on every brush event, loop through the full dataset, test each row against every active filter, rebuild the filtered subset, and redraw all views. At 80 rows (like Figure 1) this is instant. At 10,000 rows it is sluggish. At 100,000 rows it is unusable -- the brush drags at single-digit frames per second.

Crossfilter's bitmap indexing solves this. Each dimension (each filterable axis) maintains a sorted index and a bitmask. When a brush moves, crossfilter updates only the affected dimension's bitmask using binary search to find the new boundary, then ANDs the bitmasks across all dimensions. The result is the filtered set, computed in time proportional to the number of changed rows rather than the total. For a 100K-row dataset with a 12-bin brush, the typical update touches a few hundred rows instead of 100,000.

The practical implication: if your linked-views dashboard will handle more than about 10,000 rows, use crossfilter2 or route filter predicates through DuckDB-WASM as SQL WHERE clauses. The dispatch pattern from Figure 1 stays the same; only the filtering engine behind it changes.

When This Skill Hurts

  • Dashboard sprawl. The skill encourages adding views, but each view costs working memory. LLMs following this skill tend to create five or six linked charts when two or three would answer the question. More views does not mean more insight -- it means more for the viewer to track and more opportunities for confusion.
  • Coordination overhead. d3.dispatch with many listeners creates subtle bugs: feedback loops where chart A's update triggers chart B which triggers chart A again, stale closures that reference old state, race conditions between transitions. LLMs following the skill often underestimate this wiring complexity and produce dashboards that work on first load but break after a few interactions.
  • Performance cliff. Linking N views means every interaction triggers N redraws. Without debouncing or bitmap indexing (see above), brush-drag becomes laggy past roughly 10K rows. The skill's examples use 80-point datasets where this is invisible, which can mislead builders into assuming the pattern scales for free.
  • Lost narrative. Linked views are exploratory tools: they let the viewer ask their own questions. But for explanations (which is what Moonshine builds), a guided scroll-driven narrative often communicates better than a dashboard the reader must figure out on their own. If the goal is to show something specific rather than to let the viewer explore, linked views may be the wrong tool entirely.
