Calibrate Tool

Measuring whether a skill actually improves output, not just whether it runs.

Writing a skill is the easy part. Knowing whether it works is the hard part. A skill might add 800 lines of expert advice to the context window, and Claude might ignore all of it, producing the same output it would have without it. Or worse, the extra context might confuse things. The calibrate tool exists to answer a single question: did the skill make the output better?

The key insight: you cannot improve what you do not measure. Calibration sends the same prompt through Claude twice, once with the skill loaded and once without, then compares the results. The difference between the two isolates the skill's contribution.

The With/Without Comparison Protocol

The core mechanism is straightforward: send the same prompt through Claude twice, once from a directory where the skill is loaded and once from a bare directory with no skills at all, then compare the outputs. The prompt, the data, and the model version are held constant. The only variable is whether the skill was in context. Whatever differences appear in the output can be attributed to the skill rather than to chance or to the model's baseline ability.

This matters because skill authoring is prone to self-deception. You write a skill, test it on one prompt, see a nice output, and conclude the skill works. But Claude might have produced that same output without the skill. The model already knows a lot about D3. If your skill only teaches things Claude already knows, you have spent 600 tokens of context window for nothing. The with/without comparison catches this by producing the counterfactual: what would Claude have done on its own?

In practice, the eval runner at evals/run-evals.py handles the mechanics. It writes the prompt to a temporary file, runs claude -p from the project root (skills loaded) and again from an empty scratch directory (no skills), collects both HTML outputs, then runs structural checks and audit scoring on each. The outputs land in evals/results/<eval-id>/with/output.html and evals/results/<eval-id>/without/output.html. Three verdicts: BETTER (the skill adds techniques the baseline lacks), SAME (equivalent output), or WORSE (the baseline is actually better, meaning the skill confused the model). When the result is SAME or WORSE, the tool diagnoses whether the prompt was too easy, the checks too crude, or the skill itself poorly structured.

Blind Evaluation Protocol

Knowing which output used the skill biases the evaluation. If the evaluator sees that output B came from the skilled run, it will find reasons to score it higher, because the expectation is already set. This is the same problem that double-blind clinical trials solve: the person measuring the outcome must not know which treatment was applied. Calibration enforces this by stripping identifying information before the evaluator sees either output.

How the evaluator stays honest

  1. The eval runner generates both outputs in separate directories with neutral names. The evaluator does not know which version used the skill.
  2. Each output is scored independently on all four audit dimensions. The evaluator sees only the rendered HTML and its own SKILL.md criteria.
  3. Scores are compared after both evaluations are complete. Context contamination (tuning and testing in the same conversation) invalidates the results.
  4. For auditing skills, the test runs in a fresh container with no access to calibration data, memory files, or prior audit history.

Context contamination is the most common way calibration fails. If you tune a skill's wording in the same conversation where you then test it, the model has the tuning discussion in context. It knows what you are trying to achieve. It will reach for calibration targets rather than applying criteria independently. The fix is simple: tune in one conversation, evaluate in a fresh one. For auditing skills, this means a fresh container with no access to memory files, previous audit reports, or any record of the calibration discussion.

Sample Size and Measurement Noise

A single comparison tells you almost nothing. Claude's output varies from run to run even with identical prompts. One time the skill might happen to help; the next time it might not, for reasons unrelated to the skill itself (different random seeds in generation, different phrasing choices that cascade through the output). If you run one comparison and see the skilled version score 7.2 versus 6.8 for the baseline, that 0.4-point gap could easily be noise.

The practical minimum is four to six comparisons per skill. Run the same eval prompt through the with/without protocol multiple times and collect scores for each run. If the skilled version wins four out of five times, with an average gap of 0.8 or more, the skill is probably helping. If it wins three out of six, or the average gap is under 0.4, the signal is lost in the noise. The --runs flag on the eval runner handles this: python3 evals/run-evals.py --runs 5 generates five independent pairs and reports the aggregate.

The variance is not a bug. It reflects real properties of LLM generation. The same chart, evaluated five times by the same auditor with the same criteria, will produce scores that drift by roughly half a point in either direction. The figure below demonstrates this. Each row shows one audit dimension for a single chart, evaluated five times. The dots scatter across a range rather than landing on a single point. Cost-Benefit Analysis

Every skill consumes tokens. A typical skill file runs 300 to 800 tokens after compression. Those tokens displace other context that Claude could use: the user's prompt, data descriptions, conversation history, or other skills. The question is whether the quality improvement from the skill justifies the context budget it requires.

The calibration tool provides the raw data for this judgment. If a skill costs 500 tokens and raises the average audit score by 1.5 points across five comparisons, that is a good trade. If the same 500 tokens produce a 0.3-point improvement that only appears in two out of five runs, it is not worth the budget. The token cost is fixed and known. The quality benefit is uncertain and must be measured.

For skills that test as SAME, the right move is not always to delete them. Sometimes the prompt was too easy, and the skill helps on harder prompts that the eval did not test. Sometimes the skill needs restructuring so its highest-value patterns appear first, where the model is most likely to attend to them. But if a skill tests as SAME across multiple prompts of varying difficulty, it is spending context budget for no return, and the honest conclusion is to cut it or fold its content into another skill.

When This Skill Hurts

Failure modes of calibration itself
  • Small sample sizes leading to false conclusions. Running one or two comparisons and declaring a skill "effective" or "useless" is the most common mistake. LLM outputs vary enough that two data points can tell any story. A skill that lost one comparison might win the next four. A skill that won once might never win again. With fewer than four comparisons per skill, your confidence in the result should be close to zero.
  • Confounding variables. Model choice, prompt complexity, data type, and even the time of day (if API load affects generation) all influence output quality independent of the skill. If you test a skill using a scatter plot prompt on Opus and the baseline using a bar chart prompt on Sonnet, the comparison is meaningless. The protocol holds most of these constant, but you have to be careful not to introduce confounds when modifying eval prompts or switching models mid-calibration.
  • Optimizing for the auditor instead of for humans. The calibration loop uses automated auditing skills to score outputs. If you iterate on a skill until it scores well on the auditor, you have trained the skill to satisfy the auditor's criteria, not necessarily to produce visualizations that humans find clear and useful. The auditor and a human reader value overlapping but not identical things. Periodically check calibrated skills against your own judgment. If the auditor loves it but you find the output generic or overdecorated, the calibration loop has drifted.
  • Iteration overhead making improvement slow. A full calibration run with five comparisons takes time and costs money. If you re-run calibration after every small edit to a skill, the feedback loop becomes so slow that you stop iterating. The practical answer is to batch changes: make two or three edits to a skill, run calibration once, interpret the results, then batch the next round. Single-variable testing is ideal in theory but too expensive to practice on every change.
